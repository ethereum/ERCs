## Abstract

This ERC defines a standard interface for **Physical Commodity-Backed Tokens (PCBT)** — [ERC-20](./eip-20.md) tokens whose supply is directly backed 1:1 by a verified physical commodity lot (bar, ingot, barrel, batch) held in a custodian vault. Each deposited lot is certified as an on-chain non-fungible record, fractionalized into fungible ERC-20 tokens at a defined weight-per-token ratio, and redeemable back to the physical commodity through a standardized delivery queue.

The standard introduces six composable interfaces:

- `ICommodityCertificate` — on-chain lot certification with provenance (extends [ERC-721](./eip-721.md))
- `IFractionVault` — deterministic lot-to-token fractionalization gated by proof-of-reserve
- `ICommodityToken` — ERC-20 commodity token with weight math and lot-linked mint/burn
- `IRedemptionGate` — standardized physical redemption with encrypted delivery and T+2 queue
- `IProofOfReserve` — on-chain reserve verification (Chainlink PoR compatible)
- `IComplianceGate` — pluggable KYC/AML compliance for regulated jurisdictions

## Motivation

Existing commodity-backed tokens (PAXG, XAUt, DGX) share no common interface, creating four critical gaps:

**1. No DeFi interoperability.** Protocols cannot programmatically query weight, lot provenance, or redemption status across issuers. A gold lending protocol must hardcode support for each issuer individually.

**2. No standard redemption flow.** Each issuer invents its own redemption queue. Cross-protocol custody is unauditable and non-composable.

**3. No on-chain lot linkage.** Existing implementations (PAXG, XAUt) store bar-to-token linkage only in off-chain lookup tools. DeFi protocols cannot trustlessly verify which physical lots back their collateral.

**4. Class of `transferFrom` vulnerability.** The BlockSec CVE-2023 disclosure against XAUt demonstrated that a "trusted recipient" bypass in `transferFrom` is a recurring vulnerability class in commodity tokens. This standard eliminates it at the specification level.

**5. No compliance hook.** Regulated markets (SC Malaysia, UAE VARA, EU MiCA) require KYC on commodity token transfers. No standard pluggable compliance interface exists.

### Interfaces

#### `ICommodityCertificate`

Each physical lot deposited into a custodian vault MUST be certified as an on-chain record. Implementations SHOULD extend ERC-721 for interoperability.

```solidity
// SPDX-License-Identifier: CC0-1.0
pragma solidity ^0.8.20;

interface ICommodityCertificate {
    enum LotStatus { Active, PartiallyRedeemed, FullyRedeemed, Suspended }

    struct CommodityLot {
        bytes32   lotId;         // keccak256 of serialNumber
        string    serialNumber;  // physical serial number
        string    custodian;     // vault operator
        string    assayRef;      // assay/certification reference
        uint256   weightMicro;   // weight in micro-units (commodity-specific)
        uint256   purityBps;     // purity in basis points (10000 = 100.00%)
        uint256   depositedAt;   // Unix timestamp
        string    ipfsCid;       // IPFS CID of certification documents
        LotStatus status;
        uint256   remainingMicro;
    }

    event LotDeposited(
        uint256 indexed tokenId,
        bytes32 indexed lotId,
        string  serialNumber,
        uint256 weightMicro,
        uint256 purityBps,
        string  ipfsCid
    );
    event LotStatusUpdated(uint256 indexed tokenId, LotStatus newStatus);
    event RemainingUpdated(uint256 indexed tokenId, uint256 remainingMicro);
    event AuditDocUpdated(uint256 indexed tokenId, string newIpfsCid);

    function depositLot(
        address to,
        string calldata serial,
        string calldata custodian,
        string calldata assayRef,
        uint256 weightMicro,
        uint256 purityBps,
        string calldata ipfsCid
    ) external returns (uint256 tokenId);

    function getLot(uint256 tokenId) external view returns (CommodityLot memory);
    function isLotActive(uint256 tokenId) external view returns (bool);

    /// @notice Reduce remainingMicro when tokens are redeemed
    /// @dev MUST only be callable by the linked IFractionVault or IRedemptionGate
    function reduceRemaining(uint256 tokenId, uint256 microUnits) external;
    function linkFractionVault(uint256 tokenId, address vault) external;
    function updateAuditDoc(uint256 tokenId, string calldata newIpfsCid) external;
    function totalLots() external view returns (uint256);
}
```

#### `IFractionVault`

Converts a certified lot record into fungible ERC-20 commodity tokens.

```solidity
interface IFractionVault {
    event LotFractionalized(
        uint256 indexed tokenId,
        uint256 tokensIssued,
        uint256 microPerToken,
        address mintedTo
    );

    /// @notice Fractionalize a certified lot into commodity tokens
    /// @dev MUST call IProofOfReserve.verify() before minting.
    ///      MUST call ICommodityCertificate.linkFractionVault() after.
    function fractionalize(uint256 tokenId, address mintTo)
        external returns (uint256 issued);

    function microToTokens(uint256 microUnits) external view returns (uint256);
    function tokensToMicro(uint256 tokens) external view returns (uint256);
    function microPerToken() external view returns (uint256);
    function remainingCapacity(uint256 tokenId) external view returns (uint256);
    function proofOfReserve() external view returns (address);
}
```

#### `ICommodityToken`

The core ERC-20 commodity token with commodity-specific extensions.

```solidity
interface ICommodityToken {
    event MintedFromLot(address indexed to, uint256 indexed lotId, uint256 amount);
    event BurnedFromLot(address indexed from, uint256 indexed lotId, uint256 amount);

    function commodityType()  external view returns (string memory);
    function weightUnit()     external view returns (string memory);
    function microPerToken()  external view returns (uint256);

    function toMicro(uint256 baseUnits)   external view returns (uint256);
    function fromMicro(uint256 microUnits) external view returns (uint256);
    function commodityBalance(address account) external view returns (uint256);

    function lotTokensMinted(uint256 lotId) external view returns (uint256);
    function lotTokensBurned(uint256 lotId) external view returns (uint256);
    function lotNetMicro(uint256 lotId)     external view returns (uint256);

    function mintFromLot(address to, uint256 lotId, uint256 tokenAmount) external;
    function burnForRedemption(address from, uint256 lotId, uint256 tokenAmount) external;

    function calcFee(uint256 amount)   external view returns (uint256);
    function feeBasisPoints()          external view returns (uint256);
    function maxFeeAbsolute()          external view returns (uint256);

```

#### `IRedemptionGate`

Standardizes physical commodity redemption.

```solidity
interface IRedemptionGate {
    enum RedemptionStatus { Pending, Processing, Dispatched, Delivered, Cancelled }

    struct RedemptionRequest {
        address  requester;
        uint256  tokensBurned;
        uint256  microRedeemed;
        bytes32  deliveryHash;    // keccak256 of encrypted delivery details
        uint256  requestedAt;
        uint256  fulfilledAt;
        RedemptionStatus status;
        uint256  linkedLotId;
        string   trackingRef;
        uint256  flatFee;
    }

    event RedemptionRequested(
        uint256 indexed requestId,
        address indexed requester,
        uint256 tokensBurned,
        uint256 microRedeemed
    );
    event RedemptionFulfilled(
        uint256 indexed requestId,
        uint256 indexed lotId,
        string trackingRef
    );
    event RedemptionCancelled(uint256 indexed requestId);

    /// @notice Request physical redemption
    /// @dev MUST burn tokenAmount before creating the request
    function requestRedemption(uint256 tokenAmount, bytes32 deliveryHash)
        external returns (uint256 requestId);

    /// @notice Fulfil a redemption — MUST call ICommodityCertificate.reduceRemaining()
    function fulfil(uint256 requestId, uint256 lotId, string calldata trackingRef) external;

    function quoteRedemption(uint256 microUnits)
        external view returns (uint256 tokensRequired, uint256 flatFeeTokens);

    function minimumRedemption() external view returns (uint256);
    function getRequest(uint256 requestId) external view returns (RedemptionRequest memory);
    function getUserRequests(address user)  external view returns (uint256[] memory);
}
```

#### `IProofOfReserve`

On-chain reserve verification, compatible with Chainlink Proof of Reserve.

```solidity
interface IProofOfReserve {

```

Pluggable KYC/AML compliance for regulated jurisdictions.

### Required Behavior

#### Minting

An implementation MUST:
- NOT mint tokens unless `IProofOfReserve.verify(lotId, microUnits)` returns `true`
- Track `lotTokensMinted[lotId]` on-chain
- Emit `MintedFromLot(to, lotId, amount)`
- Call `ICommodityCertificate.linkFractionVault(tokenId, address(this))`

#### Transfers

An implementation MUST:
- Check `IComplianceGate.checkTransfer(from, to, amount)` if a compliance gate is set
- Apply `calcFee(amount)` and route fee to `feeTreasury`
- NOT allow `transferFrom` without verifying `allowance(from, spender) >= amount` for **all callers** with no exceptions

An implementation MUST define a hard-coded `MAX_FEE_BPS` constant that CANNOT be exceeded by any configuration function. Recommended value: `200` (2%).

#### Redemption

An implementation MUST:
- Burn tokens atomically within `requestRedemption`
- Call `ICommodityCertificate.reduceRemaining(tokenId, microUnits)` on fulfilment
- Enforce `minimumRedemption()`

#### Reserve Invariant

At all times: `totalSupply() × microPerToken ≤ IProofOfReserve.totalVerifiedReserve()`

### Fee Formula

```
fee = min(amount × feeBasisPoints ÷ 10_000, maxFeeAbsolute)
net = amount − fee
```

`MAX_FEE_BPS` MUST be a hard-coded constant. Implementations MUST revert if `feeBasisPoints` is set above it.

## Rationale

**Why six separate interfaces?** Separation of concerns enables DeFi protocols to integrate selectively. A lending protocol may only require `ICommodityToken` + `IProofOfReserve`. A compliance-focused vault may only require `IComplianceGate`. Each interface is independently useful.

**Why not extend [ERC-1155](./eip-1155.md)?** ERC-1155 is optimal for the NFT lot layer but not for the fungible ERC-20 commodity token layer. PCBT deliberately separates the two: ERC-721 handles lot provenance, ERC-20 handles fungible ownership. These are distinct concerns.

**Why mandate the `transferFrom` fix?** The BlockSec CVE-2023 disclosure showed that commodity tokens are a target for `transferFrom` bypass exploits. Standardizing the strict allowance requirement eliminates this entire vulnerability class at the specification level, rather than relying on each issuer to discover and fix it independently.

**Why on-chain lot linkage (`lotTokensMinted`)?** PAXG and XAUt only expose bar-to-token linkage through off-chain websites. This means DeFi collateral vaults cannot trustlessly verify reserves. On-chain lot tracking enables trustless, real-time reserve composition.

**Why `IComplianceGate` as a separate interface?** Regulatory requirements differ by jurisdiction. SC Malaysia (whitelist-based), EU MiCA (issuer licensing), and UAE VARA all have different KYC models. A pluggable compliance gate lets the same [ERC-8188](./eip-8188.md) implementation operate under different regulatory regimes by swapping the compliance contract without upgrading the token.

## Backwards Compatibility

ERC-8188 tokens are fully ERC-20 compatible. All additional interfaces are additive. Existing wallets, DEXes, and DeFi protocols interact with ERC-8188 tokens as standard ERC-20 tokens. The `ICommodityToken` extension is discoverable via ERC-165.

## Security Considerations

**`transferFrom` authorization.** Implementations MUST NOT allow `transferFrom` to proceed without verifying `allowance(from, spender) >= amount` for all callers. No "trusted recipient" bypass is permitted. See BlockSec CVE-2023 (XAUt).

**Proof of Reserve atomicity.** `IProofOfReserve.verify()` MUST be called within the same transaction as minting. If PoR relies on an external oracle, implementations MUST check oracle freshness (recommended maximum staleness: 3600 seconds).

**Freeze/wipe separation.** The role that can freeze addresses SHOULD be distinct from the role that can wipe balances. Balance wipes SHOULD require multi-signature authorization and a minimum time delay.

**Fee cap.** `MAX_FEE_BPS` MUST be a hard-coded constant not configurable by any role. This prevents fee extraction attacks.

**Lot over-subscription.** Minting against a lot MUST revert if `lotTokensMinted[lotId] + tokenAmount > lot.weightMicro / microPerToken`.

**Delegated transfers.** If implementing meta-transactions, implementations MUST use [EIP-712](./eip-712.md) structured data with `chainId` in the domain separator to prevent cross-chain replay attacks.
