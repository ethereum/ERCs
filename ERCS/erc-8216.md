## Abstract

This EIP defines an interface for managing slot-based equipment within [ERC-6551](./eip-6551.md) Token Bound Accounts (TBAs). It allows tokens ([ERC-721](./eip-721.md) or [ERC-1155](./eip-1155.md)) to be equipped into application-defined slots, queried as a structured loadout, and optionally permanently locked.

The interface includes single and batch operations and exposes lock status per slot via the SlotEntry struct, allowing frontends and marketplaces to render a complete character view in a single call.

## Motivation

[ERC-6551](./eip-6551.md) gives every NFT its own smart contract account, but defines no standard for what goes inside. Applications have independently built equipment, loadout, and trait systems with incompatible interfaces, making cross-application composability impossible.

A standard equipment interface enables gaming characters with equippable gear, social profiles with locked credentials, composable art with swappable accessories, and universal marketplace loadout display.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

Every compliant contract MUST implement the `IERC6551Equipment` interface and MUST register support via [ERC-165](./eip-165.md).

The [ERC-165](./eip-165.md) identifier for this interface is 0xd38f0891.

### Interface
```solidity
// SPDX-License-Identifier: CC0-1.0
pragma solidity ^0.8.24;

interface IERC6551Equipment {

    struct SlotEntry {
        bytes32 slotId;
        address tokenContract;
        uint256 tokenId;
        uint256 amount;
        bool locked;
    }

    event Equipped(bytes32 indexed slotId, address indexed tokenContract, uint256 indexed tokenId, uint256 amount);
    event Unequipped(bytes32 indexed slotId, address indexed tokenContract, uint256 indexed tokenId, uint256 amount);
    event SlotLocked(bytes32 indexed slotId, address indexed tokenContract, uint256 tokenId);

    function equip(bytes32 slotId, address tokenContract, uint256 tokenId, uint256 amount) external;
    function unequip(bytes32 slotId) external;
    function lockSlot(bytes32 slotId) external;
    function equipBatch(bytes32[] calldata slotIds, address[] calldata tokenContracts, uint256[] calldata tokenIds, uint256[] calldata amounts) external;
    function lockSlots(bytes32[] calldata slotIds) external;
    function getEquipped(bytes32 slotId) external view returns (address tokenContract, uint256 tokenId, uint256 amount);
    function getLoadout() external view returns (SlotEntry[] memory entries);
    function isSlotOccupied(bytes32 slotId) external view returns (bool);
    function isSlotLocked(bytes32 slotId) external view returns (bool);
}
```

### Slot Identifiers

Slots are identified by bytes32 values. The RECOMMENDED convention is keccak256("slot.name"). Applications define their own slot taxonomy. Applications sharing a TBA across multiple contexts SHOULD namespace slots to avoid collisions.

### Behavior

equip: MUST revert if caller is not a valid signer, slot is occupied, slot is locked, or amount is zero. MUST update state before external transfers (CEI). MUST emit Equipped event.

unequip: MUST revert if caller is not a valid signer, slot is empty, or slot is locked. MUST delete state before external transfers (CEI). MUST emit Unequipped event.

lockSlot: MUST revert if caller is not a valid signer, slot is empty, or slot is already locked. MUST set slot as permanently locked. This action is irreversible. MUST emit SlotLocked event.

equipBatch: MUST revert if array lengths do not match or any individual equip would revert. All-or-nothing semantics.

lockSlots: MUST revert if any individual lock would revert. All-or-nothing semantics.

getLoadout: Returns an array of SlotEntry structs for every occupied slot including lock status.

### Execute Layer Integrity

Implementations of `IERC6551Equipment` that are also [ERC-6551](./eip-6551.md) executable accounts (exposing an `execute()` function) MUST ensure that no execution can transfer out a token recorded in any equipment slot, regardless of whether that slot is locked.

After every successful call made via `execute()`, the implementation MUST verify, for each currently occupied slot, that the recorded token is still held by the account: for [ERC-721](./eip-721.md) tokens via `ownerOf(tokenId) == address(this)`, and for [ERC-1155](./eip-1155.md) tokens via `balanceOf(address(this), tokenId) >= amount`. If any equipped token is no longer held by the account at the conclusion of an `execute()` call, the entire execution MUST revert.

Implementations SHOULD define a dedicated error type (e.g., `SlotIntegrityViolated(bytes32 slotId)`) for this revert condition. This requirement applies uniformly to locked and unlocked slots: any token recorded as equipped is protected from being silently drained via `execute()`. Owners wishing to remove an equipped token MUST use the `unequip()` function.

### Ownership Transfer Semantics

When the parent NFT is transferred, all equipped items remain in the TBA. Unlocked slots can be modified by the new owner. Locked slots remain locked and cannot be modified.

## Rationale

bytes32 slot identifiers allow arbitrary taxonomies without a central registry. Permanent locking with no unlock provides absolute trust for marketplaces and contracts. No swap function keeps the interface minimal. Batch operations reduce mint flows from many transactions to two. bool locked in SlotEntry enables single-call character rendering. CEI pattern is mandatory to prevent reentrancy via safeTransferFrom callbacks.

The execute-layer integrity check is required because, without it, an account owner could circumvent the lock guarantees by constructing an `execute()` call that invokes `safeTransferFrom` directly on an equipped token's contract. This would drain the underlying token from the account while leaving the on-chain equipment record intact and the slot still marked as locked. An owner could lock a "founder pack" item to advertise it as permanently bonded, sell the parent NFT at a premium based on that guarantee, then drain the item before delivery — defrauding the buyer while the on-chain lock state appears unbroken. The execute-layer check transforms slot locks from advisory metadata into enforced guarantees that downstream consumers (marketplaces, indexers, buyers) can rely on.

## Backwards Compatibility

This EIP introduces no backwards compatibility issues. It extends [ERC-6551](./eip-6551.md) Token Bound Accounts with new functionality.

## Reference Implementation

A complete reference implementation with 37 Foundry tests is available at https://github.com/LordThisDrip/erc-equipment (commit `bf9728e`). The reference implementation demonstrates the execute-layer integrity check via the internal function `_verifyEquipmentInvariant()` called at the end of every `execute()` call, with a gas cost of approximately 5-10k gas per occupied slot. The test suite includes adversarial regression tests for the lock-bypass attack, initializer access control, type detection, and interface ID drift.

## Security Considerations

Implementations MUST follow checks-effects-interactions pattern. lockSlot is permanent and irreversible. Batch operations use all-or-nothing semantics. Buyers SHOULD inspect loadout before purchasing character NFTs. Marketplaces SHOULD call supportsInterface before equipment-related calls.

**Re-entrancy through the integrity check.** The execute-layer integrity check makes external calls to each equipped token's contract. A malicious token contract could attempt to re-enter the equippable account during one of these calls. Implementations MUST ensure that all state-mutating functions on the account (`equip`, `unequip`, `lockSlot`, `equipBatch`, `lockSlots`, `execute`, `initialize`) are gated by access control based on the parent NFT owner, not on the immediate `msg.sender` of the call. This pattern ensures that re-entrant calls from a malicious token contract fail the access control check, because the `msg.sender` of a re-entrant call is the token contract address, not the parent NFT owner.

**Quantity vs. provenance for ERC-1155.** The integrity check for ERC-1155 equipped tokens verifies that the account holds at least `amount` of the recorded `tokenId`. By the semantics of ERC-1155, all tokens with the same `tokenId` are fungible and indistinguishable. The check therefore provides a quantity guarantee but not a provenance guarantee — an attacker who controls both the equipped token contract and a separate exploiter contract could drain the originally-equipped tokens and replace them with newly-minted tokens of the same `tokenId` in a single transaction. Implementations and consumers SHOULD treat ERC-1155 equipment slots as providing quantity guarantees only. For provenance guarantees on individual items, use ERC-721 tokens, where each `tokenId` is unique.

**Bounded integrity check cost.** The execute-layer integrity check has a gas cost that scales linearly with the number of currently occupied slots (approximately 5-10k gas per slot). For typical loadouts of 4-10 slots this overhead is negligible. For very large loadouts (hundreds of slots) the overhead can become significant. Implementations SHOULD consider whether to enforce a maximum slot count to bound this cost. The reference implementation does not enforce such a limit, leaving the choice to consumers. Because all equip operations are gated by owner access control, this cost can only be incurred by the owner of the parent NFT — there is no third-party griefing vector.
