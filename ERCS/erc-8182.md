## Abstract

This ERC defines a standard interface for mapping off-chain entity identifiers — such as GitHub repositories, DNS domains, and package registries — to Ethereum addresses. Identifiers are `(namespace, string)` pairs hashed to a single `bytes32`. Ownership is proven through per-namespace verifier contracts, enabling a smooth migration path from oracle-signed proofs to zero-knowledge verification without changes to the registry. Multiple identifiers controlled by the same entity can be linked so they resolve to a single registrant address.

## Motivation

Protocols and AI agents increasingly need to resolve off-chain identifiers — GitHub repositories, DNS domains, npm packages — to Ethereum addresses. An agent instructed to "fund the maintainers of react" or a grants protocol allocating to `github:ethereum/go-ethereum` both need the same thing: a single, permissionless, on-chain lookup. Any agent, any framework, one `ownerOf` call.

No standard for this exists. Ethereum addresses humans well (ENS) and makes attestations about them (EAS), but off-chain *entities* have no canonical Ethereum address. Each protocol that encounters this problem builds its own resolution system:

- **Drips** maps GitHub repositories to addresses using an application-specific oracle.
- **Funding Platforms** maintains their own project registry.
- **ENS** requires proactive registration and is designed for human-readable names, not entity identifiers.

This fragmentation means every new protocol — and every agent framework — must either require entities to register before receiving funds, or build and maintain a bespoke identity backend. As agentic workflows become common, this becomes untenable: agents need a shared resolution layer they can call on-chain without API keys, without trusted backends, and without pre-registration.

This ERC proposes that shared primitive: a single registry contract per chain that any protocol or agent can read and any entity can register with. The registry holds no funds, charges no fees, and has no privileged operators beyond verifier governance.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

### Identifier Scheme

Every off-chain entity is identified by a `(namespace, canonicalString)` pair. The on-chain identifier is a single `bytes32`:

```solidity
bytes32 id = keccak256(abi.encode(namespace, canonicalString))
```

Implementations MUST use `abi.encode` (length-prefixed) rather than `abi.encodePacked` to prevent collisions between namespace/string pairs that share a common packed concatenation (e.g. `("githu", "borg/repo")` vs `("github", "org/repo")`).

Examples:

| Namespace | Canonical String | Identifies          |
| --------- | ---------------- | ------------------- |
| `github`  | `org/repo`       | A GitHub repository |
| `dns`     | `example.com`    | A DNS domain        |
| `npm`     | `package-name`   | An npm package      |

#### Normalization

Normalization rules — lowercase, trailing slash removal, Unicode handling — are namespace-specific and MUST be enforced by verifiers and frontends, not by the registry contract. The registry MUST hash whatever strings are passed to it without modification.

### Registry Interface

A compliant implementation MUST implement the following interface:

```solidity
interface IOffChainEntityRegistry {
    // ----------------------------------------------------------------
    // Events
    // ----------------------------------------------------------------

    /// @notice MUST be emitted when an identifier is successfully claimed.
    event Claimed(
        bytes32 indexed id,
        string namespace,
        string canonicalString,
        address indexed owner
    );

    /// @notice MUST be emitted when an owner voluntarily revokes their claim.
    event Revoked(
        bytes32 indexed id,
        string namespace,
        string canonicalString,
        address indexed previousOwner
    );

    /// @notice MUST be emitted when an alias is linked to a primary identifier.
    event Linked(bytes32 indexed aliasId, bytes32 indexed primaryId);

    /// @notice MUST be emitted when an alias is unlinked from its primary.
    event Unlinked(bytes32 indexed aliasId, bytes32 indexed primaryId);

    /// @notice MUST be emitted when a verifier is registered or replaced for a namespace.
    event VerifierUpdated(bytes32 indexed namespaceKey, address verifier);

    // ----------------------------------------------------------------
    // Identifier helpers
    // ----------------------------------------------------------------

    /// @notice Computes the canonical identifier for a (namespace, canonicalString) pair.
    /// @return The keccak256(abi.encode(namespace, canonicalString)) hash.
    function toId(
        string calldata namespace,
        string calldata canonicalString
    ) external pure returns (bytes32);

    /// @notice Resolves an identifier to its canonical form.
    /// @dev If `id` is an alias, returns the primary identifier.
    ///      If `id` is not an alias, returns `id` unchanged.
    ///      MUST follow at most one level of alias indirection.
    function canonicalOf(bytes32 id) external view returns (bytes32);

    /// @notice Returns the registered owner of `id`, resolving through any alias.
    /// @return The owner address, or address(0) if unclaimed.
    function ownerOf(bytes32 id) external view returns (address);

    // ----------------------------------------------------------------
    // Registration
    // ----------------------------------------------------------------

    /// @notice Claim ownership of an identifier by submitting a valid proof.
    /// @dev MUST revert if the identifier is already claimed.
    ///      MUST revert if no verifier is registered for the namespace.
    ///      MUST revert if the proof is invalid.
    ///      MUST emit Claimed on success.
    function claim(
        string calldata namespace,
        string calldata canonicalString,
        bytes calldata proof
    ) external;

    /// @notice Voluntarily revoke the caller's claim, returning the
    ///         identifier to unclaimed state.
    /// @dev MUST revert if called by anyone other than the current owner.
    ///      MUST revert if the identifier is an alias.
    ///      MUST emit Revoked on success.
    function revoke(
        string calldata namespace,
        string calldata canonicalString
    ) external;

    // ----------------------------------------------------------------
    // Linking
    // ----------------------------------------------------------------

    /// @notice Link one or more aliases to a primary identifier.
    /// @dev The caller MUST be the owner of both the primary and all alias identifiers.
    ///      The primary MUST NOT itself be an alias.
    ///      Each alias MUST NOT already be linked.
    ///      MUST emit Linked for each alias on success.
    function linkIds(
        bytes32 primaryId,
        bytes32[] calldata aliasIds
    ) external;

    /// @notice Unlink one or more aliases from their primary.
    /// @dev The caller MUST be the owner of the primary identifier.
    ///      Each alias MUST currently be linked to the specified primary.
    ///      MUST emit Unlinked for each alias on success.
    ///      After unlinking, each alias SHOULD be restored as an independently
    ///      owned identifier under the caller's address.
    function unlinkIds(
        bytes32 primaryId,
        bytes32[] calldata aliasIds
    ) external;

    // ----------------------------------------------------------------
    // Admin
    // ----------------------------------------------------------------

    /// @notice Register or replace the verifier for a namespace.
    /// @dev Existing claims under the previous verifier MUST remain valid.
    ///      MUST emit VerifierUpdated on success.
    ///      Access control for this function is implementation-defined.
    function setVerifier(
        string calldata namespace,
        address verifier
    ) external;
}
```

### Verifier Interface

The registry MUST delegate ownership verification to per-namespace verifier contracts. A compliant verifier MUST implement:

```solidity
interface IVerifier {
    /// @notice Verify that `claimant` owns the off-chain entity identified by `id`.
    /// @param id       The bytes32 identifier being claimed.
    /// @param claimant The address asserting ownership.
    /// @param proof    Encoded proof data (format is verifier-specific).
    /// @return True if the proof is valid and the claimant owns the identifier.
    function verify(
        bytes32 id,
        address claimant,
        bytes calldata proof
    ) external returns (bool);
}
```

Verifier implementations are intentionally unconstrained. The interface accommodates:

- **Oracle verifiers**: A trusted backend confirms ownership off-chain and signs an [EIP-712](./eip-712.md) proof.
- **ZK verifiers**: Ownership is proven cryptographically on-chain (e.g. zkTLS, TLSNotary, DNSSEC).

A registry deployment MAY replace a verifier for any namespace at any time. Existing claims MUST NOT be invalidated by a verifier change.

### Claim Semantics

1. The registry MUST compute the identifier as `keccak256(abi.encode(namespace, canonicalString))`.
2. The registry MUST look up the verifier registered for the namespace.
3. The registry MUST call `IVerifier.verify(id, msg.sender, proof)` and revert if it returns `false`.
4. The registry MUST record `msg.sender` as the owner of the identifier.
5. If the identifier was previously an alias (from a revoked primary), the registry MUST clear the stale alias before recording the new owner.

### Linking Semantics

1. Aliases MUST resolve through exactly one level of indirection. Chaining (an alias pointing to another alias) MUST NOT be allowed.
2. When identifiers are linked, the alias's owner record is deleted. `ownerOf(aliasId)` MUST return the primary's owner.
3. Each identifier's associated state (e.g. escrow, deposits) remains separate. Linking only affects ownership resolution.

### Revocation Semantics

1. After revocation, `ownerOf(id)` MUST return `address(0)`.
2. Any aliases pointing to the revoked identifier become stale — they also resolve to `address(0)`.
3. Stale aliases MUST be automatically cleared when someone claims that alias identifier in the future.

### Namespace Governance

Namespace registration (`npm`, `crates`, `pypi`, etc.) SHOULD be restricted to a curated set managed by the registry admin. Permissionless namespace registration invites squatting and low-quality entries. Implementations SHOULD start with an `onlyOwner` pattern for `setVerifier` and MAY migrate to a governance contract as the namespace set stabilizes.

## Rationale

### `abi.encode` over `abi.encodePacked`

`abi.encodePacked` concatenates values without length prefixes, creating ambiguity: `("githu", "borg/repo")` and `("github", "org/repo")` produce the same packed bytes. `abi.encode` is length-prefixed and unambiguous.

### No registrant transfer function

There is no function to transfer a registered identifier to another address. Changing the registrant requires revoking and re-claiming with a new proof. This ties the registered address to proven ownership at claim time, preventing social-engineering attacks where an attacker convinces a registrant to transfer without understanding the implications.

### Single-level alias indirection

Allowing chains of aliases (alias → alias → primary) would require unbounded traversal in `ownerOf`, creating gas unpredictability and a potential griefing vector. One level of indirection is sufficient for the common case (linking a GitHub repo and a DNS domain owned by the same entity) while keeping resolution O(1).

### Verifier interface simplicity

The `IVerifier` interface has a single function with no constraints on the proof format. This is intentional: it allows the same registry to support oracle-signed proofs, DNSSEC proofs, zkTLS proofs, and any future verification mechanism without interface changes.

### Normalization off-chain

Different namespaces have different canonicalization rules (GitHub is case-insensitive for organizations, DNS has punycode, npm has scoped packages). Embedding all normalization rules in the registry contract would make it large, inflexible, and expensive. Verifiers and frontends share responsibility for consistent canonicalization.

## Backwards Compatibility

This ERC introduces a new standard and does not modify any existing standards. It is designed to complement ENS (which addresses humans) and EAS (which provides attestation infrastructure).

An ENS name can resolve to an address returned by this registry. Protocols using Drips or similar application-specific registries can migrate to this standard without affecting their end-user interfaces.

## Reference Implementation

A minimal reference implementation consists of:

- `OffChainEntityRegistry.sol` — registry with claim, revoke, link, unlink
- `OracleVerifier.sol` — example [EIP-712](./eip-712.md) oracle verifier

The implementation has no external dependencies and is intentionally minimal.

## Security Considerations

### Proof replay

[EIP-712](./eip-712.md) domain separation (in oracle verifiers) binds proofs to a specific chain and registry deployment. Proofs include a claimant address and an expiry timestamp. Within the TTL window, the same proof could be re-submitted by the same claimant after a revocation. This is accepted at the contract level; verifier implementations SHOULD use short expiry windows (e.g. 1 hour) and issue fresh proofs on each request.

### Identifier collisions

The use of `abi.encode` (length-prefixed) prevents collisions between different namespace/string pairs. The `keccak256` hash is 256 bits, making accidental collision negligible.

### Oracle key compromise

A compromised oracle signing key allows claiming any unclaimed identifier in all namespaces using that verifier, until the key is rotated via `setTrustedSigner`. Signing keys SHOULD be held in an HSM with defined rotation procedures.

### Normalization inconsistency

If a verifier and a frontend disagree on the canonical form of a string (e.g. uppercase vs lowercase), the same off-chain entity could be registered under multiple identifiers. Verifier implementations SHOULD enforce strict normalization and reject non-canonical inputs.

### Stale claims and identifier permanence

Off-chain identifiers are not permanent — GitHub repositories can be transferred, domains can expire, npm packages can be unpublished. This ERC does not define a challenge mechanism for stale claims. Introducing one would add significant complexity and governance attack surface.

Instead, verifier implementations MAY enforce time-bounded claims (e.g. proofs that expire after a period, requiring periodic re-verification). A challenge or expiry mechanism is deferred to a future ERC. Protocols that cache resolved addresses SHOULD re-validate periodically.

### Stale aliases after revocation

When a primary identifier is revoked without first unlinking its aliases, those aliases become stale (resolve to `address(0)`). This is by design — the stale alias is automatically cleared when someone claims that identifier in the future.

### No registrant transfer

The lack of a transfer function means wallet rotation requires a full revoke-and-reclaim cycle, including a new verification round-trip. This is a deliberate trade-off: it prevents unauthorized transfers at the cost of friction for legitimate wallet changes.
