## Abstract

[ERC-7677](./eip-7677.md) defines the `paymasterService` capability for [EIP-5792](./eip-5792.md), allowing apps to direct wallets to a sponsorship web service. This proposal extends that capability to [EIP-8130](./eip-8130.md) by defining `payer_*` RPC methods that wallets use when constructing [EIP-8130](./eip-8130.md) transactions. Apps pass the same `paymasterService` URL they already use — the wallet selects the appropriate RPC methods based on the transaction type it is constructing. This proposal additionally defines sponsorship **balances**, so a wallet can read how much sponsorship budget or prepaid credit a sender has remaining (e.g., a subscription gas allowance) and fall back to token payment when it is exhausted.

## Motivation

[EIP-8130](./eip-8130.md) introduces native account abstraction where payers are first-class protocol participants — they co-sign the transaction using the same `verifier || data` authentication model as senders, validated with PAYER scope. This is fundamentally different from the [ERC-4337](./eip-4337.md) paymaster model:

- There is no EntryPoint contract or paymaster field injection flow.
- No stub-data/final-data two-phase dance is required — the payer simply co-signs.
- The payer signs the same transaction object that lands onchain.
- The payer can relay the transaction directly to the chain, reducing hops.
- [EIP-8130](./eip-8130.md) transactions have a native `expiry` field, enabling time-bounded sponsorship commitments enforced at the protocol level.

[ERC-7677](./eip-7677.md) defines the `pm_*` RPC methods for [ERC-4337](./eip-4337.md) user operations. Those methods do not apply to [EIP-8130](./eip-8130.md) because the two-phase `pm_getPaymasterStubData` / `pm_getPaymasterData` flow addresses concerns (gas estimation with paymaster-injected fields, stub data consistency) that do not exist in the [EIP-8130](./eip-8130.md) payer model.

However, from an app's perspective, the intent is identical: "here is a sponsorship service URL." Apps should not need to know or care whether the user's wallet constructs [ERC-4337](./eip-4337.md) user operations or [EIP-8130](./eip-8130.md) transactions. By extending [ERC-7677](./eip-7677.md)'s `paymasterService` capability rather than defining a new one, apps gain [EIP-8130](./eip-8130.md) support with zero code changes — the wallet handles the integration.

One further need motivates this extension: **sponsorship is increasingly budgeted.** Apps and payers grant a sender a *finite* amount of free gas — a monthly allowance, a per-campaign cap, or prepaid credit. Wallets need to read the remaining balance to display it to the user (e.g., "$5 of free gas left this month") and to fall back gracefully — to token payment or another payer — once it is exhausted. This proposal adds a `balance` to the terms response and an OPTIONAL `payer_getBalance` method so a wallet can surface and act on that allowance.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

### Payer Web Service Interface

We define three core JSON-RPC methods to be implemented by payer web services (`payer_getTerms`, `payer_sendTransaction`, `payer_signTransaction`) and three OPTIONAL methods for discovery and balances (`payer_getSponsorshipOptions`, `payer_getCapabilities`, `payer_getBalance`).

#### `payer_getTerms`

Returns sponsorship or token-payment terms for a transaction intent. Accepts a chain ID, sender address, intended calls, and optional hints. This method is called *before* the sender signs.

Payer web services SHOULD validate the transaction intent during `payer_getTerms` execution and reject at this stage if they would not sponsor or accept payment for the operation.

This method MAY return a `gasEstimate` with the payer's recommended gas parameters. Wallets SHOULD treat `gasEstimate` as authoritative for [EIP-8130](./eip-8130.md) transactions: a plain `eth_estimateGas` does not account for [EIP-8130](./eip-8130.md) intrinsic costs (sender and payer authentication, `account_changes` application, and the per-phase call structure), so an estimate produced for an equivalent legacy call will undercount. A wallet that estimates independently MUST add these intrinsics and MUST still satisfy the payer's `conditions.maxGasLimit`, which is the binding constraint.

This method MAY also return a `sponsor` object with a `name` field and an optional `icon` field. The `name` field is the name of the party sponsoring the transaction, and the `icon` field is a URI pointing to an image. Wallet developers MAY choose to display sponsor information to users. The `icon` string MUST be a data URI as defined in RFC-2397. The image SHOULD be a square with 96x96px minimum resolution. Since SVG images can execute Javascript, wallets MUST render SVG images using the `<img>` tag to ensure no untrusted Javascript execution can occur.

#### Token Payment

When the payer is not fully sponsoring, it returns `tokenOptions` describing accepted tokens and pricing. Each token option specifies:

- `token`, `symbol`, and `decimals` — the token's address and display metadata, so the wallet can render the fee without a separate metadata lookup.
- `maxCost` — the **canonical amount the wallet transfers in phase 0**. It is the payer's quoted cost at the gas cap (`conditions.maxGasLimit`), so transferring exactly `maxCost` always covers the transaction while the terms are valid. Wallets SHOULD transfer `maxCost` rather than computing a tighter amount themselves.
- `rate` — a machine-readable exchange rate (`numerator` token atomic units per `denominator` native wei) for wallets that recompute the amount for a smaller gas limit or that cache terms and price subsequent transactions locally (see [Terms Expiry and Caching](#terms-expiry-and-caching)). `rateDisplay` is an optional human-readable form for UI.
- `rateExpiry` — how long the rate is valid.

The wallet selects a token option and constructs the [EIP-8130](./eip-8130.md) transaction with a token transfer to the payer in phase 0 (see [Transaction Construction](#transaction-construction)). Because [EIP-8130](./eip-8130.md) call phases commit independently, the token transfer in phase 0 persists even if later phases revert — this is how the payer is compensated. The payer accepts any transaction whose phase 0 transfer is at least its required cost and validates the token before co-signing.

#### Required Calls

A payer MAY return `requiredCalls` — calls the wallet MUST include in the transaction as a condition of sponsorship. These enable payer models where sponsorship is granted in exchange for executing specific onchain operations (e.g., a commitment to a privacy pool, a protocol interaction the payer needs). The wallet MUST prepend these calls to the sender's calls so they execute in phase 0. The payer validates their presence in `payer_sendTransaction` / `payer_signTransaction` and MUST reject the transaction if they are missing or altered.

#### Recipient Address

A payer MAY return a `recipient` address indicating where value or profit from the transaction's execution should flow. This enables sponsorship models where the payer covers gas in exchange for execution profit — the payer validates (via simulation or proof) that the transaction will deliver value to `recipient` before co-signing. The payer MUST reject in `payer_sendTransaction` / `payer_signTransaction` if the requirement is not satisfied.

#### Sponsorship Balance

Sponsorship is frequently *budgeted*: a payer grants a given sender a finite amount of free gas (a periodic allowance, a per-campaign cap) or holds prepaid credit on the sender's behalf. `payer_getTerms` MAY return a `balance` object describing how much remains, and a payer MAY expose the same information without a transaction intent via [`payer_getBalance`](#payer_getbalance-optional).

A `balance` has a `kind`:

- `sponsorship` — a free budget the payer covers on the sender's behalf (e.g., a subscription gas allowance). When it reaches zero the payer stops sponsoring, and the wallet SHOULD fall back to another model (token payment or another payer).
- `credit` — spendable, prepaid value the sender holds with the payer, which the payer draws down to cover gas.

`available` is the remaining amount in atomic units of `asset`. `limit` and `spent` are optional and let the wallet render progress (e.g., "$5 of $10 used"). `resetAt` indicates when a periodic budget refills. A `balance` is **advisory**: it reflects the payer's accounting at response time and is not a commitment to sponsor any specific transaction — the binding decision is always made in `payer_getTerms` / `payer_sendTransaction`.

Eligibility is the payer's concern: the payer recognizes the `from` address (e.g., an address the user has enrolled in a subscription) and reports the applicable balance. No additional wire fields are required to express enrollment.

#### Terms Expiry and Caching

This method returns an `expiry` timestamp (absolute, in seconds) and a `ttl` (the equivalent duration in seconds from the time of the response) indicating how long the terms are valid. The two express the same lifetime; `ttl` is provided for cache layers that key on relative duration. Wallets MAY cache and reuse terms until `expiry`. After expiry, wallets MUST re-request terms — the payer address, rates, or conditions may have changed.

Caching is per intent-independent terms, not per transaction: the `payer` address, `conditions`, and each token option's `rate` do not depend on the specific calls, so a wallet that has valid cached terms SHOULD reuse them across multiple transactions **without calling `payer_getTerms` again**. For each transaction the wallet recomputes the phase 0 amount locally from the cached `rate` and that transaction's gas cost (`amount = ceil(gas_limit × max_fee_per_gas × rate.numerator / rate.denominator)`), or transfers the cached `maxCost` when it reuses the quoted gas cap. The wallet MUST re-request terms once `expiry` (or the selected option's `rateExpiry`, whichever is sooner) has passed. Token-specific `maxCost` values are only exact for the gas parameters they were quoted against; the `rate` is the cacheable primitive for re-pricing.

#### `payer_sendTransaction`

Accepts a sender-signed [EIP-8130](./eip-8130.md) transaction, co-signs it, and submits it onchain. Returns the transaction hash. The transaction MUST have the `payer` field set to the payer's address and `payer_auth` left empty. The sender's signature (`sender_auth`) covers all fields through `payer`, binding the transaction to the specific payer. The payer service computes the `payer_signature_hash` (using `AA_PAYER_TYPE`) and fills in `payer_auth` with `verifier || data`.

Payers MUST NOT mutate any sender-signed transaction fields. If the payer cannot accept the transaction as-is (e.g., gas limit too low), the payer MUST reject the request and SHOULD return actionable error data.

#### `payer_signTransaction`

Accepts a sender-signed [EIP-8130](./eip-8130.md) transaction, co-signs it, and returns the fully-signed transaction without submitting. The wallet is responsible for submission. Payers MUST NOT mutate any sender-signed transaction fields. If the payer cannot accept the transaction as-is (e.g., gas limit too low), the payer MUST reject the request and SHOULD return actionable error data.

#### `payer_getBalance` (OPTIONAL)

Returns the sender's **standing** balances — budgeted sponsorship allowances and prepaid credit — held with the payer, **without** a transaction intent. Wallets use it to show users how much free gas or credit they have and to decide up front whether to attempt sponsorship or fall back to payment. It carries no pricing — pricing is always per-quote from `payer_getTerms`.

This method returns only balances that are **intent-independent**: a `sponsorship` allowance (e.g., a subscription's monthly free-gas budget) or a `credit` balance the sender holds. It MUST NOT attempt to enumerate intent-based full sponsorship — whether a payer will fully sponsor a *specific* transaction is not a balance and is evaluated per intent in `payer_getTerms` / `payer_getSponsorshipOptions`. A payer with no standing balance for `from` returns an empty `balances` array.

`payer_getBalance` takes no `calls`. When an intent *is* available, the wallet does not use this method — it calls `payer_getTerms` (or `payer_getSponsorshipOptions`) instead. Those are already scoped to the relevant payer: in the app-directed flow that is the app-provided `paymasterService` endpoint (the app's paymaster), and the applicable balance is returned **inline** via the `balance` field on the terms result. In other words, intent-free balance lookups fan out across an aggregator's services, while intent-bearing requests narrow to the payer(s) that can serve that intent.

A payer MAY return multiple balances (e.g., a `sponsorship` allowance and a `credit` balance). When invoked through an app-provided `paymasterService`, the caller MAY pass the capability's `context` (e.g., a `policyId`) to scope the result to an app- or policy-specific allowance; the `context` is opaque to the wallet.

##### Aggregator Usage

When the endpoint is an aggregator fronting multiple payer services, each returned balance MUST carry source attribution (`payer`, `endpoint`, and SHOULD carry `name`) so the wallet can tell which service holds it. To target a single service — "give me the balance of service X" — the caller sets `payer` (the service's address) or `endpoint` (its URL); the aggregator then returns only that service's balances. Service identities are discoverable from `payer_getCapabilities`, which lists each service's `payer` and `endpoint`. To list only what the user actually holds — "do I have any credit?" — the caller sets `kind: ["credit"]`.

##### `payer_getBalance` RPC Specification

```typescript
type GetBalanceParams = {

  kind?: ("sponsorship" | "credit")[]; // filter; e.g. ["credit"] to list only credit
```

#### `payer_getSponsorshipOptions` (OPTIONAL)

Returns a ranked list of sponsorship options for a transaction intent. Options are either `full_sponsorship` (payer covers gas unconditionally) or `conditional` (payer sponsors in exchange for an action — most commonly token payment, but also required calls or recipient requirements). This method MAY be implemented by individual payers or by aggregator services.

##### `payer_getSponsorshipOptions` RPC Specification


#### `payer_getCapabilities` (OPTIONAL)

Returns a static, **intent-free** description of what a payer accepts — which chains it serves, which tokens it accepts for gas payment, whether it offers full sponsorship, and its standing conditions. Unlike `payer_getTerms` and `payer_getSponsorshipOptions`, this method takes no `from` and no `calls`: it answers "what does this payer support?" rather than "what are the terms for this specific transaction?". It carries no pricing — a `rate` is always per-quote and comes from `payer_getTerms`.

This method exists so that wallets and aggregators can discover acceptance without first constructing a probe transaction. A wallet can show "you can pay gas with USDC or cbBTC here" before the user builds an intent, and an aggregator can ingest a payer's capabilities to list it without sending speculative requests. Because the descriptor is the same regardless of the transaction, it is a portable payer profile: a payer configures what it accepts once and any aggregator can serve it.

The result includes a `ttl` (seconds) indicating how long the descriptor may be cached. Capabilities change infrequently, so wallets and aggregators SHOULD cache the descriptor and refresh it on `ttl` expiry rather than per transaction. A payer MAY filter the response to a single chain when `chainId` is supplied.

This method is discovery only. It is not a commitment: a payer MAY still reject a specific transaction in `payer_getTerms` / `payer_sendTransaction` (e.g., for policy, balance, or allowlist reasons). Inclusion of a token in `acceptedTokens` means the payer is generally willing to price it, not that every transaction paying in that token will be accepted.

### `paymasterService` Capability

This proposal reuses the `paymasterService` capability defined in [ERC-7677](./eip-7677.md). The app-side interface is identical — apps pass a URL and optional context. The wallet determines which RPC methods to call based on the transaction type it is constructing.

#### App Implementation

Apps provide wallets with a service URL using the `paymasterService` capability as part of an [EIP-5792](./eip-5792.md) `wallet_sendCalls` or [ERC-7836](./eip-7836.md) `wallet_prepareCalls` call, exactly as specified in [ERC-7677](./eip-7677.md).

The wallet will then make the appropriate RPC calls to the URL specified in the `paymasterService` capability field. For [EIP-8130](./eip-8130.md) transactions, the wallet uses the `payer_*` methods defined in this specification. For [ERC-4337](./eip-4337.md) user operations, the wallet uses the `pm_*` methods defined in [ERC-7677](./eip-7677.md).

The wallet MUST forward the capability's `context` object (e.g., a `policyId`) to the `payer_*` calls it makes — to `payer_getTerms` so the payer can apply the app's policy when quoting, and to `payer_sendTransaction` / `payer_signTransaction` at co-sign time. The `context` is opaque to the wallet.

#### Wallet Implementation

To conform to this specification, wallets constructing [EIP-8130](./eip-8130.md) transactions that wish to leverage payer-sponsored transactions:

1. MUST indicate to apps that they support the `paymasterService` capability via their response to an [EIP-5792](./eip-5792.md) `wallet_getCapabilities` call.
2. MUST call the `payer_*` methods defined in this specification (not the `pm_*` methods from [ERC-7677](./eip-7677.md)) when constructing [EIP-8130](./eip-8130.md) transactions.
3. SHOULD make calls to and use the values returned by the payer service specified in the capabilities field. Since there might be cases in which the provided payer is ultimately not used—either due to service failure or due to a user selecting a different payer—applications MUST NOT assume that the payer provided to a wallet is the entity that pays for transaction fees.

#### `wallet_sendCalls` Flow

When `wallet_sendCalls` is called with the `paymasterService` capability and the wallet is constructing an [EIP-8130](./eip-8130.md) transaction:

1. Wallet OPTIONALLY calls `payer_getTerms` at the provided URL.
2. Wallet constructs the [EIP-8130](./eip-8130.md) transaction with call phases, the `payer` field set to the payer's address, and `payer_auth` empty (see [Transaction Construction](#transaction-construction)).
3. Wallet signs the transaction (`sender_auth`).
4. Wallet calls `payer_sendTransaction` (relay) or `payer_signTransaction` (co-sign only) at the payer endpoint.
5. Payer validates the transaction, computes `payer_signature_hash`, fills in `payer_auth`, and submits or returns the co-signed transaction.

#### `wallet_prepareCalls` Flow (ERC-7836)

When `wallet_prepareCalls` is called with the `paymasterService` capability and the wallet is constructing an [EIP-8130](./eip-8130.md) transaction:

1. Wallet OPTIONALLY calls `payer_getTerms` at the provided URL to negotiate payment terms.
2. Wallet constructs the unsigned [EIP-8130](./eip-8130.md) transaction incorporating negotiated terms (gas parameters, call phases, expiry) with the `payer` field set to the payer's address.
3. Wallet computes the transaction digest and returns it to the app.
4. App signs the digest (e.g., with a session key) and calls `wallet_sendPreparedCalls`.
5. Wallet assembles the signed transaction and calls `payer_sendTransaction` or `payer_signTransaction`.

This enables apps to use session keys ([ERC-7836](./eip-7836.md)) while leveraging external payers for gas sponsorship or token payment, without the app needing to implement the payer RPC protocol.

### Wallet-Selected Payers

The `paymasterService` capability covers the case where an *app* directs the wallet to a service. It is not the only source of a payer. A wallet that wants its users to pay gas in a token — with no app involvement — selects a payer itself and uses the same `payer_*` methods against that endpoint.

The payer in this case is one the wallet trusts and configures, typically one of:

- a payer service the wallet operator runs or contracts with, configured per chain;
- the chain's block builder (L2 sequencer or L1 proposer) acting as payer directly (see [Block Builder Inclusion](#block-builder-inclusion)); or
- an option the user picks from a `payer_getSponsorshipOptions` result.

To populate the set of tokens a user can pay with — before any transaction is built — the wallet SHOULD call `payer_getCapabilities` on its configured payer(s) and cache the result for `ttl`. This lets the wallet render a token picker without sending a probe transaction; the per-token `rate` is fetched later from `payer_getTerms` once the user has chosen.

The construction, signing, and submission steps are identical to the app-directed flow; only the source of the endpoint differs. When both an app-provided `paymasterService` and a wallet-default payer are available, the wallet decides which to use — for example, preferring app sponsorship when offered and falling back to its default token-payment payer otherwise. Because `sender_auth` binds the chosen `payer`, the user always signs the exact payer that will be used.

### Transaction Construction

This section defines how wallets construct [EIP-8130](./eip-8130.md) transactions using payer terms. The [EIP-8130](./eip-8130.md) AA transaction wire format is:

```
AA_TX_TYPE || rlp([
  chain_id, sender, nonce_key, nonce_sequence, expiry,
  max_priority_fee_per_gas, max_fee_per_gas, gas_limit,
  account_changes, calls,
  payer, sender_auth, payer_auth
])
```

The `calls` field is a two-level array of **phases**: `[[call, ...], [call, ...]]` where each `call` is `[to, data]`. Phases execute in order; each phase is atomic (all-or-nothing), but completed phases persist independently.

#### Constructing the Transaction

Given terms from `payer_getTerms`, the wallet constructs the transaction as follows:

1. **Set `payer`** to the `payer` address from the terms response.
2. **Set `nonce_key` and `nonce_sequence`** — select a `nonce_key` (a parallel-nonce lane; `0` for sequential ordering) and read the next `nonce_sequence` for `(sender, nonce_key)` via `eth_getTransactionCount`.
3. **Set `expiry`** within the payer's `conditions.maxExpiry` / `conditions.minExpiry` bounds.
4. **Set gas parameters** (`max_fee_per_gas`, `max_priority_fee_per_gas`, `gas_limit`) from the payer's `gasEstimate` (recommended; see [Gas Estimation](#gas-estimation)) or the wallet's own [EIP-8130](./eip-8130.md)-aware estimation, respecting `conditions.maxGasLimit`.
5. **Construct `calls`** using call phases:

| Payer Model | Phase 0 | Phase 1 |
|---|---|---|
| Full sponsorship | — | `[user_calls]` |
| Token payment | `[token.transfer(payer, maxCost)]` | `[user_calls]` |
| Required calls | `[required_calls]` | `[user_calls]` |
| Token + required | `[token_transfer, required_calls]` | `[user_calls]` |

For token payment, the transfer amount is the selected option's `maxCost` (see [Token Payment](#token-payment)); a wallet that reuses cached terms across transactions instead recomputes it from `rate` and this transaction's gas cost. For full sponsorship with no phase 0 requirements, the wallet MAY use a single phase: `[[user_calls]]`.

**Balance-funded sponsorship** (a `sponsorship` allowance or prepaid `credit`, per [Sponsorship Balance](#sponsorship-balance)) uses the **full-sponsorship** construction — no phase 0 transfer. The transaction carries no onchain compensation; the payer is reimbursed off-chain from the sender's budget or credit ledger, which it debits when it co-signs.

6. **Leave `payer_auth` empty** — the payer fills this in after receiving the sender-signed transaction.
7. **Sign** — the wallet computes `sender_signature_hash` over all fields through `payer` (excluding `sender_auth` and `payer_auth`) and produces `sender_auth`.

#### Token Payment via Call Phases

[EIP-8130](./eip-8130.md) calls carry no ETH value — token transfers are executed through contract calls. For token payment, the wallet constructs a phase 0 call that transfers tokens from the sender to the payer:

```
phase_0 = [
  [token_address, abi.encodeCall(IERC20.transfer, (payer_address, maxCost))]
]
```

The amount is the selected token option's `maxCost` — the payer's quoted cost at the gas cap. Transferring `maxCost` is always sufficient while the terms are valid, so the wallet does not need to do its own rate arithmetic. A wallet that prices from cached terms instead computes the amount from the option's `rate` and this transaction's gas cost (see [Terms Expiry and Caching](#terms-expiry-and-caching)). Phase 0 commits independently, so the payer receives tokens regardless of whether the user's calls in phase 1 succeed or revert.

#### Payer Co-Signing

The payer receives the sender-signed transaction (with `payer_auth` empty), validates it against its terms, computes `payer_signature_hash` using `AA_PAYER_TYPE`:

```
payer_signature_hash = keccak256(AA_PAYER_TYPE || rlp([
  chain_id, sender, nonce_key, nonce_sequence, expiry,
  max_priority_fee_per_gas, max_fee_per_gas, gas_limit,
  account_changes, calls
]))
```

The payer signs this hash and fills in `payer_auth` with `verifier || data`. The payer's actor MUST have PAYER scope in its `actor_config` on the payer account.

#### Signing Services (Delegated Co-Signing)

The party that quotes terms and the key that produces `payer_auth` need not be operated by the same entity. A payer service MAY delegate co-signing to a separate **signing service** that custodies the key for a PAYER-scoped actor on the `payer` account. In this split:

- the **payer service** implements `payer_getTerms` (pricing and policy) and validates the sender-signed transaction before it is co-signed (e.g., that the phase 0 token transfer covers its cost); and
- the **signing service** holds the PAYER-scoped key, computes `payer_signature_hash`, returns `payer_auth` for the `payer` account, and MAY relay the transaction.

The `payer` field always identifies the account whose actor signs, so `sender_auth` binds the transaction to that account regardless of which operator runs the endpoint or holds the key. This lets an operator run only its own pricing logic while outsourcing key management, native-gas float, and submission to a provider — or run pricing locally and have the provider both sign and submit. A managed provider MAY also custody the `payer` account itself (with the operator configured as a PAYER-scoped actor), so the operator never handles a signing key.

#### Token Cost Transparency

Because the token cost is fixed before the sender signs, wallets can display it precisely. The displayed fee is the phase 0 transfer amount — normally the selected option's `maxCost` — rendered with the option's `decimals` and `symbol`. Because the transfer is an exact, sender-signed amount, the displayed value is the precise maximum the user spends in that token; the wallet does not need to fetch token metadata or parse a rate string to show it. A wallet that prices from cached terms derives the amount as `ceil(gas_limit × max_fee_per_gas × rate.numerator / rate.denominator)` and may show `rateDisplay` as supporting context. The quote is valid until the option's `rateExpiry`; after that the wallet MUST re-request terms. Because `maxCost` is quoted at the gas cap, it already covers minor gas movement without re-signing.

### Block Builder Inclusion

A payer need not be an external web service — the block builder (an L2 sequencer or L1 proposer) can be the payer directly. The wallet sets `payer` to the builder's address and constructs the same phase 0 token transfer; the builder pays the native gas and is reimbursed by that transfer, periodically swapping the collected token back to the native asset to replenish its float. Where the chain recognizes a builder/sentinel payer, `payer_auth` MAY be left empty and inclusion of the transaction is itself the builder's authorization; otherwise the builder co-signs `payer_auth` like any other payer.

Because the builder both pays the native gas and decides inclusion, it self-protects rather than relying on a protocol-level reservation. Before including, the builder validates off-chain that the phase 0 transfer token is one it accepts, that the amount covers `gas_limit × max_fee_per_gas × rate`, and that the sender holds the balance. A builder never includes a transaction that would not reimburse it.

#### Validity and Dropping

Nothing is committed until inclusion, so a transaction that becomes uneconomical is simply dropped with no on-chain effect or cost to the sender. Builders and mempool nodes MAY drop a transaction when:

- the phase 0 transfer **underpays** — its amount no longer covers `gas_limit × max_fee_per_gas × rate` at the current rate, or
- the sender's token **balance no longer covers** the phase 0 transfer amount.

This mirrors ordinary insufficient-fee and insufficient-balance handling: the transaction becomes non-includable rather than failing on-chain.

### Endpoint Versioning

Payer service endpoints MUST use path-based major versioning (e.g., `/v1`, `/v2`). Method names remain stable across versions. Breaking changes (new required fields, changed semantics) require a new major path version. Additive optional fields in request or response are backwards-compatible within the same version. Clients MUST ignore unknown fields. Payers MUST NOT require new optional fields within an existing version.

## Rationale

### Why Extend ERC-7677's Capability

From an app's perspective, sponsorship is sponsorship — the app passes a service URL and the wallet handles the rest. By reusing [ERC-7677](./eip-7677.md)'s `paymasterService` capability rather than defining a new one, apps gain [EIP-8130](./eip-8130.md) support without code changes. The wallet selects the correct RPC methods (`pm_*` for [ERC-4337](./eip-4337.md), `payer_*` for [EIP-8130](./eip-8130.md)) based on the transaction type it is constructing. Service providers implement both method sets at the same URL.

### Why New RPC Methods

While the capability is shared, the RPC methods must differ. [ERC-7677](./eip-7677.md)'s two-phase `pm_getPaymasterStubData` / `pm_getPaymasterData` flow exists because [ERC-4337](./eip-4337.md) paymasters inject data into the UserOperation *before* gas estimation, and the stub data must produce consistent code paths and byte lengths. In [EIP-8130](./eip-8130.md), the payer simply co-signs the complete transaction — there is no paymaster data injection, no stub values, and no code path consistency concern. The `payer_*` methods reflect this simpler model.

### Why `payer_getTerms` Is Optional

For known payers (e.g., an app's own backend), the wallet can skip `payer_getTerms` and call `payer_sendTransaction` or `payer_signTransaction` directly. The payer rejects with an actionable error code if terms are not met. This optimistic path reduces round trips for the common case. Payer services MUST return sufficient error detail (e.g., `RATE_EXPIRED` with updated terms in error data) to enable recovery without a separate terms call.

### Why an Intent-Free Capability Descriptor

`payer_getTerms` and `payer_getSponsorshipOptions` are both intent-scoped: they require a sender and concrete calls because they return a *quote* for that transaction. That is the wrong shape for two common needs. First, a wallet often wants to show which tokens a user can pay gas with *before* the user has assembled a transaction — requiring a probe transaction just to populate a token picker is awkward and leaks intent. Second, an aggregator listing many payers should not have to send speculative quote requests to every payer to know who serves a chain or accepts a token.

`payer_getCapabilities` answers the static question — "what does this payer support?" — separately from the per-transaction question — "what are the terms?". It deliberately omits pricing: a `rate` is only meaningful against a specific gas cost and freshness window, so it stays in `payer_getTerms`. Keeping the descriptor pricing-free also lets it be cached aggressively (`ttl`) and published as a portable profile: a payer declares what it accepts once, and any aggregator can ingest and list it without coupling to that payer's quoting endpoint.

### `payer_sendTransaction` vs `payer_signTransaction`

Two separate methods rather than a mode parameter, mirroring the `eth_sendTransaction` / `eth_signTransaction` convention:

- **`payer_sendTransaction`** minimizes hops — the payer co-signs and submits in one step. This is the common case for app-sponsored transactions.
- **`payer_signTransaction`** gives the wallet submission control — useful for high-value transactions, MEV-sensitive paths, or when the wallet wants to use private mempools.

Payer services SHOULD implement both methods. The choice of which to call is the wallet's, not the app's.

### Gas Estimation

Unlike a legacy call, an [EIP-8130](./eip-8130.md) transaction carries intrinsic costs that a node's `eth_estimateGas` does not model — sender and payer authentication, `account_changes` application, and the ordered, per-phase `calls` structure. Estimating the user's calls as if they were a normal transaction therefore undercounts. For this reason the payer's `gasEstimate` is the recommended source: the payer constructs and prices the full [EIP-8130](./eip-8130.md) transaction shape it will co-sign. The payer's `conditions.maxGasLimit` serves as the binding constraint — a wallet that prefers to estimate independently MAY do so as long as it includes the [EIP-8130](./eip-8130.md) intrinsics and the result satisfies the payer's conditions.

### Chain ID in Requests

Following the rationale from [ERC-7677](./eip-7677.md): requiring `chainId` as a parameter rather than mandating URL-per-chain allows payer services to offer multichain endpoints at no cost to single-chain providers.

### Terms Expiry

The top-level `expiry` on `GetTermsResult` bounds how long the entire terms response is valid — including the payer address, gas estimate, token rates, and conditions. This serves multiple purposes:

- **Payer address rotation**: A payer or payer pool may rotate addresses. The `expiry` tells the wallet when to re-fetch and potentially receive a different `payer` address.
- **Rate freshness**: Even when individual `tokenOptions` have a `rateExpiry`, the overall terms `expiry` provides a single cache key for the full response.
- **Gas estimate staleness**: Gas conditions change. The `expiry` bounds how long the wallet can rely on the payer's `gasEstimate`.

The `expiry` is distinct from `conditions.maxExpiry` — the latter constrains the transaction's onchain expiry field, while the terms `expiry` constrains how long the off-chain terms are usable.

### Required Calls

The `requiredCalls` field enables payer models where sponsorship is conditional on executing specific onchain operations. The payer returns calls the wallet must prepend (phase 0 execution), and the payer validates their presence when co-signing. This supports:

- **Privacy payers**: Sponsor gas in exchange for a commitment to a privacy pool.
- **Protocol payers**: Sponsor gas in exchange for interacting with a specific protocol.
- **Conditional sponsors**: Any model where the payer's business logic requires a specific onchain action.

The payer enforces these requirements at co-sign time — if the signed transaction does not contain the required calls, the payer rejects with `POLICY_REJECTED`.

### Recipient Address

The `recipient` field enables sponsorship models where the payer covers gas in exchange for execution profit flowing to a specified address. The payer simulates the transaction before co-signing and verifies that `recipient` will receive value. This supports solver-style payers that sponsor transactions they can extract value from, shifting gas costs from users to market participants who profit from execution.

### Expiry as a Sponsorship Primitive

[EIP-8130](./eip-8130.md) transactions have a native `expiry` field enforced at the protocol level. Payers can express time-bounded commitments via `conditions.maxExpiry` — the protocol guarantees the transaction cannot land after that time. This is strictly stronger than [ERC-4337](./eip-4337.md) where paymasters must encode and validate time bounds in their own data during execution.

### Sponsorship Balance

Budgeted sponsorship — a finite free-gas allowance or prepaid credit a payer holds for a sender — is increasingly common (e.g., subscription products that bundle gas). Surfacing the remaining `balance` lets a wallet show the user what they have left and route the next transaction accordingly: use the free allowance while it lasts, then fall back to token payment or another payer. The `balance` is deliberately **advisory** — a snapshot of the payer's accounting, not a commitment. Policy and concurrency (other in-flight transactions) can change what the payer accepts at co-sign time, so binding acceptance is always decided in `payer_getTerms` / `payer_sendTransaction`, while `payer_getBalance` stays cheap and cacheable. This mirrors how `payer_getCapabilities` is discovery, not commitment. Eligibility is left to the payer: it recognizes the enrolled `from` address and reports the applicable allowance, so no new wire fields are needed to express enrollment.

`payer_getBalance` is deliberately scoped to **standing, intent-independent** balances — sponsorship allowances and prepaid credit the user holds. Intent-based full sponsorship (will payer X cover *this* call?) is not a balance and cannot be answered without the calls, so it is left to `payer_getTerms` / `payer_getSponsorshipOptions`. This keeps "do I have any credit?" a single cheap lookup rather than a fan-out of speculative quotes. For the aggregator case, every balance carries `payer` / `endpoint` attribution so a wallet can present "$5 on Coinbase One, $12.50 on Gas Market" and route to a specific service, and the `payer` / `endpoint` request filters let a wallet ask one aggregator for just one service's balance.

## Backwards Compatibility

This proposal extends the `paymasterService` capability defined in [ERC-7677](./eip-7677.md) with new RPC methods for [EIP-8130](./eip-8130.md) transactions. Existing [ERC-4337](./eip-4337.md) wallets are unaffected — they continue to use the `pm_*` methods as before. Apps that already pass a `paymasterService` capability require no changes to support [EIP-8130](./eip-8130.md) wallets, provided their service endpoint implements the `payer_*` methods defined here.

Service providers that wish to support both wallet types SHOULD implement the `pm_*` methods from [ERC-7677](./eip-7677.md) and the `payer_*` methods from this specification at the same endpoint URL.

## Security Considerations

**API key exposure**: Payer service URLs commonly contain API keys. App developers SHOULD proxy payer requests through their own backend to keep keys secret, identical to the recommendation in [ERC-7677](./eip-7677.md). Payer services SHOULD support scoped API keys restricted to specific methods.

**Relay trust**: In relay mode, the wallet trusts the payer to submit the transaction. The transaction's `expiry` field bounds this exposure — if the payer does not submit before expiry, the transaction is dead. Wallets concerned about relay trust SHOULD use cosign mode and submit themselves.

**Payer immutability**: Payers MUST NOT alter any sender-signed fields. The sender's signature covers the exact transaction that lands onchain. Any modification invalidates the signature.

**Rate commitment**: Once a payer returns terms with a `rateExpiry`, the payer SHOULD honor those terms until expiry. Because `maxCost` is quoted at the gas cap, transferring it already absorbs normal gas movement; wallets pricing from a cached `rate` can add a small buffer and use short expiry windows to stay includable.

**Balances are advisory**: A returned `balance` is informational and MUST NOT be treated as a spend guarantee. Wallets MUST still handle `BALANCE_EXHAUSTED` and other rejections at co-sign time, since concurrent transactions or policy may invalidate a stale balance snapshot. Payers MUST enforce the actual budget server-side and MUST NOT rely on the wallet honoring the advertised balance.
