## Abstract

[ERC-7677](./eip-7677.md) defines the `paymasterService` capability for [EIP-5792](./eip-5792.md), allowing apps to direct wallets to a sponsorship web service. This proposal extends that capability to [EIP-8130](./eip-8130.md) by defining `payer_*` RPC methods that wallets use when constructing [EIP-8130](./eip-8130.md) transactions. Apps pass the same `paymasterService` URL they already use; the wallet selects the appropriate RPC methods based on the transaction type it is constructing. This proposal additionally defines sponsorship **balances**, so a wallet can read how much sponsorship budget or prepaid credit a sender has remaining (e.g., a subscription gas allowance) and fall back to token payment when it is exhausted.

## Motivation

[EIP-8130](./eip-8130.md) introduces native account abstraction where payers are first-class protocol participants that co-sign the transaction using the same `authenticator || data` model as senders, validated with PAYER scope. This differs fundamentally from the [ERC-4337](./eip-4337.md) paymaster model (no EntryPoint, no stub-data two-phase flow, the payer co-signs the exact transaction that lands onchain) and enables more efficient flows, including direct relay and protocol-enforced `expiry`.

[ERC-7677](./eip-7677.md) defines the `pm_*` RPC methods and lets an app tell a wallet "here is a sponsorship service URL" via the `paymasterService` capability. Those `pm_*` methods do not fit [EIP-8130](./eip-8130.md), so this proposal defines `payer_*` methods and reuses the same capability, so apps need no new integration: the wallet picks `payer_*` or `pm_*` based on the transaction type it is constructing.

This proposal supports three payment models: full sponsorship by an app, sponsorship drawn from a standing balance (e.g. a subscription allowance or prepaid credit), and token payment for gas.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

### Payer Web Service Interface

`payer_getTerms` and `payer_sendTransaction` are REQUIRED; `payer_signTransaction` and `payer_getSponsorshipBalance` are OPTIONAL. Relaying is near-universal (any payer can forward a signed transaction), so requiring `payer_sendTransaction` makes the common case work against every payer with no discovery; `payer_signTransaction` (co-sign and return the bytes for the wallet to submit) is the opt-in path for wallets that want submission control.

**Common case.** Most integrations need only two calls: `payer_getTerms` to obtain payment offers, then `payer_sendTransaction` to have the payer co-sign and submit the transaction. Both take a fully sender-signed transaction; the payer co-signs (fills `payer_auth`) and either submits (`payer_sendTransaction`) or, when offered, returns the co-signed bytes (`payer_signTransaction`). Because `payer_sendTransaction` is REQUIRED, this flow needs no capability probe. A wallet that wants to submit the transaction itself uses `payer_signTransaction`; each offer in `payer_getTerms`'s result advertises whether that path is available against *its* `(payer, endpoint)` via the offer's `methods` field. The remaining method is optional read-only discovery: `payer_getSponsorshipBalance`. See [`wallet_sendCalls` Flow](#wallet_sendcalls-flow).

Every responder MUST implement `payer_getTerms` and `payer_sendTransaction`. Each offer returned by `payer_getTerms` carries an **additive** list of optional `payer_*` methods supported by that offer in its `methods` field â€” the required pair is always implied and MUST NOT be restated. A wallet learns everything from the call it already makes and never needs a third hop just to discover methods, capping the transaction flow at two hops (`payer_getTerms` then a co-sign call). A wallet MUST NOT invoke an OPTIONAL method against an offer that does not list it. The `payer_*` prefix denotes the *role* of the responder (acting as a payer for an [EIP-8130](./eip-8130.md) transaction); methods are served over an HTTPS JSON-RPC endpoint.

All durations in this specification are expressed as **relative seconds** measured from the time the response is sent (`ttl`, `conditions.maxExpiry`, `RefundPolicy.window`, `PayerBalance.validFor`). Relative durations keep a payer's response portable across clients with skewed clocks; a wallet caching a value records its own receipt time and counts down from there.

#### `payer_getTerms`

Returns sponsorship or token-payment terms for a transaction intent. Called *before* the sender signs. Payer services SHOULD validate the transaction intent at this stage and reject if they would not sponsor or accept payment.

`PayerBalance` is shared by `payer_getTerms` and `payer_getSponsorshipBalance`:

```typescript
type PayerBalance = {

  // multiple services, so the wallet can tell which service holds the balance.
  payer?: `0x${string}`; // the payer/service address this balance is held with
  endpoint?: string; // that service's endpoint URL

};

  spent?: `0x${string}`; // total lifetime spent on this axis (if tracked), same units as `available`
  asset?: string; // REQUIRED when unit="asset": token contract address, "native", or an ISO-4217 currency code (e.g. "USD") for a fiat-denominated budget
  symbol?: string;
  decimals?: number;
};
```

`RefundPolicy` describes when unused gas is returned to `from`. Its **presence** on a `TokenChoice` means refunds are offered for that choice; its absence means none are:

```typescript
type RefundPolicy = {
  window: number; // upper bound in seconds until the refund is settled (e.g. 86400 â‰ˆ next day)
};
```

Refund settlement happens **via an onchain transfer** â€” the payer sends `paymentAmount` minus the transaction's actual gas cost (priced through `rate`) to `from` as a separate [ERC-20](./eip-20.md) transfer, **paying the gas for that transfer itself**, within `window` seconds of inclusion. "Onchain" here means the *mechanism* of settlement is an onchain transfer, **not** that settlement is contract-enforced: nothing in this spec compels the payer to send it, so the obligation is trust-based, at the same trust layer as the payer's quote (see [Security Considerations](#security-considerations)). Phase 0 commits independently and is never partial; refunds always settle separately, which is why `payer_sendTransaction` / `payer_signTransaction` can return `estimatedRefund` immediately without waiting for the refund tx to land.

If the payer fails to settle within `window`, the sender has no onchain recourse from this specification. Wallets SHOULD display refund amounts as expected rather than guaranteed (e.g. "up to X refunded") and MAY surface the payer's identity, refund window, and reputation to the user accordingly. Payers SHOULD make refund settlements externally observable, for example by emitting an indexable event from a known address or settling through a publicly identifiable contract, so refund streams can be monitored by users and aggregators.

##### Co-Sign Disambiguation Invariant

The wallet does not echo which offer or `TokenChoice` it selected; the payer reconstructs the selection from the co-signed transaction alone. The only signals available at co-sign are the transaction's `payer` field and its phase 0. The **co-sign fingerprint** of a selectable construction is therefore:

- **`token` construction**: `(payer, phase-0 token, phase-0 amount, phase-0 recipient)` â€” i.e. `(offer.payer, choice.token, choice.paymentAmount, choice.feeRecipient ?? offer.payer)`.
- **`sponsored` construction** (no phase 0): `(payer)` alone â€” there is nothing else onchain to distinguish it.

Within a single `payer_getTerms` response, every selectable construction MUST have a **unique co-sign fingerprint**. This is stronger than per-offer uniqueness and has two consequences a payer MUST honor:

1. **At most one `sponsored` construction per `payer` address.** A fully-paid sponsored offer and a balance-funded one build identically (same `payer`, no phase 0), so they are indistinguishable at co-sign; a payer MUST NOT return both under the same `payer`. Whether to draw from a `balance` is the payer's own co-sign-time decision (the `balance` is advisory context for the wallet, not a selectable variant). A payer that genuinely needs the wallet to commit to one or the other MUST place them under **distinct `payer` addresses**.
2. **Token `(token, paymentAmount, feeRecipient)` fingerprints MUST be distinct across the whole response per `payer`,** not merely within one offer's `tokens`. Two `token` offers from the same `payer` that quote the same token at the same amount to the same recipient â€” but differ in `refund`, `ttl`, or `conditions` â€” are indistinguishable at co-sign and MUST NOT both appear; the payer either folds them into one offer's `tokens` with distinct amounts/recipients or routes them through distinct `payer` addresses.

A `sponsored` and a `token` construction under the same `payer` never collide: the sponsored fingerprint `(payer)` has no phase 0, while the token fingerprint `(payer, token, amount, recipient)` always carries a phase-0 transfer, so the mere presence or absence of phase 0 distinguishes them. The uniqueness requirement therefore only bites *within* a kind â€” between two sponsored constructions, or between two token constructions â€” under the same `payer`.

Because `payer` is part of every fingerprint and is bound by `sender_auth`, placing colliding constructions under distinct `payer` addresses always resolves the ambiguity without an echoed selector. The constraint is naturally satisfied in practice: a refundable choice charges a higher `paymentAmount` than a non-refundable one, distinct treasuries use distinct `feeRecipient`s, and most payers expose a single sponsored construction.

#### `payer_sendTransaction`

Accepts a sender-signed [EIP-8130](./eip-8130.md) transaction, co-signs it, and submits it onchain. Returns the transaction hash. The transaction MUST have the `payer` field set to the payer's address and `payer_auth` left empty; the payer service fills in `payer_auth` with `authenticator || data` after validating the transaction. Payers MUST NOT mutate any sender-signed fields. If the payer cannot accept the transaction as-is (e.g., gas limit too low), it MUST reject and SHOULD return actionable error data.

##### `payer_sendTransaction` RPC Specification

```typescript
type SendTransactionParams = {
  signedTransaction: `0x${string}`;
  context?: Record<string, any>;
};

type SendTransactionResult = {
  transactionHash: `0x${string}`;
  tokenCharged?: {
    token: `0x${string}`;
    amount: `0x${string}`; // gross phase-0 transfer (the selected choice's `paymentAmount`), known immediately
    estimatedRefund?: `0x${string}`; // payer's estimate of the refund to settle later, when a `refund` policy applies
  };
};
```

`tokenCharged` reports the token payment for this transaction: `token` identifies the charged token and `amount` is the gross phase-0 transfer (`paymentAmount`), known immediately without waiting for inclusion. The payer identifies the selected `TokenChoice` from phase 0 of the co-signed transaction (its `(token, paymentAmount, feeRecipient)` fingerprint, kept unique per the [Co-Sign Disambiguation Invariant](#co-sign-disambiguation-invariant)); the wallet does the symmetric local match against the choice it built phase 0 with. When the choice carried a `refund`, `estimatedRefund` is the payer's estimate of what will be returned to `from`; the exact refund settles separately per the [`refund`](#field-notes) policy, so the result need not block on the transaction landing.

#### `payer_getSponsorshipBalance` (OPTIONAL)

Returns a sender's **standing** balances (budgeted sponsorship allowances and prepaid credit) held with the payer, **without** a transaction intent. Wallets use it to show users how much free gas or credit they have and to decide whether to attempt sponsorship or fall back to payment before assembling an intent.

This method returns only **intent-independent** balances: a `sponsorship` allowance (e.g., a subscription's monthly free-gas budget) or a `credit` balance the sender holds. It MUST NOT attempt to enumerate intent-based full sponsorship; whether a payer will sponsor a *specific* transaction is evaluated per intent in `payer_getTerms`. When an intent is available, use `payer_getTerms` instead; the applicable balance is returned inline via the `balance` on the matching `sponsored` offer in the terms result. A payer with no standing balance for `from` returns an empty `balances` array.

When the endpoint is an aggregator fronting multiple services, each returned balance MUST carry source attribution (`payer`, `endpoint`, and SHOULD carry `name`) so the wallet can tell which service holds it. To target a single service, set `payer` (the service's address) or `endpoint` (its URL) in params; the aggregator returns only that service's balances. A payer MAY return multiple balances (e.g., a `sponsorship` allowance and a `credit` balance). When invoked through an app-provided `paymasterService`, the caller MAY pass the capability's `context` to scope the result to an app- or policy-specific allowance.

**Discovery.** Unlike `payer_signTransaction`, this method cannot be advertised through a per-offer `methods` field: its whole purpose is intent-independent (it runs *before* the wallet has an intent, so there is no `payer_getTerms` response and no offer to read `methods` from). The discovery mechanism is therefore **attempt-and-fallback**: a wallet that wants a standing balance first simply calls `payer_getSponsorshipBalance`, and a payer that does not implement it returns JSON-RPC `-32601` (Method not found), which the wallet treats as "no standing-balance API â€” proceed to `payer_getTerms`." Wallets SHOULD cache the `-32601` outcome per endpoint to avoid re-probing. (Once the wallet is in an intent flow, the applicable balance is returned inline on the matching `sponsored` offer's `balance`, so the standalone method is only needed for the pre-intent case.)

### Payer Error Codes

This spec uses **one canonical vocabulary** for payer-domain failures across both channels â€” `sponsored_declined.code` on a `payer_getTerms` response, and `data.code` on a JSON-RPC error from `payer_sendTransaction` / `payer_signTransaction`. The string is the canonical condition name; the numeric JSON-RPC `code` is a single envelope. Consumers MUST branch on the string and MAY ignore the numeric except for routing the envelope.

JSON-RPC envelopes used by payer methods:

- `-32000` **`PAYER_REJECTED`** â€” every payer-domain rejection from `payer_sendTransaction` / `payer_signTransaction` uses this envelope, with the specific condition in `data.code` (string) and any actionable detail (e.g. suggested gas limit, updated terms, `balance`) alongside it.
- Standard JSON-RPC codes (`-32600` Invalid Request, `-32601` Method not found, `-32602` Invalid params, `-32603` Internal error, `-32700` Parse error) keep their protocol meanings and are returned for protocol-level faults, not for payer business logic.

`payer_getTerms` MUST NOT return a JSON-RPC error to signal "I cannot sponsor this intent." A query the payer evaluated successfully â€” even one that yields no offer â€” is a success response with an `options` array; sponsorship refusals ride on a `sponsored_declined` entry, and token absence is expressed by the absence of `token` offers. This keeps the optimistic single-call flow on the success path and gives the wallet structured, displayable reasons without an exception channel. `payer_getTerms` MAY still return standard JSON-RPC errors for genuine protocol faults, plus `-32000` with `data.code: "TEMPORARILY_UNAVAILABLE"` if the payer cannot even evaluate the intent.

```typescript
type PayerErrorCode =
  // Sponsorship-decline scope â€” appear on both `sponsored_declined.code` and
  // `data.code` of a `PAYER_REJECTED` envelope.
  | "POLICY_REJECTED"        // calls/sender/contract not covered by this sponsor's policy for this intent; permanent for this tx
  | "SENDER_INELIGIBLE"      // sender blocklisted or missing a required attestation; permanent until the sender changes
  | "BUDGET_EXHAUSTED"       // sponsor's budget or sender's allowance is depleted; check `balance.validFor` for refill
  | "SENDER_LIMIT_REACHED"   // per-sender cap reached (tx count or USD); transient â€” check `balance.validFor`
  | "GAS_EXCEEDS_LIMIT"      // current gas prices exceed this payer's per-tx cost ceiling; transient â€” retry when gas drops. On a `sponsored_declined` entry the payer SHOULD include the `gas` diagnostic (estimatedCost vs maxCost) so the wallet can render the gap. Forward-looking: a selectable offer's `conditions.maxCost` advertises the same ceiling.
  | "TEMPORARILY_UNAVAILABLE" // payer is degraded (e.g. rate feed down); retry shortly
  // Send-only â€” describe a fault in a *committed* transaction, so they never
  // appear on `sponsored_declined.code` (which is returned before signing).
  | "INVALID_TRANSACTION"    // malformed or invalid [EIP-8130](./eip-8130.md) transaction
  | "UNSUPPORTED_TOKEN"      // token not accepted by this payer
  | "RATE_EXPIRED"           // terms have expired; re-call `payer_getTerms`
  | "PAYMENT_INSUFFICIENT"   // phase-0 transfer below required cost (gas moved past quote); re-quote and re-sign
  | "EXPIRY_OUT_OF_BOUNDS"   // transaction `expiry` does not satisfy `conditions.maxExpiry`
  | "PAYER_BALANCE_INSUFFICIENT" // payer lacks ETH to cover gas at the protocol level
  | string; // payers MAY return their own; wallets MUST treat unknown codes as opaque, displaying `reason` when present

// The decline-scope subset â€” the only codes that may appear on a
// `DeclinedSponsorship.code` (send-only fault codes describe a committed
// transaction and so never apply pre-signing). `DeclinedSponsorship` is typed
// against this narrower set; both channels still draw from one vocabulary.
type SponsorshipDeclineCode =
  | "POLICY_REJECTED"
  | "SENDER_INELIGIBLE"
  | "BUDGET_EXHAUSTED"
  | "SENDER_LIMIT_REACHED"
  | "GAS_EXCEEDS_LIMIT"
  | "TEMPORARILY_UNAVAILABLE"
  | string; // extensible, same opaque-unknown rule as PayerErrorCode
```

The "Channel" column shows where each code MAY appear; the "Permanent?" column tells a wallet whether to retry the same intent against the same payer.

|---|---|---|---|
| `POLICY_REJECTED` | declined + send/sign | Yes | Calls or contracts rejected by this sponsor's policy for this intent. Pay with a token option or use a different payer. |

| `BUDGET_EXHAUSTED` | declined + send/sign | No | Sponsor budget or prepaid credit is depleted; `balance.validFor` indicates when it refills. |
| `SENDER_LIMIT_REACHED` | declined + send/sign | No | Per-sender cap reached (tx count or USD); `balance.validFor` indicates when it refills. |
| `GAS_EXCEEDS_LIMIT` | declined + send/sign | No | Sponsorship is available in principle but current gas exceeds this payer's per-tx ceiling. | `PAYMENT_INSUFFICIENT` | send/sign | No | Phase-0 transfer below required cost (gas moved past quote); re-quote and re-sign. |
| `EXPIRY_OUT_OF_BOUNDS` | send/sign | No | Transaction `expiry` exceeds the offer's `conditions.maxExpiry`. |
| `PAYER_BALANCE_INSUFFICIENT` | send/sign | No | Payer lacks ETH to cover gas at the protocol level. |

#### Re-Quote on `RATE_EXPIRED`

The phase-0 transfer is a plain [ERC-20](./eip-20.md) transfer: nothing onchain binds it to the quote, and gas can move past the quote between `payer_getTerms` and co-sign. The send-time check is a payer's last line of defense â€” if the quote no longer covers the transaction, the payer MUST return `PAYER_REJECTED` with `data.code: "RATE_EXPIRED"` (or `"PAYMENT_INSUFFICIENT"` when the wallet under-paid relative to a still-valid `rate`). Wallets MUST treat these as the expected loop, not an exceptional path:

1. Catch `PAYER_REJECTED` with `data.code âˆˆ { "RATE_EXPIRED", "PAYMENT_INSUFFICIENT" }`.
2. Re-call `payer_getTerms`, re-select the same offer/`TokenChoice` shape if possible, recompute `paymentAmount` (typically from the cached `rate` and the fresh `gasEstimate`), have the sender re-sign the transaction with the updated phase 0.
3. Retry `payer_sendTransaction` / `payer_signTransaction`. Implementations SHOULD bound the retry to one or two iterations to avoid loops on a degraded payer; after that, surface the failure.

Wallets pricing entirely from a cached `rate` rather than a fresh `payer_getTerms` MAY add a small headroom on `paymentAmount` (e.g. a few percent) to absorb intra-`ttl` gas drift; this trades a slightly higher upfront transfer for fewer round-trips.

This loop is the v1 mechanism for keeping a quote bound to its phase 0. Future versions MAY define an opt-in **signed-quote** profile in which the payer signs a `(token, amount, expiry, ...)` quote that the wallet (or the payer at co-sign) can verify without trusting the freshness of an unsigned price feed, removing the re-quote round trip when gas moves; that profile is out of scope for this version.

### `paymasterService` Capability

This proposal reuses the `paymasterService` capability from [ERC-7677](./eip-7677.md) unchanged: apps pass a `url` and optional `context` on an [EIP-5792](./eip-5792.md) `wallet_sendCalls` (or [ERC-7836](./eip-7836.md) `wallet_prepareCalls`) request, exactly as specified there. For [EIP-8130](./eip-8130.md) transactions the wallet uses the `payer_*` methods defined here; for [ERC-4337](./eip-4337.md) user operations it uses the `pm_*` methods from [ERC-7677](./eip-7677.md). The wallet MUST forward the opaque `context` to `payer_getTerms` and to `payer_sendTransaction` / `payer_signTransaction`.

Wallets indicate support by returning the `paymasterService` capability as `supported` from [EIP-5792](./eip-5792.md) `wallet_getCapabilities`. Because the app-provided payer may not ultimately be used (service failure or user selection), apps MUST NOT assume it is the entity that pays.

#### `wallet_sendCalls` Flow

When `wallet_sendCalls` is called with the `paymasterService` capability and the wallet is constructing an [EIP-8130](./eip-8130.md) transaction:

1. Wallet OPTIONALLY calls `payer_getTerms` at the provided URL.
2. Wallet constructs the [EIP-8130](./eip-8130.md) transaction with call phases, the `payer` field set, and `payer_auth` empty (see [Transaction Construction](#transaction-construction)).
3. Wallet signs the transaction (`sender_auth`).
4. Wallet calls `payer_sendTransaction` (relay; always available), or `payer_signTransaction` (co-sign only) when it wants to self-submit and the selected offer advertises it via its `methods` field.
5. Payer validates the transaction, fills in `payer_auth`, and submits or returns the co-signed transaction.

The same methods apply to [ERC-7836](./eip-7836.md) `wallet_prepareCalls` / `wallet_sendPreparedCalls`, with the app signing the transaction digest between steps.

### Wallet-Selected Payers

The `paymasterService` capability covers the case where an *app* directs the wallet to a service. It is not the only source of a payer. A wallet that wants its users to pay gas in a token, with no app involvement, selects a payer itself and uses the same `payer_*` methods.

Typical wallet-configured payers:

- a payer service the wallet operator runs or contracts with, configured per chain.

When both an app-provided `paymasterService` and a wallet-default payer are available, the wallet decides which to use, for example preferring app sponsorship when offered and falling back to its default token-payment payer otherwise. Because `sender_auth` binds the chosen `payer`, the user always signs the exact payer that will be used.

### Aggregation

A wallet may have several payer sources for one intent â€” for example a sequencer that accepts stablecoins, an app-provided `paymasterService` URL that may sponsor, and a subscription service the user holds an allowance with. There are two ways to combine them, both using only the methods above.

**Client-side fan-out.** The wallet calls `payer_getTerms` on each candidate (and `payer_getSponsorshipBalance` where it wants a standing balance first), then ranks the results itself and proceeds with one. Because `sender_auth` binds the chosen `payer`, the wallet commits to exactly one payer when it signs. This needs nothing beyond `payer_getTerms` and is the path to use when the wallet wants to surface a chooser across payers to the user.

**Aggregator endpoint.** A single service MAY front multiple downstream payers behind one URL and implement the same methods, so a wallet (or app) configures just one endpoint. For an intent the aggregator evaluates its downstreams and returns one `payer_getTerms` result. Because every offer is self-contained, carrying its own `payer`, `endpoint`, and `methods`, the aggregator MAY either collapse to a single chosen offer or return several offers sourced from *different* downstream payers, each pointing at the downstream that co-signs it (or proxied through the aggregator). The wallet picks one offer (and, for a `token` offer, one `TokenChoice` from its `tokens`) and co-signs against that offer's `payer`/`endpoint`. For standing balances, `payer_getSponsorshipBalance` returns multiple `PayerBalance` entries carrying `payer` / `endpoint` / `name` source attribution, so the wallet can show what the user holds across services without a per-service probe.

Because the offered terms are **not a total order** â€” a lower `paymentAmount` with no refund can be worse than a higher one that refunds unused gas, refund settlement is trust-based, and refund window and payer reputation also matter â€” an aggregator that returns a single offer MUST rank according to a documented policy (e.g. "cheapest guaranteed", "lowest expected cost assuming the refund settles", or "most trusted payer") rather than implying a canonical "best". An aggregator that instead returns several offers leaves that ranking to the wallet/user, which is the right choice when the user should weigh the tradeoffs. Either way, since each offer carries its own `payer`, a single response can present competing offers from *different* payers as separate `options` entries; competing terms from *one* payer for the same token â€” refundable versus non-refundable â€” are expressed as multiple entries in that offer's `tokens` sublist, distinguished by their phase-0 fingerprint per the [Co-Sign Disambiguation Invariant](#co-sign-disambiguation-invariant). When an aggregator surfaces variants that would collide at co-sign under one payer (e.g. two sponsored constructions, or identical token transfers with different refund terms), it MUST route them through distinct `payer` addresses. Whichever offer the wallet picks, `sender_auth` binds that offer's `payer`, so the user signs the exact payer that will be used.

### Transaction Construction

The [EIP-8130](./eip-8130.md) AA transaction wire format is:

```
AA_TX_TYPE || rlp([
  chain_id, sender, nonce_key, nonce_sequence, expiry,
  max_priority_fee_per_gas, max_fee_per_gas, gas_limit,
  account_changes, calls,
  payer, sender_auth, payer_auth
])
```

`calls` is a two-level array of phases: `[[call, ...], [call, ...]]` where each `call` is `[to, data]`. Phases execute in order; each phase is atomic (all-or-nothing), but completed phases persist independently.

Note on `value`: [EIP-8130](./eip-8130.md) calls carry no protocol value; each onchain `call` is `[to, data]` with `msg.value == 0`. The wallet realizes any non-zero intent `value` through the account's wallet bytecode (e.g., an `executeBatch` call that performs the value-bearing CALL). An intent call whose `value` is `0` or omitted is encoded directly as `[to, data]`.

#### Constructing the Transaction

Given terms from `payer_getTerms`, the wallet first picks one selectable `PaymentOption` from `options`; if the picked offer is a `kind: "token"` offer, the wallet additionally picks one `TokenChoice` from its `tokens`. The transaction is then built entirely from that offer (and, for token payment, that `TokenChoice`):

1. **Set `payer`** to the selected offer's `payer`.
2. **Set `nonce_key` and `nonce_sequence`**: select a `nonce_key` (a parallel-nonce lane; `0` for sequential ordering) and read the next `nonce_sequence` for `(sender, nonce_key)`. Note that `eth_getTransactionCount` has no `nonce_key` parameter and so only returns the sequence for the default lane (`nonce_key == 0`); reading any other lane requires an [EIP-8130](./eip-8130.md)-specific accessor for `(sender, nonce_key)`, which wallets using parallel lanes MUST use instead.
3. **Set `expiry`** to any value up to the offer's `conditions.maxExpiry` (RECOMMENDED: `conditions.maxExpiry`). This is the protocol-enforced onchain transaction lifetime; the payer rejects a transaction whose `expiry` exceeds `maxExpiry` at co-sign.
4. **Set gas parameters** (`max_fee_per_gas`, `max_priority_fee_per_gas`, `gas_limit`) from the response's top-level `gasEstimate` when present (the same recommendation applies to every offer; `gasLimit` already accounts for the worst-case phase-0 transfer so it works for both `kind: "sponsored"` and `kind: "token"` without further estimation), otherwise from the wallet's own [EIP-8130](./eip-8130.md)-aware estimation. The values MUST respect the picked offer's `conditions.maxGasLimit` and `conditions.maxCost`.
5. **Construct `calls`** using call phases:

| Offer `kind` | Phase 0 | Phase 1 |
|---|---|---|
| `sponsored` | (none) | `[user_calls]` |
| `token` | `[token.transfer(choice.feeRecipient ?? offer.payer, choice.paymentAmount)]` | `[user_calls]` |

A `kind: "sponsored"` offer uses the full-sponsorship construction (no phase 0); a `kind: "token"` offer uses the token-payment construction with the selected `TokenChoice`. For token payment, the transfer amount is the choice's `paymentAmount`, and the destination is its `feeRecipient` when present, otherwise the offer's `payer`. A wallet reusing a cached offer re-selects the same `TokenChoice` by its `(token, paymentAmount, feeRecipient)` tuple â€” which, together with the offer's `payer`, forms the co-sign fingerprint the [Co-Sign Disambiguation Invariant](#co-sign-disambiguation-invariant) requires to be unique across the whole response â€” recomputes the amount from that choice's `rate`, and uses that choice's `feeRecipient`. For full sponsorship with no phase 0 requirements, the wallet MAY use a single phase: `[[user_calls]]`.

**Balance-funded sponsorship** is a `kind: "sponsored"` offer that carries a `balance` (a `sponsorship` allowance or prepaid `credit`); it uses the same full-sponsorship construction with no phase 0 transfer. The payer is reimbursed off-chain from the sender's budget when it co-signs.

6. **Leave `payer_auth` empty**: the payer fills this in after receiving the sender-signed transaction.
7. **Sign**: compute `sender_signature_hash` over all fields through `payer` (excluding `sender_auth` and `payer_auth`) and produce `sender_auth`.

#### Token Payment via Call Phases

[EIP-8130](./eip-8130.md) calls carry no ETH value; token transfers are executed through contract calls. For token payment, the wallet constructs a phase 0 call that transfers tokens from the sender to the destination on the selected `TokenChoice` (its `feeRecipient` when present, otherwise the offer's `payer`):

```
offer     = terms.options[i]        // the `kind: "token"` offer the wallet picked
choice    = offer.tokens[j]         // the `TokenChoice` the wallet picked
recipient = choice.feeRecipient ?? offer.payer
phase_0   = [
  [choice.token, abi.encodeCall(IERC20.transfer, (recipient, choice.paymentAmount))]
]
```

This uses `IERC20.transfer` (a push from the sender) rather than `transferFrom`, so **no token approvals are required**. This is a significant simplification over [ERC-4337](./eip-4337.md) paymaster flows, which require the sender to pre-approve the paymaster contract (or inject an `approve()` call into the UserOperation's calldata) before the paymaster can pull payment in `postOp`.

Transferring `paymentAmount` to `recipient` is always sufficient while terms are valid. Phase 0 commits independently, so the recipient receives tokens regardless of whether the user's calls in phase 1 succeed or revert; this is how the payer is compensated.

#### Payer Co-Signing

The payer receives the sender-signed transaction (with `payer_auth` empty), validates it against its terms, computes `payer_signature_hash` using `AA_PAYER_TYPE` (see [EIP-8130](./eip-8130.md) Signature Payload), and fills in `payer_auth` with `authenticator || data`. The payer's actor MUST have PAYER scope in its `actor_config` on the payer account.

A payer service MAY delegate co-signing to a separate **signing service** that custodies the PAYER-scoped key, for example to separate pricing and policy logic from key management and submission. The `payer` field always identifies the account whose actor signs, so `sender_auth` binds the transaction to that account regardless of which operator runs the endpoint.

#### Token Cost Transparency

Because the token cost is fixed before the sender signs, wallets can display it precisely. The displayed fee is the phase 0 transfer amount: `paymentAmount` rendered with the selected `TokenChoice`'s `decimals` and `symbol`. A wallet pricing from a cached offer derives the amount as `ceil(gas_limit Ã— max_fee_per_gas Ã— rate.numerator / rate.denominator)` and MAY derive a human-readable rate from `rate` and `symbol` for supporting context. When `usdRate` is present (the payment token's USD price, 6-decimal fixed-point), the wallet can additionally display an approximate USD cost as `paymentAmount Ã— usdRate / (10^decimals Ã— 10^6)` â€” `paymentAmount / 10^decimals` is the token amount and `usdRate / 10^6` is its USD price, so the product is the USD cost. For the running example (`paymentAmount` `0x30D40` = 0.2 USDC, `usdRate` `0xF4240` = $1.00) this is `0.2 Ã— $1.00 = $0.20`. When no `refund` is offered, this is the final charge; when a `refund` policy is present, it is an upper bound and the wallet MAY indicate that unused gas is refunded. The quote is valid until the offer's `ttl`; after that the wallet MUST re-request terms.

### Endpoint Versioning

Payer service endpoints MUST use path-based major versioning (e.g., `/v1`, `/v2`). Method names remain stable across versions. Breaking changes (new required fields, changed semantics) require a new major path version. Additive optional fields in request or response are backwards-compatible within the same version. Clients MUST ignore unknown fields. Payers MUST NOT require new optional fields within an existing version.

## Rationale

### Why New RPC Methods

While the `paymasterService` capability is shared, the RPC methods must differ. [ERC-7677](./eip-7677.md)'s two-phase `pm_getPaymasterStubData` / `pm_getPaymasterData` flow exists because [ERC-4337](./eip-4337.md) paymasters inject data into the UserOperation before gas estimation, and the stub data must produce consistent code paths and byte lengths. In [EIP-8130](./eip-8130.md), the payer simply co-signs the complete transaction; there is no paymaster data injection, no stub values, and no code path consistency concern. The `payer_*` methods reflect this simpler model. Service providers that support both wallet types SHOULD implement both method sets at the same endpoint URL.

### `payer_sendTransaction` vs `payer_signTransaction`

Two methods mirroring the `eth_sendTransaction` / `eth_signTransaction` convention:

- **`payer_sendTransaction`** minimizes hops: the payer co-signs and submits in one step. This is the common case for app-sponsored transactions.
- **`payer_signTransaction`** gives the wallet submission control, useful for high-value transactions, MEV-sensitive paths, or when the wallet wants to use private mempools.

They are kept as two methods rather than one method with a `submit`/`relay` flag because their return types genuinely diverge â€” a `transactionHash` versus the co-signed transaction bytes â€” and a merged method would only relocate the "is wallet-side submission available?" question from *which method exists* to *which parameter value is supported*, turning a statically-knowable fact into a runtime error-and-fallback. Two methods let the wallet learn the path up front from the picked offer's `methods` field and never make a call it knows will fail.

`payer_sendTransaction` is REQUIRED and `payer_signTransaction` is OPTIONAL because the two paths are not symmetric in practice:

- **Relaying is near-universal and cheap.** Any payer can forward a fully-signed transaction to a public node, so requiring `payer_sendTransaction` lets the common flow work against every payer with no discovery step.
- **Returning co-signed bytes is the rarer capability.** A payer that co-signs just-in-time â€” a sequencer or builder that produces `payer_auth` as part of block construction â€” has no bytes to hand back before inclusion and can only relay. Independently, returning bytes opens a window between the payer committing its signature and the transaction landing, during which the sender's phase-0 token balance can move, the quote can go stale, or the bytes can be broadcast to a competing builder; relay-only collapses that window. Risk-sensitive payers (especially token-payment ones) therefore tend to withhold `payer_signTransaction`.

So `payer_sendTransaction` is the de-facto baseline and `payer_signTransaction` is the opt-in for wallets that need submission control. The choice between them is the wallet's, not the app's. A relay-only sequencer that wants exclusive inclusion simply omits `payer_signTransaction`, so the wallet never receives broadcastable bytes (see [Sequencer-Operated Payer Implementations](#sequencer-operated-payer-implementations)).

### `feeRecipient` Placement

`feeRecipient` is **per token**, not per payer. The destination of phase 0's `token.transfer` is properly a token-level routing decision: a payer accepting both USDC and a native-token wrapper may route USDC to a stablecoin treasury and the native wrapper to a different operator wallet; revenue-share splits with apps are negotiated per integration and per token. Hoisting `feeRecipient` to the payer-level (or to the offer level) would force a single recipient for the payer's full token set, which collides with these patterns. It lives on `TokenChoice` so each token entry in a `token` offer can carry its own destination; the wallet uses the `feeRecipient` of the choice it picked as the phase-0 transfer destination, and absent that, falls back to the offer's `payer`.

The recipient is per-intent in practice as well as per-token: app-sponsored token-payment policies route fees to different treasuries by app or policy, builder-operated payers may route through different operator addresses depending on inclusion lane, and revenue-share splits with apps are negotiated per integration. Carrying `feeRecipient` on the `payer_getTerms` quote lets a payer return the right destination for each specific intent without the wallet needing a separate discovery hop, and a wallet building from cached terms has everything it needs to construct phase 0.

### Sequencer-Operated Payer Implementations

The wallet protocol is identical for every payer, but a payer operated by the chain's sequencer (or by a party with privileged inclusion) has operational levers worth noting as one possible implementation pattern; this is non-normative and many variations are possible:

- **Drop-on-failure**: because the operator builds the block, it MAY simulate phase 0 against pending state and omit the transaction entirely if phase 0 would revert, avoiding nonce consumption on a failed payment transfer.
- **Exclusive inclusion**: a payer that wants its co-signed transactions includable only by the sequencer that signed them simply does **not** implement the OPTIONAL `payer_signTransaction` (returns the signed bytes), serving only the REQUIRED `payer_sendTransaction` (returns a hash). The wallet never receives the serialized transaction, so no third party can broadcast it to a competing builder; this is also useful for avoiding mempool exposure for MEV reasons.
- **Pattern-aware policies**: a sequencer-payer MAY apply broader heuristics, for example dropping transactions whose phase 0 fails in patterns consistent with probing or griefing against a particular token set.

These are operator choices, not wire-protocol additions. From a wallet's perspective a sequencer-operated payer is just a payer; its accepted tokens and methods surface through the same `payer_getTerms` response (in each offer's `tokens` and `methods` fields) as any other payer.

## Security Considerations

**API key exposure**: Payer service URLs commonly contain API keys. App developers SHOULD proxy payer requests through their own backend to keep keys secret, identical to the recommendation in [ERC-7677](./eip-7677.md). Payer services SHOULD support scoped API keys restricted to specific methods.

**Relay trust**: In relay mode, the wallet trusts the payer to submit the transaction. The transaction's `expiry` field bounds this exposure; if the payer does not submit before expiry, the transaction is dead. Wallets concerned about relay trust SHOULD use cosign mode and submit themselves.

**Rate commitment**: Once a payer returns terms, it SHOULD honor them until the quote's `ttl`. Because `paymentAmount` is quoted against the worst-case top-level `gasEstimate`, transferring it already absorbs normal gas movement; wallets pricing from a cached `rate` can add a small buffer and use short expiry windows to stay includable.

**Balances are advisory**: A returned `balance` MUST NOT be treated as a spend guarantee. Wallets MUST handle `BUDGET_EXHAUSTED` and other rejections at co-sign time, since concurrent transactions or policy may invalidate a stale balance snapshot. Payers MUST enforce the actual budget server-side and MUST NOT rely on the wallet honoring the advertised balance.

**Phase-0 settlement risk (token payment)**: Between `payer_getTerms` and inclusion, a sender's token balance can change (front-running, concurrent transfers, or intentional griefing) and leave the payer with a reverted phase-0 transfer and no compensation. Payers SHOULD put mitigations in place appropriate to their risk tolerance, for example sender risk scoring, allowlists for trusted senders or tokens, deposit-history requirements, and velocity limits. Builder-operated payers MAY additionally drop phase-0 failures pre-inclusion (see [Sequencer-Operated Payer Implementations](#sequencer-operated-payer-implementations)).

**Refund settlement is trust-based**: A `RefundPolicy` on a `TokenChoice` commits the payer to return `paymentAmount` minus actual gas cost to `from` as a separate transfer within `window` seconds â€” but this spec has no onchain enforcement. The payer transfers from its own balance, paying the gas itself, and a sender has no programmatic recourse if the refund does not arrive. `tokenCharged.estimatedRefund` is the payer's estimate, not a binding receipt. Wallets MUST display refund amounts as expected rather than guaranteed (e.g. "up to X refunded"), SHOULD surface the payer's identity and refund window so the user can weigh the trust assumption, and SHOULD treat a missed refund as a signal that lowers their ranking of that payer for future intents. Payers SHOULD make refund settlements externally observable (an indexable event from a known address or settlement contract) so refund streams are auditable. Onchain refund settlement (e.g. an escrow contract that holds `paymentAmount` and releases overage to `from` permissionlessly) is a meaningful extension and a candidate for a future profile of this spec.
