## Abstract

[ERC-7677](./eip-7677.md) defines the `paymasterService` capability for [EIP-5792](./eip-5792.md), allowing apps to direct wallets to a sponsorship web service. This proposal extends that capability to [EIP-8130](./eip-8130.md) by defining `payer_*` RPC methods that wallets use when constructing [EIP-8130](./eip-8130.md) transactions. Apps pass the same `paymasterService` URL they already use; the wallet selects the appropriate RPC methods based on the transaction type it is constructing. This proposal additionally defines sponsorship **balances**, so a wallet can read how much sponsorship budget or prepaid credit a sender has remaining (e.g., a subscription gas allowance) and fall back to token payment when it is exhausted.

## Motivation

[EIP-8130](./eip-8130.md) introduces native account abstraction where payers are first-class protocol participants that co-sign the transaction using the same `authenticator || data` model as senders, validated with PAYER scope. This differs fundamentally from the [ERC-4337](./eip-4337.md) paymaster model (no EntryPoint, no stub-data two-phase flow, the payer co-signs the exact transaction that lands onchain) and enables more efficient flows, including direct relay and protocol-enforced `expiry`.

[ERC-7677](./eip-7677.md) defines the `pm_*` RPC methods and lets an app tell a wallet "here is a sponsorship service URL" via the `paymasterService` capability. Those `pm_*` methods do not fit [EIP-8130](./eip-8130.md), so this proposal defines `payer_*` methods and reuses the same capability, so apps need no new integration: the wallet picks `payer_*` or `pm_*` based on the transaction type it is constructing.

This proposal supports three payment models: full sponsorship by an app, sponsorship drawn from a standing balance (e.g. a subscription allowance or prepaid credit), and token payment for gas.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

### Payer Web Service Interface

`payer_getTerms` and `payer_sendTransaction` are REQUIRED; `payer_signTransaction` and `payer_getSponsorshipBalance` are OPTIONAL. Relaying is near-universal (any payer can forward a signed transaction), so requiring `payer_sendTransaction` makes the common case work against every payer with no discovery; `payer_signTransaction` (co-sign and return the bytes for the wallet to submit) is the opt-in path for wallets that want submission control.

**Common case.** Most integrations need only two calls: `payer_getTerms` to obtain payment offers, then `payer_sendTransaction` to have the payer co-sign and submit the transaction. Both take a fully sender-signed transaction; the payer co-signs (fills `payer_auth`) and either submits (`payer_sendTransaction`) or, when offered, returns the co-signed bytes (`payer_signTransaction`). Because `payer_sendTransaction` is REQUIRED, this flow needs no capability probe. A wallet that wants to submit the transaction itself uses `payer_signTransaction`; each offer in `payer_getTerms`'s result advertises whether that path is available against *its* `(payer, endpoint)` via the offer's `methods` field. The remaining method is optional read-only discovery: `payer_getSponsorshipBalance`. See [`wallet_sendCalls` Flow](#wallet_sendcalls-flow).

Every responder MUST implement `payer_getTerms` and `payer_sendTransaction`. Each offer returned by `payer_getTerms` carries the full set of `payer_*` methods callable against that offer in its `methods` field, so a wallet learns everything from the call it already makes and never needs a third hop just to discover methods, capping the transaction flow at two hops (`payer_getTerms` then a co-sign call). A wallet MUST NOT invoke an OPTIONAL method against an offer that does not advertise it. The `payer_*` prefix denotes the *role* of the responder (acting as a payer for an [EIP-8130](./eip-8130.md) transaction); methods are served over an HTTPS JSON-RPC endpoint.

All durations in this specification are expressed as **relative seconds** measured from the time the response is sent (`ttl`, `conditions.maxExpiry`, `RefundPolicy.window`, `PayerBalance.validFor`). Relative durations keep a payer's response portable across clients with skewed clocks; a wallet caching a value records its own receipt time and counts down from there.

#### `payer_getTerms`

Returns sponsorship or token-payment terms for a transaction intent. Called *before* the sender signs. Payer services SHOULD validate the transaction intent at this stage and reject if they would not sponsor or accept payment.

##### `payer_getTerms` RPC Specification

```typescript
// [params]
type GetTermsParams = {
  chainId: `0x${string}`;
  from: `0x${string}`;
  calls: {
    to: `0x${string}`;
    value?: `0x${string}`;
    data?: `0x${string}`;
  }[];
  gasLimit?: `0x${string}`;
  preferredTokens?: `0x${string}`[];
  context?: Record<string, any>; // opaque app context (e.g. policyId) from the paymasterService capability
};

type GetTermsResult = {
  options: PaymentOption[]; // payment offers for this intent, best-first. Each offer is self-contained (its own payer, endpoint, methods, ttl, gas, conditions). A `sponsored` entry is either a real offer or carries a `code` explaining why sponsorship was declined (informational, not selectable). An empty array means the payer has nothing to offer (e.g. unknown sender, unsupported chain).
};

// A payment offer for this intent's gas. Each offer is a complete, self-
// contained quote: its own co-signer, endpoint, RPC surface, validity, gas
// estimate, and conditions. There is no top-level inheritance: everything an
// offer needs sits on the offer. Offers are grouped at the right grain â€” one
// sponsored offer, one token-payment offer covering N accepted tokens â€” so the
// shared `(payer, ttl, gasEstimate, conditions, â€¦)` of N token quotes is
// written once on the offer and the per-token differences (`paymentAmount`,
// `rate`, `refund`, `feeRecipient`, `id`) live in its `tokens` sublist.
type PaymentOption = SponsoredOffer | TokenPaymentOffer;

type BaseOffer = {
  payer: `0x${string}`; // co-signing account for THIS offer; the wallet sets it on the transaction
  endpoint?: string; // co-sign URL for this offer; OPTIONAL, defaults to the called URL (see field note)
  methods?: string[]; // payer_* methods callable against this offer's `(payer, endpoint)`. When present MUST include payer_getTerms and payer_sendTransaction. Lets the wallet pick its submission path (e.g. payer_signTransaction when offered) directly from the offer, capping the flow at two hops. Absent means only payer_getTerms and payer_sendTransaction
  ttl: number; // how long this quote (rates included) stays valid and may be cached/reused, in seconds
  gasEstimate?: {
    gasLimit: `0x${string}`; // for THIS offer's calls â€” a token-payment offer includes its phase-0 transfer in the budget, so its gasLimit exceeds an equivalent sponsored offer's; when a token offer covers multiple tokens, the payer quotes a conservative worst-case across them
    maxFeePerGas: `0x${string}`;
    maxPriorityFeePerGas: `0x${string}`;
  };
  conditions?: {
    maxExpiry?: number; // relative, seconds from current time
    maxGasLimit?: `0x${string}`;
  };
  sponsor?: { // display info for this offer's sponsoring party
    name: string;
    icon?: string;
  };
};

// A `sponsored` offer is either a real offer (BaseOffer fields populated, the
// wallet MAY select it) or a declined entry (carries `code`, is informational,
// MUST NOT be selected). The two shapes share `kind: "sponsored"`;
// selectability is determined by the presence or absence of `code`.
type SponsoredOffer =
  | (BaseOffer & {
      kind: "sponsored";
      balance?: PayerBalance; // present when this sponsorship draws from an allowance/credit ("$5 of $10"); absent means fully paid by the sponsor
    })
  | {
      kind: "sponsored";
      code: SponsorshipDeclineCode; // stable string enum; informational only â€” wallets MUST NOT select an offer that carries `code`
      reason?: string; // human-readable detail for display/logs (e.g. "Monthly budget exhausted, resets 2026-07-01")
      balance?: PayerBalance; // optional context â€” e.g. the exhausted allowance, whose `validFor` tells the wallet when to retry
      sponsor?: { name: string; icon?: string }; // display info for the would-be sponsor, so the wallet can say which party declined
    };

// Extensible: payers MAY return their own strings; wallets MUST treat unknown
// codes as opaque (display `reason` when present, otherwise treat as a generic
// decline). The well-known values mirror the most common policy outcomes a
// wallet wants to brand differently.
type SponsorshipDeclineCode =
  | "POLICY_REJECTED"        // calls/sender/contract not covered by this sponsor's policy; permanent for this intent
  | "SENDER_INELIGIBLE"      // sender blocklisted or missing a required attestation; permanent until the sender changes
  | "BUDGET_EXHAUSTED"       // sponsor's budget or the sender's allowance is depleted; check `balance.validFor`
  | "SENDER_LIMIT_REACHED"   // per-sender cap reached (tx count or USD); check `balance.validFor`
  | "GAS_EXCEEDS_LIMIT"      // gas price exceeds the sponsor's per-tx cost ceiling; transient â€” retry when gas drops
  | "TEMPORARILY_UNAVAILABLE" // sponsor is degraded (e.g. rate feed down); retry shortly
  | string;

// A token-payment offer: a single co-signer accepting one or more tokens for
// gas under shared terms. Per-token differences (`paymentAmount`, `rate`,
// `refund`, `feeRecipient`, `id`) live on each `TokenChoice`; everything else
// is on the offer.
type TokenPaymentOffer = BaseOffer & {
  kind: "token";
  tokens: TokenChoice[]; // â‰¥1 accepted tokens. Multiple entries MAY share `token` to offer distinct terms (e.g. refundable vs non-refundable); when they do, each MUST carry a distinct `id`
};

type TokenChoice = {
  id?: string; // stable identifier; REQUIRED to disambiguate when two choices share the same `token` (e.g. refundable vs non-refundable); the wallet selects/caches/re-prices against it
  token: `0x${string}`;
  symbol: string;
  decimals: number;
  paymentAmount: `0x${string}`; // token amount the wallet transfers in phase 0 to `feeRecipient` (or the offer's `payer` when absent), quoted at the offer's gas cap
  feeRecipient?: `0x${string}`; // optional phase-0 destination; defaults to the offer's `payer`. Per-token by design: different tokens may route to different treasuries or operator addresses
  rate: {
    numerator: `0x${string}`; // token atomic units
    denominator: `0x${string}`; // per this many native wei
  };
  usdRate?: `0x${string}`; // optional: native token price in USD, 6 decimal fixed-point (e.g. "0x75BCD15" â‰ˆ $1234.56); allows wallets to display the cost in USD without a separate price feed
  refund?: RefundPolicy;
};
```

`PayerBalance` is shared by `payer_getTerms` and `payer_getSponsorshipBalance`:

```typescript
type PayerBalance = {
  kind: "sponsorship" | "credit";
  available: string; // remaining amount in base units of `asset`
  limit?: string; // total budget or cap, same units as `available`
  spent?: string; // total lifetime spent (if tracked), same units as `available`
  asset: string; // denomination of available/limit/spent: token contract address, "native", or an ISO-4217 currency code (e.g. "USD") for a fiat-denominated budget
  symbol?: string;
  decimals?: number;
  validFor?: number; // relative seconds the current snapshot remains accurate before its state changes â€” for a `sponsorship` this is when the periodic budget refills; for a `credit` this is when the balance lapses to zero. The wallet uses `kind` to choose the verb when rendering ("resets in"/"expires in").
  // Source attribution: REQUIRED when returned by an aggregator fronting
  // multiple services, so the wallet can tell which service holds the balance.
  payer?: `0x${string}`; // the payer/service address this balance is held with
  endpoint?: string; // that service's endpoint URL
  name?: string; // human-readable service name, e.g. "Coinbase One"
};
```

`RefundPolicy` describes when unused gas is returned to `from`. Its **presence** on a `TokenChoice` means refunds are offered for that choice; its absence means none are:

```typescript
type RefundPolicy = {
  window: number; // upper bound in seconds until the refund is settled (e.g. 86400 â‰ˆ next day)
};
```

Refund settlement is **onchain**: the payer transfers `paymentAmount` minus the transaction's actual gas cost (priced through `rate`) to `from` as a separate [ERC-20](./eip-20.md) transfer, **paying the gas for that transfer itself**, within `window` seconds of inclusion. Phase 0 commits independently and is never partial; refunds always settle separately, which is why `payer_sendTransaction` / `payer_signTransaction` can return `estimatedRefund` immediately without waiting for the refund tx to land.

If the payer fails to settle within `window`, the sender has no onchain recourse from this specification: the refund obligation is trust-based and operates at the same trust layer as the payer's quote in the first place. Wallets SHOULD display refund amounts as expected rather than guaranteed (e.g. "up to X refunded") and MAY surface the payer's identity, refund window, and reputation to the user accordingly. Payers SHOULD make refund settlements externally observable, for example by emitting an indexable event from a known address or settling through a publicly identifiable contract, so refund streams can be monitored by users and aggregators.

##### Field Notes

| Field | Notes |
|---|---|
| `options` | The payment offers for this intent, best-first. Each is `kind: "sponsored"` (full-sponsorship construction, no phase 0) or `kind: "token"` (phase-0 token transfer, with one or more accepted tokens listed in `tokens`). Each offer is a **self-contained quote**: its own `payer`, `endpoint`, `methods`, `ttl`, `gasEstimate`, `conditions`, and `sponsor`. There is no top-level inheritance. A `sponsored` offer with no `balance` is fully paid by the sponsor; with a `balance` it draws from an allowance or prepaid credit. A `sponsored` entry MAY instead carry a `code` explaining why sponsorship was declined â€” informational only, **not selectable**; see the `sponsored` decline note. An offer is grouped at the right grain so that N accepted tokens written as `tokens: TokenChoice[]` share the offer's `(payer, ttl, gasEstimate, conditions, â€¦)` once; per-token differences live on each `TokenChoice`. Token absence is fully expressed by the absence of a `token` offer (or an offer whose `tokens` array is empty, which a payer SHOULD avoid). The wallet picks one selectable offer (and, for a `token` offer, one `TokenChoice`) and builds the transaction entirely from that pair. |
| `sponsored` decline | A `sponsored` offer carrying `code` is informational, never a selectable offer; wallets MUST ignore it for transaction construction and SHOULD surface it in the UI. `code` is a stable string enum (see [Sponsorship Decline Codes](#sponsorship-decline-codes)); payers MAY return their own values and wallets MUST treat unknown codes as opaque, displaying `reason` when present. `balance` (optional) carries context â€” most usefully the exhausted allowance, whose `validFor` tells the wallet when sponsorship may return without inventing a separate `retryAfter`. `sponsor` (optional) identifies the would-be sponsor so the wallet can say "Coinbase One â€” free gas used up, resets in 3 days." A declined `sponsored` entry MAY appear alongside a selectable `token` offer. |
| `payer` / `endpoint` (per offer) | `payer` is the co-signing account for that offer â€” the address the wallet sets on the transaction when it picks the offer, and the default phase-0 token-transfer destination when a `TokenChoice.feeRecipient` is absent. Different offers MAY name different payers (e.g. a token sponsor co-signing from a different account than full sponsorship). `endpoint` is the co-sign URL and is OPTIONAL: it defaults to the called URL; set it only when this offer is co-signed elsewhere, chiefly aggregator downstream routing. |
| `methods` (per offer) | Authoritative list of `payer_*` methods callable against this offer's `(payer, endpoint)`. When present MUST include `payer_getTerms` and `payer_sendTransaction`. A responder that implements `payer_signTransaction` (or other optional methods) SHOULD list them here so a self-submitting wallet stays within the two-hop maximum (`payer_getTerms` then a co-sign call). Absent means only `payer_getTerms` and `payer_sendTransaction`. `methods` is per offer because an offer that routes co-signing to a different `endpoint` MAY expose a different surface; in the common case where every offer shares the called endpoint, payers repeat the same list. |
| `ttl` (per offer) | How long that offer's quote (rates included) stays valid and may be cached and reused to build transactions, in seconds. After `ttl` the wallet MUST re-call `payer_getTerms` for that offer. Independent of the transaction's onchain `expiry`, which is governed by the offer's `conditions.maxExpiry`: a payer may let a quote be cached for an hour while still requiring each transaction to carry a 30-second `expiry`. Token offers typically carry a shorter `ttl` than sponsorship offers because their `rate` tracks the market. |
| `gasEstimate` (per offer) | Authoritative gas parameters for the [EIP-8130](./eip-8130.md) transaction *as built for that offer*. It is per offer because a `token` offer prepends a phase-0 transfer, so its `gasLimit` exceeds an equivalent `sponsored` offer's; when a token offer covers multiple tokens the payer quotes a conservative worst-case `gasLimit` across them, and each `TokenChoice.paymentAmount` (quoted at the gas cap) naturally absorbs the small per-token slack. The binding cap is that offer's `conditions.maxGasLimit`. When both `gasEstimate.gasLimit` and `conditions.maxGasLimit` are present, payers MUST set `gasEstimate.gasLimit â‰¤ conditions.maxGasLimit`; wallets MUST treat `conditions.maxGasLimit` as the binding cap and reject an offer in which the invariant is violated. When `conditions` (and thus `maxGasLimit`) is absent, `gasEstimate.gasLimit` is the effective ceiling. |
| `conditions` (per offer) | Constraints the wallet MUST satisfy for that offer: `maxGasLimit` (binding gas cap) and `maxExpiry` (upper bound on the transaction's relative `expiry`; the wallet sets the onchain `expiry` to any value up to `maxExpiry`, RECOMMENDED `maxExpiry`, and the payer rejects anything above it at co-sign). |
| `tokens` (on a `token` offer) | The accepted tokens under this offer's shared terms; MUST have â‰¥1 entry. Each `TokenChoice` carries `paymentAmount` (the exact phase-0 transfer, quoted at the offer's gas cap; transferring it always covers the transaction while the offer's `ttl` is valid), `rate` (the cacheable re-pricing primitive â€” payers SHOULD fold their own `payer_auth` and operational overhead into `rate` / `paymentAmount`), and optionally `usdRate` (for USD display without a separate feed; shares the offer's `ttl` and MUST NOT be used after it), `refund`, `feeRecipient`, and `id`. `feeRecipient` defaults to the offer's `payer` when absent; different tokens MAY route to different recipients (e.g. fees split by token treasury). Multiple entries MAY share the same `token` to offer distinct terms for it â€” most usefully a refundable choice (higher `paymentAmount`, with `refund`) alongside a cheaper non-refundable one â€” since `paymentAmount` alone is not a total order once refunds are considered. When two entries share a `token` they MUST each carry a distinct `id`, and the wallet selects, caches, and re-prices against that `id`. |
| `refund` | When present on a `TokenChoice`, the payer returns `paymentAmount` minus the transaction's actual gas cost (priced back through `rate`) to `from` via a separate transfer within `window` seconds of inclusion. When absent, `paymentAmount` is final. The phase-0 transfer is never partial; refunds always settle separately. |
| `balance` (on a `sponsored` offer) | Remaining sponsorship budget (`kind: "sponsorship"`) or prepaid credit (`kind: "credit"`) backing that offer; `available` / `limit` / `spent` let the wallet render progress (e.g. "$5 of $10 used"). **Advisory** only: the binding decision is made in `payer_getTerms` / `payer_sendTransaction`. `validFor` is **relative** seconds â€” for a sponsorship it counts down to the next refill, for a credit to lapse â€” and the wallet picks the verb from `kind`. `payer` / `endpoint` / `name` carry source attribution for aggregator responses and SHOULD be omitted otherwise. |
| `sponsor` (per offer) | Optional display info for that offer's sponsoring party. `icon` MUST be a data URI (RFC-2397), minimum 96Ã—96px; wallets MUST render SVG icons via `<img>` to prevent untrusted script execution. |
| `context` | Opaque app context threaded unchanged through `payer_getTerms`, `payer_getSponsorshipBalance`, `payer_sendTransaction`, and `payer_signTransaction`. Keys with a leading underscore (`_*`) are RESERVED for future payer-protocol use and MUST NOT be used by apps or policies. App- and policy-defined keys SHOULD be namespaced (e.g. `myApp.policyId`) to avoid collisions when multiple services consume the same `context`. Wallets MUST forward `context` verbatim without inspecting or rewriting its contents. |

#### `payer_sendTransaction`

Accepts a sender-signed [EIP-8130](./eip-8130.md) transaction, co-signs it, and submits it onchain. Returns the transaction hash. The transaction MUST have the `payer` field set to the payer's address and `payer_auth` left empty; the payer service fills in `payer_auth` with `authenticator || data` after validating the transaction. Payers MUST NOT mutate any sender-signed fields. If the payer cannot accept the transaction as-is (e.g., gas limit too low), it MUST reject and SHOULD return actionable error data.

##### `payer_sendTransaction` RPC Specification

```typescript
type SendTransactionParams = {
  signedTransaction: `0x${string}`;
  context?: Record<string, any>;
};

type SendTransactionResult = {
  transactionHash: `0x${string}`;
  tokenCharged?: {
    token: `0x${string}`;
    amount: `0x${string}`; // gross phase-0 transfer (the selected choice's `paymentAmount`), known immediately
    estimatedRefund?: `0x${string}`; // payer's estimate of the refund to settle later, when a `refund` policy applies
  };
};
```

`tokenCharged` reports the token payment for this transaction: `token` identifies the charged token and `amount` is the gross phase-0 transfer (`paymentAmount`), known immediately without waiting for inclusion. The wallet already knows which `TokenChoice` it built phase 0 with, so the payer does not echo `id` â€” when two same-token choices exist (e.g. refundable vs not) the wallet disambiguates locally by matching the choice it picked. When the choice carried a `refund`, `estimatedRefund` is the payer's estimate of what will be returned to `from`; the exact refund settles separately per the [`refund`](#field-notes) policy, so the result need not block on the transaction landing.

#### `payer_getSponsorshipBalance` (OPTIONAL)

Returns a sender's **standing** balances (budgeted sponsorship allowances and prepaid credit) held with the payer, **without** a transaction intent. Wallets use it to show users how much free gas or credit they have and to decide whether to attempt sponsorship or fall back to payment before assembling an intent.

This method returns only **intent-independent** balances: a `sponsorship` allowance (e.g., a subscription's monthly free-gas budget) or a `credit` balance the sender holds. It MUST NOT attempt to enumerate intent-based full sponsorship; whether a payer will sponsor a *specific* transaction is evaluated per intent in `payer_getTerms`. When an intent is available, use `payer_getTerms` instead; the applicable balance is returned inline via the `balance` on the matching `sponsored` offer in the terms result. A payer with no standing balance for `from` returns an empty `balances` array.

When the endpoint is an aggregator fronting multiple services, each returned balance MUST carry source attribution (`payer`, `endpoint`, and SHOULD carry `name`) so the wallet can tell which service holds it. To target a single service, set `payer` (the service's address) or `endpoint` (its URL) in params; the aggregator returns only that service's balances. A payer MAY return multiple balances (e.g., a `sponsorship` allowance and a `credit` balance). When invoked through an app-provided `paymasterService`, the caller MAY pass the capability's `context` to scope the result to an app- or policy-specific allowance.

### Error Codes

Payer services MUST use the following error codes when rejecting requests. Error responses SHOULD include a `data` field with actionable details (e.g., suggested gas limit, updated terms). These codes occupy the `-32000` to `-32099` range JSON-RPC 2.0 reserves for implementation-defined server errors; they MUST NOT reuse the codes JSON-RPC reserves for protocol errors (`-32600` Invalid Request, `-32601` Method not found, `-32602` Invalid params, `-32603` Internal error, `-32700` Parse error), which payers continue to use with their standard meanings.

The "Applies to" column lists which methods return each code as a JSON-RPC error.

| Code | Message | Applies to | Permanent? | Description |
|------|---------|------------|------------|-------------|
| -32001 | `INVALID_TRANSACTION` | send, sign | n/a | Malformed or invalid [EIP-8130](./eip-8130.md) transaction |
| -32002 | `UNSUPPORTED_TOKEN` | send, sign | Yes | Token not accepted by this payer |
| -32003 | `RATE_EXPIRED` | send, sign | No | Terms have expired; re-call `payer_getTerms` |
| -32004 | `PAYMENT_INSUFFICIENT` | send, sign | No | Phase-0 transfer below required cost (gas moved up since quote) |
| -32005 | `EXPIRY_OUT_OF_BOUNDS` | send, sign | No | Transaction `expiry` does not satisfy `conditions.maxExpiry` |
| -32006 | `POLICY_REJECTED` | send, sign | Yes | Calls or contracts rejected by this sponsor's policy for this intent. Permanent for this tx; do not retry; pay with a token option or use a different payer |
| -32007 | `PAYER_BALANCE_INSUFFICIENT` | send, sign | No | Payer lacks ETH to cover gas at the protocol level |
| -32008 | `SENDER_INELIGIBLE` | send, sign | Yes | Sender is rejected entirely (blocklisted, or missing a required attestation); permanent until the sender changes |
| -32009 | `BUDGET_EXHAUSTED` | send, sign | No | Sender's sponsorship budget or prepaid credit is depleted; check `balance.validFor` for when it refills |
| -32010 | `GAS_EXCEEDS_LIMIT` | send, sign | No | Sponsorship is available in principle but current gas prices exceed this payer's per-tx cost ceiling. Transient; retry when gas is lower |
| -32011 | `SENDER_LIMIT_REACHED` | send, sign | No | Per-sender cap reached (transaction count or USD sponsored); transient â€” check `balance.validFor` |
| -32012 | `TEMPORARILY_UNAVAILABLE` | send, sign | No | Payer is degraded (e.g. rate feed down); retry shortly |

#### Sponsorship Decline Codes

`payer_getTerms` MUST NOT return a JSON-RPC error to signal "I cannot sponsor this intent." A query that the payer evaluated successfully â€” even one that yields no offer â€” is a success response with an `options` array. Sponsorship refusals ride on a **declined `sponsored` option** (an entry carrying `code`, see [SponsoredOption](#payer_getterms-rpc-specification)); token absence is fully expressed by the absence of `token` entries. This keeps the optimistic single-call flow on the success path and gives the wallet structured, displayable reasons without an exception channel. JSON-RPC errors remain for invalid requests (`-32602`), parse errors (`-32700`), method-not-found (`-32601`), and internal errors (`-32603`); a payer also MAY return `-32012 TEMPORARILY_UNAVAILABLE` if it cannot even evaluate the intent right now.

The well-known `code` values are the strings in the [Error Codes](#error-codes) table above that apply to the sponsorship-decline scope (`POLICY_REJECTED`, `SENDER_INELIGIBLE`, `BUDGET_EXHAUSTED`, `SENDER_LIMIT_REACHED`, `GAS_EXCEEDS_LIMIT`, `TEMPORARILY_UNAVAILABLE`). The same vocabulary appears in the `data.code` field of JSON-RPC errors returned by `payer_sendTransaction` / `payer_signTransaction` when an optimistic caller skips `payer_getTerms`, so a wallet sees the same string whether the decline arrives in a `getTerms` body or a send-time error envelope. `code` is open: payers MAY return their own strings, and wallets MUST treat unknown values as opaque, displaying `reason` when present and treating the entry as a generic decline otherwise.

Codes that are **send-only** by nature (`INVALID_TRANSACTION`, `UNSUPPORTED_TOKEN`, `RATE_EXPIRED`, `PAYMENT_INSUFFICIENT`, `EXPIRY_OUT_OF_BOUNDS`, `PAYER_BALANCE_INSUFFICIENT`) describe a fault in a *committed* transaction and never appear on a declined `sponsored` option, because `payer_getTerms` runs before the wallet has signed anything.

### `paymasterService` Capability

This proposal reuses the `paymasterService` capability from [ERC-7677](./eip-7677.md) unchanged: apps pass a `url` and optional `context` on an [EIP-5792](./eip-5792.md) `wallet_sendCalls` (or [ERC-7836](./eip-7836.md) `wallet_prepareCalls`) request, exactly as specified there. For [EIP-8130](./eip-8130.md) transactions the wallet uses the `payer_*` methods defined here; for [ERC-4337](./eip-4337.md) user operations it uses the `pm_*` methods from [ERC-7677](./eip-7677.md). The wallet MUST forward the opaque `context` to `payer_getTerms` and to `payer_sendTransaction` / `payer_signTransaction`.

Wallets indicate support by returning the `paymasterService` capability as `supported` from [EIP-5792](./eip-5792.md) `wallet_getCapabilities`. Because the app-provided payer may not ultimately be used (service failure or user selection), apps MUST NOT assume it is the entity that pays.

#### `wallet_sendCalls` Flow

When `wallet_sendCalls` is called with the `paymasterService` capability and the wallet is constructing an [EIP-8130](./eip-8130.md) transaction:

1. Wallet OPTIONALLY calls `payer_getTerms` at the provided URL.
2. Wallet constructs the [EIP-8130](./eip-8130.md) transaction with call phases, the `payer` field set, and `payer_auth` empty (see [Transaction Construction](#transaction-construction)).
3. Wallet signs the transaction (`sender_auth`).
4. Wallet calls `payer_sendTransaction` (relay; always available), or `payer_signTransaction` (co-sign only) when it wants to self-submit and the selected offer advertises it via its `methods` field.
5. Payer validates the transaction, fills in `payer_auth`, and submits or returns the co-signed transaction.

The same methods apply to [ERC-7836](./eip-7836.md) `wallet_prepareCalls` / `wallet_sendPreparedCalls`, with the app signing the transaction digest between steps.

### Wallet-Selected Payers

The `paymasterService` capability covers the case where an *app* directs the wallet to a service. It is not the only source of a payer. A wallet that wants its users to pay gas in a token, with no app involvement, selects a payer itself and uses the same `payer_*` methods.

Typical wallet-configured payers:

- a payer service the wallet operator runs or contracts with, configured per chain.

When both an app-provided `paymasterService` and a wallet-default payer are available, the wallet decides which to use, for example preferring app sponsorship when offered and falling back to its default token-payment payer otherwise. Because `sender_auth` binds the chosen `payer`, the user always signs the exact payer that will be used.

### Aggregation

A wallet may have several payer sources for one intent â€” for example a sequencer that accepts stablecoins, an app-provided `paymasterService` URL that may sponsor, and a subscription service the user holds an allowance with. There are two ways to combine them, both using only the methods above.

**Client-side fan-out.** The wallet calls `payer_getTerms` on each candidate (and `payer_getSponsorshipBalance` where it wants a standing balance first), then ranks the results itself and proceeds with one. Because `sender_auth` binds the chosen `payer`, the wallet commits to exactly one payer when it signs. This needs nothing beyond `payer_getTerms` and is the path to use when the wallet wants to surface a chooser across payers to the user.

**Aggregator endpoint.** A single service MAY front multiple downstream payers behind one URL and implement the same methods, so a wallet (or app) configures just one endpoint. For an intent the aggregator evaluates its downstreams and returns one `payer_getTerms` result. Because every offer is self-contained, carrying its own `payer`, `endpoint`, and `methods`, the aggregator MAY either collapse to a single chosen offer or return several offers sourced from *different* downstream payers, each pointing at the downstream that co-signs it (or proxied through the aggregator). The wallet picks one offer (and, for a `token` offer, one `TokenChoice` from its `tokens`) and co-signs against that offer's `payer`/`endpoint`. For standing balances, `payer_getSponsorshipBalance` returns multiple `PayerBalance` entries carrying `payer` / `endpoint` / `name` source attribution, so the wallet can show what the user holds across services without a per-service probe.

Because the offered terms are **not a total order** â€” a lower `paymentAmount` with no refund can be worse than a higher one that refunds unused gas, refund settlement is trust-based, and refund window and payer reputation also matter â€” an aggregator that returns a single offer MUST rank according to a documented policy (e.g. "cheapest guaranteed", "lowest expected cost assuming the refund settles", or "most trusted payer") rather than implying a canonical "best". An aggregator that instead returns several offers leaves that ranking to the wallet/user, which is the right choice when the user should weigh the tradeoffs. Either way, since each offer carries its own `payer`, a single response can present competing offers from *different* payers as separate `options` entries; competing terms from *one* payer for the same token â€” refundable versus non-refundable â€” are expressed as multiple entries in that offer's `tokens` sublist (distinguished by `id`). Whichever offer the wallet picks, `sender_auth` binds that offer's `payer`, so the user signs the exact payer that will be used.

### Transaction Construction

The [EIP-8130](./eip-8130.md) AA transaction wire format is:

```
AA_TX_TYPE || rlp([
  chain_id, sender, nonce_key, nonce_sequence, expiry,
  max_priority_fee_per_gas, max_fee_per_gas, gas_limit,
  account_changes, calls,
  payer, sender_auth, payer_auth
])
```

`calls` is a two-level array of phases: `[[call, ...], [call, ...]]` where each `call` is `[to, data]`. Phases execute in order; each phase is atomic (all-or-nothing), but completed phases persist independently.

Note on `value`: [EIP-8130](./eip-8130.md) calls carry no protocol value; each onchain `call` is `[to, data]` with `msg.value == 0`. The wallet realizes any non-zero intent `value` through the account's wallet bytecode (e.g., an `executeBatch` call that performs the value-bearing CALL). An intent call whose `value` is `0` or omitted is encoded directly as `[to, data]`.

#### Constructing the Transaction

Given terms from `payer_getTerms`, the wallet first picks one selectable `PaymentOption` from `options`; if the picked offer is a `kind: "token"` offer, the wallet additionally picks one `TokenChoice` from its `tokens`. The transaction is then built entirely from that offer (and, for token payment, that `TokenChoice`):

1. **Set `payer`** to the selected offer's `payer`.
2. **Set `nonce_key` and `nonce_sequence`**: select a `nonce_key` (a parallel-nonce lane; `0` for sequential ordering) and read the next `nonce_sequence` for `(sender, nonce_key)` via `eth_getTransactionCount`.
3. **Set `expiry`** to any value up to the offer's `conditions.maxExpiry` (RECOMMENDED: `conditions.maxExpiry`). This is the protocol-enforced onchain transaction lifetime; the payer rejects a transaction whose `expiry` exceeds `maxExpiry` at co-sign.
4. **Set gas parameters** (`max_fee_per_gas`, `max_priority_fee_per_gas`, `gas_limit`) from the offer's `gasEstimate` (recommended) or the wallet's own [EIP-8130](./eip-8130.md)-aware estimation, respecting the offer's `conditions.maxGasLimit`. A `kind: "token"` offer's `gasEstimate.gasLimit` already accounts for its phase-0 transfer (quoted as a conservative worst-case across the offer's `tokens`).
5. **Construct `calls`** using call phases:

| Offer `kind` | Phase 0 | Phase 1 |
|---|---|---|
| `sponsored` | (none) | `[user_calls]` |
| `token` | `[token.transfer(choice.feeRecipient ?? offer.payer, choice.paymentAmount)]` | `[user_calls]` |

A `kind: "sponsored"` offer uses the full-sponsorship construction (no phase 0); a `kind: "token"` offer uses the token-payment construction with the selected `TokenChoice`. For token payment, the transfer amount is the choice's `paymentAmount`, and the destination is its `feeRecipient` when present, otherwise the offer's `payer`. A wallet reusing a cached offer re-selects the same `TokenChoice` by `id` (REQUIRED whenever two choices share a `token`), recomputes the amount from that choice's `rate`, and uses that choice's `feeRecipient`. For full sponsorship with no phase 0 requirements, the wallet MAY use a single phase: `[[user_calls]]`.

**Balance-funded sponsorship** is a `kind: "sponsored"` offer that carries a `balance` (a `sponsorship` allowance or prepaid `credit`); it uses the same full-sponsorship construction with no phase 0 transfer. The payer is reimbursed off-chain from the sender's budget when it co-signs.

6. **Leave `payer_auth` empty**: the payer fills this in after receiving the sender-signed transaction.
7. **Sign**: compute `sender_signature_hash` over all fields through `payer` (excluding `sender_auth` and `payer_auth`) and produce `sender_auth`.

#### Token Payment via Call Phases

[EIP-8130](./eip-8130.md) calls carry no ETH value; token transfers are executed through contract calls. For token payment, the wallet constructs a phase 0 call that transfers tokens from the sender to the destination on the selected `TokenChoice` (its `feeRecipient` when present, otherwise the offer's `payer`):

```
offer     = terms.options[i]        // the `kind: "token"` offer the wallet picked
choice    = offer.tokens[j]         // the `TokenChoice` the wallet picked (by `id` when present)
recipient = choice.feeRecipient ?? offer.payer
phase_0   = [
  [choice.token, abi.encodeCall(IERC20.transfer, (recipient, choice.paymentAmount))]
]
```

This uses `IERC20.transfer` (a push from the sender) rather than `transferFrom`, so **no token approvals are required**. This is a significant simplification over [ERC-4337](./eip-4337.md) paymaster flows, which require the sender to pre-approve the paymaster contract (or inject an `approve()` call into the UserOperation's calldata) before the paymaster can pull payment in `postOp`.

Transferring `paymentAmount` to `recipient` is always sufficient while terms are valid. Phase 0 commits independently, so the recipient receives tokens regardless of whether the user's calls in phase 1 succeed or revert; this is how the payer is compensated.

#### Payer Co-Signing

The payer receives the sender-signed transaction (with `payer_auth` empty), validates it against its terms, computes `payer_signature_hash` using `AA_PAYER_TYPE` (see [EIP-8130](./eip-8130.md) Signature Payload), and fills in `payer_auth` with `authenticator || data`. The payer's actor MUST have PAYER scope in its `actor_config` on the payer account.

A payer service MAY delegate co-signing to a separate **signing service** that custodies the PAYER-scoped key, for example to separate pricing and policy logic from key management and submission. The `payer` field always identifies the account whose actor signs, so `sender_auth` binds the transaction to that account regardless of which operator runs the endpoint.

#### Token Cost Transparency

Because the token cost is fixed before the sender signs, wallets can display it precisely. The displayed fee is the phase 0 transfer amount: `paymentAmount` rendered with the selected `TokenChoice`'s `decimals` and `symbol`. A wallet pricing from a cached offer derives the amount as `ceil(gas_limit Ã— max_fee_per_gas Ã— rate.numerator / rate.denominator)` and MAY derive a human-readable rate from `rate` and `symbol` for supporting context. When `usdRate` is present, the wallet can additionally display an approximate USD cost as `paymentAmount Ã— usdRate / (10^decimals Ã— 10^6)`. When no `refund` is offered, this is the final charge; when a `refund` policy is present, it is an upper bound and the wallet MAY indicate that unused gas is refunded. The quote is valid until the offer's `ttl`; after that the wallet MUST re-request terms.

### Endpoint Versioning

Payer service endpoints MUST use path-based major versioning (e.g., `/v1`, `/v2`). Method names remain stable across versions. Breaking changes (new required fields, changed semantics) require a new major path version. Additive optional fields in request or response are backwards-compatible within the same version. Clients MUST ignore unknown fields. Payers MUST NOT require new optional fields within an existing version.

## Rationale

### Why New RPC Methods

While the `paymasterService` capability is shared, the RPC methods must differ. [ERC-7677](./eip-7677.md)'s two-phase `pm_getPaymasterStubData` / `pm_getPaymasterData` flow exists because [ERC-4337](./eip-4337.md) paymasters inject data into the UserOperation before gas estimation, and the stub data must produce consistent code paths and byte lengths. In [EIP-8130](./eip-8130.md), the payer simply co-signs the complete transaction; there is no paymaster data injection, no stub values, and no code path consistency concern. The `payer_*` methods reflect this simpler model. Service providers that support both wallet types SHOULD implement both method sets at the same endpoint URL.

### `payer_sendTransaction` vs `payer_signTransaction`

Two methods mirroring the `eth_sendTransaction` / `eth_signTransaction` convention:

- **`payer_sendTransaction`** minimizes hops: the payer co-signs and submits in one step. This is the common case for app-sponsored transactions.
- **`payer_signTransaction`** gives the wallet submission control, useful for high-value transactions, MEV-sensitive paths, or when the wallet wants to use private mempools.

They are kept as two methods rather than one method with a `submit`/`relay` flag because their return types genuinely diverge â€” a `transactionHash` versus the co-signed transaction bytes â€” and a merged method would only relocate the "is wallet-side submission available?" question from *which method exists* to *which parameter value is supported*, turning a statically-knowable fact into a runtime error-and-fallback. Two methods let the wallet learn the path up front from the picked offer's `methods` field and never make a call it knows will fail.

`payer_sendTransaction` is REQUIRED and `payer_signTransaction` is OPTIONAL because the two paths are not symmetric in practice:

- **Relaying is near-universal and cheap.** Any payer can forward a fully-signed transaction to a public node, so requiring `payer_sendTransaction` lets the common flow work against every payer with no discovery step.
- **Returning co-signed bytes is the rarer capability.** A payer that co-signs just-in-time â€” a sequencer or builder that produces `payer_auth` as part of block construction â€” has no bytes to hand back before inclusion and can only relay. Independently, returning bytes opens a window between the payer committing its signature and the transaction landing, during which the sender's phase-0 token balance can move, the quote can go stale, or the bytes can be broadcast to a competing builder; relay-only collapses that window. Risk-sensitive payers (especially token-payment ones) therefore tend to withhold `payer_signTransaction`.

So `payer_sendTransaction` is the de-facto baseline and `payer_signTransaction` is the opt-in for wallets that need submission control. The choice between them is the wallet's, not the app's. A relay-only sequencer that wants exclusive inclusion simply omits `payer_signTransaction`, so the wallet never receives broadcastable bytes (see [Sequencer-Operated Payer Implementations](#sequencer-operated-payer-implementations)).

### `feeRecipient` Placement

`feeRecipient` is **per token**, not per payer. The destination of phase 0's `token.transfer` is properly a token-level routing decision: a payer accepting both USDC and a native-token wrapper may route USDC to a stablecoin treasury and the native wrapper to a different operator wallet; revenue-share splits with apps are negotiated per integration and per token. Hoisting `feeRecipient` to the payer-level (or to the offer level) would force a single recipient for the payer's full token set, which collides with these patterns. It lives on `TokenChoice` so each token entry in a `token` offer can carry its own destination; the wallet uses the `feeRecipient` of the choice it picked as the phase-0 transfer destination, and absent that, falls back to the offer's `payer`.

The recipient is per-intent in practice as well as per-token: app-sponsored token-payment policies route fees to different treasuries by app or policy, builder-operated payers may route through different operator addresses depending on inclusion lane, and revenue-share splits with apps are negotiated per integration. Carrying `feeRecipient` on the `payer_getTerms` quote lets a payer return the right destination for each specific intent without the wallet needing a separate discovery hop, and a wallet building from cached terms has everything it needs to construct phase 0.

### Sequencer-Operated Payer Implementations

The wallet protocol is identical for every payer, but a payer operated by the chain's sequencer (or by a party with privileged inclusion) has operational levers worth noting as one possible implementation pattern; this is non-normative and many variations are possible:

- **Drop-on-failure**: because the operator builds the block, it MAY simulate phase 0 against pending state and omit the transaction entirely if phase 0 would revert, avoiding nonce consumption on a failed payment transfer.
- **Exclusive inclusion**: a payer that wants its co-signed transactions includable only by the sequencer that signed them simply does **not** implement the OPTIONAL `payer_signTransaction` (returns the signed bytes), serving only the REQUIRED `payer_sendTransaction` (returns a hash). The wallet never receives the serialized transaction, so no third party can broadcast it to a competing builder; this is also useful for avoiding mempool exposure for MEV reasons.
- **Pattern-aware policies**: a sequencer-payer MAY apply broader heuristics, for example dropping transactions whose phase 0 fails in patterns consistent with probing or griefing against a particular token set.

These are operator choices, not wire-protocol additions. From a wallet's perspective a sequencer-operated payer is just a payer; its accepted tokens and methods surface through the same `payer_getTerms` response (in each offer's `tokens` and `methods` fields) as any other payer.

## Security Considerations

**API key exposure**: Payer service URLs commonly contain API keys. App developers SHOULD proxy payer requests through their own backend to keep keys secret, identical to the recommendation in [ERC-7677](./eip-7677.md). Payer services SHOULD support scoped API keys restricted to specific methods.

**Relay trust**: In relay mode, the wallet trusts the payer to submit the transaction. The transaction's `expiry` field bounds this exposure; if the payer does not submit before expiry, the transaction is dead. Wallets concerned about relay trust SHOULD use cosign mode and submit themselves.

**Rate commitment**: Once a payer returns terms, it SHOULD honor them until the quote's `ttl`. Because `paymentAmount` is quoted at the gas cap, transferring it already absorbs normal gas movement; wallets pricing from a cached `rate` can add a small buffer and use short expiry windows to stay includable.

**Balances are advisory**: A returned `balance` MUST NOT be treated as a spend guarantee. Wallets MUST handle `BUDGET_EXHAUSTED` and other rejections at co-sign time, since concurrent transactions or policy may invalidate a stale balance snapshot. Payers MUST enforce the actual budget server-side and MUST NOT rely on the wallet honoring the advertised balance.

**Phase-0 settlement risk (token payment)**: Between `payer_getTerms` and inclusion, a sender's token balance can change (front-running, concurrent transfers, or intentional griefing) and leave the payer with a reverted phase-0 transfer and no compensation. Payers SHOULD put mitigations in place appropriate to their risk tolerance, for example sender risk scoring, allowlists for trusted senders or tokens, deposit-history requirements, and velocity limits. Builder-operated payers MAY additionally drop phase-0 failures pre-inclusion (see [Sequencer-Operated Payer Implementations](#sequencer-operated-payer-implementations)).
