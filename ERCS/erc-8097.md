## Abstract

The *In-Ground Asset Token* standard provides a unified, machine-verifiable framework for representing natural resource assets (gold, copper, lithium, rare earths, aggregates, hydrocarbons, and other subsurface materials) on Ethereum.

This standard defines five canonical metadata objects — IRO, IGO, ICO, IEXO, and IEO — that together represent the complete verifiable knowledge record of any in-ground asset, from geological survey through ESG compliance. It specifies a Solidity interface, event definitions, and normative requirements for each object including immutability constraints, Competent Person attestation via [EIP-712](./eip-712.md) signatures, and monotonic depletion tracking.

## Motivation

In-ground assets represent trillions of dollars of value in proven and probable mineral reserves globally. These assets are presently represented as static, unverifiable PDFs (JORC reports, NI 43-101 filings, ESG disclosures) with no canonical digital identity, no machine-readable standard, and no mechanism to verify that data is current.

This creates three compounding problems for capital markets and on-chain applications:

1. **No canonical digital identity**: Unlike equities (XBRL) or real estate (cadastral records), mining assets have no shared standard defining what a mine is as a data object. Every asset is a bespoke PDF.

2. **ESG is self-reported and unverifiable**: GHG figures, water usage, and tailings integrity are declared annually. There is no continuous, tamper-resistant record of what is actually occurring at the mine site.

3. **RWA tokenisation is blocked**: Real-world asset tokenisation of mining assets requires trusted, structured, verifiable data at the geological layer. This standard provides that layer.

Existing token standards ([ERC-20](./eip-20.md), [ERC-721](./eip-721.md), [ERC-1155](./eip-1155.md), [ERC-3643](./eip-3643.md)) do not address geological metadata, Competent Person attestation, depletion-aware state transitions, or jurisdictional compliance gating. This ERC defines the minimum interface to support these requirements.

## Specification

### Terminology and External Standards

This specification references the following international mineral reporting frameworks. These standards define the authoritative geological classification, reporting, and compliance requirements that this ERC encodes into machine-verifiable form.

| Abbreviation | Full name | Reference |
|---|---|---|
| JORC | Australasian Joint Ore Reserves Committee Code | jorc.org/code |
| NI 43-101 | Canadian National Instrument 43-101 | securities-administrators.ca |
| SAMREC | South African Mineral Reporting Codes | samcode.co.za |
| PERC | Pan-European Reserves and Resources Reporting Committee | percreserves.eu |
| S-K 1300 | US Securities and Exchange Commission mining disclosure standard | sec.gov |
| CP/QP | Competent Person (JORC/SAMREC/PERC) or Qualified Person (NI 43-101 / S-K 1300) — a named professional with specified credentials who signs the technical report | Per the relevant reporting standard above |

In this document, the terms "CP" and "QP" are used interchangeably and both refer to the qualified professional whose identity and signature are required under the applicable reporting standard.

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

### Overview

An in-ground asset implementation maintains five metadata objects per deposit, each with a distinct update lifecycle:

| Object | Code | Mutability | Primary update trigger |
|---|---|---|---|
| In-Ground Resource Object | IRO | Immutable after anchoring | Technical report publication |
| In-Ground Geology Object | IGO | Versioned via CP/QP signature | Updated geological survey or report |
| In-Ground Compliance Object | ICO | Updatable by authorised address | Licence renewal, CP change |
| In-Ground Extraction Object | IEXO | Monotonically decreasing | Extraction event (IoT sensor or manual) |
| In-Ground Environmental Object | IEO | Updatable | ESG disclosure, rehabilitation milestone |

### Data Structures

```solidity
// SPDX-License-Identifier: CC0-1.0
pragma solidity ^0.8.20;

/// @notice Canonical resource quantity — anchored once, never mutated.
struct IRO {
    /// @notice keccak256 of canonical deposit name concatenated with jurisdiction identifier
    bytes32 depositId;
    /// @notice ISO 4217-style commodity code (e.g. "AU" for gold, "CU" for copper, "LI" for lithium)
    string  commodity;
    /// @notice Reporting standard identifier: one of "JORC2012", "NI43-101", "SAMREC", "PERC", "SKS1300"
    string  reportingStandard;
    /// @notice Total in-ground quantity at time of anchoring, expressed in the smallest unit
    ///         (e.g. milligrams for gold, kilograms for bulk commodities)
    uint256 totalInGround;
    /// @notice Resource classification: 0 = Inferred, 1 = Indicated, 2 = Measured
    uint8   resourceClass;
    /// @notice SHA-256 hash of the source technical report PDF as bytes32
    bytes32 anchorHash;
    /// @notice block.timestamp at time of first anchoring — set once, never changed
    uint256 anchoredAt;
}

/// @notice Geological metadata — versioned, must be signed by the CP/QP.
struct IGO {
    /// @notice Must match IRO.depositId
    bytes32 depositId;
    /// @notice ISO 3166-1 alpha-2 country code (e.g. "AU", "CA", "ZA")
    string  country;
    /// @notice Deposit style as used in the reporting standard (e.g. "Orogenic Gold", "Porphyry Copper")
    string  depositStyle;
    /// @notice Host rock lithology (e.g. "granite", "basalt", "sediment")
    string  hostRock;
    /// @notice Ethereum address of the signing CP/QP
    address cpAddress;
    /// @notice EIP-712 signature over the IGOAttestation struct fields (excluding cpSignature itself)
    ///         signed by cpAddress
    bytes   cpSignature;
    /// @notice UNIX timestamp of the most recent CP/QP site visit documented in the report
    uint256 siteVisitDate;
    /// @notice SHA-256 hash of the technical report as bytes32
    bytes32 reportHash;
    /// @notice Starts at 1 on first anchoring, increments by 1 on each superseding update
    uint256 version;
}

/// @notice Compliance and tenure record — must be validated before transfers are enabled.
struct ICO {
    /// @notice Must match IRO.depositId
    bytes32 depositId;
    /// @notice ISO 3166-1 alpha-2 country code concatenated with a colon and the
    ///         jurisdiction-specific licence identifier
    ///         (e.g. "AU:M70/1388" for Mining Licence M70/1388 in Australia)
    string  jurisdictionId;
    /// @notice Must match IGO.cpAddress for the current version
    address cpAddress;
    /// @notice Set to true by verifyAttestation() when cpSignature in IGO is verified
    ///         as a valid EIP-712 signature from cpAddress. Must be true before transfers.
    bool    attestationValid;
    /// @notice UNIX timestamp of licence expiry. Set to 0 if the licence has no expiry date.
    uint256 licenceExpiresAt;
    /// @notice SHA-256 hash of the indigenous consent documentation, or bytes32(0) if
    ///         not applicable or not yet obtained
    bytes32 indigenousConsentHash;
}

/// @notice Extraction and depletion tracking — monotonically decreasing.
struct IEXO {
    /// @notice Must match IRO.depositId
    bytes32 depositId;
    /// @notice Remaining in-ground quantity in the same units as IRO.totalInGround.
    ///         MUST be <= IRO.totalInGround. MUST only decrease. MUST NOT increase.
    uint256 remainingInGround;
    /// @notice block.timestamp of the most recent recordDepletion() call
    uint256 lastDepletionAt;
    /// @notice Cumulative quantity extracted since first anchoring
    ///         Invariant: totalDepleted + remainingInGround == IRO.totalInGround (at all times)
    uint256 totalDepleted;
}

/// @notice Environmental, social, and governance record.
struct IEO {
    /// @notice Must match IRO.depositId
    bytes32 depositId;
    /// @notice Whether an Environmental Impact Assessment has been submitted to the relevant authority
    bool    eiaSubmitted;
    /// @notice Whether documented water rights are on file
    bool    waterRightsPresent;
    /// @notice Whether a tailings storage facility management plan is on file
    bool    tailingsPlanPresent;
    /// @notice SHA-256 hash of the most recent GHG (Scope 1 and Scope 2) disclosure,
    ///         or bytes32(0) if no disclosure is on record
    bytes32 ghgReportHash;
    /// @notice Value in wei of the rehabilitation bond posted with the relevant authority.
    ///         Set to 0 if no bond has been posted or is not required.
    uint256 rehabilitationBond;
    /// @notice block.timestamp of the most recent IEO update
    uint256 lastUpdated;
}
```

### Interface

```solidity
// SPDX-License-Identifier: CC0-1.0
pragma solidity ^0.8.20;

/// @title IERC-8097
/// @notice Interface for the In-Ground Asset Token standard
/// @dev Implementations MUST emit the specified events on every state change.
///      Implementations MUST revert with the specified errors on invalid inputs.
interface IERC8097 {

    // ── Events ─────────────────────────────────────────────────────────

    /// @notice Emitted when a new resource is anchored for the first time.
    /// @param depositId     Unique identifier for the deposit
    /// @param commodity     Commodity code (e.g. "AU")
    /// @param totalInGround Total resource quantity at anchoring
    /// @param anchorHash    SHA-256 hash of the source technical report
    event ResourceAnchored(
        bytes32 indexed depositId,
        string commodity,
        uint256 totalInGround,
        bytes32 anchorHash
    );

    /// @notice Emitted when a depletion event is recorded against a deposit.
    /// @param depositId         Unique identifier for the deposit
    /// @param amountDepleted    Quantity extracted in this event
    /// @param remainingInGround Remaining quantity after this depletion
    /// @param recorder          Address that called recordDepletion()
    event DepletionRecorded(
        bytes32 indexed depositId,
        uint256 amountDepleted,
        uint256 remainingInGround,
        address indexed recorder
    );

    /// @notice Emitted when a CP/QP provides or updates a geological attestation.
    /// @param depositId     Unique identifier for the deposit
    /// @param cpAddress     Ethereum address of the signing CP/QP
    /// @param reportHash    SHA-256 hash of the signed technical report
    /// @param siteVisitDate UNIX timestamp of the CP/QP site visit
    /// @param version       IGO version number after this update
    event CPAttestation(
        bytes32 indexed depositId,
        address indexed cpAddress,
        bytes32 reportHash,
        uint256 siteVisitDate,
        uint256 version
    );

    /// @notice Emitted when the ICO jurisdiction or attestation status is updated.
    /// @param depositId        Unique identifier for the deposit
    /// @param jurisdictionId   Updated jurisdiction identifier string
    /// @param licenceExpiresAt UNIX timestamp of licence expiry, or 0
    event JurisdictionUpdated(
        bytes32 indexed depositId,
        string jurisdictionId,
        uint256 licenceExpiresAt
    );

    /// @notice Emitted when the IEO environmental record is updated.
    /// @param depositId          Unique identifier for the deposit
    /// @param ghgReportHash      SHA-256 hash of the updated GHG disclosure
    /// @param eiaSubmitted       Whether an EIA has been submitted
    /// @param rehabilitationBond Current rehabilitation bond value in wei
    event ESGUpdated(
        bytes32 indexed depositId,
        bytes32 ghgReportHash,
        bool eiaSubmitted,
        uint256 rehabilitationBond
    );

    // ── Errors ──────────────────────────────────────────────────────────

    /// @notice Thrown when attempting to modify an already-anchored IRO
    error AlreadyAnchored(bytes32 depositId);

    /// @notice Thrown when attempting to anchor a deposit with zero totalInGround
    error ZeroResourceQuantity();

    /// @notice Thrown when a depletion amount would cause remainingInGround to underflow
    error DepletionExceedsRemaining(bytes32 depositId, uint256 requested, uint256 remaining);

    /// @notice Thrown when EIP-712 signature verification fails for a CP/QP attestation
    error InvalidCPSignature(bytes32 depositId, address cpAddress);

    /// @notice Thrown when a function requires a valid attestation (ICO.attestationValid == true)
    ///         but none is present
    error AttestationNotValid(bytes32 depositId);

    /// @notice Thrown when attempting to operate on a depositId that has not been anchored
    error DepositNotFound(bytes32 depositId);

    // ── Anchoring ───────────────────────────────────────────────────────

    /// @notice Anchor a new resource object for the first time.
    ///         MUST revert with AlreadyAnchored if IRO for depositId already exists.
    ///         MUST revert with ZeroResourceQuantity if iro.totalInGround == 0.
    ///         MUST set iro.anchoredAt to block.timestamp.
    ///         MUST emit ResourceAnchored.
    /// @param iro The IRO to anchor. iro.depositId MUST be unique.
    function anchorResource(IRO calldata iro) external;

    // ── Geology ─────────────────────────────────────────────────────────

    /// @notice Record or update a CP/QP geological attestation.
    ///         MUST revert with DepositNotFound if IRO for igo.depositId does not exist.
    ///         MUST revert with InvalidCPSignature if EIP-712 signature verification fails.
    ///         On first call: sets IGO.version to 1.
    ///         On subsequent calls: MUST increment IGO.version by 1.
    ///         MUST emit CPAttestation.
    /// @param igo The IGO to record. igo.cpSignature MUST be a valid EIP-712 signature
    ///            from igo.cpAddress over the IGO fields (excluding cpSignature itself).
    function updateGeology(IGO calldata igo) external;

    // ── Compliance ──────────────────────────────────────────────────────

    /// @notice Verify the current CP/QP attestation and set ICO.attestationValid.
    ///         MUST revert with DepositNotFound if no IGO exists for depositId.
    ///         MUST revert with InvalidCPSignature if signature verification fails.
    ///         On success: MUST set ICO.attestationValid = true.
    /// @param depositId The deposit to verify.
    /// @return valid True if attestation was successfully verified.
    function verifyAttestation(bytes32 depositId) external returns (bool valid);

    /// @notice Update the compliance object for a deposit.
    ///         MUST revert with DepositNotFound if no IRO exists for depositId.
    ///         MUST emit JurisdictionUpdated.
    /// @param ico Updated ICO. ico.depositId MUST match an existing anchored IRO.
    function updateCompliance(ICO calldata ico) external;

    // ── Extraction ──────────────────────────────────────────────────────

    /// @notice Record an extraction (depletion) event.
    ///         MUST revert with DepositNotFound if no IRO exists for depositId.
    ///         MUST revert with AttestationNotValid if ICO.attestationValid is false.
    ///         MUST revert with DepletionExceedsRemaining if amount > IEXO.remainingInGround.
    ///         MUST set IEXO.remainingInGround -= amount.
    ///         MUST set IEXO.totalDepleted += amount.
    ///         MUST set IEXO.lastDepletionAt = block.timestamp.
    ///         MUST emit DepletionRecorded.
    ///         Invariant after call: IEXO.totalDepleted + IEXO.remainingInGround == IRO.totalInGround.
    /// @param depositId The deposit to record depletion against.
    /// @param amount    Quantity depleted in this event, in same units as IRO.totalInGround.
    function recordDepletion(bytes32 depositId, uint256 amount) external;

    // ── Environmental ───────────────────────────────────────────────────

    /// @notice Update the environmental and ESG record for a deposit.
    ///         MUST revert with DepositNotFound if no IRO exists for ieo.depositId.
    ///         MUST emit ESGUpdated.
    /// @param ieo Updated IEO.
    function updateEnvironmental(IEO calldata ieo) external;

    // ── Views ────────────────────────────────────────────────────────────

    /// @notice Returns the immutable IRO for a deposit.
    ///         MUST revert with DepositNotFound if not found.
    function getIRO(bytes32 depositId) external view returns (IRO memory);

    /// @notice Returns the current IGO for a deposit.
    ///         MUST revert with DepositNotFound if not found.
    function getIGO(bytes32 depositId) external view returns (IGO memory);

    /// @notice Returns the current ICO for a deposit.
    ///         MUST revert with DepositNotFound if not found.
    function getICO(bytes32 depositId) external view returns (ICO memory);

    /// @notice Returns the current IEXO for a deposit.
    ///         MUST revert with DepositNotFound if not found.
    function getIEXO(bytes32 depositId) external view returns (IEXO memory);

    /// @notice Returns the current IEO for a deposit.
    ///         MUST revert with DepositNotFound if not found.
    function getIEO(bytes32 depositId) external view returns (IEO memory);

    /// @notice Convenience function — returns IEXO.remainingInGround.
    ///         MUST revert with DepositNotFound if not found.
    function remainingInGround(bytes32 depositId) external view returns (uint256);
}
```

### EIP-712 Domain and Signing

CP/QP attestation signatures in the IGO struct MUST be produced using [EIP-712](./eip-712.md) structured data signing.

The [EIP-712](./eip-712.md) domain MUST be:

```solidity
EIP712Domain({
    name: "ERC8097",
    version: "1",
    chainId: <current chain ID>,
    verifyingContract: <address of the implementing contract>
})
```

The signed type hash for IGO attestation is:

```solidity
bytes32 constant IGO_TYPEHASH = keccak256(
    "IGOAttestation(bytes32 depositId,string country,string depositStyle,"
    "string hostRock,address cpAddress,uint256 siteVisitDate,"
    "bytes32 reportHash,uint256 version)"
);
```

The digest that the CP/QP signs is:

```solidity
bytes32 digest = _hashTypedDataV4(keccak256(abi.encode(
    IGO_TYPEHASH,
    igo.depositId,
    keccak256(bytes(igo.country)),
    keccak256(bytes(igo.depositStyle)),
    keccak256(bytes(igo.hostRock)),
    igo.cpAddress,
    igo.siteVisitDate,
    igo.reportHash,
    igo.version
)));
```

Verification MUST use `ECDSA.recover(digest, igo.cpSignature)` and MUST revert with `InvalidCPSignature` if the recovered address does not equal `igo.cpAddress`.

### Deposit Identifier

The `depositId` MUST be computed as:

```solidity
bytes32 depositId = keccak256(abi.encodePacked(
    canonicalDepositName,  // bytes: canonical name of the deposit (e.g. "Boddington Gold Mine")
    jurisdictionId         // bytes: jurisdiction identifier (e.g. "AU:M70/1388")
));
```

The canonical deposit name MUST be the name as it appears in the technical report. The `depositId` is deterministic and MUST be consistent across all objects for the same deposit.

### Transfer Gating

Implementations that represent tradeable ownership of in-ground assets MUST gate transfers on `ICO.attestationValid == true`. This requirement exists because all five referenced reporting standards require an active CP/QP attestation as a precondition for any public market representation of the asset.

Implementations MAY additionally gate transfers on `ICO.licenceExpiresAt > block.timestamp` where a non-zero expiry is set.

## Rationale

**Why five separate objects rather than one struct?**
Each object has a fundamentally different update lifecycle. The IRO is immutable once anchored — it represents the certified reserve at the time of the technical report and must not change. The IEXO changes continuously as ore is extracted. The IEO changes as ESG obligations are met or updated. If all fields were in a single struct, an implementation would need to overwrite the immutable resource quantity every time an ESG document was updated, creating the risk of accidental or fraudulent resource inflation. Separation enforces immutability at the type level.

**Why SHA-256 hashing for report anchoring rather than IPFS CIDs?**
SHA-256 is the cryptographic hash function already used in JORC, NI 43-101, and SEC S-K 1300 reporting workflows. Mining industry participants already compute and publish SHA-256 hashes of technical reports when filing with the ASX, TSX, and SEC. IPFS CIDs introduce a content-addressing dependency that requires the document to be hosted on IPFS, which is not required by any existing reporting standard and creates availability risk. SHA-256 anchoring allows any party who obtains the report PDF to verify the hash independently.

**Why require CP/QP identity at the standard level rather than leaving it to implementors?**
All five referenced reporting standards legally mandate a named Competent Person or Qualified Person as a precondition for public market disclosure. Making this optional at the standard level would permit compliant [ERC-8097](./eip-8097.md) implementations that omit the legal requirement entirely, creating regulatory and liability ambiguity for institutional users. By requiring [EIP-712](./eip-712.md) signing, the standard makes CP/QP identity a verifiable on-chain property.

**Why monotonic depletion rather than a balance-adjustment model?**
Ore cannot be physically restored to the ground once extracted. A monotonically decreasing constraint reflects this physical reality and prevents a class of fraudulent "reserve additions" — inflating the `remainingInGround` field to misrepresent asset status to capital markets. The invariant `totalDepleted + remainingInGround == totalInGround` is checkable by any on-chain observer at any time without trusting the operator.

**Why require `ICO.attestationValid` for transfers rather than for anchoring?**
Anchoring a resource object — the first act of creating the on-chain record — should be permissioned only by the contract deployer or an authorized address. Transfer gating on attestation validity is where the CP/QP requirement becomes economically significant: it means no economic benefit can be extracted from the token until a qualified professional has signed the geological record.

**Why use [EIP-712](./eip-712.md) for CP/QP signing rather than a simple `ecrecover` over a raw hash?**
[EIP-712](./eip-712.md) structured data signing allows the CP/QP to see exactly what they are signing in a hardware wallet or signing interface, including the deposit ID, country, deposit style, and report hash. Signing a raw hash provides no such transparency and creates signing ambiguity that could be exploited to obtain signatures on unintended data.

**Why is `jurisdictionId` a formatted string rather than an enum or a numeric code?**
Licence identifiers vary substantially across mining jurisdictions: Australian mining licences use formats like `M70/1388`, Canadian claim numbers use alphanumeric codes, and African jurisdictions often use permit numbers with country-specific prefixes. A single enum or numeric space would require centralized maintenance as new jurisdictions are added. The `"ISO_CODE:LICENCE_ID"` format is self-describing, human-readable, and does not require a registry.

**What about IGA-T, IGA-X, and IGA-C token extensions?**
These optional token models (fixed-supply reserve tokens, extraction-linked dynamic supply, and ESG/rehabilitation credit modules respectively) are intentionally out of scope for this [ERC-8097](./eip-8097.md). They will be specified in separate ERCs that reference [ERC-8097](./eip-8097.md) as a dependency. Defining them in this ERC would expand the normative surface area significantly and is not required to define the core metadata standard.

## Backwards Compatibility

This ERC introduces a new interface and does not modify any existing ERC. There are no backwards compatibility considerations.

Implementations MAY choose to extend [ERC-20](./eip-20.md), [ERC-721](./eip-721.md), or [ERC-1155](./eip-1155.md) and add [ERC-8097](./eip-8097.md) compliance as an additional interface layer — for example, representing fractionalized ownership of a verified in-ground asset as [ERC-20](./eip-20.md) tokens, or representing unique mine titles as [ERC-721](./eip-721.md) tokens. In each case, [ERC-8097](./eip-8097.md) provides the geological metadata and depletion-tracking layer; the token standard provides the transfer and ownership layer.

Implementations that require identity-gated or compliance-gated transfers MAY integrate [ERC-3643](./eip-3643.md) (the T-REX permissioned token standard) as a transfer control layer. [ERC-3643](./eip-3643.md) defines on-chain identity registries and compliance modules that are architecturally aligned with the jurisdictional and CP/QP attestation gating requirements of this ERC (`ICO.attestationValid`, `ICO.licenceExpiresAt`). Such integration is OPTIONAL and out of scope for this specification. [ERC-3643](./eip-3643.md) is not a dependency of this standard.

- `updateGeology` uses OpenZeppelin `ECDSA.recover` for [EIP-712](./eip-712.md) signature verification

## Security Considerations

**CP/QP key compromise**: If the private key of a CP/QP is compromised, an attacker can produce fraudulent geological attestations. Implementations SHOULD support CP/QP key rotation via `updateGeology` with a new `cpAddress` and SHOULD emit events that allow off-chain monitoring services to detect unexpected attestation updates.

**Replay attacks on `recordDepletion`**: The `recordDepletion` function does not include a nonce. If the access control role is assigned to an automated IoT relay, implementors MUST implement replay protection in the relay layer (e.g. requiring a monotonically increasing sequence number in the relay's signed message).

**`depositId` collision**: The `depositId` is a `keccak256` hash of the deposit name and jurisdiction identifier. Implementors MUST ensure that the canonical deposit name and jurisdiction identifier are normalised consistently (e.g. consistent casing and whitespace removal) to avoid two legitimate deposits resolving to the same `depositId`.

**Jurisdiction identifier spoofing**: The `jurisdictionId` string is free-form and not verified on-chain against any authoritative registry. Off-chain consumers of this data MUST independently verify that the stated licence number corresponds to an active, valid mining licence in the stated jurisdiction before relying on `ICO.attestationValid` as a compliance signal.

**`totalInGround` denomination**: The unit of `totalInGround` and `remainingInGround` is not enforced by the standard. Implementors MUST document the denomination clearly (e.g. milligrams for gold, short tonnes for bulk commodities) in their deployment documentation to prevent unit confusion that could lead to economic misrepresentation.

**Immutability of IRO**: Once anchored, the IRO cannot be changed. A technical report that is subsequently found to be materially incorrect will result in a permanent on-chain record of the original figures alongside any corrected figures anchored under a new reporting cycle. Implementors and data consumers MUST treat the most recent valid IGO version as the authoritative geological record.
