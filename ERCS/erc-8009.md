## Abstract

Introduce a singleton, stateless contract that proxies arbitrary smart-contract calls and enforces user-supplied outcome constraints. The proxy supports **absolute post-balance thresholds** and **balance-difference (delta) checks** over native ETH and [ERC-20](./eip-20.md) balances. Calls that violate the declared constraints MUST revert.

A separate, non-normative *periphery layer* is expected to handle user approvals, permits, metadata, and hardware wallet integration, while the core proxy only executes balance operations and checks.

## Motivation

Current hardware wallets struggle to present meaningful transaction information when interacting with unknown or newly deployed smart contracts. Without prior knowledge of a contract’s ABI or address, these devices can only display raw hexadecimal calldata – an unreadable and unsafe experience for most users. This practice, known as blind signing, leaves users vulnerable to unintentionally approving malicious or incorrect transactions. This vulnerability has already been exploited in high-value attacks.

Instead of parsing transaction calldata, the proposed approach focuses on verifying expected balance differences after execution. This proposal introduces an alternative path to clear signing – defined here as the ability for users to verify the outcome of a transaction before signing, without requiring ABI knowledge. By routing calls through a singleton proxy contract that enforces these expectations on-chain, hardware wallets can reliably display the “balance after transaction” or “balance change” values to the user, taken from the transaction parameters. This enables a permissionless, scalable, and understandable transaction confirmation experience, even for unknown contracts.

## Specification

The keywords "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

### Contract Overview

A singleton, stateless permissionless proxy contract deployed once per network.

The contract accepts transaction execution requests containing:

1. **Target call data** — Encoded calldata for the intended interaction.
2. **Approvals** — A sequence of instructions that either:
   - approve ERC-20 allowances from the proxy to the target contract, or
   - transfer tokens from the proxy to the target contract,
   executed **before** the target call.
3. **Withdrawals** — Optional ETH/[ERC-20](./eip-20.md) transfers executed **after** the target call (from the proxy to specified recipients).
4. **Balance rules** — One of:
   - **Post-balance rules:** minimum absolute balances after execution; or
   - **Balance-difference rules:** minimum required deltas between pre- and post-execution balances.

The contract MUST NOT store persistent state. Multiple independent users MAY reuse the same instance without interference. The proxy **assumes that any ERC-20 tokens it operates on have already been transferred into it** (e.g., by a periphery contract).

### Interface

The `BalanceProxy` MUST implement the following interface:

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.27;

/// @title IBalanceProxy
/// @notice Interface for the BalanceProxy contract
/// @dev This interface is used to proxy calls to a target contract with specified balances and approvals
interface IBalanceProxy {
    /// @notice Struct to represent a balance or value of a specific token by target address
    /// @param target Target address whose balance is checked or modified
    /// @param token Token address (zero address for native ETH)
    /// @param balance Signed balance or delta:
    ///        - For post-balance checks: required minimum absolute balance (may be encoded as signed but interpreted as abs())
    ///        - For diff checks: required minimum signed delta (after - before >= balance)
    struct Balance {
        address target;
        address token;
        int256 balance;
    }

    /// @notice Approval instruction from the proxy to the target
    /// @param balance Token and amount; the proxy MUST operate from its own balance
    /// @param useTransfer If true, proxy transfers tokens to target; if false, proxy approves target to pull tokens
    struct Approval {
        Balance balance;
        bool useTransfer;
    }

    /// @notice Error when actual diff != expected
    /// @param token    Token address
    /// @param target   Target address
    /// @param expected Expected signed diff
    /// @param actual   Actual signed diff
    error UnexpectedBalanceDiff(
        address token,
        address target,
        int256 expected,
        int256 actual
    );

    /// @notice Error thrown when a balance is insufficient for an absolute check
    /// @param token Token address
    /// @param target Target address
    /// @param required Required (possibly signed) balance value
    /// @param actual Actual balance
    error InsufficientBalance(
        address token,
        address target,
        int256 required,
        uint256 actual
    );

    /// @notice Error thrown when an approval is targeting a different contract than the call
    /// @param token Token address
    /// @param target Target encoded into Approval
    error MaliciousApproveTarget(address token, address target);

    /// @notice Error thrown when an approval specifies a negative amount
    /// @param amount Signed amount that was expected to be non-negative
    error NegativeApprovalAmount(int256 amount);

    /// @notice Error thrown when a call fails
    /// @param target Target address
    /// @param data Data passed to the target contract
    /// @param returnData Return data from the target contract
    error CallFailed(address target, bytes data, bytes returnData);

    /// @notice Proxy call to a target contract with specified post-balance checks and approvals
    /// @param postBalances Balances to check after the call
    /// @param approvals Approvals or transfers (from proxy balance) to apply before the call
    /// @param target Target contract to call
    /// @param data Data to pass to the target contract
    /// @param withdrawals Withdrawals to make after the call (from proxy to recipients)
    /// @return Result of the call
    function proxyCall(
        Balance[] memory postBalances,
        Approval[] memory approvals,
        address target,
        bytes memory data,
        Balance[] memory withdrawals
    ) external payable returns (bytes memory);

    /// @notice Proxy call to a target contract with balance-difference checks and approvals
    /// @param diffs Required minimum signed balance differences (after - before >= diffs[i].balance)
    /// @param approvals Approvals or transfers (from proxy balance) to apply before the call
    /// @param target Target contract to call
    /// @param data Data to pass to the target contract
    /// @param withdrawals Withdrawals to make after the call (from proxy to recipients)
    /// @return Result of the call
    function proxyCallDiffs(
        Balance[] memory diffs,
        Approval[] memory approvals,
        address target,
        bytes memory data,
        Balance[] memory withdrawals
    ) external payable returns (bytes memory);
}
