## Abstract

This ERC defines a standard for deterministic deposit addresses tied to identity registries. Every identifier in a registry conforming to [ERC-8185](./eip-8185.md) has a pre-computable Ethereum address that can receive funds before the identifier is claimed. Once an entity registers, accumulated funds are withdrawn to the registrant. The deposit address is stable across escrow implementation upgrades and does not depend on the entity's participation.

## Motivation

[ERC-8185](./eip-8185.md) maps off-chain identifiers to Ethereum addresses, but only after the entity has registered. The most compelling use cases require sending value *before* the entity has engaged with Ethereum.

An AI agent instructed to "fund every dependency in this project" needs to compute deposit addresses locally and transfer tokens immediately — no coordination, no trusted backend, no waiting for maintainers to register. A grants protocol allocating to `github:org/repo`, a dependency funding tool routing to every package in a supply chain, or a DAO escrowing a payment to a domain all need the same property: an address that exists before the entity does.

This requires a deposit address that is a pure function of the identifier and the factory, computable by anyone without an on-chain call. The entity claims the funds whenever they register.

This differs from ENS, which requires proactive registration before the name resolves to anything. It also differs from simple `mapping(bytes32 => uint256)` escrow patterns, which cannot receive funds via standard `ERC20.transfer` calls and require depositors to interact with a custom contract.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

### Overview

The system consists of two components:

1. **Escrow Factory** — deploys one escrow proxy per identifier at a deterministic address via CREATE2.
2. **Claimable Escrow** — per-identifier proxy that holds funds and releases them to the registrant.

### Escrow Factory Interface

A compliant escrow factory MUST implement:

```solidity
interface IEscrowFactory {
    /// @notice MUST be emitted when an escrow proxy is deployed for an identifier.
    event EscrowDeployed(bytes32 indexed id, address escrow);

    /// @notice Returns the deterministic deposit address for `id`.
    /// @dev    This MUST be a pure computation based on the factory address,
    ///         the proxy initcode, and the identifier. It MUST NOT require
    ///         any on-chain state to compute.
    ///         The returned address MUST NOT change if the escrow implementation
    ///         is upgraded.
    function predictAddress(bytes32 id) external view returns (address);

    /// @notice Deploys the escrow proxy for `id` if not yet deployed.
    /// @dev    MUST revert if the escrow for `id` has already been deployed.
    ///         MUST use CREATE2 with `id` as the salt.
    ///         MUST emit EscrowDeployed on success.
    ///         Anyone MAY call this function — it is not access-controlled.
    function deployEscrow(bytes32 id) external returns (address escrow);
}
```

### Deterministic Address Computation

The deposit address for an identifier MUST be deterministic and MUST be computable off-chain using the CREATE2 formula:

```
address = keccak256(0xff ++ factoryAddress ++ salt ++ keccak256(initcode))[12:]
```

Where:
- `factoryAddress` is the address of the escrow factory (or registry, if the factory is embedded).
- `salt` is the `bytes32` identifier from [ERC-8185](./eip-8185.md).
- `initcode` is the proxy creation code concatenated with its constructor arguments.

The deposit address MUST remain stable across escrow implementation upgrades. Implementations using a beacon proxy pattern (where the proxy delegates to a beacon that points to the current implementation) satisfy this requirement, since upgrading the beacon does not change the proxy's initcode or address.

### Claimable Escrow Interface

Each per-identifier escrow proxy MUST implement:

```solidity
interface IClaimableEscrow {
    /// @notice MUST be emitted when funds are withdrawn to the registrant.
    event Withdrawn(address indexed token, address indexed to, uint256 amount);

    /// @notice Withdraw all `token` balance held by this escrow to the
    ///         current registered owner of the associated identifier.
    /// @dev    MUST revert if the identifier has not been claimed
    ///         (i.e. the registry returns address(0) for the identifier).
    ///         MUST revert if the total balance of `token` is zero.
    ///         Anyone MAY call this function — funds MUST always go to the
    ///         registrant, never to the caller.
    /// @param  token The ERC-20 token address to withdraw.
    function withdraw(address token) external;
}
```

### Funding Paths

An escrow proxy MUST support receiving funds via standard `ERC20.transfer` to the proxy address. Implementations MAY additionally support other funding paths (e.g. Splits Warehouse deposits) as long as `withdraw` sweeps all accumulated balances.

### Native ETH

Implementations SHOULD support native ETH deposits by including a `receive()` function on the escrow proxy. When native ETH is held, `withdraw` SHOULD be callable with a sentinel token address (RECOMMENDED: `0xEeeeeEeeeEeEeeEeEeEeeEEEeeeeEeeeeeeeEEeE`) to sweep the ETH balance to the registrant. The proxy's deterministic address can receive ETH before deployment via `selfdestruct` or coinbase transactions, so implementations SHOULD handle pre-existing ETH balances.

### Escrow Lifecycle

1. **Before deployment**: The deposit address is computable by anyone. Funds sent to this address via `ERC20.transfer` accumulate at the address. The escrow proxy does not need to be deployed for the address to receive [ERC-20](./eip-20.md) tokens.
2. **Deployment**: Anyone may call `deployEscrow(id)` to deploy the proxy. The factory MAY also deploy the escrow lazily during `claim` in the associated registry.
3. **Before claim**: The escrow holds funds. `withdraw` MUST revert because the identifier has no registered owner.
4. **After claim**: Anyone may call `withdraw(token)` to send accumulated funds to the registrant.
5. **After revocation**: If the registrant revokes their claim, `withdraw` MUST revert again (no owner). Funds remain at the escrow address and are claimable by whoever claims the identifier next.

### Ownership Change Behavior

If ownership of an identifier changes (revoke and re-claim by a different entity), accumulated funds at the escrow address go to the *current* registrant at the time `withdraw` is called. The deposit address is stable; the destination is determined at withdrawal time.

## Rationale

### Beacon proxy pattern

Using a beacon proxy (rather than a minimal proxy or a direct deployment) allows upgrading the escrow logic for all identifiers simultaneously without changing any deposit address. The deposit address depends only on the factory address, the beacon address, and the identifier salt — not on the implementation address.

### Anyone can call `withdraw`

Allowing anyone to trigger withdrawal (with funds always going to the registrant) enables keepers and frontends to sweep on behalf of registrants. This reduces friction for entities that may not be familiar with Ethereum transaction mechanics.

### Anyone can call `deployEscrow`

Escrow deployment is permissionless because the deposit address must be usable before the entity has interacted with Ethereum. A funder, a frontend, or the registry itself may deploy the escrow. The deterministic address ensures the result is the same regardless of who deploys.

### Separate from the registry

This ERC is architecturally independent from [ERC-8185](./eip-8185.md). The registry holds no funds and has no opinion about how money flows. This separation allows:

- Protocols that only need identity resolution to depend on the registry alone.
- The escrow module to be upgraded or replaced without modifying the registry.
- Alternative escrow implementations (e.g. streaming, vesting, multi-token) to be built against the same registry.

An implementation MAY embed the escrow factory into the registry contract for convenience. The interfaces remain the same.

## Backwards Compatibility

This ERC is compatible with any [ERC-20](./eip-20.md) token. It does not modify any existing standards.

Implementations that integrate with the Splits Warehouse ([ERC-6909](./eip-6909.md)-based) can sweep both Warehouse deposits and direct [ERC-20](./eip-20.md) transfers in a single `withdraw` call, but this integration is OPTIONAL and not required for compliance.

## Reference Implementation

A minimal reference implementation consists of:

- `EscrowFactory.sol` — factory using [EIP-1167](./eip-1167.md) minimal proxies
- `ClaimableEscrow.sol` — escrow holding [ERC-20](./eip-20.md) tokens

The implementation has no external dependencies and uses [EIP-1167](./eip-1167.md) clones for simplicity.

## Security Considerations

### Escrow upgradeability

The beacon owner can upgrade the escrow implementation for all proxies simultaneously. This is a critical trust assumption — a malicious upgrade could redirect funds. The beacon owner key SHOULD be held by a multisig or governance contract with appropriate controls.

### Counterfactual address griefing

Deposit addresses are public and computable by anyone. An adversary could pre-deploy escrow proxies for identifiers at a gas cost, without any effect on correctness — the address and accumulated funds are unaffected. This is a nuisance, not a security issue.

### Funds go to current owner

After a revoke-and-reclaim cycle, funds accumulated before revocation go to the new claimant. This is by design — the escrow is tied to the identifier, not to any particular owner. Entities SHOULD withdraw funds before revoking their claim.

### [ERC-20](./eip-20.md) token compatibility

The escrow relies on `IERC20.balanceOf` and `safeTransfer`. Tokens with non-standard transfer behavior (fee-on-transfer, rebasing) may cause discrepancies between expected and actual withdrawal amounts. Implementations SHOULD document any limitations with non-standard tokens.

### Native ETH edge cases

Native ETH can arrive at the escrow address before the proxy is deployed (via `selfdestruct` or coinbase transactions). Implementations SHOULD account for pre-existing ETH balances when the proxy is first deployed. The sentinel address pattern (`0xEeee...eEEeE`) avoids conflating ETH withdrawals with [ERC-20](./eip-20.md) zero-address edge cases.
