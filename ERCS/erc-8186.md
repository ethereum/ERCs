## Abstract

This ERC defines a standard for deterministic smart accounts tied to identity registries. Every identifier in a registry conforming to ERC-8185 has a pre-computable Ethereum address that can receive funds before the identifier is claimed. Once registered, the owner can execute arbitrary calls through the account — withdrawing tokens, interacting with protocols, or calling any contract. The account address is stable across implementation upgrades and does not depend on the entity's participation.

## Motivation

ERC-8185 maps off-chain identifiers to Ethereum addresses, but only after the entity has registered. The most compelling use cases require sending value *before* the entity has engaged with Ethereum.

An AI agent instructed to "fund every dependency in this project" needs to compute deposit addresses locally and transfer tokens immediately — no coordination, no trusted backend, no waiting for maintainers to register. A grants protocol allocating to `github:org/repo`, a dependency funding tool routing to every package in a supply chain, or a DAO sending a payment to a domain all need the same property: an address that exists before the entity does.

Once the entity claims their identifier, they need more than just token withdrawal. They may need to interact with DeFi protocols, claim airdrops, withdraw from a Splits Warehouse, interact with vesting contracts, or perform any other on-chain action through their identity's address. A fixed `withdraw` function would require protocol-level upgrades every time a new integration pattern emerges.

This ERC defines a minimal smart account — one `execute` function gated to the registered owner — that turns every identity into a general-purpose on-chain actor.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

### Overview

The system consists of two components:

1. **Account Factory** — deploys one account proxy per identifier at a deterministic address via CREATE2.
2. **Identity Account** — per-identifier proxy that the registered owner can execute arbitrary calls through.

### Account Factory Interface

A compliant account factory MUST implement:

```solidity
interface IAccountFactory {
    /// @notice MUST be emitted when an account proxy is deployed for an identifier.
    event AccountDeployed(bytes32 indexed id, address account);

    /// @notice Returns the deterministic address for the account of `id`.
    /// @dev    This MUST be a pure computation based on the factory address,
    ///         the proxy initcode, and the identifier. It MUST NOT require
    ///         any on-chain state to compute.
    ///         The returned address MUST NOT change if the account implementation
    ///         is upgraded.
    function predictAddress(bytes32 id) external view returns (address);

    /// @notice Deploys the account proxy for `id` if not yet deployed.
    /// @dev    MUST revert if the account for `id` has already been deployed.
    ///         MUST use CREATE2 with `id` as the salt.
    ///         MUST emit AccountDeployed on success.
    ///         Anyone MAY call this function — it is not access-controlled.
    function deployAccount(bytes32 id) external returns (address account);
}
```

### Deterministic Address Computation

The account address for an identifier MUST be deterministic and MUST be computable off-chain using the CREATE2 formula:

```
address = keccak256(0xff ++ factoryAddress ++ salt ++ keccak256(initcode))[12:]
```

Where:
- `factoryAddress` is the address of the account factory (or registry, if the factory is embedded).
- `salt` is the `bytes32` identifier from ERC-8185.
- `initcode` is the proxy creation code concatenated with its constructor arguments.

The account address MUST remain stable across implementation upgrades. Implementations using a beacon proxy pattern (where the proxy delegates to a beacon that points to the current implementation) satisfy this requirement, since upgrading the beacon does not change the proxy's initcode or address.

### Identity Account Interface

Each per-identifier account proxy MUST implement:

```solidity
interface IIdentityAccount {
    /// @notice Execute an arbitrary call through this account.
    /// @dev    MUST revert if `msg.sender` is not the registered owner of
    ///         the associated identifier (as returned by the registry's `ownerOf`).
    ///         MUST revert if the call to `target` fails.
    /// @param  target The contract to call.
    /// @param  data   The calldata to send.
    /// @param  value  The ETH value to forward.
    /// @return The return data from the call.
    function execute(address target, bytes calldata data, uint256 value)
        external
        returns (bytes memory);
}
```

### Funding Paths

An account proxy MUST support receiving funds via standard `ERC20.transfer` to the proxy address. The account SHOULD include a `receive()` function to accept native ETH.

### Account Lifecycle

1. **Before deployment**: The account address is computable by anyone. Funds sent to this address via `ERC20.transfer` or ETH transfer accumulate at the address. The account proxy does not need to be deployed for the address to receive tokens.
2. **Deployment**: Anyone may call `deployAccount(id)` to deploy the proxy. The factory MAY also deploy the account lazily during `claim` in the associated registry.
3. **Before claim**: The account holds funds. `execute` MUST revert because the identifier has no registered owner.
4. **After claim**: The registered owner may call `execute` to interact with any contract — withdraw ERC-20 tokens, claim airdrops, interact with DeFi protocols, or perform any other on-chain action.
5. **After revocation**: If the registrant revokes their claim, `execute` MUST revert again (no owner). Funds remain at the account address and are controllable by whoever claims the identifier next.

### Ownership Change Behavior

If ownership of an identifier changes (revoke and re-claim by a different entity), accumulated funds at the account address become controllable by the *current* registrant. The account address is stable; the controller is determined by the registry at execution time.

## Rationale

### `execute` over `withdraw`

A fixed `withdraw(token)` function requires protocol-level upgrades to support new integration patterns (Splits Warehouse, streaming protocols, vesting contracts). The `execute` pattern delegates this flexibility to the owner, who can encode any call they need. This turns the identity account into a lightweight smart account — owned not by a private key, but by whoever controls the off-chain identity.

### Owner-only execution

Unlike a permissionless `withdraw` (where anyone can push funds to the owner), `execute` is restricted to the registered owner. This is necessary because `execute` can interact with arbitrary contracts — a permissionless version would allow anyone to make calls on behalf of the identity, which is a critical security risk.

### Beacon proxy pattern

Using a beacon proxy (rather than a minimal proxy or a direct deployment) allows upgrading the account logic for all identifiers simultaneously without changing any account address. The account address depends only on the factory address, the beacon address, and the identifier salt — not on the implementation address.

### Anyone can call `deployAccount`

Account deployment is permissionless because the account address must be usable before the entity has interacted with Ethereum. A funder, a frontend, or the registry itself may deploy the account. The deterministic address ensures the result is the same regardless of who deploys.

### Separate from the registry

This ERC is architecturally independent from ERC-8185. The registry holds no funds and has no opinion about how money flows. This separation allows:

- Protocols that only need identity resolution to depend on the registry alone.
- The account module to be upgraded or replaced without modifying the registry.
- Alternative account implementations to be built against the same registry.

An implementation MAY embed the account factory into the registry contract for convenience. The interfaces remain the same.

### Relationship to ERC-6551 (Token Bound Accounts)

[ERC-6551](./eip-6551.md) solves a structurally similar problem — "thing X owns a smart account" — for NFTs. The ownership check in ERC-6551 is `IERC721.ownerOf(tokenId)`; in this ERC it is `IOffChainEntityRegistry.ownerOf(bytes32)`. The `execute` function signature is nearly identical.

This ERC does not use ERC-6551 for three reasons:

1. **The registry is not an NFT.** Identifier ownership is revocable, non-transferable, and exists conceptually before anyone claims it. Layering ERC-721 semantics on top (soulbound minting on claim, burning on revoke) would add complexity without benefit, and would confuse tooling that expects NFTs to be transferable.
2. **Beacon upgradeability.** ERC-6551 accounts are bound to a fixed implementation at creation time. This ERC requires upgrading all accounts simultaneously (e.g. to fix a bug or add functionality) without changing any account address — a property that the beacon proxy pattern provides and ERC-6551 does not.
3. **Minimal interface.** ERC-6551 accounts implement `token()`, `state()`, `isValidSigner()`, and `execute` with an `operation` parameter for DELEGATECALL support. Identity accounts need only `execute` with CALL semantics. A smaller interface reduces attack surface and implementation burden.

Implementations that prefer ERC-6551 compatibility MAY expose the ERC-6551 account interfaces in addition to `IIdentityAccount`. The two are not mutually exclusive.

## Backwards Compatibility

This ERC is compatible with any ERC-20 token. It does not modify any existing standards.

## Security Considerations

### Account upgradeability

The beacon owner can upgrade the account implementation for all proxies simultaneously. This is a critical trust assumption — a malicious upgrade could add functions that bypass ownership checks. The beacon owner key SHOULD be held by a multisig or governance contract with appropriate controls.

### Arbitrary execution risk

The `execute` function allows the owner to make any call from the account's address. This is by design, but means the owner can interact with malicious contracts. This risk is mitigated by the fact that only the registered owner (who proved control of the off-chain entity) can call `execute`.

### Reentrancy

Since `execute` can call arbitrary contracts, implementations SHOULD consider reentrancy protection. A `nonReentrant` modifier on `execute` prevents a called contract from re-entering the account.

### Counterfactual address griefing

Account addresses are public and computable by anyone. An adversary could pre-deploy account proxies for identifiers at a gas cost, without any effect on correctness — the address and accumulated funds are unaffected. This is a nuisance, not a security issue.

### Funds go to current owner

After a revoke-and-reclaim cycle, funds accumulated before revocation become controllable by the new claimant. This is by design — the account is tied to the identifier, not to any particular owner. Entities SHOULD withdraw funds before revoking their claim.

### ERC-20 token compatibility

The account relies on the owner encoding correct calldata for token interactions. Tokens with non-standard transfer behavior (fee-on-transfer, rebasing) may cause unexpected results. This is consistent with standard smart account behavior.

### Native ETH edge cases

Native ETH can arrive at the account address before the proxy is deployed (via `selfdestruct` or coinbase transactions). Implementations SHOULD handle pre-existing ETH balances by including a `receive()` function on the account proxy.
