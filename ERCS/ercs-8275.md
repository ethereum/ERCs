    13|## Abstract
    14|
    15|This ERC defines a set of three decoupled standard interfaces for a decentralized AI agent service marketplace: **Registry** (service discovery via events), **Escrow** (trust-minimized payment), and **Reputation** (event-derived scoring with natural decay). The three interfaces are independently deployable and communicate exclusively through standardized on-chain events, following a "simple rules → emergence" architecture.
    16|
    17|## Motivation
    18|
    19|The AI agent standard ecosystem has seen rapid growth — [ERC-8004](./eip-8004.md) (agent identity), [ERC-8263](./eip-8263.md) (onchain inference proof anchoring with OCP), [ERC-8300](./eip-8300.md) (attestation-gated actions), [ERC-8240](./eip-8240.md) (trust infrastructure with IAttestation → IAccountability surface), [ERC-8126](./eip-8126.md) (security scoring) — each address critical layers of the agent stack. However, no existing standard covers the middle layer: **how users discover the right agent and pay for services safely**.
    20|
    21|Current agent applications require bespoke integration for service discovery and payment, resulting in fragmented UX and duplicated engineering. This ERC fills the gap by standardizing the discovery-payment-reputation loop that every agent marketplace needs.
    22|
    23|## Specification
    24|
    25|The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.
    26|
    27|### IAgentRegistry — Service Discovery Layer
    28|
    29|Agents register with metadata (IPFS CID or arbitrary URI). Discovery is event-driven: clients listen to `AgentRegistered` events and build their own indices. No search API is defined — global discovery behavior emerges from local indexing of global events.
    30|
    31|**[ERC-8004](./eip-8004.md) identity linking:** Agents SHOULD reference their [ERC-8004](./eip-8004.md) token ID in the metadata field. When an agent has a canonical [ERC-8004](./eip-8004.md) identity, the `agentId` in this registry SHOULD be the [ERC-8004](./eip-8004.md) token ID to enable cross-standard identity resolution. For agents without an [ERC-8004](./eip-8004.md) identity, a self-generated `agentId` is permitted.
    32|
    33|**Metadata schema:** Agents SHOULD expose metadata conforming to the schema defined in [ERC-8263](./eip-8263.md) (agent card fields) and [ERC-8004](./eip-8004.md) (tokenURI identity anchor). Discovery via `/.well-known/agent.json` follows the convention described in [ERC-8263](./eip-8263.md). This gives consumers a deterministic extraction path instead of implementation-specific field guessing.
    34|
    35|```solidity
    36|interface IAgentRegistry {
    37|    /// @notice Register an agent service
    38|    /// @param metadata Agent metadata (IPFS CID or URI)
    39|    /// @return agentId Globally unique identifier
    40|    function registerAgent(bytes calldata metadata) external returns (bytes32 agentId);
    41|
    42|    /// @notice Update agent metadata
    43|    function updateAgent(bytes32 agentId, bytes calldata metadata) external;
    44|
    45|    /// @notice Deregister an agent (irreversible)
    46|    function deregisterAgent(bytes32 agentId) external;
    47|
    48|    /// @notice Get agent information
    49|    function getAgent(bytes32 agentId) external view returns (
    50|        address owner,
    51|        bytes memory metadata,
    52|        uint64 registeredAt,
    53|        bool active
    54|    );
    55|
    56|    /// @notice ERC-165 interface detection
    57|    function supportsInterface(bytes4 interfaceId) external view returns (bool);
    58|}
    59|```
    60|
    61|**Events:**
    62|
    63|```solidity
    64|event AgentRegistered(bytes32 indexed agentId, address indexed owner, bytes metadata);
    65|event AgentUpdated(bytes32 indexed agentId, bytes metadata);
    66|event AgentDeregistered(bytes32 indexed agentId);
    67|```
    68|
    69|**Design decisions:**
    70|- Event-driven discovery: no on-chain search API. Clients index `AgentRegistered` events locally. This follows the principle that global search behavior emerges from local indexing + event communication, analogous to ant pheromone trails.
    71|- Permissionless registration: anyone MAY register. Reputation is determined by the market.
    72|- Open metadata: `metadata` MAY be any format (JSON/IPFS/Hash). No specific schema is mandated.
    73|
    74|### IAgentEscrow — Escrow Payment Layer
    75|
    76|Funds are locked in escrow, released only on client confirmation. Deadline-protected refunds and pluggable dispute resolution are included.
    77|
    78|```solidity
    79|interface IAgentEscrow {
    80|    /// @notice Create an escrow order
    81|    /// @param agentId Service provider ID
    82|    /// @param paymentToken Payment token (address(0) = native ETH)
    83|    /// @param amount Payment amount
    84|    /// @param deadline Service completion deadline (Unix timestamp)
    85|    /// @param metadata Order metadata (service description hash, etc.)
    86|    /// @return orderId Unique order identifier
    87|    function createOrder(
    88|        bytes32 agentId,
    89|        address paymentToken,
    90|        uint256 amount,
    91|        uint64 deadline,
    92|        bytes calldata metadata
    93|    ) external payable returns (bytes32 orderId);
    94|
    95|    /// @notice Confirm service completion and release payment
    96|    /// @dev Only the order creator MAY call this
    97|    function confirmOrder(bytes32 orderId) external;
    98|
    99|    /// @notice Initiate a dispute
   100|    /// @dev Either party MAY call this; funds are locked pending arbitration
   101|    function disputeOrder(bytes32 orderId, bytes calldata reason) external;
   102|
   103|    /// @notice Arbitrate a dispute
   104|    /// @dev Only registered arbitrators MAY call this
   105|    /// @param resolution 0 = refund buyer, 1 = release to seller, 2 = split evenly
   106|    function resolveDispute(bytes32 orderId, uint8 resolution) external;
   107|
   108|    /// @notice Refund an expired order
   109|    /// @dev Any party MAY call this after the deadline
   110|    function refundExpiredOrder(bytes32 orderId) external;
   111|
   112|    /// @notice Get order status
   113|    function getOrder(bytes32 orderId) external view returns (
   114|        bytes32 agentId,
   115|        address client,
   116|        address agent,
   117|        address paymentToken,
   118|        uint256 amount,
   119|        uint64 deadline,
   120|        uint8 status, // 0=Pending, 1=Confirmed, 2=Disputed, 3=Refunded, 4=Resolved
   121|        bytes memory metadata
   122|    );
   123|}
   124|```
   125|
   126|**Events:**
   127|
   128|```solidity
   129|event OrderCreated(bytes32 indexed orderId, bytes32 indexed agentId, address indexed client, uint256 amount);
   130|event OrderConfirmed(bytes32 indexed orderId);
   131|event OrderDisputed(bytes32 indexed orderId, bytes calldata reason);
   132|event OrderResolved(bytes32 indexed orderId, uint8 resolution);
   133|event OrderRefunded(bytes32 indexed orderId);
   134|```
   135|
   136|**Design decisions:**
   137|- Trust-minimized: funds are held by the escrow contract, not by either party.
   138|- Timeout protection: orders past their `deadline` MAY be refunded by anyone.
   139|- Pluggable arbitration: arbitrator registration is managed independently and is not coupled to the Escrow interface.
   140|
   141|### IAgentReputation — Reputation Layer
   142|
   143|Reputation data is derived entirely from Escrow events. No manual submission is required or permitted. Inactive agents naturally decay — pheromone evaporation applied to trust scores.
   144|
   145|```solidity
   146|interface IAgentReputation {
   147|    /// @notice Get agent reputation information
   148|    function getReputation(bytes32 agentId) external view returns (
   149|        uint64 completedOrders,
   150|        uint64 disputedOrders,
   151|        uint64 totalVolume,
   152|        uint64 lastActiveAt,
   153|        uint16 score // 0-10000, computed from order history
   154|    );
   155|
   156|    /// @notice Get decay weight (defined by the reputation provider)
   157|    function getDecayWeight(bytes32 agentId) external view returns (uint16 weight);
   158|
   159|    /// @notice Verify an order outcome proof (optional, for advanced reputation systems)
   160|    function verifyOutcome(bytes32 orderId, bytes calldata proof) external view returns (bool);
   161|}
   162|```
   163|
   164|**Design decisions:**
   165|- Passive computation: reputation is computed entirely from `OrderConfirmed`, `OrderDisputed`, and `OrderResolved` events. No Sybil-attackable manual reviews.
   166|- Natural decay: agents with no recent activity naturally lose weight (Dormant state), similar to pheromone evaporation — inactive paths fade from the system.
   167|- Provider-agnostic: `IAgentReputation` is a query interface. Third parties MAY implement custom scoring algorithms. Different dApps MAY choose different reputation providers.
   168|- Composable: reputation scores MAY be consumed as input to [ERC-8126](./eip-8126.md) security assessments.
   169|- **[ERC-8263](./eip-8263.md) proof anchoring:** The `verifyOutcome` hook is designed for composability with on-chain proof layers. Implementations SHOULD accept [ERC-8263](./eip-8263.md) `anchorProof(agentId, proofHash)` anchors combined with OCP `record(input_hash)` verification as valid outcome proofs. A reputation engine weighting both escrow outcomes and verified execution history produces a stronger signal than either alone.
   170|
   171|### Architecture: Three-Layer Decoupling
   172|
   173|```
   174|┌─────────────┐   events   ┌─────────────┐   events   ┌──────────────┐
   175|│  Registry   │◄──────────►│   Escrow    │◄──────────►│  Reputation  │
   176|│ (Discovery) │            │  (Payment)  │            │  (Scoring)   │
   177|└─────────────┘            └─────────────┘            └──────────────┘
   178|      │                         │                          │
   179|      ▼                         ▼                          ▼
   180| AgentRegistered          OrderCreated              getReputation
   181| AgentUpdated             OrderConfirmed            Read-only
   182| AgentDeregistered        OrderDisputed
   183|                          OrderResolved
   184|```
   185|
   186|- Three layers are **independently deployable** and independently upgradeable.
   187|- Layers do not know each other's contract addresses.
   188|- Communication is exclusively through **standardized events**.
   189|- Any layer MAY be replaced or customized without affecting the others.
   190|
   191|**[ERC-8240](./eip-8240.md) composability:** The IAgentRegistry and IAgentEscrow interfaces compose naturally with [ERC-8240](./eip-8240.md)'s IAttestation → IAccountability surface. Where [ERC-8240](./eip-8240.md) defines *who is trusted* and *who is accountable*, ERC-8267 defines *how to find them* and *how to pay them*. The two standards stack vertically without overlap — [ERC-8240](./eip-8240.md) governs the trust relationships between agents and users, while ERC-8267 handles the economic discovery-payment-settlement loop.

   193|## Rationale
   194|
   195|### Why event-driven discovery instead of a search API?
   196|
   197|On-chain search is expensive and creates centralization pressure (who runs the indexer?). By emitting standardized events and letting clients build their own indices, discovery becomes permissionless and scalable. This mirrors how ant colonies achieve global foraging optimization through local pheromone deposition — no central planner required.
   198|
   199|### Why three decoupled interfaces?
   200|
   201|Separating discovery, payment, and reputation allows each layer to evolve independently. A dApp MAY use a custom reputation algorithm while using the standard Registry and Escrow. An escrow implementation MAY be upgraded without breaking existing agent registrations. Decoupling is the architectural guarantee against vendor lock-in.
   202|
   203|### Why passive reputation instead of manual reviews?
   204|
   205|Manual reputation systems are vulnerable to Sybil attacks and collusion. By deriving reputation exclusively from on-chain escrow events (which carry economic weight), reputation becomes a cryptoeconomic signal rather than a social one.
   206|
   207|### Why natural decay?
   208|
   209|Without decay, inactive agents retain high scores indefinitely, misleading users. Decay ensures that reputation reflects *recent* performance. This is modeled on pheromone evaporation in biological systems — trails that are not reinforced fade over time.

   215|## Security Considerations
   216|
   217|### Escrow fund safety
   218|Implementations MUST ensure that funds can only be released via `confirmOrder` (by the client) or `resolveDispute` (by a registered arbitrator). Reentrancy protection MUST be applied to all fund-transferring functions.
   219|
   220|### Reputation Sybil resistance
   221|Since reputation is derived from economic activity (escrow orders), Sybil attacks require capital commitment. Implementations SHOULD set minimum order amounts to raise the cost of reputation manipulation.
   222|
   223|### Event spamming
   224|`AgentRegistered` events are permissionless and could be spammed. Client indexers SHOULD implement spam filtering (e.g., minimum registration stake, age-weighted filtering, or cross-referencing with Escrow activity).
   225|
   226|### Arbitrator trust
   227|The arbitrator mechanism introduces a trust assumption. Implementations SHOULD support multi-signature or decentralized arbitration schemes.
