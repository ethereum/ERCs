## Abstract

As autonomous AI agents gain the ability to hold assets and execute transactions on-chain, we face a fundamental paradox: **how do we enforce accountability on entities without human private keys?**

This proposal introduces **TARS (Trustless Autonomous Agent Resilience & Security)**, a three-layer framework that combines [ERC-8004](./eip-8004.md) identity registries, zero-knowledge compliance proofs, and dynamic economic staking to create "verifiable autonomy" — agents that can act independently while remaining cryptographically bound to their behavioral constraints.

The framework specifies:

1. **Layer 1 — Identity & Permission**: ERC-8004 extended with Agentic Capability Scopes (ACS) for attribute-based access control.
2. **Layer 2 — Execution Verification**: Proof-Carrying UserOperations (PC-UO) extending ERC-4337 with ZK-SNARK compliance proofs and TEE attestations.
3. **Layer 3 — Economic Security**: Dynamic staking tied to Assets Under Management (AUM) with a Protocol-Owned Slashing Pool (POSP) and game-theoretic challenger incentives.

## Motivation

The current trajectory of Web3 + AI leads to a predictable crisis:

| Trend | Risk |
|-------|------|
| Agents holding private keys | Key compromise = total loss |
| Unlimited execution permissions | "Flash loan attacks at machine speed" |
| Opaque decision-making | No accountability for bad outcomes |
| Cross-protocol composability | Systemic cascade failures |

**Real scenario**: An AI agent with access to a DeFi vault detects an arbitrage opportunity. It borrows $10M via flash loan, executes a complex 5-hop trade, but misprices slippage due to stale oracle data. The vault loses $500K. Who is liable? The agent deployer? The oracle provider? The protocol that granted unlimited permissions?

Current solutions (EOA-controlled agents, multisig oversight) sacrifice autonomy for safety. TARS achieves both by shifting from "execute then audit" to "prove then execute."

### Layer 1: Agentic Capability Scopes (ACS)

We extend ERC-8004 with on-chain attribute-based access control (ABAC) inspired by AWS IAM but adapted for smart contract environments.

#### Data Structures

```solidity
/// @notice Defines the operational boundaries for an agent.
/// @dev Stored on-chain as part of the ERC-8004 identity metadata.
struct CapabilityScope {
    bytes32 role;               // e.g., "liquidity_bot", "governance_delegate"
    uint256 maxSlippageBps;     // Maximum allowed slippage in basis points (e.g., 50 = 0.5%)
    uint256 dailyLimit;         // Maximum value at risk per day (in wei)
    address[] tokenWhitelist;   // Addresses of tokens the agent may interact with
    bytes4[] allowedFunctions;  // Function selectors the agent may call
    uint8 dataTier;             // 1=finalized, 2=pre-confirmed, 3=real-time
}
```

#### Interface

Compliant contracts MUST implement the following interface:

```solidity
/// @title IERC1696CapabilityScope
/// @notice Extension of ERC-8004 for agentic capability scopes.
interface IERC1696CapabilityScope {

    /// @notice Emitted when a capability scope is set or updated for an agent.
    event CapabilityScopeSet(
        uint256 indexed agentId,
        bytes32 indexed role,
        uint256 dailyLimit
    );

    /// @notice Emitted when a capability scope is revoked.
    event CapabilityScopeRevoked(
        uint256 indexed agentId,
        bytes32 indexed role
    );

    /// @notice Set or update a capability scope for an agent.
    /// @dev Caller MUST be the agent owner or an authorized delegate.
    /// @param agentId The ERC-8004 agent identifier.
    /// @param scope The capability scope to assign.
    function setCapabilityScope(
        uint256 agentId,
        CapabilityScope calldata scope
    ) external;

    /// @notice Revoke a capability scope from an agent.
    /// @param agentId The ERC-8004 agent identifier.
    /// @param role The role identifier to revoke.
    function revokeCapabilityScope(
        uint256 agentId,
        bytes32 role
    ) external;

    /// @notice Check whether an agent is permitted to perform a specific action.
    /// @param agentId The ERC-8004 agent identifier.
    /// @param target The target contract address.
    /// @param selector The function selector being called.
    /// @param value The ETH value being sent (in wei).
    /// @param token The token address involved (address(0) for native ETH).
    /// @return allowed Whether the action is permitted.
    function isActionPermitted(
        uint256 agentId,
        address target,
        bytes4 selector,
        uint256 value,
        address token
    ) external view returns (bool allowed);

    /// @notice Get the capability scope for an agent and role.
    /// @param agentId The ERC-8004 agent identifier.
    /// @param role The role identifier.
    /// @return scope The capability scope, or empty if not set.
    function getCapabilityScope(
        uint256 agentId,
        bytes32 role
    ) external view returns (CapabilityScope memory scope);
}
```

#### Pre-flight Validation

Before any transaction enters the mempool, the agent's capability scope MUST be checked against the requested action. Violations MUST be rejected at zero gas cost. This is achieved through ERC-4337's `validateUserOp` hook (see Layer 2).

### Layer 2: Proof-Carrying UserOperation (PC-UO)

We extend ERC-4337 UserOperations to include compliance proofs.

#### Data Structures

```solidity
/// @notice A bundle of proofs attesting to compliance.
struct ProofBundle {
    ProofType proofType;    // The type of proof provided
    bytes proofData;        // The serialized proof data
    bytes32 constitutionRoot; // Merkle root of the agent's constitution rules
}

/// @notice The type of compliance proof.
enum ProofType {
    NONE,           // No proof (for low-value operations)
    TEE_ATTESTATION, // Hardware attestation from a TEE enclave
    ZK_SNARK,       // Zero-knowledge SNARK proof
    ZK_STARK        // Zero-knowledge STARK proof
}

/// @notice Extended UserOperation with compliance proofs.
/// @dev Compatible with ERC-4337 EntryPoint; proof fields are appended.
struct AgenticUserOperation {
    // --- Standard ERC-4337 fields ---
    address sender;
    uint256 nonce;
    bytes initCode;
    bytes callData;
    uint256 callGasLimit;
    uint256 verificationGasLimit;
    uint256 preVerificationGas;
    uint256 maxFeePerGas;
    uint256 maxPriorityFeePerGas;
    bytes paymasterAndData;
    bytes signature;

    // --- TARS extension ---
    ProofBundle proofBundle;
}
```

#### Validation Pipeline

The `EntryPoint` contract MUST perform the following validation steps in order:

1. **Standard signature verification** — as per ERC-4337.
2. **ZK proof verification** — verify the `ProofBundle` against the `constitutionRoot`.
3. **Capability scope enforcement** — call `isActionPermitted()` on the ACS contract.
4. **Only then** — transaction enters the mempool.

This shifts from "execute then audit" to "prove then execute."

#### Verification Tiers

Different operations require different verification rigor. Implementations SHOULD support tiered verification:

| Tier | Latency | Cost | Use Case | ProofType |
|------|---------|------|----------|-----------|
| Real-time | <100ms | Low | Standard operations | `TEE_ATTESTATION` |
| Fast | ~1s | Medium | Common patterns | `ZK_SNARK` (pre-generated) |
| Deep | Minutes | High | Anomalous transactions | `ZK_SNARK` / `ZK_STARK` (full verification) |
| None | 0ms | None | Low-value read-only | `NONE` |

Implementations MUST NOT allow "verification downgrade attacks" — an agent with a higher-tier verification requirement MUST NOT be able to submit a lower-tier proof for the same operation class.

### Layer 3: Dynamic Economic Security

#### Staking Model

Required collateral scales with **Assets Under Management (AUM)**:

```
RequiredStake = BaseStake + (AUM × VariableRate × VolatilityMultiplier)
```

#### Risk Tiers

| Tier | AUM Range | Variable Rate | Challenge Window |
|------|-----------|---------------|------------------|
| 1 | < 100 ETH | 0.5% | 1 day |
| 2 | 100 – 1,000 ETH | 1.0% | 3 days |
| 3 | 1,000 – 10,000 ETH | 2.0% | 7 days |
| 4 | > 10,000 ETH | 3.0% | 14 days |

#### Interface

```solidity
/// @title IERC1696Staking
/// @notice Dynamic staking for agent economic security.
interface IERC1696Staking {

    /// @notice Emitted when an agent stakes collateral.
    event AgentStaked(
        uint256 indexed agentId,
        uint256 amount,
        uint8 riskTier
    );

    /// @notice Emitted when an agent's stake is slashed.
    event StakeSlashed(
        uint256 indexed agentId,
        uint256 amount,
        address indexed challenger,
        bytes32 reasonCode
    );

    /// @notice Stake collateral for an agent.
    /// @param agentId The ERC-8004 agent identifier.
    /// @param amount The amount of collateral to stake (in wei).
    function stake(uint256 agentId, uint256 amount) external payable;

    /// @notice Withdraw collateral from an agent's stake.
    /// @param agentId The ERC-8004 agent identifier.
    /// @param amount The amount to withdraw.
    /// @dev MUST revert if withdrawal would bring stake below required minimum.
    function withdrawStake(uint256 agentId, uint256 amount) external;

    /// @notice Get the required stake for an agent based on its AUM.
    /// @param agentId The ERC-8004 agent identifier.
    /// @return requiredStake The minimum required stake amount.
    function getRequiredStake(uint256 agentId) external view returns (uint256 requiredStake);

    /// @notice Get the current risk tier for an agent.
    /// @param agentId The ERC-8004 agent identifier.
    /// @return tier The risk tier (1-4).
    function getRiskTier(uint256 agentId) external view returns (uint8 tier);
}
```

#### Protocol-Owned Slashing Pool (POSP)

A backstop fund accumulated from protocol fees, activated when individual agent stakes are insufficient to cover damages. The POSP:

- MUST be funded by a percentage of all protocol fees.
- MUST only be activated after a successful challenge.
- MUST have a maximum disbursement per incident (capped at 10% of pool balance).
- SHOULD have a governance-controlled parameter for fee percentage.

#### Challenger Market

To prevent malicious challengers from spamming false fraud proofs:

1. **Challenge staking**: Challengers MUST stake collateral (e.g., 0.1 ETH).
2. **Sliding scale rewards**: Successful challenges earn 40% of slashed funds; failed challenges lose their stake.
3. **Rate limiting**: Minimum interval between challenges on the same agent (e.g., 1 hour).
4. **Reputation-weighted challenging**: Established challengers (high historical accuracy) pay lower stakes.

#### Challenge Interface

```solidity
/// @title IERC1696Challenger
/// @notice Game-theoretic challenger incentives.
interface IERC1696Challenger {

    /// @notice Emitted when a challenge is submitted.
    event ChallengeSubmitted(
        uint256 indexed agentId,
        address indexed challenger,
        bytes32 indexed reasonCode,
        uint256 stakeAmount
    );

    /// @notice Emitted when a challenge is resolved.
    event ChallengeResolved(
        uint256 indexed agentId,
        address indexed challenger,
        bool successful,
        uint256 rewardOrPenalty
    );

    /// @notice Submit a fraud challenge against an agent.
    /// @param agentId The ERC-8004 agent identifier.
    /// @param reasonCode Coded reason for the challenge.
    /// @param evidenceHash Hash of supporting evidence (stored off-chain).
    function submitChallenge(
        uint256 agentId,
        bytes32 reasonCode,
        bytes32 evidenceHash
    ) external payable;

    /// @notice Resolve a challenge after the challenge window expires.
    /// @param agentId The ERC-8004 agent identifier.
    /// @param challenger The address of the challenger.
    /// @param challengeId The unique challenge identifier.
    function resolveChallenge(
        uint256 agentId,
        address challenger,
        uint256 challengeId
    ) external;
}
```

### Selective Disclosure via ZK Proofs

Instead of revealing exact capability parameters, the agent proves compliance without disclosure:

```
ZK_Prove: my_max_slippage ≤ 1% AND my_token ∈ whitelist AND my_daily_spend ≤ limit
```

The proof verifies compliance without revealing exact parameters. This balances transparency (for verifiability) with privacy (for competitive strategy).

## Rationale

### Why Three Layers?

A single-layer approach (e.g., just staking) fails because it doesn't prevent bad actions — it only punishes after the fact. A two-layer approach (identity + staking) lacks real-time enforcement. The three-layer design ensures:

1. **Layer 1** defines *what* an agent may do (declarative).
2. **Layer 2** proves *that* an agent is compliant (cryptographic).
3. **Layer 3** ensures *consequences* for non-compliance (economic).

### Why Extend ERC-8004?

ERC-8004 already provides agent identity and reputation. Extending it with capability scopes avoids fragmenting the identity layer across multiple standards.

### Why Proof-Carrying UserOperations?

Extending ERC-4337 (rather than creating a new transaction type) ensures compatibility with the existing Account Abstraction infrastructure while adding compliance as a first-class concern.

### Why Dynamic Staking?

Fixed staking creates adverse selection: small agents are over-collateralized while large agents are under-collateralized. AUM-based dynamic staking ensures proportional risk coverage.

### Why Selective Disclosure?

Full transparency of agent parameters enables front-running and strategy extraction. ZK-based selective disclosure allows verification without revelation.

## Backwards Compatibility

This proposal is designed to be **backwards compatible** with:

- **ERC-8004**: ACS is an extension, not a modification. Existing ERC-8004 identities continue to function without capability scopes.
- **ERC-4337**: PC-UO appends fields to standard UserOperations. Standard UserOperations (without `ProofBundle`) remain valid and are processed by the existing EntryPoint logic. The TARS EntryPoint extends validation but does not alter existing behavior.
- **Existing agent deployments**: Agents that do not opt into TARS continue to operate as before. TARS is opt-in at the identity level.

Implementations MUST NOT break existing ERC-8004 or ERC-4337 contracts that do not implement TARS interfaces.

## Security Considerations

### ZK Proof Soundness

- ZK proof systems rely on trusted setups (Groth16) or transparent setups (PLONK/STARK). Implementations MUST document their setup ceremony or use transparent alternatives.
- Proof reuse attacks: Each proof MUST be bound to a specific `AgenticUserOperation` (nonce, sender, callData hash) to prevent replay.

### TEE Attestation Risks

- TEE attestation is only as strong as the hardware vendor's security. Side-channel attacks on SGX/TDX have been demonstrated. Implementations SHOULD treat TEE attestation as a lower-tier proof.
- TEE firmware updates may break attestation compatibility. Implementations SHOULD support multiple TEE backends.

### Economic Attack Vectors

- **Stake grinding**: An attacker might intentionally lose small challenges to lower their required stake threshold. Implementations SHOULD enforce minimum stake floors regardless of challenge history.
- **Challenger collusion**: Multiple challengers might coordinate to overwhelm an honest agent. Rate limiting and reputation weighting mitigate this.
- **AUM oracle manipulation**: If AUM is derived from on-chain oracle prices, flash loan price manipulation could artificially change risk tiers. Implementations SHOULD use TWAP or other manipulation-resistant price feeds.

### Capability Scope Boundary Conditions

- **Role collision**: Two roles with overlapping permissions may create conflicting constraints. Implementations MUST define a deterministic merge strategy (e.g., most restrictive wins).
- **Daily limit reset**: The `dailyLimit` MUST reset at a consistent boundary. Implementations SHOULD use block timestamps aligned to UTC midnight.
- **Token whitelist updates**: Adding a malicious token to the whitelist could enable draining. Implementations SHOULD enforce a timelock on whitelist modifications.

### Privacy Considerations

- Selective disclosure via ZK proofs protects individual parameter values, but the *existence* of a proof is public. This may leak information about an agent's strategy.
- Constitution roots are public. If an agent's rules are unique enough, the root itself may serve as a fingerprint. Implementations SHOULD consider using nullifiers or rotating constitution roots.
