## Abstract

This ERC defines an optional `integrity` field for [ERC-7730](./eip-7730.md) clear signing descriptor files. The field provides signer identification so that wallets can verify which parties have attested to a descriptor's correctness.

The `integrity` object is a map keyed by [CAIP-10](https://github.com/ChainAgnostic/CAIPs/blob/main/CAIPs/caip-10.md) account identifiers. Each value contains a signer type (`"EOA"` or `"contract"`) and an [EIP-191](./eip-191.md) personal-message signature (for EOA signers) or an [ERC-1271](./eip-1271.md) signature (for contract signers). The hash is not stored — verifiers compute it by Keccak-256 hashing the RFC 8785 canonicalized descriptor content (excluding the `integrity` field itself).

Multiple signers — such as a wallet vendor, an auditor, and a registry operator — can independently attest to the descriptor without altering its content. The `integrity` field is strictly additive: its presence or absence MUST NOT change how a wallet interprets the descriptor's formatting instructions.

## Motivation

[ERC-7730](./eip-7730.md) defines a JSON format that enables wallets to display human-readable transaction details before a user signs. Wallets fetch these descriptor files from registries and rely on them to format transaction data correctly. If an attacker can contribute a misleading descriptor to a registry, the wallet may display deceptive information, undermining the security guarantees that clear signing is designed to provide.

The primary threat motivating this proposal is **registry or repository poisoning.** ERC-7730 descriptors may be contributed to shared registries by multiple parties. A malicious contributor could submit a subtly incorrect descriptor — for example, one that hides a token approval amount — which passes cursory review but misleads end users. Signatures from trusted parties (wallet vendors, auditors) allow consumers to distinguish vetted descriptors from unvetted ones.

Existing mechanisms do not address these scenarios adequately:

- **ERC-7730 context binding** verifies that a descriptor targets the correct contract address and chain, but it does not indicate whether the display formatting has been reviewed by a trusted party. A descriptor can have valid context binding while still containing misleading field labels or hidden parameters.
- **HTTPS** protects data in transit but does not authenticate the content itself or identify who vouches for its correctness.
- **Git commit signatures** prove authorship within a repository but do not survive extraction. Once a descriptor file is served via an API or downloaded as a standalone file, the git signature is lost.

This ERC provides a self-contained attestation mechanism that travels with the descriptor file regardless of how it is distributed, allowing wallets to verify which trusted parties have reviewed and signed a descriptor.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

### `integrity` Field Schema

An ERC-7730 descriptor MAY include a top-level `integrity` field. The field is a JSON object used as a map, where each key is a [CAIP-10](https://github.com/ChainAgnostic/CAIPs/blob/main/CAIPs/caip-10.md) account identifier and each value is a signature entry object:

```json
{
  "integrity": {
    "<CAIP-10 account ID>": {
      "signerType": "EOA" | "contract",
      "signature": "0x<hex-encoded signature>"
    }
  }
}
```

#### Signature Entry Fields

Each value in the `integrity` map is an object with the following fields:

| Field        | Type     | Required | Description                                                                                  |
|--------------|----------|----------|----------------------------------------------------------------------------------------------|
| `signerType` | `string` | REQUIRED | Signer account type. MUST be `"EOA"` or `"contract"`.                                       |
| `signature`  | `string` | REQUIRED | `0x`-prefixed hex-encoded signature bytes.                                                   |

The map key is the signer's CAIP-10 account identifier (e.g., `eip155:1:0xAbC...123`). This naturally prevents duplicate signer entries.

### Scope Rules

- The `integrity` field is OPTIONAL. Its absence is not an error and MUST NOT be treated as one.
- When present, the `integrity` field MUST NOT alter the interpretation of any other field in the descriptor. The descriptor's formatting instructions are evaluated identically whether or not `integrity` is present.
- Implementations that do not support this ERC will ignore the `integrity` field, consistent with standard JSON extensibility.

### Multi-Signer Semantics

The `integrity` map allows multiple independent parties to attest to the same descriptor content. Each entry is verified independently.

- Each signature entry MUST be verified independently. A failure in one entry MUST NOT invalidate other entries or the descriptor itself.
- Wallets are responsible for maintaining their own set of trusted signers. It is up to each wallet to decide which signers to trust, whether to require signatures at all, and how and when to warn users about verification results.

### Hash Computation

To compute the integrity hash of a descriptor:

1. Let `D` be the parsed JSON descriptor object.
2. If `D` contains a top-level key `"integrity"`, remove it to produce `D'`. Otherwise, let `D' = D`.
3. Serialize `D'` to a byte string using [RFC 8785 (JCS — JSON Canonicalization Scheme)](https://www.rfc-editor.org/rfc/rfc8785).
4. Compute the Keccak-256 hash of the resulting byte string.
5. Encode the hash as a `0x`-prefixed lowercase hexadecimal string (66 characters total).

Implementations MUST use Keccak-256 as specified in the Ethereum Yellow Paper (i.e., the pre-standardization variant used throughout the Ethereum ecosystem), NOT the NIST SHA3-256 standard.

### Message Format

The message to be signed is a UTF-8 encoded string with the following format:

```abnf
message = label LF "hash=" hash-value LF
label   = %s"ERC-7730 descriptor"
hash-value = "0x" 64HEXDIG
LF      = %x0A
HEXDIG  = DIGIT / "a" / "b" / "c" / "d" / "e" / "f"
```

The message label reads "ERC-7730 descriptor" because the signature attests to an ERC-7730 descriptor file, not to this ERC itself.

Expressed as a template:

```
ERC-7730 descriptor\nhash=0x{hash}\n
```

where `{hash}` is the 64-character lowercase hexadecimal Keccak-256 hash from the previous section.

### Signature Verification

#### EOA Signers (`signerType: "EOA"`)

1. Compute the integrity hash as defined in [Hash Computation](#hash-computation).
2. Construct the message string as defined in [Message Format](#message-format).
3. Compute the [EIP-191](./eip-191.md) personal message hash: `keccak256("\x19Ethereum Signed Message:\n" + len(message) + message)`.
4. Recover the signer address from the signature using ECDSA recovery.
5. Compare the recovered address to the address component of the map key (CAIP-10 identifier). Verification succeeds if and only if they match.

#### Contract Signers (`signerType: "contract"`)

1. Compute the integrity hash, construct the message string, and compute the [EIP-191](./eip-191.md) personal message hash as for EOA verification.
2. Parse the map key as a CAIP-10 identifier to extract the chain ID and contract address.
3. Call [ERC-1271](./eip-1271.md) `isValidSignature(hash, signature)` on the contract at the extracted address on the referenced chain, where `hash` is the EIP-191 personal message hash from step 1.
4. Verification succeeds if and only if the call returns the ERC-1271 magic value (`0x1626ba7e`).

If the referenced chain is unavailable (e.g., no RPC endpoint configured), the verification result for that entry is "inconclusive".

### Signing Algorithm

To sign a descriptor (producer-side):

1. Compute the integrity hash as defined in [Hash Computation](#hash-computation).
2. Construct the message string as defined in [Message Format](#message-format).
3. Sign the message using [EIP-191](./eip-191.md) personal message signing with the signer's private key.
4. Construct the signer's CAIP-10 identifier.
5. If the descriptor does not yet have an `integrity` field, create an empty object.
6. Add or replace the entry at the signer's CAIP-10 key with an object containing `signerType` and `signature`.

Adding a new signer does not affect existing signatures. Since the `integrity` field is excluded from hash computation, inserting a new key in the map does not invalidate other entries.

## Rationale

### Why Keccak-256

Keccak-256 is the hash function used throughout the Ethereum ecosystem — in the EVM, in address derivation, in EIP-191, and in ABI encoding. Every wallet and Ethereum library already implements it. Using a different hash function (e.g., SHA-256) would introduce an unnecessary dependency for wallets that do not otherwise need it.

### Why RFC 8785 (JSON Canonicalization Scheme)

JSON objects have no defined key ordering, and JSON serializers differ in whitespace, Unicode escaping, and number formatting. To produce a deterministic byte string for hashing, a canonicalization scheme is required. RFC 8785 is a published standard that defines deterministic JSON serialization with well-specified rules for key ordering, number formatting, and string escaping. It is more widely implemented and better specified than ad-hoc approaches like "sort keys and minimize whitespace."

### Why EIP-191 and Not EIP-712

[EIP-712](./eip-712.md) typed data signing is designed for on-chain verifiable structured data with a domain separator. It requires defining a type schema and domain — complexity that provides no benefit here, since the signed data is a simple message verified off-chain. [EIP-191](./eip-191.md) personal message signing is simpler, widely supported by every wallet, and sufficient for this use case. Domain separation is achieved through the message label ("ERC-7730 descriptor") rather than through an EIP-712 domain struct.

### Why CAIP-10

[CAIP-10](https://github.com/ChainAgnostic/CAIPs/blob/main/CAIPs/caip-10.md) provides a standardized format for identifying accounts across multiple chains. Since ERC-7730 descriptors may target contracts on any EVM-compatible chain, and since contract signers (ERC-1271) require specifying which chain to query, a chain-agnostic account identifier is necessary. CAIP-10 is the established standard for this purpose.

### Why No Stored Hash

The integrity hash is deterministic — given the same descriptor content, it always produces the same value. Storing it in the file would be redundant: verifiers must recompute it anyway to verify signatures. Omitting the stored hash eliminates a consistency problem (stored hash vs. computed hash mismatch) and simplifies the schema. The hash is computed once during verification and used to check each signature independently.

### Why Optional

Making the `integrity` field mandatory would break backwards compatibility with all existing ERC-7730 descriptors and tools. An optional field allows incremental adoption: producers can begin adding integrity data while consumers that do not yet support verification continue to function normally.

## Backwards Compatibility

This ERC is fully backwards compatible with [ERC-7730](./eip-7730.md).

- The `integrity` field is optional and additive. Existing ERC-7730 descriptors remain valid without modification.
- Wallets and tools that do not recognize the `integrity` field will ignore it per standard JSON extensibility behavior. Their functionality is unaffected.
- No changes to the ERC-7730 specification are required to adopt this ERC. The `integrity` field is simply an additional top-level key.
- Adoption can be incremental: descriptor producers can begin adding `integrity` data independently of consumer support. Consumers can begin verifying `integrity` when present without requiring all descriptors to include it.

## Security Considerations

### Threat Model

| Threat                         | Description                                                                                                   | Mitigation                                                                                             |
|--------------------------------|---------------------------------------------------------------------------------------------------------------|--------------------------------------------------------------------------------------------------------|
| Descriptor modification        | Attacker modifies a signed descriptor's content.                                                              | Recomputed hash differs from the hash used at signing time; all signatures fail verification.          |
| Signer impersonation           | Attacker forges a signature entry with a trusted signer's CAIP-10 identifier.                                 | ECDSA signature verification fails; recovered address will not match claimed signer.                   |
| Descriptor swap (downgrade)    | Attacker replaces a signed descriptor with a modified unsigned version, stripping the `integrity` field to hide the modification. | Wallets that maintain trusted signer lists can detect the absence of expected signatures.               |

### Descriptor Swap (Downgrade Attack)

An attacker who can replace a descriptor (e.g., through registry poisoning) may also strip the `integrity` field to remove evidence that a signed version previously existed, effectively downgrading the descriptor's security level. The modified descriptor would appear as a standard unsigned ERC-7730 file. This combination — content modification plus integrity stripping — is the primary attack vector this ERC helps defend against.

This ERC does not prevent such swaps by itself. Wallets that maintain trusted signer lists for known contracts can detect the absence of expected signatures and act accordingly. This is analogous to certificate pinning: this ERC provides the verification mechanism; the wallet's trust policy determines what level of verification is required.

### Wallet Trust Responsibility

A valid signature proves that a specific key holder attested to the descriptor content. It does not prove that the key holder is trustworthy or that the descriptor is correct. Wallets are responsible for maintaining the set of signer keys they trust. Trust decisions — which signers to accept, whether to require multiple signers, how to handle unknown signers — are wallet policy concerns, not protocol concerns.

### Explicit Non-Protections

This ERC does **not** protect against the following:

- **Malicious or compromised signers.** If a trusted signer acts maliciously or their private key is stolen, they can sign an incorrect descriptor. This is a key management and governance problem, not a protocol problem. Key rotation and revocation mechanisms are out of scope.
- **Semantic attacks.** A descriptor may be technically correct (valid context binding, valid signatures) but semantically misleading (e.g., labeling a field "Amount" when it represents something else). Integrity verification confirms who attested to the descriptor; it does not validate the descriptor's semantic correctness.

### Implementation Pitfalls

- **Keccak-256 vs. SHA3-256.** Ethereum uses the pre-standardization Keccak-256. The NIST-standardized SHA3-256 produces different output for the same input. Implementations MUST use Keccak-256.
- **RFC 8785 compliance.** Partial or incorrect implementations of RFC 8785 (e.g., incorrect number serialization or Unicode handling) will produce different canonical forms and cause hash mismatches. Implementations SHOULD use well-tested RFC 8785 libraries and validate against the RFC 8785 test vectors.
- **ERC-1271 chain availability.** Contract signer verification requires an RPC connection to the referenced chain. Implementations MUST handle chain unavailability gracefully and MUST NOT treat it as a verification failure.
- **CAIP-10 parsing.** The CAIP-10 format is `namespace:chainId:address`. Implementations MUST parse this correctly and MUST validate that the namespace is `eip155` for EVM chains. Malformed CAIP-10 identifiers SHOULD cause the entry to be skipped with a warning.

### Privacy Considerations

Signer addresses in the `integrity` field are publicly visible in the descriptor file. Signer identities are not confidential — they are intentionally disclosed to enable trust decisions. However, participants should be aware that signing a descriptor publicly links their address to the attestation.
