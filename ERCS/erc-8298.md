## Abstract

Every dApp that wants to invoke an AI agent today defines its own task format, and every agent must implement a separate adapter per dApp. This ERC addresses that N×M fragmentation by defining three minimal standard interfaces for on-chain AI agent task dispatch: `AgentTask` (task definition), `IAgentCaller` (dispatch with a `CallAgent` event), and `IAgentHandler` (result and proof callbacks).

A key addition beyond simple request–response is a composable workflow model. Each task carries an `agentWorkflowHash` — a cryptographic commitment to a workflow definition that specifies the execution steps (executor selection, input preparation, multi-agent orchestration, output consensus). The `IAgentHandler` callbacks `onAgentStep` and `onAgentProve` map directly onto these workflow steps, enabling a continuum from the simplest single-agent call to distributed multi-round consensus protocols, without changing the base interface.

## Motivation

The on-chain AI agent ecosystem has made real progress across several layers: agent identity ([ERC-8004](./erc-8004.md)), inference proof verification, proof anchoring, agentic commerce ([ERC-8183](./erc-8183.md)). But there is one foundational primitive still missing — a standard for *how* a smart contract invokes an AI agent and receives its output.

Today, every dApp that wants to call an AI agent defines its own task format, and every agent must implement a separate adapter per dApp:

```
dApp A defines its own task format  →  agent X adapts
dApp B defines a different format   →  agent X adapts again
dApp C defines yet another format   →  agent X adapts again
...
agent Y must do the same for A, B, C  →  N×M integration complexity
```

This is the same fragmentation the inference proof verification layer solved. The task layer has the same problem one level up.

The on-chain AI agent stack maps naturally to six base-layer primitives, analogous to blockchain's foundational properties:

| Primitive   | Blockchain Analogue                    | ERC                        |
|-------------|----------------------------------------|----------------------------|
| Identity    | Address                                | ERC-8004                   |
| **Execution** | Smart Contract (definition + invocation) | **This ERC**             |
| Verify      | Consensus                              | inference proof verification |
| Anchor      | On-chain State                         | proof anchoring            |
| Settlement  | Value Transfer                         | agentic settlement         |
| Prove       | Logs / Audit Trail                     | OCP + WYRIWE               |

There are also two layers built on top of this base:

```
  ┌── Ecosystem Layer ─────────────────────────────────────────┐
  │                                                                      │
  │  ┌───────────────┐   ┌───────────────┐   ┌─────────────────────┐  │
  │  │   ERC-8183    │   │   ERC-8273    │   │       ERC-8257        │  │
  │  │ Labor Market  │   │Authorization  │   │    Skill Registry     │  │
  │  └───────────────┘   └───────────────┘   └─────────────────────┘  │
  │                                                                      │
  └─────────────────────────────────┤ all depend on
  ┌── Base Layer ───────────────▼────────────────────────┐
  │                                                                       │
  │  Identity → Execution → Verify  → Anchor   → Settlement → Prove       │
  │  ERC-8004   [This ERC]  Verifier   Anchoring  Settlement  OCP+WYRIWE  │
  │                                                                       │
  └───────────────────────────────────────────────────────────────────────┘
```

Execution is the last missing brick in the base layer. Without it, every ecosystem ERC is forced to define its own task format rather than composing on a shared primitive — ERC-8183 uses a free-form `description` string, [ERC-8001](./erc-8001.md) uses opaque `executionData` bytes, ERC-8004 delegates entirely to off-chain A2A/MCP.

### AgentTask — Task Definition

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

struct AgentTask {
    bytes32 taskId;             // Unique task identifier; caller-supplied
    bytes32 agentWorkflowHash;  // keccak256 of the workflow definition (CID or inline JSON)
    bytes   agentWorkflow;      // Workflow definition plaintext; OPTIONAL (hash is authoritative)
    address agentHandler;       // Contract implementing IAgentHandler
    uint256 deadline;           // Task expiry as Unix timestamp
}
```

**`agentWorkflowHash`** is the cryptographic commitment to the complete workflow definition for this task. It MUST be computed as `keccak256` of the canonical workflow document (see [Agent Workflow Definition](#agent-workflow-definition)). It is written into the `CallAgent` event and is immutable for the lifetime of the task. All subsequent `onAgentStep` and `onAgentProve` calls MUST be consistent with the workflow committed here.

**`taskId`** is caller-supplied. The caller is responsible for uniqueness within its context.

### IAgentCaller — Task Dispatch

```solidity
interface IAgentCaller {

    /// @notice Emitted when an agent task is dispatched.
    /// @param taskId            Unique task identifier
    /// @param requester         Address that called callAgent()
    /// @param agentWorkflowHash Commitment to the workflow definition
    /// @param agentWorkflow     Workflow plaintext; may be empty
    /// @param inputHash         keccak256 of the caller-supplied input
    /// @param input             Input plaintext; may be empty
    /// @param agentHandler      IAgentHandler contract address
    /// @param deadline          Task expiry timestamp
    event CallAgent(
        bytes32 indexed taskId,
        address indexed requester,
        bytes32         agentWorkflowHash,
        bytes           agentWorkflow,
        bytes32         inputHash,
        bytes           input,
        address         agentHandler,
        uint256         deadline,
        bytes32         genesisStepHash  // root of the stepHash chain for this task; see StepHash Chain
    );

    /// @notice Dispatch an agent task.
    /// @param task             The task definition; task.taskId is caller-supplied
    /// @param inputHash        keccak256 of the input
    /// @param input            Input plaintext; MAY be empty bytes
    /// @return genesisStepHash Root of the stepHash chain for this task; see StepHash Chain
    function callAgent(
        AgentTask calldata task,
        bytes32           inputHash,
        bytes    calldata input
    ) external returns (bytes32 genesisStepHash);
}
```

### IAgentHandler — Workflow Callbacks

```solidity
interface IAgentHandler {

    /// @notice Called by an agent to advance the workflow by one step.
    ///         MAY be called multiple times for the same task; step + isFinal carry
    ///         the execution position and finality.
    ///
    /// @param taskId         Task identifier
    /// @param step           Workflow step index (0-based)
    /// @param prevStepHashes Steps this step builds on. The first step of each execution
    ///                       branch MUST contain [genesisStepHash]. For aggregation steps
    ///                       MAY contain multiple entries. MUST be sorted ascending.
    /// @param agentId        ERC-8004 agent identity; AgentVerifier MUST verify msg.sender matches agentId
    /// @param inputHash      keccak256 of this step's input. For the first step (prevStepHashes=[genesisStepHash])
    ///                       MUST equal CallAgent.inputHash. For subsequent steps MUST equal the outputHash
    ///                       of the preceding step(s) this step builds on.
    /// @param outputHash     keccak256 of this step's output; MAY be zero if no output yet
    /// @param output         This step's output plaintext; MAY be empty
    /// @param isFinal        true = this is the task's final result
    /// @return stepHash      Unique identifier for this step; see StepHash Chain
    function onAgentStep(
        bytes32            taskId,
        uint64             step,
        bytes32[] calldata prevStepHashes,
        uint256            agentId,
        bytes32            inputHash,
        bytes32            outputHash,
        bytes     calldata output,
        bool               isFinal
    ) external returns (bytes32 stepHash);

    /// @notice Called to submit a cryptographic proof for a specific step.
    ///         MAY be called for any step, including intermediate steps.
    ///         MAY be called by a third party, not only by the agent.
    ///         A single proof MAY cover the identified step and its DAG predecessors.
    ///
    /// @param taskId      Task identifier
    /// @param inputHash   MUST match the inputHash of the identified step
    /// @param outputHash  MUST match the outputHash of the identified step
    /// @param proof       Raw proof bytes; encoding is verifier-specific
    /// @param verifier    IAgentVerifiable address (ERC-8274) used to verify this proof
    /// @param stepHash    Identifies the step this proof attests to; MAY cover DAG predecessors
    function onAgentProve(
        bytes32        taskId,
        bytes32        inputHash,
        bytes32        outputHash,
        bytes calldata proof,
        address        verifier,
        bytes32        stepHash
    ) external;
}
```

**Minimum path.** The simplest conforming sequence is `callAgent()` → `onAgentStep(step=0, isFinal=true, prevStepHashes=[genesisStepHash])` → `onAgentProve(stepHash=<final stepHash>)`. Because the final step is always sync, `onAgentProve` for the final `stepHash` is REQUIRED; all intermediate steps and their proofs are OPTIONAL.

### Workflow Design Primitives

The interfaces above are kept minimal by design. Three orthogonal primitives are sufficient to express any on-chain agent coordination pattern representable as a provable DAG of steps — including single-agent post-hoc attestation, pre-action gating, parallel fanout with aggregation, commit-reveal consensus, BFT voting, and sequential multi-agent pipelines — without changing the base interface. For scenario walkthroughs, see [Workflow Scenarios](#workflow-scenarios) in the Rationale; for concrete implementations, see [Reference Implementation](#reference-implementation).

#### Data Structure: DAG Execution Trace via StepHash

Every step produces a `stepHash` that links to its predecessors, forming a tamper-evident execution trace across the workflow.

```
  callAgent()
       │
       ▼
 genesisStepHash
       │
       ├──────────────────────────┐
       ▼                          ▼
  stepHash_A                 stepHash_B        ← fanout: competing agents,
  (selected)                 (abandoned)          prevStepHashes=[genesis]
       │
       ├───────────────────┐
       ▼                   ▼
  stepHash_C           stepHash_D             ← parallel branches,
       │                   │                     prevStepHashes=[A]
       └──────────┬────────┘
                  ▼
            stepHash_E  (isFinal=true)        ← aggregation,
                                                 prevStepHashes=[C, D]
```

The workflow definition (`agentWorkflow`) MAY specify loops — iterative refinement, retry-on-failure, or multi-round consensus. The step DAG, however, is always acyclic: each step represents an actual execution result, and it depends on prior actual execution results — not on nodes defined in the workflow graph. Loops at the workflow level are expressed as new forward steps in the DAG.

A `stepHash` MUST satisfy the following properties:

1. **Uniqueness** — it MUST uniquely identify the step result within the task; two distinct results MUST NOT produce the same `stepHash`.
2. **Determinism** — given the same inputs, the same `stepHash` MUST always be produced.
3. **Predecessor binding** — it MUST commit to `prevStepHashes`, so no step can rewrite its ancestry after the fact.

The exact hash computation is left to the `IAgentHandler` implementation. Implementations MAY include additional fields (e.g. `msg.sender`, a nonce, or a timestamp) to meet their uniqueness requirements. The following formulas are RECOMMENDED as a baseline:

**Genesis stepHash** — produced by `callAgent()`:

```
genesisStepHash = keccak256(abi.encode(
    taskId,
    requester,
    agentWorkflowHash,
    inputHash
))
```

**Step stepHash** — produced by `onAgentStep()`:

```
stepHash = keccak256(abi.encode(
    taskId,
    step,
    agentId,
    inputHash,
    outputHash,
    keccak256(abi.encodePacked(prevStepHashes))
))
```

When `prevStepHashes` contains multiple entries, they MUST be sorted ascending before hashing to ensure determinism regardless of submission order. The array naturally supports linear chains, fanout, and multi-parent aggregation — as illustrated in the diagram above.

#### Execution Mode: Sync/Async Proof Resolution

Each workflow step operates in one of two proof resolution modes, declared in the workflow definition committed to in `agentWorkflowHash`.

```
  [async mode (default)]              [sync mode (gating)]

  stepHash_A                          stepHash_A
       │                                   │
       ▼                                   ▼
  stepHash_B                         onAgentProve(A) ✓ ← must pass
       │                                   │
       ▼                                   ▼
  stepHash_C  (isFinal=true)         stepHash_B
       │                                   │
       ▼                                   ▼
  onAgentProve(A)  (post-hoc)        onAgentProve(B) ✓ ← must pass
       │                                   │
       ▼                                   ▼
  onAgentProve(B)  (post-hoc)        stepHash_C  (isFinal=true)
       │                                   │
       ▼                                   ▼
  onAgentProve(C) ✓                  onAgentProve(C) ✓
```

**Async mode (default).** `onAgentStep` MAY proceed without a preceding `onAgentProve` for the same step. Proofs arrive after execution and provide an auditable record. This is the default for all steps unless the workflow definition declares otherwise.

**Sync mode (gating).** When a step is declared gating in the workflow definition:

1. A handler MUST NOT accept an `onAgentStep` whose `prevStepHashes` includes a gating `stepHash` until `onAgentProve` for that `stepHash` has been submitted and resolved.
2. A non-approving verdict for a gating step MUST cause the handler to reject any subsequent `onAgentStep` that lists that `stepHash` in `prevStepHashes`.
3. `onAgentProve` at a gating step MUST be called with `inputHash` and `outputHash` matching the gating step's values.

**Final step invariant.** The step with `isFinal=true` is always treated as sync. A handler MUST NOT consider a task complete until `onAgentProve` for the final `stepHash` has been submitted and resolved with an approving verdict.

Step transition enforcement SHOULD be implemented inside the `AgentVerifier`. This ERC defines the callback interface; the `AgentVerifier` is where gating rules are checked and step transitions are guarded.

#### Proof Coverage: One Proof, Multiple Steps

Proof coverage and execution mode are two independent dimensions.

A single `onAgentProve` call MAY cover any number of steps. The `stepHash` parameter identifies the primary step being proven; the proof payload MAY additionally attest to any set of DAG predecessor steps in the same submission. The `AgentVerifier` determines which steps a given proof covers and verifies accordingly.

```
  [per-step proves]                    [one batched proof]

  genesisStepHash                      genesisStepHash
       │                                    │
       ▼                                    ▼
  stepHash_A ──► onAgentProve(A)       stepHash_A
       │                                    │
       ▼                                    ▼
  stepHash_B ──► onAgentProve(B)       stepHash_B
       │                                    │
       ▼                                    ▼
  stepHash_C ──► onAgentProve(C)       stepHash_C  (isFinal=true)
  (isFinal=true)                            │
                                            ▼
                                      onAgentProve(stepHash=C)
                                      └─ covers A, B, and C
```

This enables batch verification across an entire execution path — for example, a ZK proof attesting to all steps in a sequential pipeline from genesis to final result, or a WYRIWE attestation covering both the input preparation step and the inference step in one submission.

The two dimensions combine independently, yielding four valid configurations:

| | Per-step prove | Batched prove (one proof, N steps) |
|---|---|---|
| **Async** | each step gets its own post-hoc prove; none block execution | one proof submitted post-hoc after multiple steps complete |
| **Sync (gating)** | each step's prove must pass before the next step proceeds | a single batched proof must pass before the gated step proceeds |

### Agent Workflow Definition

`agentWorkflowHash` is a cryptographic commitment to a workflow definition document. The document SHOULD be stored on a content-addressed system (e.g. IPFS) so it is retrievable and verifiable from the hash alone.

The exact document format is not specified by this ERC and is left as an open question for community input. Regardless of format, the document MUST express:

- **Behavioral instructions** — what the agent is configured to do; semantics are agent-implementation-specific and opaque to this ERC.
- **Model or agent type** — which model or agent class handles the task; kept off-chain to avoid on-chain registry governance overhead.
- **Step definitions** — which steps are active, their execution mode (sync/async), and any parameters required by the `AgentVerifier`.

The workflow state machine — which steps are required, which transitions are valid, and which invariants must hold — is managed by the `AgentVerifier`. This ERC does not define state storage or transition logic.

## Rationale

### Workflow Scenarios

A simple request–response model is sufficient for the most basic agent invocations, but real-world deployments quickly encounter problems a single callback cannot express. The three workflow design primitives (DAG Execution Trace via StepHash, Execution Mode: Sync/Async, Proof Coverage: One Proof Multiple Steps) are sufficient to express any of these patterns without changing the base interface. The following scenarios demonstrate how.

**Scenario 0 — All in One Step**

The simplest conforming workflow: the caller dispatches a task via `callAgent()`, the agent executes and returns the result in a single `onAgentStep(step=0, isFinal=true)`, and a proof is submitted via `onAgentProve`. No multi-step sequencing, no node selection, no consensus round — just one execution, one result, one proof. None of the scenarios below are mandatory; this is the baseline from which complexity is added only as needed.

**Scenario 1 — Input Provenance**

The input a caller commits to on-chain (`inputHash`) and the input a model actually processes may differ due to sanitization transforms, template injection, or RAG context assembly. The stepHash chain makes this gap provable:

- **WYRIWE attestation:** an agent submits a step whose `outputHash` equals the sanitized input hash. All subsequent inference steps list this step's `stepHash` in `prevStepHashes`, proving on-chain that inference happened after the WYRIWE attestation and used the attested input.
- **OCP temporal anchoring:** same pattern — the anchoring commitment is submitted as a step; downstream steps reference its `stepHash`.
- **Pre-action gate:** the input preparation step is declared gating (sync mode); the verifier must approve before any inference step can proceed.

**Scenario 2 — Node Selection**

In open agent networks, multiple agents compete to execute a task. All four common selection mechanisms share the same on-chain pattern: competing agents each submit a step with `prevStepHashes=[genesisStepHash]`, producing distinct `stepHash` values. The downstream orchestrator selects which `stepHash` to reference in the next step's `prevStepHashes`, effectively choosing the winner. Unselected branches are never referenced and go silent — directly analogous to PoW block production.

- **Competitive bidding (price-based):** each agent submits a step whose `output` contains its bid. The orchestrator selects the lowest-bid `stepHash` and advances the workflow from it.
- **VRF-based random selection:** each agent submits a step whose `output` contains a VRF proof. The orchestrator verifies all VRF outputs and selects the winner's `stepHash` deterministically.
- **Reputation-weighted assignment:** agents submit steps whose `agentId` is their ERC-8004 identity. The `AgentVerifier` selects the step from the agent with the highest reputation score.
- **Stake-to-claim with timeout rotation:** the first agent to submit a step claiming the task locks it; if `isFinal=true` is not reached before `deadline`, a new round begins with a fresh `genesisStepHash`.

**Scenario 3 — Multi-Agent Orchestration**

- **Sequential chain (A→B→C):** each step's `outputHash` becomes the next step's `inputHash`, and the next step lists the prior step's `stepHash` in `prevStepHashes`. Pure stepHash chaining.
- **Parallel fanout with aggregation:** parallel agents each submit a step from the same parent `prevStepHashes`. The aggregation step lists all of their `stepHash` values in its own `prevStepHashes[]` array; the `AgentVerifier` validates that all referenced predecessor steps completed before accepting the aggregation step.
- **Conditional routing:** agent A completes; agents B and C each observe A's `stepHash` and independently decide whether to reference it. Whichever one proceeds includes A's `stepHash` in `prevStepHashes`; the other branch produces no step and goes silent. No explicit routing mechanism required.
- **Inter-agent communication:** intermediate state is passed via the `output` field; the recipient's step references the sender's `stepHash` in `prevStepHashes`, creating an auditable on-chain record of each handoff.

**Scenario 4 — Output Consensus**

- **Commit-reveal:** in the commit phase, each node submits a step whose `outputHash = keccak256(result ‖ nonce)`. In the reveal phase, each node submits a second step with `prevStepHashes=[its own commit stepHash]` and `output = result ‖ nonce`. This protects against last-mover advantage while keeping the reveal traceable on-chain.
- **BFT multi-round voting:** each round, multiple nodes submit steps. The next round's steps reference the prior round's winning `stepHash` in `prevStepHashes`, forming a verifiable per-round record analogous to Tendermint pre-vote / pre-commit rounds.
- **Optimistic finalization with challenge window:** the final step is submitted and enters a challenge window; sync mode ensures the task is not marked complete until `onAgentProve` for the final `stepHash` is submitted and approved.

### Input Verification and WYRIWE Integration

WYRIWE addresses the declared-vs-attested input gap by introducing a provable chain of custody:

```
declared inputHash  (CallAgent event, on-chain commitment)
        ↓
sanitization transform  (applied by inference gateway, specified by CID)
        ↓
attested inputHash  (WYRIWE attestation, proven by gateway signature / ZK / TEE)
```

When a workflow includes a WYRIWE step, the agent submits the WYRIWE attestation via `onAgentStep(output=<attestation>, isFinal=false)` before inference begins. Verifiers consuming `onAgentProve` MUST key on the attested `inputHash` from the WYRIWE attestation, not the declared `inputHash` from `CallAgent`.

The `agentWorkflowHash` pre-commits the authorized sanitization pipeline CID. Because this commitment is on-chain and immutable, the gateway cannot substitute an unauthorized transform after seeing the input. The WYRIWE attestation then proves the gateway actually applied the committed pipeline — closing the loop between caller authorization and execution reality.

**No-transform invariant.** When no sanitization transform is applied, the attested `inputHash` MUST equal the declared `inputHash`. This equality is a conformance condition enforced by the verifier, not a derived assumption — an implementation that skips WYRIWE attestation on the grounds that "nothing changed" provides no proof that nothing changed.

### Step State Management

This ERC defines the callback interface for workflow steps but does not specify how step state is tracked — which steps have been completed, which transitions are valid, and what invariants must hold before a step can advance. Different agent implementations may handle this in fundamentally different ways: some may enforce strict sequencing on-chain, others may treat step ordering as advisory and validate only the final result, and others still may delegate state tracking entirely to off-chain infrastructure.

The recommended approach is to manage step state inside the `AgentVerifier` implementation. `IAgentVerifier` is explicitly designed to be stateful — it wraps application-level context about which agents are authorized, which tasks are in progress, and what verification policy applies. This makes it the natural home for workflow state: the `AgentVerifier` can track which `onAgentStep` calls have been received per task, enforce that a WYRIWE attestation step completes before inference proceeds in a pre-action workflow, and validate that consensus rounds reach the required threshold before `isFinal=true` is accepted.

Keeping state management in `AgentVerifier` rather than in this ERC preserves the dispatch interface's neutrality — the same `onAgentStep` and `onAgentProve` callbacks work regardless of verification policy — and allows different verification regimes (optimistic, ZK, TEE, multisig, BFT) to enforce their own step semantics without requiring changes to the base interfaces.

### Pre-action Verification versus Post-hoc Attestation

Two fundamentally different verification shapes exist for agent output, and each has valid use cases.

**Post-hoc attestation** is the simpler path: the agent executes inference, submits a result via `onAgentStep(isFinal=true)`, and a proof is submitted afterward via `onAgentProve`. The verifier attests that the output was correctly produced — but the handler has already received the result before verification completes. This is appropriate for most use cases: the proof provides an auditable record, and any fraud is detectable after the fact.

**Pre-action gating** inverts this: the verifier issues a verdict on the *input* before inference begins, and only a passing verdict allows execution to proceed. This is necessary when the consequences of acting on an unauthorized or malformed input are irreversible — for example, a task that authorizes an on-chain token transfer or triggers an external action. In these cases, post-hoc detection is insufficient because the action has already occurred.

The design tension is real. Pre-action gating requires more contract interactions (a verdict round-trip before inference), separates the verdict from the inference step, and adds latency. It is also a better fit for distributed inference models where multiple nodes independently commit to verdicts before any of them executes — verdict reuse across multiple inference candidates becomes possible. Post-hoc attestation requires fewer interactions, composes more naturally with existing agent frameworks, and keeps the execution path simple.

This ERC does not mandate either shape. Both are expressible through the same interface: post-hoc uses `onAgentStep(isFinal=true)` followed by `onAgentProve`; pre-action uses a gating step (`onAgentStep(isFinal=false)` in sync mode) to submit the verdict commitment before inference, then `onAgentStep(isFinal=true)` for the result. The choice belongs to the workflow definition and the `AgentVerifier` policy, not to the dispatch interface.

## Backwards Compatibility

This ERC introduces new interfaces with no conflicts with existing ERCs. The inference proof verification, agentic settlement, agent identity ([ERC-8004](./erc-8004.md)), and input provenance standards integrate with this ERC as optional dependencies; none require modification.

The minimum conforming path — `callAgent()` → `onAgentStep(step=0, isFinal=true)` — is equivalent in behavior to the simplest existing patterns and requires no additional infrastructure.

## Security Considerations

**`inputHash` integrity depends on handler verification.** The caller computes `inputHash = keccak256(input)` off-chain and passes it alongside the optional `input` plaintext. Because the ERC cannot mandate behavior of off-chain agents, the on-chain `IAgentHandler` implementation SHOULD verify `keccak256(input) == inputHash` on receipt if plaintext is provided. Any mismatch is also detectable by replaying the `CallAgent` event. Callers that include plaintext `input` in the event accept that it is publicly visible in the mempool before the task executes; sensitive inputs SHOULD be withheld from the event and delivered to the agent through an off-chain channel.

**`agentWorkflowHash` pre-commits the execution protocol.** Because the full workflow definition — including the WYRIWE sanitization pipeline CID — is committed at dispatch time, an agent cannot substitute an unauthorized transform after seeing the user input. Any deviation between the committed workflow and the executed steps is detectable by the `AgentVerifier`.

**Callers SHOULD use content-addressed workflow documents.** Storing the workflow document at a mutable URL allows the document to change after `agentWorkflowHash` is committed. Callers SHOULD use IPFS CIDs or equivalent content-addressed references so the document is retrievable and verifiable by anyone.

**Handlers SHOULD guard against task replay.** `taskId` is caller-supplied and is not deduplicated by this ERC. A handler that processes the same `taskId` twice may produce duplicate effects. Handlers SHOULD track which `taskId` values have been processed and reject duplicate `onAgentStep` or `onAgentProve` calls for the same task.

**Deadline expiry is the only built-in revocation mechanism.** This ERC does not define revocation semantics for the model or pipeline referenced by the workflow definition. If the workflow's target model or inference provider becomes unavailable or unsafe after a task is dispatched, the task expires naturally at `deadline`. Callers SHOULD set short deadlines for high-risk tasks.

### Contract Implementation

Minimal Solidity implementations of `IAgentCaller` and `IAgentHandler`:

```solidity
contract AgentCaller is IAgentCaller {
    function callAgent(
        AgentTask calldata task,
        bytes32 inputHash,
        bytes calldata input
    ) external returns (bytes32 genesisStepHash) {
        genesisStepHash = keccak256(abi.encode(
            task.taskId,
            msg.sender,
            task.agentWorkflowHash,
            inputHash
        ));
        emit CallAgent(
            task.taskId,
            msg.sender,
            task.agentWorkflowHash,
            task.agentWorkflow,
            inputHash,
            input,
            task.agentHandler,
            task.deadline,
            genesisStepHash
        );
    }
}
```

```solidity
contract AgentHandler is IAgentHandler {
    mapping(bytes32 => bytes32) public finalStepHash; // taskId => final stepHash
    mapping(bytes32 => bool)    public taskComplete;  // taskId => proven complete

    function onAgentStep(
        bytes32            taskId,
        uint64             step,
        bytes32[] calldata prevStepHashes,
        uint256            agentId,
        bytes32            inputHash,
        bytes32            outputHash,
        bytes     calldata output,
        bool               isFinal
    ) external returns (bytes32 stepHash) {
        stepHash = keccak256(abi.encode(
            taskId, step, agentId, inputHash, outputHash,
            keccak256(abi.encodePacked(prevStepHashes))
        ));
        if (isFinal) {
            finalStepHash[taskId] = stepHash;
        }
    }

    function onAgentProve(
        bytes32        taskId,
        bytes32        inputHash,
        bytes32        outputHash,
        bytes calldata proof,
        address        verifier,
        bytes32        stepHash
    ) external {
        // IAgentVerifier is defined in ERC-8274; this interface is not redefined here
        bool approved = IAgentVerifier(verifier).verify(
            taskId, inputHash, outputHash, proof, stepHash
        );
        if (approved && stepHash == finalStepHash[taskId]) {
            taskComplete[taskId] = true;
        }
    }
}
```

### Workflow Orchestration

The following outlines the call sequence for each scenario.

---

**Scenario 0 — All in One Step**

A single agent executes the task and returns a result in one shot; no workflow complexity, no intermediate steps.

```
callAgent()
     │
     ▼
genesisStepHash
     │
     ▼
onAgentStep(step=0, isFinal=true)
     │
     ▼
onAgentProve(stepHash=step_0) ✓
     │
     ▼
  task complete
```

---

**Scenario 1 — Input Provenance (WYRIWE + pre-action gate)**

Proves on-chain that inference used the attested (sanitized) input, not the raw declared input; the sync gate blocks inference from proceeding until the attestation is verified.

```
callAgent()
     │
     ▼
genesisStepHash
     │
     ▼
onAgentStep(step=0, isFinal=false)        ← WYRIWE attestation
  outputHash = sanitizedInputHash
     │
     ▼
onAgentProve(stepHash=wyriwe_step) ✓      ← sync gate: must pass
     │
     ▼
onAgentStep(step=1, isFinal=true)         ← inference
  prevStepHashes=[wyriwe_stepHash]
  inputHash = sanitizedInputHash
     │
     ▼
onAgentProve(stepHash=inference_step) ✓
     │
     ▼
  task complete
```

---

**Scenario 2 — Node Selection (competing submissions)**

Agents bid for the task and stake a bond simultaneously; after the bid window closes, the winning agent executes and must submit the result within a deadline — failing to do so causes the bond to be slashed and triggers a new bid round.

```
callAgent()
     │
     ▼
genesisStepHash
     ├──────────────────┬──────────────────┐
     ▼                  ▼                  ▼
onAgentStep(A)    onAgentStep(B)    onAgentStep(C)   ← bid + bond staked
stepHash_A        stepHash_B        stepHash_C
(bid=100)         (bid=80)          (bid=120)

                  [bid window closes: B wins]
                       │
                       ▼
               onAgentStep(step=1, isFinal=true)     ← B executes within deadline
               prevStepHashes=[stepHash_B]
                       │
              ┌────────┴────────┐
              ▼                 ▼
     onAgentProve ✓          deadline missed
     bond released           bond slashed → new bid round
     + bid payment received
```

---

**Scenario 3 — Multi-Agent Orchestration**

Sequential chain (A→B→C): each agent's output becomes the next agent's input, with a traceable handoff at every step.

```
callAgent()
     │
     ▼
onAgentStep(A, step=0)
stepHash_A, outputHash_A
     │
     ▼  inputHash = outputHash_A
onAgentStep(B, step=1)
prevStepHashes=[stepHash_A]
stepHash_B, outputHash_B
     │
     ▼  inputHash = outputHash_B
onAgentStep(C, step=2, isFinal=true)
prevStepHashes=[stepHash_B]
     │
     ▼
onAgentProve(stepHash=step_C) ✓
```

Parallel fanout with aggregation: multiple agents work in parallel on the same input; an aggregation step merges their results into a single final output.

```
callAgent()
     │
     ▼
genesisStepHash
     ├──────────────────┐
     ▼                  ▼
onAgentStep(A)    onAgentStep(B)        ← parallel
stepHash_A        stepHash_B
     │                  │
     └────────┬─────────┘
              ▼
onAgentStep(AGG, isFinal=true)
prevStepHashes=[stepHash_A, stepHash_B]
              │
              ▼
onAgentProve(stepHash=AGG) ✓
```

---

**Scenario 4 — Output Consensus**

Commit-reveal: agents first commit to a hidden result, then reveal simultaneously; prevents any agent from copying another's answer after seeing it.

```
callAgent()
     │
     ▼
genesisStepHash
     ├──────────────────┐
     ▼                  ▼
onAgentStep(A_commit)  onAgentStep(B_commit)    ← commit phase
outputHash=H(result_A‖nonce_A)  outputHash=H(result_B‖nonce_B)
stepHash_cA            stepHash_cB
     │                  │
     ▼                  ▼
onAgentStep(A_reveal)  onAgentStep(B_reveal)    ← reveal phase
prevStepHashes=[stepHash_cA]  prevStepHashes=[stepHash_cB]
output=result_A‖nonce_A       output=result_B‖nonce_B
     │                  │
     └────────┬─────────┘
              ▼
onAgentStep(FINALIZE, isFinal=true)             ← select majority
              │
              ▼
onAgentProve(stepHash=FINALIZE) ✓
```
