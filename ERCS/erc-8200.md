## Abstract

[ERC-8200](./eip-8200.md) defines `IERC8200`, a standard on-chain interface for registering and discovering QUIC/HTTP3 agent transport endpoints. Any contract, scheduler, or off-chain client can call `discoverAgents()` to resolve the fastest reachable peer without relying on a centralised directory or proprietary API. Registration optionally includes a recoverable ETH or [ERC-20](./eip-20.md) bond whose fee fraction deters Sybil entries; the exact fee recipient is implementation-defined and MUST be configurable.

## Motivation

Autonomous agents operating across decentralised networks must locate their peers at the transport layer. Today there is no standard on-chain interface for this: each protocol invents a proprietary discovery mechanism, creating fragmentation and lock-in.

HTTP/3 + QUIC eliminates TCP head-of-line blocking and enables 0-RTT connection resumption. eBPF/XDP kernel-bypass further reduces round-trip latency to 8–20 µs end-to-end. By anchoring transport metadata on-chain via a common interface, ERC-8200 enables:

* **Composable endpoint discovery** — any contract or on-chain scheduler can call `discoverAgents(kernelBypassOnly, maxLatencyUs, offset, limit)` to resolve peers at execution time, with no off-chain infrastructure dependency.
* **Trustless peer selection** — latency SLA declarations (`latencyBudgetUs`) are on-chain and auditable; routers can filter by declared capability without trusting a third-party index.
* **Transport-mode diversity** — Standard QUIC, eBPF kernel-bypass, and WebTransport datagram modes coexist in a single registry, discoverable through one interface.
* **0-RTT session optimisation** — the `zeroRTT` flag enables connection-reuse paths for agent workflows.
* **Sybil resistance** — an optional recoverable bond discourages spam registrations; the fee mechanism is deliberately minimal and implementation-defined so the standard is not chain-specific.

**Example:** an on-chain intent-layer contract selects the execution agent for a swap by calling `discoverAgents(true, 50, 0, 5)` (kernel-bypass agents with ≤ 50 µs declared SLA, first page). No DNS lookup, no off-chain API call — the routing decision is fully reproducible on-chain.

This standard addresses a gap not covered by any existing ERC: there is no application-layer interface for on-chain discovery of agent transport endpoints with capability signalling.

## Specification

The key words "MUST", "MUST NOT", "SHOULD", "MAY" in this document are to be interpreted as described in RFC 2119.

### Constants

| Name | Value | Description |
|---|---|---|
| `QUIC_FEE_BPS` | `100` | Default registration fee in basis points (1 %) |

> **Note:** The address that receives the registration fee (`feeRecipient`) is implementation-defined. Implementations MUST make `feeRecipient` configurable (e.g., via constructor parameter or governance-controlled setter) and MUST NOT hardcode a chain-specific address.

### HTTP Header Convention

Compliant agents MUST set the following HTTP/3 response header when communicating via QUIC:

```
Opacus-QUIC: mode=<standard|kernel-bypass|datagram>; zero-rtt=<0|1>; latency-us=<us>
```

Example:
```
Opacus-QUIC: mode=kernel-bypass; zero-rtt=1; latency-us=12
```

### Enums

```solidity
enum TransportMode {
    Standard,      // Regular QUIC / HTTP3 (Quinn, msquic, etc.)
    KernelBypass,  // eBPF + XDP zero-copy, sub-20 µs end-to-end
    Datagram       // WebTransport datagram mode (browser-compatible)
}
```

### Structs

```solidity
struct Endpoint {
    address       agent;
    string        quicEndpoint;      // e.g. "quic://agent.example.com:4433"
    string        grpcEndpoint;      // e.g. "https://agent.example.com:443" (gRPC-H3)
    TransportMode mode;              // Authoritative transport indicator; KernelBypass implies eBPF/XDP capability
    bool          zeroRTT;           // supports QUIC 0-RTT early data
    uint32        latencyBudgetUs;   // declared SLA in microseconds (0 = best-effort)
    uint64        registeredAt;      // UNIX timestamp (block.timestamp) set once on creation; MUST NOT change thereafter
    uint64        updatedAt;         // UNIX timestamp (block.timestamp) of most recent mutation; equals registeredAt on creation; updated on every successful mutation
}
```

### Interface

```solidity
interface IERC8200 {
    function registerEndpoint(
        string  calldata quicEndpoint,
        string  calldata grpcEndpoint,
        TransportMode    mode,
        bool             zeroRTT,
        uint32           latencyBudgetUs,
        address          token,
        uint256          grossAmount
    ) external payable;

    function updateEndpoint(
        string  calldata quicEndpoint,
        string  calldata grpcEndpoint,
        TransportMode    mode,
        bool             zeroRTT,
        uint32           latencyBudgetUs
    ) external;

    function deregisterEndpoint() external;
    function getEndpoint(address agent) external view returns (Endpoint memory);
    function getKernelBypassAgents(uint256 offset, uint256 limit) external view returns (address[] memory agents, uint256 total);
    function discoverAgents(bool kernelBypassOnly, uint32 maxLatencyUs, uint256 offset, uint256 limit) external view returns (Endpoint[] memory endpoints, uint256 total);
}
```

> **Note on kernel-bypass:** `TransportMode.KernelBypass` is the sole authoritative indicator of eBPF/XDP capability. All kernel-bypass filtering MUST use `mode == TransportMode.KernelBypass`. There is no separate `kernelBypass` boolean field.

### Methods

#### `registerEndpoint`

```solidity
function registerEndpoint(
    string  calldata quicEndpoint,
    string  calldata grpcEndpoint,
    TransportMode    mode,
    bool             zeroRTT,
    uint32           latencyBudgetUs,
    address          token,
    uint256          grossAmount
) external payable;
```

The `registerEndpoint` function handles both native ETH and [ERC-20](./eip-20.md) deposits, distinguished by `token`:

- `token == address(0)` **MUST** indicate a native ETH deposit. `msg.value` **MUST** equal `grossAmount`. MUST NOT perform any [ERC-20](./eip-20.md) calls.
- `token != address(0)` **MUST** indicate an [ERC-20](./eip-20.md) deposit. `msg.value` **MUST** equal zero. MUST pull funds via `IERC20(token).transferFrom(msg.sender, address(this), grossAmount)` and MUST revert on failure.

In both cases:

- **MUST** compute `feeAmount = (grossAmount + 99) / 100` (1 % rounded up using integer division) and forward `feeAmount` to `feeRecipient` atomically.
- **MUST** compute `bondAmount = grossAmount - feeAmount` and lock `bondAmount` as a recoverable bond.
- **MUST** revert if `quicEndpoint` is empty.
- **MUST** revert if the caller is already registered.
- **MUST** set `registeredAt = block.timestamp` and `updatedAt = block.timestamp`.
- **MUST** emit `EndpointRegistered(agent, quicEndpoint, mode)`.

#### `updateEndpoint`

```solidity
function updateEndpoint(
    string  calldata quicEndpoint,
    string  calldata grpcEndpoint,
    TransportMode    mode,
    bool             zeroRTT,
    uint32           latencyBudgetUs
) external;
```

MUST only be callable by the registered agent (`msg.sender` matches stored record).  
MUST NOT collect any fee.  
MUST set `updatedAt = block.timestamp`.  
MUST emit `EndpointUpdated(agent, quicEndpoint, mode)`.

#### `deregisterEndpoint`

```solidity
function deregisterEndpoint() external;
```

MUST release the locked `bondAmount` back to `msg.sender`.  
MUST delete the endpoint record.  
MUST emit `EndpointDeregistered(agent)`.  
MUST use a reentrancy guard.

#### `getEndpoint`

```solidity
function getEndpoint(address agent) external view returns (Endpoint memory);
```

MUST return the full `Endpoint` struct for the given agent.

#### `getKernelBypassAgents`

```solidity
function getKernelBypassAgents(uint256 offset, uint256 limit)
    external view returns (address[] memory agents, uint256 total);
```

MUST return agents where `mode == TransportMode.KernelBypass`, paginated by `offset`/`limit`.  
Results MUST be ordered by ascending `registeredAt` (registration order) to ensure consistent pagination.

#### `discoverAgents`

```solidity
function discoverAgents(
    bool   kernelBypassOnly,
    uint32 maxLatencyUs,
    uint256 offset,
    uint256 limit
) external view returns (Endpoint[] memory endpoints, uint256 total);
```

MUST filter by `mode == TransportMode.KernelBypass` if `kernelBypassOnly == true`.  
MUST filter by `latencyBudgetUs <= maxLatencyUs` when `maxLatencyUs > 0`.  
`offset` / `limit` provide pagination.  
Results MUST be ordered by ascending `registeredAt` (registration order) to ensure consistent, non-duplicating pagination across calls.

Compliant contracts MUST implement the `IERC8200` interface defined by this standard and MUST return `true` from `supportsInterface` for `type(IERC8200).interfaceId`.

## Rationale

* **Separate quic + grpc endpoints** — Some agents expose both a raw QUIC socket and a gRPC-over-H3 multiplexed socket; keeping them distinct avoids client-side parsing.
* **latencyBudgetUs as uint32** — 32 bits covers 0–4 294 seconds, sufficient for any real-world SLA while saving storage vs `uint256`.
* **Bond-not-stake** — Fully refundable (minus 1 % fee) to avoid liquidity lockup while still providing Sybil resistance.
* **Single kernel-bypass indicator** — `TransportMode.KernelBypass` is the sole authoritative source, eliminating inconsistent state from a redundant boolean.
* **Header convention** — The `Opacus-QUIC` header (without deprecated `X-` prefix per RFC 6648) enables HTTP middleware and proxies to enforce transport policies without querying the chain on every request.
* **Configurable fee recipient** — Hardcoding a chain-specific address prevents cross-chain deployment; implementations configure it via constructor or governance.
* **Registration-order pagination** — Ordering by `registeredAt` ascending ensures deterministic, stable pagination even as new entries are added.

## Backwards Compatibility

No existing ERC is modified. ERC-8200 is composable with any agent registry standard: implementations MAY store a registered `quicEndpoint` URI alongside agent identity records defined by other standards.

## Security Considerations

* **Reentrancy** — `deregisterEndpoint` releases ETH/[ERC-20](./eip-20.md) and MUST be protected by a reentrancy guard.
* **Endpoint validation** — `quicEndpoint` is stored as an opaque string; consumers MUST validate the URI format off-chain before connecting.
* **Kernel-bypass privilege** — A `mode == KernelBypass` claim is self-reported and cannot be verified on-chain. Consumers SHOULD perform a live latency probe before routing critical tasks to self-declared kernel-bypass agents.
* **Fee recipient access control** — Implementations MUST protect `feeRecipient` mutation with appropriate access control (e.g., owner-only or time-locked governance). The fee recipient is intentionally not specified in this standard to avoid chain-specific addresses that would prevent cross-chain deployment.
* **0-RTT replay** — Agents that set `zeroRTT=true` MUST implement replay protection (e.g., Ticket Age, Anti-Replay Cache) at the transport layer as specified in RFC 9001 §8.
* **Bond token risk** — When `token != address(0)`, implementations SHOULD only accept pre-approved ERC-20 tokens or validate that the token contract behaves as expected to prevent malicious callbacks.
