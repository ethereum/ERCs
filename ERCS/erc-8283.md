## Abstract

This ERC defines an on-chain protocol-agnostic registry that maps structured *context IDs*, derived from [ERC-7730](./eip-7730.md) `context` binding information, to ERC-7730 descriptors and their attestations, represented as their respective URIs and IDs.

Descriptors' URIs may reference any content-addressed scheme (`ipfs://`, `ar://`, `bzz://`, `magnet:`, etc.) and serve as a suggested retrieval mechanism for the wallet.

Each descriptor is backed by an attester-signed attestation artifact stored off-chain, tagged with a namespaced attestation format — typically a standard EAS off-chain attestation, but MAY be any format an attester chooses to sign. The registry itself creates no attestations and depends on no external attestation protocol: it tracks only an attestation ID pointer per context ID and descriptor schema major version, plus a native revocation record, leaving attestation production and verification entirely to attesters and wallets.

## Motivation

[ERC-7730](./eip-7730.md) defines a JSON descriptor format that enriches smart contract calls and [EIP-712](./eip-712.md) messages with human-readable display metadata. Each descriptor carries a `context` section that binds it to specific contracts or EIP-712 domains and serves as a key for the descriptor lookup when displaying the actual signature requests to the users.

The maintenance and distribution of descriptors' registry is an important component of the Clear Signing solution and would greatly benefit from being standardized across wallets without compromising on security or decentralization.

1. **On-chain context-keyed discovery.** Wallets can trustlessly find which descriptor applies to a given contract interaction or signature request without querying any off-chain repository service. This improves the reliability and accessibility of the Clear Signing descriptor discovery process.

2. **Protocol-independent attestation tracking.** Descriptors can be backed by any attestation an attester is willing to sign, not a scheme baked into the registry or any single external protocol. This removes the need for wallet providers to maintain their own services mapping descriptors to attestations, and lets attesters adopt new signature schemes — including post-quantum ones — as they become necessary, without any change to the registry.

3. **Protocol-agnostic descriptor resolution.** Attesters are free to serve the descriptors via IPFS, Swarm, BitTorrent, or any other content-addressed scheme. This removes any bias towards any currently dominating protocol which might become obsolete during the lifetime of the Clear Signing ERC, which is supposed to be indefinite.

4. **Unified revocation and upgrade path.** An on-chain mechanism for an attester to atomically replace their active attestation with either a newer descriptor version or a revocation message. This removes the bottleneck on the speed of issuing a revocation, which is inevitable in any independent-review-based central registry.

## Specification

### Definitions

| Term                 | Definition                                                                                                            |
|----------------------|-----------------------------------------------------------------------------------------------------------------------|
| **descriptor**       | An ERC-7730 JSON file as defined in ERC-7730                                                                          |
| **descriptorHash**   | The [ERC-8176](./eip-8176.md) descriptor hash: `keccak256` of the fully resolved canonicalized descriptor             |
| **contextId**        | A `bytes32` value canonically derived from an ERC-7730 `context` binding, used as the registry lookup key             |
| **contextTypeTag**   | A `bytes32` equal to `keccak256("erc7730.context.<typename>")`, namespacing a context ID derivation rule              |
| **attestationFormatTag** | A `bytes32` equal to `keccak256("erc7730.attestation.<format>")`, namespacing the shape of the off-chain artifact referenced by a `DescriptorInfo`'s attestation MirrorList entry |
| **active attestation** | The single `attestationId` an attester currently has active for a given `(contextId, schemaMajor)` slot, together with the descriptor hash it attests to |
| **schemaMajor**      | The MAJOR version of the ERC-7730 descriptor schema a descriptor's `$schema` key declares, used as an opaque non-zero `uint256` lane in the active-attestation key |
| **attester**         | An `address` on whose behalf at least one attestation has been registered                                             |
| **attestationId**    | The identifier of an attester's active attestation for a given `(contextId, schemaMajor)` slot: a standard EAS off-chain attestation UID when the declared `format` is `ATTESTATION_FORMAT_EAS_OFFCHAIN`; an attester-chosen opaque identifier for any other format |
| **MirrorList**       | A `string[]` of retrieval URIs for a descriptor file or a signed attestation blob, stored once globally and referenced by its content hash |
| **mirrorListId**     | `keccak256(abi.encode(uris))` — the content hash identifying a MirrorList                                             |
| **deployEventTopic** | The factory contract's deploy event signature `topic[0]` in Ethereum logs, e.g. `keccak256("TokenDeployed(address)")` |

Descriptor hashes are computed exactly as defined in [ERC-8176](./eip-8176.md) "Descriptor Hash Computation".

#### Current Tag Constants

```
CONTEXT_TAG_CONTRACT   = keccak256("erc7730.context.contract")
CONTEXT_TAG_EIP712_DEP = keccak256("erc7730.context.eip712.deployment")
CONTEXT_TAG_EIP712_DS  = keccak256("erc7730.context.eip712.domainseparator")
CONTEXT_TAG_FACTORY    = keccak256("erc7730.context.factory")
```

#### Current Attestation Format Tags

```
ATTESTATION_FORMAT_EAS_OFFCHAIN = keccak256("erc7730.attestation.eas.offchain")
```

Attestation format tags follow the same namespacing scheme as context tags, declaring the shape of the artifact a `DescriptorInfo`'s attestation reference points to. New formats — including vendor-specific ones, such as a hardware wallet's proprietary device-friendly encoding — MAY be minted permissionlessly off-chain by any attester with zero registry changes: the registry stores and returns the tag without interpreting it.

### **Context ID** Derivation

The **Context ID** keys are computed as:

```
contextId = keccak256(abi.encode(contextTypeTag, param1, param2, …))
```

The `contextTypeTag` prefix namespaces keys so that different context binding types cannot collide regardless of parameter values.

This scheme is forward-extensible: future additions of new context binding types in ERC-7730 define new tag strings and parameter lists. The registry contract requires no upgrade when a new context tag is introduced, and wallets can derive the new key type using the new context and continue querying the registry.

All context IDs are computed as `keccak256(abi.encode(contextTypeTag, params...))`, where the tag and parameter list depend on the ERC-7730 context binding type. A descriptor covering *N* deployments MUST be registered under *N* context IDs — one per deployment entry.

`abi.encode` is type-sensitive, so the parameter types below are normative: every parameter occupies a full 32-byte word with Solidity's standard padding.

| ERC-7730 context type            | `params` in `abi.encode`                                                |
|----------------------------------|-------------------------------------------------------------------------|
| `context.contract.deployments`   | `uint256 chainId`, `address contract`                                   |
| `context.contract.factory`       | `uint256 chainId`, `address factoryAddress`, `bytes32 deployEventTopic` |
| `context.eip712.deployments`     | `uint256 chainId`, `address verifyingContract`                          |
| `context.eip712.domainSeparator` | `bytes32 domainSeparator`                                               |

### Schema major versioning

ERC-7730 versions its descriptor schema; a descriptor's `$schema` key declares the exact version it is authored against, and a MAJOR version bump signals a breaking schema change that older wallet implementations cannot process.

Active attestations are therefore slotted per `(contextId, schemaMajor)`: each registered descriptor declares the MAJOR version of its `$schema` as `DescriptorInfo.schemaMajor` (a non-zero `uint256`, reverting with `ZeroSchemaMajor` otherwise), and descriptors of different schema MAJORs occupy separate slots that never displace each other. This is what makes a breaking-version migration possible without a coordinated ecosystem flip: an attester registers the new MAJOR's descriptors alongside the old MAJOR's, wallets on old firmware keep resolving the old lane, and the old rendition is retired later with an ordinary revocation listing its context IDs.

The lane is deliberately minimal:

- Only the MAJOR version is a registry concept. MINOR and PATCH releases are backward compatible by definition, so updating to one is an ordinary revoke-and-replace within the same slot, and older wallets keep processing the result.
- The registry never interprets `schemaMajor` — it is an opaque key component, exactly like a context ID. It is attester-declared and unverifiable on-chain (see "Schema major is declared, not verified" in Security Considerations).
- Integrity needs no on-chain version data at all: the `$schema` key is part of the descriptor content covered by `descriptorHash`.

Wallet rules:

- A wallet passes exactly the schema MAJORs it implements to `resolveDescriptors` — it never sees, and never fetches, descriptors it cannot process.
- When the same `(attester, contextId)` resolves under several implemented MAJORs, the wallet SHOULD prefer the highest one.
- A wallet MUST reject a fetched descriptor whose `$schema` MAJOR differs from the lane it was found under, exactly as it rejects a descriptor whose `context` section does not match the transaction.

### Data Model

The registry maintains the following mappings:

```solidity
// The attestation ID currently active for the given attester, context ID and schema
// MAJOR lane. Atomically replaced on each registration call; slots of different
// schema MAJORs never displace each other.
mapping(address attester => mapping(bytes32 contextId => mapping(uint256 schemaMajor => bytes32))) attestationIds;

// Extra metadata for an attestation, stored exactly once per attestation ID.
struct AttestationDetails {
    bytes32 descriptorHash;          // the attested descriptor hash
    uint256 schemaMajor;             // the declared schema MAJOR lane; opaque to the registry
    bytes32 format;                  // the declared attestationFormatTag; opaque to the registry
}
mapping(address attester => mapping(bytes32 attestationId => AttestationDetails)) attestationDetails;

// The timestamp at which 'attester' revoked 'attestationId', or 0 if never revoked.
mapping(address attester => mapping(bytes32 attestationId => uint64)) revokedAt;

// Per-attester pointer to the MirrorList for a given descriptor.
mapping(address attester => mapping(bytes32 descriptorHash => bytes32)) descriptorMirrorListIds;

// Per-attester pointer to the MirrorList for a given attestation ID.
mapping(address attester => mapping(bytes32 attestationId => bytes32)) attestationMirrorListIds;

// Global store of MirrorLists, written once per unique URI set.
// A "mirror list" is an array of URIs for fetching the descriptor file or an
// off-chain attestation blob, supplied by the given attester.
// Key = keccak256(abi.encode(uris)).
mapping(bytes32 mirrorListId => string[]) mirrorLists;

// EIP-712 nonce shared by all relayed calls (registration batches, revocation batches,
// MirrorList updates and profile updates).
mapping(address attester => uint256) nonces;

// Self-declared profile document URI per attester ("business card").
// Display-only metadata, never trust input; empty when unset.
mapping(address attester => string) attesterProfileURIs;
```

The descriptor hash is read from `attestationDetails`; revocation status is read from `revokedAt`. There is no expiration field — see "No expiration mechanism" in Rationale.

`revocations` passed to `createAttestations` is a `RevocationEntry[]` — an attestation ID paired with the context IDs to clear immediately wherever they still point to it — not a bare ID list; see "Registration authorization" below.

The `format` field records the attester-declared `attestationFormatTag` for the artifact behind the attestation ID; the registry stores and returns it without interpreting it — verification of the artifact against its declared format is entirely a wallet/vendor-backend responsibility.

`attestationMirrorListId` reuses the `mirrorLists` content-addressed storage rather than a dedicated mapping — an attestation MirrorList is resolved through the exact same reference-or-inline logic as a descriptor's own MirrorList (see `MirrorListRef` in Interface), it just holds retrieval URIs for signed attestation blobs instead of descriptor JSON files.

MirrorLists are immutable and are stored once globally and can be shared by multiple attesters.

### Interface

The full normative interface is provided in [IClearSigningRegistry.sol](../assets/erc-8283/IClearSigningRegistry.sol).

#### `createAttestations`

The registry's write function. It registers a batch of descriptors backed by attester-signed attestations in a single transaction, revoking and clearing any previously active attestations that are being replaced.

The attester produces the signed attestation artifact locally, stores it wherever it chooses, and hands the registry the pre-computed attestation ID together with an *attestation MirrorList* — the same kind of retrieval-URI list already used for descriptors, pointed at signed attestation blobs instead of descriptor JSON files. Each attestation declares a namespaced `format` tag (see "Current Attestation Format Tags"): typically a standard [EIP-712](./eip-712.md)-signed EAS off-chain attestation (`ATTESTATION_FORMAT_EAS_OFFCHAIN`), but MAY be any other format an attester chooses to sign — see "Attestation format independence" in Rationale.

Both a descriptor's own MirrorList (`DescriptorInfo.descriptorMirrorListURIs`) and the batch's shared attestation MirrorList (`attestationMirrorListURIs`) are given as a `MirrorListRef`, supporting the same two flows:

```solidity
struct MirrorListRef {
    bytes32 id;
    string[] uris;
}
```

- **Reference flow** (`id` set, `uris` empty): `id` MUST reference a MirrorList already published — by any address, in any prior transaction — otherwise the call reverts with `UnknownMirrorList`. Mirror operation may thus be a separate role from attestation: an infrastructure operator maintains the published lists while attesters only ever sign 32-byte list IDs.
- **Inline flow** (`uris` non-empty, `id` zero): the list is published idempotently as part of the registration and its content hash becomes the effective `id`. A first registration therefore needs no separate publication transaction. Declaring a non-zero `id` together with inline `uris` reverts with `RedundantMirrorListId`, as the id is derived from the uris. A `MirrorListRef` carrying neither an `id` nor `uris` reverts with `EmptyMirrorListRef`.

```solidity
function createAttestations(
    address           attester,
    DescriptorInfo[]  calldata descriptors,
    RevocationEntry[] calldata revocations,
    MirrorListRef     calldata attestationMirrorListURIs,
    bytes             calldata signature
) external;
```

Each `DescriptorInfo` entry carries its own attestation reference directly — a non-zero `attestationId` and a non-zero `format` tag (zero values revert with `ZeroAttestationId` and `ZeroAttestationFormat` respectively) — alongside its descriptor identity, its declared `schemaMajor` (see "Schema major versioning") and MirrorList pointer; there is no separate parallel array to keep in sync. For attestations that are standard EAS off-chain attestation objects, `format` SHOULD be `ATTESTATION_FORMAT_EAS_OFFCHAIN`; other values indicate a vendor-specific or otherwise non-EAS artifact shape. Entries within a single batch MAY declare different `format` tags while still sharing one `attestationMirrorListId` — the container/`index.json` convention below already supports per-attestation-ID heterogeneous content, so a batch mixing formats needs no special-casing.

`attestationMirrorListURIs` is resolved exactly once per call (reference or inline, per the flows above) through the same content-addressed MirrorList storage used for descriptor MirrorLists, and every descriptor in the batch shares the resulting `attestationMirrorListId` pointer. A batch of attestations minted together thus shares a single attestation MirrorList: the shared URIs resolve either to a single attestation blob directly or to a shared multi-file container keyed by attestation ID (see "MirrorList entry resolution" below), so a wallet can retrieve one attestation out of a batch without downloading the whole batch. A recurring batch that keeps reusing the same attestation MirrorList only needs to supply its `id` (reference flow) — the `uris` never need to be resupplied once published.

The registry does not validate the attestation's signature or content, regardless of its declared format. Wallets MUST fetch the attestation blob via the attestation MirrorList and verify it per the procedure appropriate to its declared `format` (attestation ID recomputation, schema and signature checks per ERC-8176 for `ATTESTATION_FORMAT_EAS_OFFCHAIN`) before trusting the record. Revocation status is part of every `resolveDescriptors` entry (`revokedAt`) and independently queryable via `getRevocationTimestamp(attester, attestationId)` (see Security Considerations).

Attestation IDs are single-use: an ID that was ever registered — or ever revoked, even without a registration — under the same attester reverts with `AttestationIdAlreadyUsed`. A new attestation, even for the exact same descriptor, MUST carry a fresh attestation ID (see "Attestation IDs are single-use" in Security Considerations). Each registered attestation emits `AttestationRegistered` exactly once, carrying its write-once metadata (descriptor hash, schema MAJOR, format tag), alongside one `AttestationUpdated` per affected `(contextId, schemaMajor)` slot.

`revocations` MAY be empty when no active attestations are replaced. When the attester already has an active attestation for any of the supplied `(contextId, schemaMajor)` slots, its attestation ID MUST already be recorded as revoked (`getRevocationTimestamp` non-zero) at the moment the slot's pointer is written — either via a `RevocationEntry` in this same `revocations` array (processed before any descriptor) or via an earlier call — otherwise the call reverts with `MissingRevocation`. Because the check runs at each pointer write, a batch that claims the same slot twice also reverts: the second write would displace the first, unrevoked attestation. Each entry in `revocations` is recorded as revoked under the attester's own address, and its `contextIds` are cleared immediately wherever they still point to that attestation ID within the attestation's own schema MAJOR lane (read from the attestation's stored metadata) — a `RevocationEntry`'s `contextIds` MAY be empty (the context IDs covered by the new descriptors in this same batch get updated automatically regardless; an entry's `contextIds` matter for context IDs the replacement no longer covers).

Wallets MUST verify the fetched attestation per ERC-8176 "Verifying an Attestation" — schema, descriptor hash match, and revocation status — before trusting an active attestation record.

##### Registration authorization

Unless `msg.sender == attester`, the call MUST carry a `signature`: an [EIP-712](./eip-712.md) signature by the attester over the structure

```
ClearSigningRegistrationBatch(
    DescriptorInfo[] descriptors,
    bytes32 attestationMirrorListId,
    RevocationEntry[] revocations,
    uint256 nonce
)
DescriptorInfo(
    bytes32 descriptorHash,
    uint256 schemaMajor,
    bytes32[] contextIds,
    bytes32 descriptorMirrorListId,
    bytes32 attestationId,
    bytes32 format
)
RevocationEntry(
    bytes32 attestationId,
    bytes32[] contextIds
)
```

The signature is verified via ECDSA recovery for EOA attesters and via [ERC-1271](./eip-1271.md) `isValidSignature` for contract attesters — the registry has no external dependency that would restrict this, so multisig and governance-contract attesters work identically to EOAs here.

The registry maintains a single per-attester nonce, consumed by every relayed call (registration batches, revocation batches and MirrorList updates alike) and exposed via `getNonce`, making each signature single-use. An attester can cancel an outstanding signature by calling `invalidateNonce()`, which consumes the current nonce without executing anything (see `invalidateNonce` below).

`attestationMirrorListId` is the *resolved* id of `attestationMirrorListURIs` — either the reference `id` supplied directly, or `keccak256(abi.encode(uris))` when supplied inline — so the signer computes the same value the registry independently derives when resolving the list, regardless of which flow is used. It is bound into the signed struct because without it, a relayer could substitute a different attestation MirrorList than the one the attester intended, silently breaking retrieval for the batch. `revocations` is bound into the same struct for the same reason: without it, a relayer could add or drop revocation entries the attester never signed off on.

#### `revokeAttestation(bytes32 attestationId, bytes32[] contextIds)`

Revokes `attestationId` under `msg.sender`, and clears `contextIds` immediately wherever they still point to it within the attestation's own schema MAJOR lane (read from its stored metadata) — combining revocation and cleanup into a single transaction. Self-service: an attester can revoke an attestation directly, independent of any registration batch. `contextIds` MAY be empty for a revoke-only call that skips clearing — matching a batch `RevocationEntry`, whose `contextIds` MAY likewise be empty.

A context ID whose active attestation has since moved on to a different attestation ID (raced by an intervening registration) is silently skipped rather than reverting the whole call.

Emits `AttestationRevoked(msg.sender, attestationId, block.timestamp)`, and `AttestationUpdated` for each context ID actually cleared.

#### `revokeAttestations(address attester, RevocationEntry[] revocations, bytes signature)`

The batch and relayed counterpart of `revokeAttestation`: revokes every entry's attestation ID under `attester` and clears its listed context IDs, with the same per-entry semantics as above. `revocations` MUST be non-empty, reverting with `EmptyRevocations` otherwise.

Unless `msg.sender == attester`, `signature` MUST be a valid EIP-712 signature by the attester over

```
ClearSigningRevocationBatch(
    RevocationEntry[] revocations,
    uint256 nonce
)
RevocationEntry(
    bytes32 attestationId,
    bytes32[] contextIds
)
```

consuming the attester's nonce exactly like a relayed registration batch.

Emergency revocation therefore never requires the attester to hold ETH or to attach a descriptor registration: a single signature handed to any relayer revokes a batch of attestations and clears their context IDs.

#### `invalidateNonce()`

Consumes the caller's current EIP-712 nonce without executing anything, cancelling any outstanding signature over that nonce. Emits `NonceInvalidated(msg.sender, newNonce)`. Since self-submitted calls do not consume nonces, this is the mechanism for an attester to withdraw a signed-but-not-yet-relayed batch they no longer want executed.

#### `getRevocationTimestamp(address attester, bytes32 attestationId) returns (uint64)`

Returns the timestamp at which `attester` revoked `attestationId`, or 0 if never revoked. Populated by a direct `revokeAttestation` call, a `revokeAttestations` batch, or a registration batch that displaced an active attestation backed by it — all are recorded identically, under the real attester's address, and all are equally authoritative (see "Attestation revocation semantics" in Security Considerations).

The timestamp is write-once: revoking an already-revoked attestation ID keeps the original timestamp and emits no additional `AttestationRevoked` event.

#### `resolveDescriptors()`

The primary wallet-facing query entry point: resolves all active attestations for the given attesters across the given context IDs and schema MAJOR lanes in a single call. A wallet passes exactly the schema MAJORs it implements in `schemaMajors` (see "Schema major versioning").

For every resolved attestation record the returned `ResolvedDescriptor` structure carries all the necessary data: the attester, the context ID, the descriptor hash, the schema MAJOR lane it was found under, the backing attestation ID, its revocation timestamp (`revokedAt`, 0 if never revoked), the declared attestation `format`, the attestation MirrorList contents (`attestationMirrorListUris`), and the descriptor MirrorList contents.
Results are ordered by `attesters` first, then `contextIds`, then `schemaMajors`; slots with no active attestation produce no entry.

The descriptor hash and `format` are read from registry storage, and `attestationMirrorListUris` carries the MirrorList for retrieving the signed attestation blob. A non-zero `revokedAt` marks a context ID that an incomplete cleanup left pointing at a revoked attestation; wallets MUST treat such an entry as revoked (see "Attestation revocation semantics" in Security Considerations).

##### URI prefix filtering

Both `resolveDescriptors` and `getMirrorListById` take an `allowedPrefixes` parameter: a list of raw string prefixes, e.g. `["ipfs:", "https:"]`.
A URI is returned only if it starts with at least one of the prefixes; an empty array means "no filter, return everything".
The prefixes are plain string prefixes, not a scheme enum — `"ipfs:"` filters by scheme, but a caller can equally pass `"data:application/json;base64,"` to ask only for base64 inline JSON, or a scheme plus content-hash prefix to match a specific document. The contract has no notion of "scheme"; it only performs prefix comparison.

Filtering happens on-chain, inside the view function, before a URI is ever copied into the return array.
A rejected entry is never fully loaded and never reaches the `eth_call` response, so the cost of discarding it is bounded by the length of the prefixes being matched, not by the size of the data being filtered out.
This is what makes it safe to mix large inline `data:` entries into a MirrorList (see "MirrorList URI schemes" in Rationale): callers that do not ask for `data:` never pay to receive them.

An active attestation is included in `resolved` based on whether it exists, not on whether any of its URIs survive the filter, so a caller still sees `descriptorHash` and `attestationId` for it even if none of its mirrors matched the requested prefixes.
The same `allowedPrefixes` list is applied to both the descriptor MirrorList and the attestation MirrorList of each active attestation, as a wallet's fetch capabilities do not differ between the two.

A wallet derives its candidate context IDs locally — the contract key for a calldata transaction, the deployment and domain-separator keys for an EIP-712 message, and factory keys for factory contracts the wallet supports — and obtains the attested descriptor hashes, their attestation validity, and the retrieval URIs for every schema MAJOR it implements in a single `eth_call`.
The wallet remains responsible for applying its trust policy across attesters and for verifying the fetched descriptor bytes against `descriptorHash` and the descriptor's `context` section against the transaction.

#### `publishMirrorLists(string[][] uriLists) returns (bytes32[] mirrorListIds)`

Stores a batch of retrieval URI lists on-chain and returns their content hashes `keccak256(abi.encode(uris))`.
A first publication emits `MirrorListPublished` carrying the full URI list (see "Events and state reconstruction").
A MirrorList whose content has already been published is not re-written and emits no event, and identical URI sets occupy storage exactly once regardless of how many descriptors or attesters reference them.

Any address may publish MirrorLists, and publication may be done independently of attestation: a pinning service, mirror operator, or any other party can maintain the published lists that attesters may reference by ID.
Registrants that prefer a single transaction can instead supply the URIs inline in `createAttestations` (see above).

#### `getMirrorListById(bytes32 mirrorListId, string[] allowedPrefixes)`

Returns the URI list for a given MirrorList ID, filtered by `allowedPrefixes` (see "URI prefix filtering" above; an empty array returns the full list). Returns an empty array if the ID is unknown.

This getter is the point-lookup counterpart to following `MirrorListPublished` events.
An attester's current MirrorList for a descriptor is available through `resolveDescriptors` and through the MirrorList update events.

#### `updateDescriptorMirrorList() & updateAttestationMirrorList()`

Allows an attester to update the MirrorList for existing descriptors or attestations without requiring a new attestation or registration batch.

```solidity
function updateDescriptorMirrorList(
    address attester,
    bytes32[] calldata descriptorHashes,
    MirrorListRef calldata descriptorMirrorListRef,
    bytes calldata signature
) external;

function updateAttestationMirrorList(
    address attester,
    bytes32[] calldata attestationIds,
    MirrorListRef calldata attestationMirrorListRef,
    bytes calldata signature
) external;
```

`mirrorListRef` supports both `MirrorListRef` flows (see "Reference flow" / "Inline flow" above): rotating to an already-published list needs only its `id`, while rotating to a brand-new one publishes it inline and rotates to it atomically in the same call — the two-transaction dance of `publishMirrorLists` followed by `updateDescriptorMirrorList` is no longer required for a MirrorList an attester hasn't used before.

Every key MUST reference a descriptor hash (respectively attestation ID) the attester has registered before, reverting with `UnknownDescriptor` (respectively `UnknownAttestationId`) otherwise — the update functions rotate existing pointers and never create orphaned ones.

The EIP-712 `signature` (if `msg.sender != attester`) covers:

```
DescriptorMirrorListUpdate(
    bytes32[] descriptorHashes,
    bytes32 descriptorMirrorListId,
    uint256 nonce
)

AttestationMirrorListUpdate(
    bytes32[] attestationIds,
    bytes32 attestationMirrorListId,
    uint256 nonce
)
```

`descriptorMirrorListId` (or `attestationMirrorListId`) here is the *resolved* id — identical whether `mirrorListRef` took the reference or inline flow — so the signed struct is unaffected by which flow the caller used.

#### `setAttesterProfileURI(address attester, string profileURI, bytes signature)` & `getAttesterProfileURI(address attester)`

An attester MAY publish a profile — a "business card" — as a single URI pointing at a self-hosted JSON document. The registry stores only the pointer: the contract stays schema-free, and profile edits never require re-attestation. An empty `profileURI` clears the profile; setting a value identical to the current one changes nothing and emits nothing. Updates emit `AttesterProfileUpdated` and are relayable like every other attester action — unless `msg.sender == attester`, `signature` MUST be a valid EIP-712 signature by the attester over

```
AttesterProfileUpdate(
    string profileURI,
    uint256 nonce
)
```

consuming the attester's nonce.

The profile document is a JSON object. Version 1, defined by this ERC, has three REQUIRED members; a consumer MUST treat a document missing any of them as invalid and render nothing from it:

- `version` — the integer `1`. Consumers MUST NOT attribute meaning to a document declaring a version they do not implement.
- `attesters` — an array of `0x`-prefixed hexadecimal attester addresses this profile covers, compared case-insensitively. Before attributing a profile to an attester, a consumer MUST verify that the attester's address appears in this array: without this back-reference, any address could point its profile URI at someone else's legitimate document and borrow their identity. One document MAY serve several attester addresses run by the same operator.
- `name` — a human-readable display name. Self-declared — see "Attester profiles are self-declared display data" in Security Considerations before rendering it.

All other members are OPTIONAL, and consumers MUST ignore members they do not recognize. RECOMMENDED members:

- `url` — the attester's website.
- `registryRepository` — the profile's registry-domain payload: the attester's canonical full-descriptor collection (a git repository, an IPFS folder, an ENS-named location), letting the on-chain registry point back at the off-chain distribution channel it indexes.
- `securityContact` — a URI for security disclosures.
- `keyRotationPolicy` — a URI describing the attester's key rotation procedures.

```json
{
  "version": 1,
  "attesters": ["0x00000000000000000000000000000000DeaDBeef"],
  "name": "Example Attester Inc.",
  "url": "https://attester.example.com",
  "registryRepository": "https://github.com/example/erc7730-descriptors",
  "securityContact": "mailto:security@attester.example.com",
  "keyRotationPolicy": "https://attester.example.com/erc7730-key-rotation"
}
```

The profile is deliberately mutable and content-unpinned — a business card's freshness matters more than immutability — and `AttesterProfileUpdated` events preserve the full change history for auditors.

**Verification (bidirectional domain binding).** The on-chain pointer is authenticated in one direction only: the attester's key vouches for the URI. The REQUIRED `attesters` array provides the reverse direction: the document names the addresses it belongs to. When the profile is served over `https:`, a passing `attesters` check is a mutual attestation between the address and the serving domain — the same pattern as `security.txt` — and a consumer MAY present such a profile as *verified*. For content-addressed transports (`ipfs:` and the like) there is no domain to bind, so the `attesters` check still prevents identity borrowing but the profile remains unverified self-declaration. Either way, profile data is never trust input — see "Attester profiles are self-declared display data" in Security Considerations.

For the identity and branding layer (display name, avatar, social handles), attesters SHOULD additionally set a primary ENS name with standard text records — ENS names are unique and disputable in a way self-declared strings are not, and wallets and explorers already render them. The profile URI and ENS compose: ENS answers "who is this address", the profile's `registryRepository` answers "where else do this attester's descriptors live", and neither is required for the other.

### MirrorLists

A MirrorList is a `string[]` of retrieval URIs for a descriptor file or a signed attestation blob.
URIs may use any content-addressed scheme: `ipfs://`, `ar://`, `bzz://`, `magnet:`, etc.
The list is identified by `keccak256(abi.encode(uris))` and stored once globally in the registry.

MirrorLists are published via `publishMirrorLists` or inline during registration, and referenced by ID; each attester's per-descriptor state is a single 32-byte pointer to the content-addressed list.
When multiple descriptors or attesters use the same retrieval endpoints, the URI strings exist exactly once in contract storage regardless of how many attestations reference them.

Retrieval via the published URIs is never the source of trust.
Wallets MUST recompute the retrieved descriptor's `descriptorHash` per ERC-8176 and verify that it equals the registered `descriptorHash` before using the descriptor, and MUST verify a retrieved attestation blob against its registered attestation ID.

#### MirrorList entry resolution

Each MirrorList entry — descriptor or attestation alike — resolves to one of two things:

1. **The target content directly**: an ERC-7730 descriptor JSON file, or a signed attestation blob. This is the default case.
2. **A shared multi-file container whose entry point is an `index.json`** manifest mapping content keys — `descriptorHash` for descriptors, attestation ID for attestation blobs — to the specific member within that container.

There is no universal "folder" concept across `ipfs:`, `git:`, `https:`, `ftp:`, `magnet:`, `bzz:` and other schemes — an IPFS UnixFS directory, a git tree, an HTTP directory listing, a member of a multi-file torrent, and a Swarm manifest are all different addressing mechanisms.
The registry does not try to unify them: it only requires that the *published URI itself* — whatever it takes to construct it for that scheme — resolve to fetchable bytes that are either the target content or an `index.json` manifest.

A wallet's resolution algorithm is therefore uniform regardless of transport:

1. Fetch the MirrorList URI.
2. If the result validates as the expected content — the descriptor hash matches `descriptorHash`, or the blob is a validly signed attestation for the expected attestation ID — use it directly.
3. Otherwise, if the result parses as an `index.json` manifest, look up the expected key (`descriptorHash` or attestation ID) and fetch the member it points to.

This is what makes shared containers viable for batches: an attester bulk-publishing signed attestations for many attestation IDs (or descriptors for many hashes) can point them all at one container URI keyed by an `index.json`, so a single deduplicated MirrorList entry serves arbitrarily many content items.

### Events and state reconstruction

Every write to wallet-facing registry state is accompanied by an event carrying the written values, so a consumer following the registry's event log from contract deployment can reconstruct the complete wallet-facing state without a single `eth_call`. Implementations MUST preserve this property. Events fire only when state actually changes: idempotent republication of a MirrorList and no-op MirrorList pointer updates emit nothing.

| State                                          | Reconstructed from             |
|------------------------------------------------|--------------------------------|
| Active attestations (`attestationIds`)         | `AttestationUpdated`           |
| Attestation metadata (`attestationDetails`)    | `AttestationRegistered`        |
| Revocation timestamps (`revokedAt`)            | `AttestationRevoked`           |
| Global MirrorList store (`mirrorLists`)        | `MirrorListPublished`          |
| Descriptor MirrorList pointers                 | `DescriptorMirrorListUpdated`  |
| Attestation MirrorList pointers                | `AttestationMirrorListUpdated` |
| Attester profile URIs (`attesterProfileURIs`)  | `AttesterProfileUpdated`       |

The per-attester nonce is deliberately excluded: it is replay-protection bookkeeping rather than wallet-facing content, it cannot be derived from logs (relayed and direct calls are indistinguishable there), and it is queryable via `getNonce`.

## Rationale

### Self-contained revocation

Depending on an external protocol for attestation lifecycle management ties the registry's own trust assumptions to that protocol's design choices. Concretely: EAS's delegated attestation flow verifies signatures via plain ECDSA recovery only, with no path to another scheme without an EAS protocol upgrade, and EAS never stores the original signature past attestation creation — so a wallet cannot independently re-verify an EAS-backed on-chain attestation without trusting an RPC provider or implementing a full consensus light client. Neither property is under this registry's control, and neither is acceptable for a registry meant to remain useful indefinitely.

Revocation is therefore tracked natively: `revokeAttestation`, `revokeAttestations` and `getRevocationTimestamp` are the registry's own bookkeeping, with no dependency on any external attestation protocol. Revocation and cleanup are always a single atomic step, whether triggered directly (`revokeAttestation(attestationId, contextIds)`), as a standalone signed batch (`revokeAttestations`), or as part of a registration batch's `RevocationEntry[]` displacing an active attestation on the attester's behalf — each path's own authorization chain (`msg.sender` or the batch `signature`) proves the revocation is legitimate, so all paths are equally authoritative (see "Attestation revocation semantics" in Security Considerations).

This also means the registry places no constraint on how an attestation itself is produced or verified: content authenticity is entirely a matter between an attester and the wallets that trust them, per the declared `format` tag (see "Attestation format independence" below). The registry only tracks which attestation ID is currently active for a context ID, and whether it has been revoked.

### No expiration mechanism

An earlier design included an `expirationTime` on each attestation, cleared automatically once past. That requires whoever evaluates it to know the current time — which is a real problem for the wallets this registry is designed around. A hardware wallet with no RTC and no network connection has no trustworthy way to learn "what time is it now": it would need either an external trusted time source, or to read `block.timestamp` through the same untrusted RPC channel this design otherwise goes out of its way to avoid depending on (see "Self-contained revocation" above, and the light-client discussion this registry's design is built around). Unlike revocation — a discrete fact, checkable once via a single query and stable until explicitly changed — expiration requires re-deriving "is this still valid right now" on every single use, with no way to cache or pre-verify it.

The registry therefore supports only explicit, discrete revocation. An attestation is either active or revoked; there is no time-based decay. An attester who wants to time-box an attestation's validity is free to encode that inside the signed artifact itself and have wallets that *do* have a trusted clock enforce it client-side — the registry just never has an opinion on it.

### Version-agnostic registry with one opaque version lane

The registry deliberately stores no schema version data beyond the `schemaMajor` key component. Version integrity needs nothing on-chain — the `$schema` key is part of the descriptor content covered by `descriptorHash` — and MINOR/PATCH releases are backward compatible, so they are invisible to the registry by design.

The `schemaMajor` lane exists for exactly one reason: breaking MAJOR releases need a migration window in which two renditions of the same descriptor are simultaneously discoverable under the same context, because wallet firmware — hardware wallets especially — adopts new schema versions over years, not weeks. A single-slot registry would force a destructive swap and a coordinated ecosystem flip; the extra key dimension lets both renditions live side by side at near-zero marginal cost, since a breaking schema release produces new descriptor files, new hashes and new attestations regardless.

Two alternatives were rejected. Encoding the major into versioned context type tags (`erc7730.context.contract.v3`) needs no contract changes but hashes two orthogonal concerns into one opaque identifier, forks the derivation table per major, and leaves the version invisible in storage, queries and events. Keeping versioning entirely in the retrieval layer via an attested per-release manifest (major to descriptor hash) preserves the contract but breaks single-call resolution — the wallet learns which descriptor hash it needs only after fetching the manifest, forcing a second `eth_call` for the mirror list — and either freezes retrieval URIs into the attested manifest (breaking free mirror rotation) or leaves them unreachable in the first query. An explicit, typed, queryable dimension costs one struct field and one query parameter, and keeps every other mechanism untouched.

### Content-addressed MirrorLists

With tens of thousands of descriptors and dozens of attesters, sharing identical retrieval endpoints may save a lot of on-chain storage.

A per-attester state is reduced to a single 32-byte pointer.

Any attester can attest to an existing descriptor by providing only their signatures and the `mirrorListId` — no URI data needs to be re-transmitted or stored.

Publishing MirrorLists is decoupled from attestation, so running mirrors can be a separate role: infrastructure operators maintain the published URI lists while attesters only ever sign 32-byte list IDs.

Once a MirrorList is published, it cannot be modified.

Changing an attester's active MirrorList for a descriptor requires calling `updateDescriptorMirrorList` with a new `MirrorListRef` and a valid EIP-712 signature from the attester.

This decouples the storage endpoint pointer from the descriptor attestation, allowing attesters to rotate or update broken infrastructure links (e.g., changing IPFS gateways) instantly for thousands of descriptors without re-attesting the exact same descriptor hashes. The on-chain verifiable record is maintained via the EIP-712 signature.

### MirrorList URI schemes

The RECOMMENDED baseline schemes for MirrorList entries are `ipfs:` and `https:`; `http:`, `git:`, `ftp:`, `ar:` (Arweave) and `data:` are OPTIONAL.

Because URI matching happens on-chain through the `allowedPrefixes` parameter, no "wallets MUST skip unrecognized schemes" client-side rule is needed: an unrecognized scheme simply is not among the prefixes a wallet sends, so it is filtered out before it ever leaves the contract.

A MirrorList entry MAY be an [RFC 2397](https://www.rfc-editor.org/rfc/rfc2397) `data:` URI carrying the full ERC-7730 descriptor JSON inline — e.g. `data:application/json,<percent-encoded content>` or `data:application/json;base64,<base64 content>` — making the descriptor fully available on-chain with no external retrieval at all.
This works through the existing inline publication flow with no write-path changes, at a storage cost of roughly 20,000 gas per 32 bytes of new data: a several-KB descriptor runs roughly 3–4 million gas (about 0.03 ETH at 10 gwei on mainnet, negligible on an L2).
It is intended selectively for top-priority protocols rather than as the default entry type, and the on-chain prefix filtering is precisely what makes mixing a few large `data:` entries into an otherwise normal MirrorList safe: callers that do not ask for `data:` never pay to receive it.

No new content-verification logic is needed for any scheme: wallets already MUST hash fetched descriptor content and compare it against the attested `descriptorHash` regardless of transport, which covers `data:` URIs the same as `https:` or `ipfs:` ones.

### Attestation format independence

The registry never inspects an attestation's shape or signature — only its declared `format` tag, attestation ID, and the MirrorList pointing at the actual signed artifact. This keeps the registry's own attack surface and gas cost minimal (no on-chain signature verification of attestations at all) while placing no constraint on what attesters can sign.

The default and RECOMMENDED format is a standard EAS off-chain attestation (`ATTESTATION_FORMAT_EAS_OFFCHAIN`) — preserving the EAS ecosystem's [EIP-712](./eip-712.md) format, UID scheme, ERC-8176 schema and tooling, without any EAS write call or its roughly 200,000 gas cost per attestation.

The format tag also accommodates artifacts that are not EAS-shaped at all. A resource-constrained device — for example a hardware wallet that cannot parse JSON on-device — may instead sign its own device-native binary encoding of a descriptor and register the resulting attestation ID with a vendor-specific `format` tag (see "Current Attestation Format Tags"). Supporting such a format costs nothing beyond minting a new tag off-chain; the vendor's own tooling and device firmware are solely responsible for producing and verifying it.

There's a second reason format independence matters: any single hardcoded signature scheme is eventually a liability. ECDSA won't necessarily be a safe assumption forever; a quantum computer breaking secp256k1 is the obvious case. A registry meant to stay useful indefinitely shouldn't be stuck with whatever scheme it originally launched with. Because the registry never looks inside a `DescriptorInfo`'s `format`, an attester can switch to a post-quantum signature scheme on their own, tag it accordingly, and wallets just verify it per that tag — no registry upgrade required.

The registry deliberately performs no verification of the attestation signature or content: wallets MUST verify the fetched blob anyway, so a registry-side check would add gas cost without adding trust.

### Curated, vendor-specific consumption

The registry's value does not require every wallet to consume it directly, nor does it require universal on-chain or on-device coverage of the long tail of contracts. A hardware wallet vendor with device or bandwidth constraints MAY instead operate its own backend that periodically queries `resolveDescriptors` for a small, curated subset of high-volume protocols, converts the resulting ERC-7730 descriptors into its own device-friendly encoding (see "Attestation format independence" for the accompanying `format` tag), and ships only that subset to its devices.

Such a vendor still benefits from the registry's context-keyed discovery, attester attestations, and revocation for the protocols it chooses to curate, without needing on-device JSON parsing or an incentive scheme covering descriptors it does not.

The event-sourcing guarantee (see "Events and state reconstruction") is what makes such wrapping a first-class consumption model rather than an afterthought: a vendor backend, an indexer, or a static snapshot generator maintains a complete, verifiable copy of the registry by following logs alone — no archive node and no per-item `eth_call` backfill — and anyone can deterministically regenerate the same mirror from the same chain.

### Attester profile as a single URI pointer

The registry stores no profile content on-chain — only the pointer. A registry whose purpose is preventing misleading human-readable displays should not itself hold self-declared display strings: a free-text name field becomes impersonation bait the moment any consumer renders it, and an on-chain profile schema cannot evolve once the contract is deployed. Keeping the content in a document behind a single URI leaves the contract schema-free, and a self-hosted document supports the bidirectional domain binding check — a verification mechanism a bare on-chain string cannot have.

The profile does not reuse the MirrorList machinery either: a profile is one mutable document per attester, not deduplicated content-addressed data shared across attesters, and pinning it by content hash would make every profile edit an on-chain ceremony for data whose freshness matters more than its immutability.

Identity and branding are ENS's job, not this registry's — see the ENS recommendation in the `setAttesterProfileURI` section. The profile URI exists for the registry-domain pointers (canonical descriptor repository, security contact, key rotation policy) and for attesters who do not want an ENS dependency.

### Registration gas costs

The cost of populating the registry is dominated by MirrorList storage and per-context bookkeeping, not by attestation creation — the registry never writes an attestation itself.

A descriptor with two context IDs costs roughly 90,000 gas once its descriptor MirrorList is already published: a single `bytes32` attestation ID write per context ID, plus a once-per-attestation-ID metadata record (descriptor hash, schema major, attestation MirrorList pointer, and format tag) instead of any per-context URI string storage.

`createAttestations` scales as `O(N·M + R)` in the size of a batch — `N` descriptors, `M` context IDs per descriptor, `R` revocation entries — dominated end to end by the storage writes above rather than by any array search: checking that a displaced attestation was revoked is a single storage read per `(contextId, schemaMajor)` slot (`getRevocationTimestamp`-equivalent), independent of `R`, and processing `revocations` itself is a single pass. No sorting or search step is needed on either array.

Four properties keep this cost manageable and predictable:

1. Registrations and MirrorList publications are batched: a release is at most one transaction (inline MirrorLists) and one registration signature regardless of descriptor count, amortizing per-transaction overhead.

2. Subsequent attesters attesting to an already-registered descriptor skip the MirrorList storage entirely (publication is idempotent), reducing their cost further.

3. The attestation MirrorList is content-addressed and deduplicated across every context ID and descriptor that shares it; a batch reusing an already-published attestation MirrorList supplies only its `id` (reference flow, see `MirrorListRef`) and pays no publication cost at all.

4. Updating the MirrorList for existing descriptors uses the `updateDescriptorMirrorList` function, bypassing re-registration entirely, making infrastructure rotations virtually free.

These figures are order-of-magnitude estimates only: actual costs depend on URI lengths, the number of deployments per descriptor, and prevailing gas prices.

### Mainnet-only deployment

The registry is deployed on Ethereum mainnet only.

Context IDs encode `chainId`, so a single registry can serve wallets operating on any EVM chain.

The ERC-7730 descriptors format is designed to cover all deployments on all EVM-compatible chains.

## Backwards Compatibility

This ERC introduces a new contract interface and makes no modifications to ERC-7730 or ERC-8176.

ERC-7730 descriptor files require no changes. The context section fields used for context ID derivation are already present in all compliant descriptors.

## Security Considerations

### Malicious registration

Any address can call `createAttestations` with their own attestation.

Wallets only query active attestations for their configured `trustedAttesters`.

An unknown attacker's address will never appear in a trusted list, so their attestations are invisible to wallets.

Registration is permissionless precisely because it is harmless without a trusted attester's attestation.

### Attester key compromise

If a trusted attester's key is compromised, an attacker can create a malicious attestation and call `createAttestations`.

Attesters SHOULD use secure setup multisig accounts or governance contracts rather than EOAs — the registry has no external dependency that would prevent this; contract accounts work identically to EOAs everywhere in this contract (registration signatures verify via ERC-1271, and both `revokeAttestation` and direct registration calls are plain `msg.sender`-authenticated).

Wallet vendors SHOULD publish their trusted attester addresses and key rotation procedures. Attesters SHOULD publish their own security contact and key rotation policy via their attester profile (see `setAttesterProfileURI`).

### Attestation IDs are single-use

The registry enforces that an `attestationId` is never reused across registrations by the same attester: `createAttestations` reverts with `AttestationIdAlreadyUsed` if the ID was ever registered — or ever revoked, including a revocation of an ID that was never registered — under that attester. This keeps both the attestation metadata (`descriptorHash`, `format`) and the revocation timestamp write-once: once wallets have seen an attestation ID, its meaning can never silently change, and a revoked ID can never come back to life. A new attestation — even for the exact same descriptor — MUST carry a fresh attestation ID.

An `attestationId` MUST NOT be zero (`bytes32(0)`), as zero represents the absence of an attestation in the registry state.

### Atomicity of active-attestation replacement and revocation

When `createAttestations` replaces an existing active attestation, the registry requires that every displaced attestation ID has already been recorded as revoked — via a `RevocationEntry` in the same call's `revocations` array (processed before any descriptor) or via any earlier call. The check is a single `getRevocationTimestamp`-equivalent storage read performed at the moment each `(contextId, schemaMajor)` slot pointer is written, not a search of `revocations`, so its cost is independent of how many entries `revocations` contains — and, because it runs at write time, it also catches displacement by a duplicate slot within the same batch. This prevents a permissionless relay from silently updating an active attestation while leaving it unrevoked. `revocations` is itself bound into `signature`, so a relayer cannot add or drop entries the attester didn't sign off on either.

### Schema major is declared, not verified

`DescriptorInfo.schemaMajor` is attester-declared metadata, exactly like `contextIds` and `format`: the registry cannot verify it against the off-chain descriptor's actual `$schema` key. An attester could — maliciously or mistakenly — register a descriptor under the wrong lane. Wallets MUST reject a fetched descriptor whose `$schema` MAJOR differs from the lane it was resolved under, exactly as they reject a descriptor whose `context` section does not match the transaction, so a mis-declared lane results in the descriptor being discarded rather than misinterpreted.

### Attestation revocation semantics

Revocation is recorded natively by the registry, always under the real attester's address — whether via a direct `revokeAttestation` call (`msg.sender` is definitionally the attester), via a signed `revokeAttestations` batch, or via a registration batch that displaces an active attestation on the attester's behalf (each batch's own `signature`/`MissingRevocation` checks already establish that the attester authorized it). There is no informational or unauthenticated revocation path: every entry recorded via `getRevocationTimestamp` is either the attester's own direct call or backed by their verified signature.

Revocation and cleanup happen atomically in every path: `revokeAttestation(attestationId, contextIds)` clears exactly the context IDs it's given, and a batch's `RevocationEntry.contextIds` clears any extra ones the replacement descriptors don't already cover. If a caller's `contextIds` doesn't name every context ID an attestation ID actually backs (an incomplete enumeration), the omitted context ID stays stale — still pointing at the now-revoked attestation ID — until a later registration replaces it or another revocation call names it explicitly. This is a cosmetic staleness, not a safety issue: every `resolveDescriptors` entry carries its `revokedAt` timestamp, and wallets MUST reject entries with a non-zero `revokedAt` regardless of whether the corresponding context ID has been cleared.

### Attestation format is not verified content

The `format` tag on a `DescriptorInfo`'s attestation reference is attester-declared metadata, exactly like `contextTypeTag` on a context ID: the registry stores and returns it without interpreting, validating, or verifying it against the artifact it names. Declaring `format = ATTESTATION_FORMAT_EAS_OFFCHAIN` does not cause the registry to check that the fetched blob is actually a well-formed EAS off-chain attestation, and declaring any other tag does not exempt an attester from producing a genuinely verifiable artifact.

Wallets and vendor backends MUST apply the verification procedure appropriate to the declared `format` before trusting the fetched blob — attestation ID recomputation, signature and schema checks for `ATTESTATION_FORMAT_EAS_OFFCHAIN` per ERC-8176, or the vendor's own equivalent procedure for any other declared format. An unrecognized `format` value SHOULD be treated as unverifiable and rejected, exactly as a wallet should reject any attestation blob it cannot validate.

### Attester profiles are self-declared display data

An attester profile is unverified self-declaration: anyone can register any address and point its profile at a document claiming any name. Profile data MUST NOT be used as trust input — wallets select attesters exclusively by their configured trusted address lists, never by profile contents.

Two checks bound the damage a fabricated profile can do. The REQUIRED `attesters` back-reference (see `setAttesterProfileURI`) MUST be verified before attributing a profile to an address, closing the identity-borrowing attack where an attacker points their profile URI at a reputable operator's genuine document. Beyond that, consumers SHOULD render profile data only for attesters they already trust, or present it as *verified* only when the bidirectional domain binding holds (an `https:`-served document whose `attesters` array names the address). A profile failing these checks SHOULD be displayed as unverified or not at all — a self-declared "Ledger" string shown next to an attestation is precisely the class of misleading display this registry exists to prevent.

### Incorrect context ID bindings

The registry cannot verify `contextIds` against the off-chain JSON descriptor, so an attester could — maliciously or mistakenly — associate a descriptor for contract A under the context ID for contract B.
Wallet-side validation of the descriptor's `context` section against the actual transaction (a MUST, see above) ensures such a mis-binding results in the descriptor being rejected rather than misleading formatting being shown.

Because `signature` commits to the exact descriptors, context IDs, MirrorList pointer, and revocation entries for the whole batch in one signed structure, there is no partial signature a relayer could observe and re-bind to different context IDs or retrieval endpoints — unlike a design where the attestation signature and the registry-specific binding are separate artifacts.
