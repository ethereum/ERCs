## Abstract

This ERC defines an on-chain protocol-agnostic registry that maps structured *context IDs*, derived from [ERC-7730](./eip-7730.md) `context` binding information, to ERC-7730 descriptors and their attestations represented as their respective URIs and UIDs.

Descriptors' URIs may reference any content-addressed scheme (`ipfs://`, `ar://`, `bzz://`, `magnet:`, etc.) and serve as a suggested retrieval mechanism for the wallet.

Attestations backing on-chain registrations are Ethereum Attestation Service records per [ERC-8176](./eip-8176.md); off-chain registrations declare a namespaced attestation format tag describing the shape of the artifact retrieved via the MirrorList, which is typically a standard EAS off-chain attestation but MAY be any format an attester chooses to sign.
The registry additionally provides context-aware on-chain discoverability that EAS does not offer.

## Motivation

[ERC-7730](./eip-7730.md) defines a JSON descriptor format that enriches smart contract calls and [EIP-712](./eip-712.md) messages with human-readable display metadata. Each descriptor carries a `context` section that binds it to specific contracts or EIP-712 domains and serves as a key for the descriptor lookup when displaying the actual signature requests to the users.

The maintenance and distribution of descriptors' registry is an important component of the Clear Signing solution and would greatly benefit frombeing standardized across wallets without compromising on security or decentralization.

1. **On-chain context-keyed discovery.** Wallets can trustlessly find which descriptor applies to a given contract interaction or signature request without querying any off-chain repository service. This improves the riliability and accessability of the Clear Signing descriptor discovery process.

2. **Unified on-chain attestation surface.** [ERC-8176](./eip-8176.md) defines EAS-based attestation records, but EAS contracts have no suitable attestation lookup mechanisms. This removes the need for wallet providers to maintain their own services mapping descriptors to their attestations, and makes any entity maintaining singular trusted registries of clear signing descriptors completely redundant.

3. **Protocol-agnostic descriptor resolution.** Attesters are free to serve the descriptors via IPFS, Swarm, BitTorrent, or any other content-addressed scheme. This removes any bias towards any currently dominating protocol which might become obsolete during the lifetime of the Clear Signing ERC, which is supposed to be indefinate.

4. **Unified revocation and upgrade path.** An on-chain mechanism for an attester to atomically replace their active attestation with either a newer descriptor version or a revocation message. This removes the bottleneck on the speed of issuing a revocation, which is inevitable in any independent-review-based central registry.

## Specification

### Definitions

| Term                 | Definition                                                                                                            |
|----------------------|-----------------------------------------------------------------------------------------------------------------------|
| **descriptor**       | An ERC-7730 JSON file as defined in ERC-7730                                                                          |
| **descriptorHash**   | The ERC-8176 descriptor hash: `keccak256` of the fully resolved canonicalized descriptor                              |
| **contextId**        | A `bytes32` value canonically derived from an ERC-7730 `context` binding, used as the registry lookup key             |
| **contextTypeTag**   | A `bytes32` equal to `keccak256("erc7730.context.<typename>")`, namespacing a context ID derivation rule              |
| **attestationFormatTag** | A `bytes32` equal to `keccak256("erc7730.attestation.<format>")`, namespacing the shape of the off-chain artifact referenced by an `OffchainAttestation`'s MirrorList entry |
| **active slot**      | The single `attestationId` an attester currently has active for a given `contextId`; the attested descriptor hash is carried in the attestation's data |
| **attester**         | An `address` whose ERC-8176 EAS attestation has been submitted to the registry                                        |
| **attestationId**    | The identifier of an attester's active attestation for a given slot: an EAS UID for on-chain attestations and for off-chain attestations declaring the EAS off-chain format; an attester-chosen opaque identifier for off-chain attestations declaring any other format |
| **MirrorList**       | A `string[]` of retrieval URIs for a descriptor file or a signed off-chain attestation blob, stored once globally and referenced by its content hash |
| **mirrorListId**     | `keccak256(abi.encode(uris))` — the content hash identifying a MirrorList                                             |
| **deployEventTopic** | The factory contract's deploy event signature `topic[0]` in Ethereum logs, e.g. `keccak256("TokenDeployed(address)")` |

Descriptor hashes are computed exactly as defined in [ERC-8176](./eip-8176.md) "Descriptor Hash Computation".

#### Current Tag Constants

```
CONTEXT_TAG_CONTRACT   = keccak256("erc7730.context.contract")
CONTEXT_TAG_EIP712_DEP = keccak256("erc7730.context.eip712.deployment")
CONTEXT_TAG_EIP712_DS  = keccak256("erc7730.context.eip712.domainseparator")
CONTEXT_TAG_FACTORY    = keccak256("erc7730.context.factory")
```

#### Current Attestation Format Tags

```
ATTESTATION_FORMAT_EAS_ONCHAIN  = keccak256("erc7730.attestation.eas.onchain")
ATTESTATION_FORMAT_EAS_OFFCHAIN = keccak256("erc7730.attestation.eas.offchain")
```

Attestation format tags follow the same namespacing scheme as context tags, declaring the shape of the artifact an `OffchainAttestation` entry points to (or, for on-chain attestations, a fixed constant reported for symmetry — on-chain slots are always EAS-backed and carry no format ambiguity). New formats — including vendor-specific ones, such as a hardware wallet's proprietary device-friendly encoding — MAY be minted permissionlessly off-chain by any attester with zero registry changes: the registry stores and returns the tag without interpreting it.

### **Context ID** Derivation

The **Context ID** keys are computed as:

```
contextId = keccak256(abi.encode(contextTypeTag, param1, param2, …))
```

The `contextTypeTag` prefix namespaces keys so that different context binding types cannot collide regardless of parameter values.

This scheme is forward-extensible: future additions of new context binding types in ERC-7730 define new tag strings and parameter lists. The registry contract requires no upgrade when a new context tag is introduced, and wallets can derive the new key type using the new context and contiue querying the registry.

All context IDs are computed as `keccak256(abi.encode(contextTypeTag, params...))`, where the tag and parameter list depend on the ERC-7730 context binding type. A descriptor covering *N* deployments MUST be registered under *N* context IDs — one per deployment entry.

| ERC-7730 context type            | `params` in `abi.encode`                        |
|----------------------------------|-------------------------------------------------|
| `context.contract.deployments`   | `chainId`, `contract`                           |
| `context.contract.factory`       | `chainId`, `factoryAddress`, `deployEventTopic` |
| `context.eip712.deployments`     | `chainId`, `verifyingContract`                  |
| `context.eip712.domainSeparator` | `domainSeparator`                               |

### Data Model

The registry maintains the following mappings:

```solidity
// The ERC-8176 EAS UID of the active attestation for the given attester and context ID.
// Atomically replaced on each registration call.
// Holds either an on-chain attestation UID or a pre-computed off-chain attestation UID.
mapping(address attester => mapping(bytes32 contextId => bytes32)) attestationUIDs;

// Extra metadata for off-chain attestations, stored exactly once per UID.
// Never written on the on-chain path, keeping on-chain slot updates to a single UID write.
struct OffchainDetails {
    bytes32 descriptorHash;          // the attested descriptor hash
    bytes32 attestationMirrorListId; // points into 'mirrorLists', like a descriptor's mirrorListId
    uint64  expirationTime;          // from the attestation message; 0 = no expiration
    bytes32 format;                  // the declared attestationFormatTag; opaque to the registry
}
mapping(address attester => mapping(bytes32 uid => OffchainDetails)) offchainDetails;

// Per-attester pointer to the MirrorList for a given descriptor.
mapping(address attester => mapping(bytes32 descriptorHash => bytes32)) mirrorListId;

// Global store of MirrorLists, written once per unique URI set.
// A "mirror list" is an array of URIs for fetching the descriptor file or an
// off-chain attestation blob, supplied by the given attester.
// Key = keccak256(abi.encode(uris)).
mapping(bytes32 mirrorListId => string[]) mirrorLists;

// EIP-712 nonce for relayed off-chain registration batches.
mapping(address attester => uint256) registrationNonce;
```

For on-chain attestations the attested descriptor hash, expiration time and revocation time are carried in the EAS attestation and read from the EAS contract at query time; they are not duplicated in registry storage.
For off-chain attestations no EAS record exists, so the descriptor hash and expiration time are read from `offchainDetails` instead.

The `format` field records the attester-declared `attestationFormatTag` for the artifact behind the UID; the registry stores and returns it without interpreting it — verification of the artifact against its declared format is entirely a wallet/vendor-backend responsibility.

`attestationMirrorListId` reuses the `mirrorLists` content-addressed storage rather than a dedicated mapping — an attestation MirrorList is published and deduplicated exactly like a descriptor's `mirrorListId`, it just holds retrieval URIs for signed attestation blobs instead of descriptor JSON files.

MirrorLists are immutable and are stored once globally and can be shared by multiple attesters.

### Interface

The full normative interface is provided in [IClearSigningRegistry.sol](../assets/erc-8283/IClearSigningRegistry.sol).

Both write functions below take the displaced attestations to revoke as a single `RevocationBatch revocations` argument:

```solidity
struct RevocationBatch {
    MultiDelegatedRevocationRequest[] onchain;
    bytes32[] offchain;
}
```

`onchain` and `offchain` MAY independently be empty when no active slot of that kind is being replaced; see "Registration authorization" below for what each MUST contain when a slot is displaced.

#### `createDescriptorAttestations`

The primary write function. It registers a batch of descriptors in a single transaction by first revoking any previously active attestations that are being replaced — `eas.multiRevokeByDelegation(revocations.onchain)` for displaced on-chain attestations and `eas.revokeOffchain` for each entry of `revocations.offchain` — then calls `eas.multiAttestByDelegation(attestations)` to create all attestations on-chain.

Each descriptor references a MirrorList, and two flows are supported for providing its content:

- **Reference flow** (`mirrorListId` set, `mirrorListUris` empty): `mirrorListId` MUST reference a MirrorList already published — by any address, in any prior transaction — otherwise the call reverts with `UnknownMirrorList`. Mirror operation may thus be a separate role from attestation: an infrastructure operator maintains the published lists while attesters only ever sign 32-byte list IDs.
- **Inline flow** (`mirrorListUris` non-empty, `mirrorListId` zero): the list is published idempotently as part of the registration and its content hash becomes the effective MirrorList ID. A first registration therefore needs no separate publication transaction. Declaring a non-zero `mirrorListId` together with inline URIs reverts with `RedundantMirrorListId`, as the ID is derived from the URIs.

In both flows the relayed [EIP-712](./eip-712.md) registration signature covers the *effective* MirrorList ID.
Any address may permissionlessly call this function, and the attester identity is taken from `attestations[0].attester`.

`attestations[0]` MUST use the ERC-8176 schema UID and MUST contain exactly one `data` entry per descriptor: `attestations[0].data[i]` is the active attestation for `descriptors[i]`.

Each entry's `data` MUST be exactly 32 bytes long and ABI-decode to `bytes32` equal to the corresponding `descriptorHash`, and each entry MUST be revocable: the registry requires revocation of a displaced attestation, so an irrevocable active attestation would permanently freeze the affected slots.

All entries in `attestations[1..N]` are supplementary and are passed through to EAS without registry-level validation.

The registry stores only the attestation UID per `(attester, contextId)` slot.
The attested descriptor hash is carried in the attestation's data, verified at registration time as described above, and is read from the EAS contract when queried.

`revocations.onchain` and `revocations.offchain` MAY be empty when no active slots are replaced. When the attester already has an active attestation for any of the supplied context IDs, the corresponding attestation UID MUST appear in the matching revocation set — `revocations.onchain` if the displaced slot holds an on-chain attestation, `revocations.offchain` if it holds an off-chain one; otherwise the call reverts with `MissingRevocation`.
This rule is identical on both write functions, making migration between the two attestation paths symmetric: a context ID registered off-chain can be re-registered with an on-chain attestation and vice versa.

Wallets MUST verify the returned attestation per ERC-8176 "Verifying an Attestation" — schema, descriptor hash match, expiration, and revocation status — before trusting an active attestation record.

##### Registration authorization

The EAS delegated attestation signatures cover only the EAS request fields; they do not cover the context IDs or MirrorLists, which are chosen by the transaction sender.
Unless `msg.sender == attestations[0].attester`, the call MUST therefore carry a `registrationSignature`: an [EIP-712](./eip-712.md) signature by the attester over the structure

```
ClearSigningRegistrationBatch(
    DescriptorInfo[] descriptors,
    bytes32 attestationSignaturesHash
)
DescriptorInfo(
    bytes32 descriptorHash,
    bytes32[] contextIds,
    bytes32 mirrorListId
)
```

The signature is verified via ECDSA recovery for EOA attesters and via [ERC-1271](./eip-1271.md) `isValidSignature` for contract attesters.
When `msg.sender == attestations[0].attester`, `registrationSignature` SHOULD be empty.

A single registration signature authorizes the entire batch, so an attester performs one extra signing operation per registration transaction regardless of how many descriptors it covers.
Binding the batch to the EAS delegated attestation signatures makes the registration signature single-use without any additional nonce state: EAS consumes the attester's nonces when the attestations are submitted, so none of the signatures can be replayed.
This prevents a third party from taking an attester's EAS attestation signatures and registering the descriptors under attacker-chosen context IDs or retrieval endpoints.

#### `createOffchainDescriptorAttestations`

The write function for the off-chain attestation path.

The attester produces the signed attestation artifact locally, stores it wherever it chooses, and hands the registry the pre-computed attestation UID together with an *attestation MirrorList* — the same kind of retrieval-URI list already used for descriptors, pointed at signed attestation blobs instead of descriptor JSON files. Each attestation declares a namespaced `format` tag (see "Current Attestation Format Tags"): typically a standard [EIP-712](./eip-712.md)-signed EAS off-chain attestation (`ATTESTATION_FORMAT_EAS_OFFCHAIN`), preserving the EAS format, UID scheme, ERC-8176 schema and tooling, but MAY be any other format an attester chooses to sign — see "Off-chain attestation path" in Rationale.
No EAS write call is made for the attestations, so the per-descriptor attestation write cost is eliminated regardless of declared format.

```solidity
function createOffchainDescriptorAttestations(
    address                attester,
    DescriptorInfo[]       calldata descriptors,
    OffchainAttestation[]  calldata attestations,
    string[]               calldata attestationMirrorListUris,
    bytes                  calldata registrationSignature,
    RevocationBatch        calldata revocations
) external returns (bytes32[] memory attestationIds);
```

`attestations` MUST contain exactly one entry per descriptor, carrying the pre-computed off-chain attestation UID, the expiration time from the attestation message, and a non-zero `format` tag; a zero `format` reverts with `ZeroAttestationFormat`. For attestations that are standard EAS off-chain attestation objects, `format` SHOULD be `ATTESTATION_FORMAT_EAS_OFFCHAIN`; other values indicate a vendor-specific or otherwise non-EAS artifact shape. Entries within a single batch MAY declare different `format` tags while still sharing one `attestationMirrorListId` — the container/`index.json` convention below already supports per-UID heterogeneous content, so a batch mixing formats needs no special-casing.

`attestationMirrorListUris` is published exactly once per call through the same content-addressed MirrorList storage used for descriptor MirrorLists, and every descriptor in the batch shares the resulting `attestationMirrorListId` pointer.
A batch of attestations minted together thus shares a single attestation MirrorList: the shared URIs resolve either to a single attestation blob directly or to a shared multi-file container keyed by attestation UID (see "MirrorList entry resolution" below), so a wallet can retrieve one attestation out of a batch without downloading the whole batch.
A later call reusing identical URIs is idempotent and re-writes nothing, exactly like any MirrorList publication.
Descriptor MirrorLists follow the same reference/inline flows as `createDescriptorAttestations`.

The registry does not validate the off-chain attestation's signature or content, regardless of its declared format.
Wallets MUST fetch the attestation blob via the attestation MirrorList and verify it per the procedure appropriate to its declared `format` — for `ATTESTATION_FORMAT_EAS_OFFCHAIN`, UID recomputation, schema, attester signature, descriptor hash match, and expiration per ERC-8176 — before trusting the record, which covers attestation correctness without any registry-side check.
Revocation status of an off-chain attestation is queried via `eas.getRevokeOffchain(attester, uid)` (see Security Considerations).

Replacement of active slots mirrors the on-chain path: every active attestation displaced by the batch MUST be explicitly revoked, and the call reverts with `MissingRevocation` otherwise.
The check is routed by the kind of slot being displaced — a displaced on-chain attestation UID MUST appear in `revocations.onchain` (an EAS delegated revocation batch, executed via `eas.multiRevokeByDelegation`), while a displaced off-chain attestation UID MUST appear in `revocations.offchain`, which the registry records via `eas.revokeOffchain` (roughly 20,000 gas each).

##### Registration authorization (off-chain path)

The off-chain path makes no EAS write call, so no EAS nonce is consumed and the on-chain path's replay protection does not apply.
The registry instead maintains its own `registrationNonce` per attester, consumed on each relayed call and exposed via `getRegistrationNonce`.

Unless `msg.sender == attester`, the call MUST carry a `registrationSignature`: an [EIP-712](./eip-712.md) signature by the attester over the structure

```
ClearSigningOffchainRegistrationBatch(
    DescriptorInfo[] descriptors,
    OffchainAttestation[] attestations,
    bytes32 attestationMirrorListId,
    uint256 nonce
)
DescriptorInfo(
    bytes32 descriptorHash,
    bytes32[] contextIds,
    bytes32 mirrorListId
)
OffchainAttestation(
    bytes32 uid,
    uint64 expirationTime,
    bytes32 format
)
```

`attestationMirrorListId` is `keccak256(abi.encode(attestationMirrorListUris))` — the signer computes it locally, exactly the value the registry independently derives when publishing the list.
It is bound into the signed struct for the same reason `mirrorListId` is bound on the on-chain path: without it, a relayer could substitute a different attestation MirrorList than the one the attester intended, silently breaking retrieval for the batch.
Signing the full `OffchainAttestation` structs likewise prevents a relayer from submitting falsified `expirationTime` values, which would otherwise subvert expiration clearing.

#### `resolveDescriptors()`

The primary wallet-facing query entry point: resolves all active attestations for the given attesters across the given context IDs in a single call.

For every resolved attestation record the returned `ResolvedDescriptor` structure carries all the necessary data: the attester, the context ID, the descriptor hash, the backing attestation UID, the attestation's `expirationTime` and `revocationTime`, the declared attestation `format`, the attestation MirrorList contents (`attestationMirrorListUris`), and the descriptor MirrorList contents.
Results are ordered by `attesters` first, then `contextIds`; empty slots produce no entry.

For slots backed by an on-chain attestation, `expirationTime` and `revocationTime` are read from EAS at query time, `attestationMirrorListUris` is empty, and `format` is always the fixed `ATTESTATION_FORMAT_EAS_ONCHAIN` constant, reported for symmetry since on-chain attestations carry no format ambiguity.
For slots backed by an off-chain attestation, the descriptor hash, `expirationTime` and `format` are read from registry storage, `attestationMirrorListUris` carries the MirrorList for retrieving the signed attestation blob, and `revocationTime` is always 0 — off-chain revocation status requires a separate `eas.getRevokeOffchain(attester, uid)` call.

##### URI prefix filtering

Both `resolveDescriptors` and `getMirrorListById` take an `allowedPrefixes` parameter: a list of raw string prefixes, e.g. `["ipfs:", "https:"]`.
A URI is returned only if it starts with at least one of the prefixes; an empty array means "no filter, return everything".
The prefixes are plain string prefixes, not a scheme enum — `"ipfs:"` filters by scheme, but a caller can equally pass `"data:application/json;base64,"` to ask only for base64 inline JSON, or a scheme plus content-hash prefix to match a specific document. The contract has no notion of "scheme"; it only performs prefix comparison.

Filtering happens on-chain, inside the view function, before a URI is ever copied into the return array.
A rejected entry is never fully loaded and never reaches the `eth_call` response, so the cost of discarding it is bounded by the length of the prefixes being matched, not by the size of the data being filtered out.
This is what makes it safe to mix large inline `data:` entries into a MirrorList (see "MirrorList URI schemes" in Rationale): callers that do not ask for `data:` never pay to receive them.

A slot is included in `resolved` based on whether it exists, not on whether any of its URIs survive the filter, so a caller still sees `descriptorHash` and `attestationId` for a slot even if none of its mirrors matched the requested prefixes.
The same `allowedPrefixes` list is applied to both the descriptor MirrorList and the attestation MirrorList of each slot, as a wallet's fetch capabilities do not differ between the two.

A wallet derives its candidate context IDs locally — the contract key for a calldata transaction, the deployment and domain-separator keys for an EIP-712 message, and factory keys for factory contracts the wallet supports — and obtains the attested descriptor hashes, their attestation validity, and the retrieval URIs in a single `eth_call`.
The wallet remains responsible for applying its trust policy across attesters and for verifying the fetched descriptor bytes against `descriptorHash` and the descriptor's `context` section against the transaction.

#### `clearRevokedAttestations()`

Clears active slots whose backing EAS attestation has been revoked or has expired.

The function is permissionless: anyone may clean up stale slots, allowing a watchdog service to sweep multiple attesters in a single transaction.
`contextIds[i]` lists the context IDs to clear for `attesters[i]`; the two arrays MUST have the same length, and neither `attesters` nor any per-attester context ID list may be empty.

A slot is cleared when it is non-empty and its backing attestation is revoked (`revocationTime != 0`) or expired (`expirationTime != 0` and in the past).
For slots backed by an off-chain attestation, revocation is checked via `eas.getRevokeOffchain(attester, uid)` and expiration against the `expirationTime` recorded in registry storage at registration time.
All other slots are skipped rather than causing a revert, so a large sweep cannot be blocked by a single slot changing state between the sweep's preparation and its inclusion in a block.
The function returns the number of attestation records actually cleared, allowing a watchdog to determine via simulation whether a sweep is worth submitting.

For each cleared slot, an `AttestationUpdated` event is emitted with `bytes32(0)` as the new descriptor hash and attestation ID.

#### `publishMirrorLists(string[][] uriLists) returns (bytes32[] mirrorListIds)`

Stores a batch of retrieval URI lists on-chain and returns their content hashes `keccak256(abi.encode(uris))`.
A MirrorList whose content has already been published is not re-written and emits no event, and identical URI sets occupy storage exactly once regardless of how many descriptors or attesters reference them.

Any address may publish MirrorLists, and publication may be done independently of attestation: a pinning service, mirror operator, or any other party can maintain the published lists that attesters may reference by ID.
Registrants that prefer a single transaction can instead supply the URIs inline in `createDescriptorAttestations` (see above).

#### `getMirrorListById(bytes32 mirrorListId, string[] allowedPrefixes)`

Returns the URI list for a given MirrorList ID, filtered by `allowedPrefixes` (see "URI prefix filtering" above; an empty array returns the full list). Returns an empty array if the ID is unknown.

Because the `MirrorListPublished` event carries only the ID, this getter is the standalone way to read a published list's contents.
An attester's current MirrorList for a descriptor is available through `resolveDescriptors` and through `MirrorListUpdated` events.

#### `updateMirrorList()`

Allows an attester to update the MirrorList for existing descriptors without requiring a new EAS attestation.

```solidity
function updateMirrorList(
    address attester,
    bytes32[] calldata descriptorHashes,
    bytes32 mirrorListId,
    bytes calldata signature
) external;
```

The EIP-712 `signature` (if `msg.sender != attester`) covers:

```
MirrorListUpdate(
    bytes32[] descriptorHashes,
    bytes32 mirrorListId
)
```

### MirrorLists

A MirrorList is a `string[]` of retrieval URIs for a descriptor file or a signed off-chain attestation blob.
URIs may use any content-addressed scheme: `ipfs://`, `ar://`, `bzz://`, `magnet:`, etc.
The list is identified by `keccak256(abi.encode(uris))` and stored once globally in the registry.

MirrorLists are published via `publishMirrorLists` or inline during registration, and referenced by ID; each attester's per-descriptor state is a single 32-byte pointer to the content-addressed list.
When multiple descriptors or attesters use the same retrieval endpoints, the URI strings exist exactly once in contract storage regardless of how many attestations reference them.

Retrieval via the published URIs is never the source of trust.
Wallets MUST recompute the retrieved descriptor's `descriptorHash` per ERC-8176 and verify that it equals the registered `descriptorHash` before using the descriptor, and MUST verify a retrieved off-chain attestation blob against its registered UID.

#### MirrorList entry resolution

Each MirrorList entry — descriptor or attestation alike — resolves to one of two things:

1. **The target content directly**: an ERC-7730 descriptor JSON file, or a signed off-chain attestation blob. This is the default case.
2. **A shared multi-file container whose entry point is an `index.json`** manifest mapping content keys — `descriptorHash` for descriptors, attestation UID for attestation blobs — to the specific member within that container.

There is no universal "folder" concept across `ipfs:`, `git:`, `https:`, `ftp:`, `magnet:`, `bzz:` and other schemes — an IPFS UnixFS directory, a git tree, an HTTP directory listing, a member of a multi-file torrent, and a Swarm manifest are all different addressing mechanisms.
The registry does not try to unify them: it only requires that the *published URI itself* — whatever it takes to construct it for that scheme — resolve to fetchable bytes that are either the target content or an `index.json` manifest.

A wallet's resolution algorithm is therefore uniform regardless of transport:

1. Fetch the MirrorList URI.
2. If the result validates as the expected content — the descriptor hash matches `descriptorHash`, or the blob is a validly signed attestation for the expected UID — use it directly.
3. Otherwise, if the result parses as an `index.json` manifest, look up the expected key (`descriptorHash` or UID) and fetch the member it points to.

This is what makes shared containers viable for batches: an attester bulk-publishing signed attestations for many UIDs (or descriptors for many hashes) can point them all at one container URI keyed by an `index.json`, so a single deduplicated MirrorList entry serves arbitrarily many content items.

## Rationale

### EAS as the sole attestation authority

Having two parallel attestation mechanisms, native and EAS, would mean the same information is stored in two places.

EAS provides on-chain `attest()` and `revoke()` with full lifecycle guarantees.

The registry adds the context-keyed discovery surface that EAS itself does not provide.

For the same reason, the registry stores only the attestation UID per active slot: the attested descriptor hash is already carried in the attestation's data — verified at registration time — and is read from EAS when queried, rather than being duplicated in registry storage.

This principle does not extend to the off-chain path, which never carried EAS-verified guarantees in the first place: the registry already performs no signature or content verification of an off-chain attestation blob regardless of its shape (see "Off-chain attestation path" below). Tagging that blob's declared `format` therefore does not introduce a second verified attestation mechanism — it makes an already-opaque, wallet-verified pointer self-describing, without adding any on-chain trust.

### Content-addressed MirrorLists

With tens of thousands of descriptors and dozens of attesters, sharing identical retrieval endpoints may save a lot of on-chain storage.

A per-attester state is reduced to a single 32-byte pointer.

Any attester can attest to an existing descriptor by providing only their signatures and the `mirrorListId` — no URI data needs to be re-transmitted or stored.

Publishing MirrorLists is decoupled from attestation, so running mirrors can be a separate role: infrastructure operators maintain the published URI lists while attesters only ever sign 32-byte list IDs.

Once a MirrorList is published, it cannot be modified.

Changing an attester's active MirrorList for a descriptor requires calling `updateMirrorList` with a new `mirrorListId` and a valid EIP-712 signature from the attester.

This decouples the storage endpoint pointer from the EAS descriptor attestation, allowing attesters to rotate or update broken infrastructure links (e.g., changing IPFS gateways) instantly for thousands of descriptors without the massive gas overhead of re-attesting the exact same descriptor hashes on EAS. The on-chain verifiable record is maintained via the EIP-712 signature.

### MirrorList URI schemes

The RECOMMENDED baseline schemes for MirrorList entries are `ipfs:` and `https:`; `http:`, `git:`, `ftp:`, `ar:` (Arweave) and `data:` are OPTIONAL.

Because URI matching happens on-chain through the `allowedPrefixes` parameter, no "wallets MUST skip unrecognized schemes" client-side rule is needed: an unrecognized scheme simply is not among the prefixes a wallet sends, so it is filtered out before it ever leaves the contract.

A MirrorList entry MAY be an [RFC 2397](https://www.rfc-editor.org/rfc/rfc2397) `data:` URI carrying the full ERC-7730 descriptor JSON inline — e.g. `data:application/json,<percent-encoded content>` or `data:application/json;base64,<base64 content>` — making the descriptor fully available on-chain with no external retrieval at all.
This works through the existing inline publication flow with no write-path changes, at a storage cost of roughly 20,000 gas per 32 bytes of new data: a several-KB descriptor runs roughly 3–4 million gas (about 0.03 ETH at 10 gwei on mainnet, negligible on an L2).
It is intended selectively for top-priority protocols rather than as the default entry type, and the on-chain prefix filtering is precisely what makes mixing a few large `data:` entries into an otherwise normal MirrorList safe: callers that do not ask for `data:` never pay to receive it.

No new content-verification logic is needed for any scheme: wallets already MUST hash fetched descriptor content and compare it against the attested `descriptorHash` regardless of transport, which covers `data:` URIs the same as `https:` or `ipfs:` ones.

### Off-chain attestation path

The on-chain and off-chain registration paths are alternatives sharing the same slot storage, discovery surface and MirrorList mechanism; the difference is where the EAS attestation lives.

The off-chain path preserves the EAS ecosystem compatibility — the attestation blob is a standard EAS off-chain attestation with the same [EIP-712](./eip-712.md) format, UID scheme, ERC-8176 schema and tooling — while eliminating the roughly 200,000 gas EAS write per attestation.
This enables cost-sensitive attesters to cover large descriptor sets that would be uneconomical to attest on-chain, at the cost of the wallet fetching and verifying the attestation blob itself.

The off-chain path also accommodates artifacts that are not EAS-shaped at all. A resource-constrained device — for example a hardware wallet that cannot parse JSON on-device — may instead sign its own device-native binary encoding of a descriptor and register the resulting UID with a vendor-specific `format` tag (see "Current Attestation Format Tags"). Because the registry never inspects the artifact's shape or signature on either path, supporting such a format costs nothing beyond minting a new tag off-chain; the vendor's own tooling and device firmware are solely responsible for producing and verifying it.

Revocation remains on-chain in both paths: an off-chain attestation is revoked via `eas.revokeOffchain` (roughly 20,000 gas), so a compromised or withdrawn off-chain attestation can be invalidated with the same finality and observability as an on-chain one.

The registry deliberately performs no verification of the off-chain attestation signature or content: wallets MUST verify the fetched blob anyway (exactly as they MUST verify on-chain attestations per ERC-8176), so a registry-side check would add gas cost without adding trust.

### Curated, vendor-specific consumption

The registry's value does not require every wallet to consume it directly, nor does it require universal on-chain or on-device coverage of the long tail of contracts. A hardware wallet vendor with device or bandwidth constraints MAY instead operate its own backend that periodically queries `resolveDescriptors` for a small, curated subset of high-volume protocols, converts the resulting ERC-7730 descriptors into its own device-friendly encoding (see "Off-chain attestation path" for the accompanying `format` tag), and ships only that subset to its devices.

Such a vendor still benefits from the registry's context-keyed discovery, attester attestations, and revocation for the protocols it chooses to curate, without needing on-device JSON parsing or an incentive scheme covering descriptors it does not.

### Registration gas costs

The cost of populating the registry is dominated by components external to it.
A batched `createDescriptorAttestations` call costs roughly 350,000–400,000 gas per descriptor: approximately 200,000 gas for the EAS attestation, approximately 150,000 gas for the one-time storage of a content-addressed MirrorList, and the remainder for calldata and registry-level validation.
Each context ID adds roughly 25,000 gas for the registry slot and the event.

As a round reference point: registering 10,000 descriptors averaging two deployments each requires on the order of 4 billion gas — about 40 ETH at a 10 gwei gas price, or roughly 0.004 ETH per descriptor.

Four properties keep this cost manageable:

1. It is a one-time bootstrap cost. Subsequent releases only re-attest descriptors whose content changed.

2. Registrations and MirrorList publications are batched: a release is at most two transactions (one, when MirrorLists are supplied inline) and one registration signature regardless of descriptor count, amortizing per-transaction overhead.

3. Subsequent attesters attesting to an already-registered descriptor skip the MirrorList storage entirely (publication is idempotent), reducing their cost by roughly a third.

4. Costs scale linearly with descriptor and deployment counts, so attesters can prioritize high-value protocols and expand coverage incrementally.

5. Updating the MirrorList for existing descriptors uses the `updateMirrorList` function, completely bypassing EAS, making infrastructure rotations virtually free compared to re-attesting.

The off-chain attestation path reduces the cost much further.
A descriptor with two context IDs that costs roughly 250,000 gas on the on-chain path once its descriptor MirrorList is already published (the EAS attestation plus two registry slots) drops to roughly 90,000 gas on the off-chain path: the EAS write is eliminated entirely, the attestation MirrorList is published once per batch regardless of batch size (and is content-addressed and deduplicated across every context ID and descriptor that shares it), and each slot pays for a single `bytes32` UID write plus the once-per-UID off-chain metadata record (descriptor hash, attestation MirrorList pointer, expiration time, and format tag) instead of any per-slot URI string storage.

These figures are order-of-magnitude estimates only: actual costs depend on the EAS contract implementation, URI lengths, the number of deployments per descriptor, and prevailing gas prices.

### Mainnet-only deployment

The registry is deployed on Ethereum mainnet only.

Context IDs encode `chainId`, so a single registry can serve wallets operating on any EVM chain.

The ERC-7730 descriptors format is designed to cover all deployments on all EVM-compatible chains.

## Backwards Compatibility

This ERC introduces a new contract interface and makes no modifications to ERC-7730 or ERC-8176.

ERC-7730 descriptor files require no changes. The context section fields used for context ID derivation are already present in all compliant descriptors.

## Security Considerations

### Malicious registration

Any address can call `createDescriptorAttestations` with their own EAS attestation.

Wallets only query active slots for their configured `trustedAttesters`.

An unknown attacker's address will never appear in a trusted list, so their slot is invisible to wallets.

Registration is permissionless precisely because it is harmless without a trusted attester's attestation.

### Attester key compromise

If a trusted attester's key is compromised, an attacker can create a malicious EAS attestation and call `createDescriptorAttestations`.

Attesters SHOULD use secure setup multisig accounts or governance contracts rather than EOAs.

Wallet vendors SHOULD publish their trusted attester addresses and key rotation procedures.

### EAS attestation revoked after slot update (staleness)

An attester may revoke their EAS attestation directly after calling `createDescriptorAttestations`. The registry's active slot continues to point to the revoked attestation until the attester replaces it or anyone calls `clearRevokedAttestations`.

Wallets MUST check the revocation and expiration status of the backing attestation (per ERC-8176) and so are not affected by stale slots; clearing is a housekeeping mechanism, not a safety requirement.

### Atomicity of slot replacement and revocation

When `createDescriptorAttestations` or `createOffchainDescriptorAttestations` replaces an existing active slot, the registry requires that every displaced attestation UID is present in the supplied `RevocationBatch` (`revocations.onchain` for displaced on-chain attestations, `revocations.offchain` for displaced off-chain ones). This prevents a permissionless relay from silently updating a slot while leaving a stale active attestation on EAS.

### Off-chain attestation revocation semantics

When `createOffchainDescriptorAttestations` displaces an off-chain slot, the registry calls `eas.revokeOffchain(uid)`, which records the revocation under the *registry's* address as the revoker.
Since anyone can route arbitrary UIDs through this call as part of their own registrations, such records are informational — a public, timestamped displacement trail — and MUST NOT be treated as an authoritative revocation of another attester's attestation.

Wallets MUST determine an off-chain attestation's validity from the registry slot still pointing at the UID and from `eas.getRevokeOffchain(attester, uid)` — a revocation recorded by the attester themselves.
An attester can thus directly and immediately revoke an off-chain attestation by calling `eas.revokeOffchain` from their own address, without interacting with the registry; `clearRevokedAttestations` picks such revocations up when sweeping stale slots.

### Attestation format is not verified content

The `format` tag on an `OffchainAttestation` is attester-declared metadata, exactly like `contextTypeTag` on a context ID: the registry stores and returns it without interpreting, validating, or verifying it against the artifact it names. Declaring `format = ATTESTATION_FORMAT_EAS_OFFCHAIN` does not cause the registry to check that the fetched blob is actually a well-formed EAS off-chain attestation, and declaring any other tag does not exempt an attester from producing a genuinely verifiable artifact.

Wallets and vendor backends MUST apply the verification procedure appropriate to the declared `format` before trusting the fetched blob — UID recomputation, signature and schema checks for `ATTESTATION_FORMAT_EAS_OFFCHAIN` per ERC-8176, or the vendor's own equivalent procedure for any other declared format. An unrecognized `format` value SHOULD be treated as unverifiable and rejected, exactly as a wallet should reject any off-chain attestation blob it cannot validate.

### Incorrect or hijacked context ID bindings

The registry cannot verify `contextIds` against the off-chain JSON descriptor, so an attester could — maliciously or mistakenly — associate a descriptor for contract A under the context ID for contract B.
Wallet-side validation of the descriptor's `context` section against the actual transaction (a MUST, see above) ensures such a mis-binding results in the descriptor being rejected rather than misleading formatting being shown.

Without further protection, a third party observing an attester's EAS delegated attestation signature (for example, in the public mempool) could front-run the registration and bind the descriptor to attacker-chosen context IDs or retrieval endpoints, consuming the attester's EAS nonce and blocking the honest registration.
The mandatory `registrationSignature` (see Registration authorization) closes this vector: the transaction sender cannot assert any `contextIds` or `mirrorListId` that the attester did not explicitly sign.
