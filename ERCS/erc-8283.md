## Abstract

This ERC defines an on-chain protocol-agnostic registry that maps structured *context IDs*, derived from [ERC-7730](./eip-7730.md) `context` binding information, to ERC-7730 descriptors and their attestations represented as their respective URIs and UIDs.

Descriptors' URIs may reference any content-addressed scheme (`ipfs://`, `ar://`, `bzz://`, `magnet:`, etc.) and serve as a suggested retrieval mechanism for the wallet.

Attestations are backed by the Ethereum Attestation Service per [ERC-8176](./eip-8176.md).
The registry additionally provides context-aware on-chain discoverability that EAS does not offer.

## Motivation

[ERC-7730](./eip-7730.md) defines a JSON descriptor format that enriches smart contract calls and [EIP-712](./eip-712.md) messages with human-readable display metadata. Each descriptor carries a `context` section that binds it to specific contracts or EIP-712 domains and serves as a key for the descriptor lookup when displaying the actual signature requests to the users.

The maintenance and distribution of descriptors' registry is an important component of the Clear Signing solution and would greatly benefit frombeing standardized across wallets without compromising on security or decentralization.

1. **On-chain context-keyed discovery.** Wallets can trustlessly find which descriptor applies to a given contract interaction or signature request without querying any off-chain repository service. This improves the riliability and accessability of the Clear Signing descriptor discovery process.

2. **Unified on-chain attestation surface.** [ERC-8176](./eip-8176.md) defines EAS-based attestation records, but EAS contracts have no suitable attestation lookup mechanisms. This removes the need for wallet providers to maintain their own services mapping descriptors to their attestations, and makes any entity maintaining singular trusted registries of clear signing descriptors completely redundant.

3. **Protocol-agnostic descriptor resolution.** Attesters are free to serve the descriptors via IPFS, Swarm, BitTorrent, or any other content-addressed scheme. This removes any bias towards any currently dominating protocol which might become obsolete during the lifetime of the Clear Signing ERC, which is supposed to be indefinate.

4. **Unified revocation and upgrade path.** An on-chain mechanism for an attester to atomically replace their endorsement with either a newer descriptor version or a revocation message. This removes the bottleneck on the speed of issuing a revocation, which is inevitable in any independent-review-based central registry.

## Specification

### Definitions

| Term                 | Definition                                                                                                            |
|----------------------|-----------------------------------------------------------------------------------------------------------------------|
| **descriptor**       | An ERC-7730 JSON file as defined in ERC-7730                                                                          |
| **descriptorHash**   | The ERC-8176 descriptor hash: `keccak256` of the fully resolved canonicalized descriptor                              |
| **contextId**        | A `bytes32` value canonically derived from an ERC-7730 `context` binding, used as the registry lookup key             |
| **contextTypeTag**   | A `bytes32` equal to `keccak256("erc7730.context.<typename>")`, namespacing a context ID derivation rule              |
| **active slot**      | The single `attestationId` an attester currently endorses for a given `contextId`; the endorsed descriptor hash is carried in the attestation's data |
| **attester**         | An `address` whose ERC-8176 EAS attestation has been submitted to the registry                                        |
| **attestationId**    | The EAS UID of the active ERC-8176 attestation backing an attester's endorsement                                      |
| **MirrorList**       | A `string[]` of retrieval URIs for a descriptor file, stored once globally and referenced by its content hash         |
| **mirrorListId**     | `keccak256(abi.encode(uris))` — the content hash identifying a MirrorList                                             |
| **deployEventTopic** | The factory contract's deploy event signature `topic[0]` in Ethereum logs, e.g. `keccak256("TokenDeployed(address)")` |

Descriptor hashes are computed exactly as defined in [ERC-8176](./eip-8176.md) "Descriptor Hash Computation".

#### Current Tag Constants

```
CONTEXT_TAG_CONTRACT   = keccak256("erc7730.context.contract")
CONTEXT_TAG_EIP712_DEP = keccak256("erc7730.context.eip712.deployment")
CONTEXT_TAG_EIP712_DS  = keccak256("erc7730.context.eip712.domainseparator")
CONTEXT_TAG_FACTORY    = keccak256("erc7730.context.factory")
```

### **Context ID** Derivation

The **Context ID** keys are computed as:

```
contextId = keccak256(abi.encode(contextTypeTag, param1, param2, …))
```

The `contextTypeTag` prefix namespaces keys so that different context binding types cannot collide regardless of parameter values.

This scheme is forward-extensible: future additions of new context binding types in ERC-7730 define new tag strings and parameter lists. The registry contract requires no upgrade when a new context tag is introduced, and wallets can derive the new key type using the new context and contiue querying the registry.

All context IDs are computed as `keccak256(abi.encode(contextTypeTag, params...))`, where the tag and parameter list depend on the ERC-7730 context binding type. A descriptor covering *N* deployments MUST be registered under *N* context IDs — one per deployment entry.

| ERC-7730 context type            | `params` in `abi.encode`                        |
|----------------------------------|-------------------------------------------------|
| `context.contract.deployments`   | `chainId`, `contract`                           |
| `context.contract.factory`       | `chainId`, `factoryAddress`, `deployEventTopic` |
| `context.eip712.deployments`     | `chainId`, `verifyingContract`                  |
| `context.eip712.domainSeparator` | `domainSeparator`                               |

### Data Model

The registry maintains the following mappings:

```solidity
// The ERC-8176 EAS UID of the active attestation for the given attester and context ID.
// Atomically replaced on each call to 'createDescriptorAttestations'.
// The endorsed descriptor hash is carried in the attestation's data and is not
// duplicated in registry storage.
mapping(address attester => mapping(bytes32 contextId => bytes32)) attestationId;

// Per-attester pointer to the MirrorList for a given descriptor.
mapping(address attester => mapping(bytes32 descriptorHash => bytes32)) mirrorListId;

// Global store of MirrorLists, written once per unique URI set.
// A "mirror list" is an array of URIs for fetching the descriptor file, supplied by the given attester.
// Key = keccak256(abi.encode(uris)).
mapping(bytes32 mirrorListId => string[]) mirrorLists;
```

MirrorLists are immutable and are stored once globally and can be shared by multiple attesters.

### Interface

The full normative interface is provided in [IClearSigningRegistry.sol](../assets/erc-8283/IClearSigningRegistry.sol).

#### `createDescriptorAttestations`

The primary write function. It registers a batch of descriptors in a single transaction by calling `eas.multiRevokeByDelegation(revocations)` to revoke any previously active attestations that are being replaced, then calls `eas.multiAttestByDelegation(attestations)` to create all attestations on-chain.

Each registration endorses a MirrorList, and two flows are supported for providing its content:

- **Reference flow** (`mirrorListId` set, `mirrorListUris` empty): `mirrorListId` MUST reference a MirrorList already published — by any address, in any prior transaction — otherwise the call reverts with `UnknownMirrorList`. Mirror operation may thus be a separate role from attestation: an infrastructure operator maintains the published lists while attesters only ever sign 32-byte list IDs.
- **Inline flow** (`mirrorListUris` non-empty, `mirrorListId` zero): the list is published idempotently as part of the registration and its content hash becomes the effective MirrorList ID. A first registration therefore needs no separate publication transaction. Declaring a non-zero `mirrorListId` together with inline URIs reverts with `RedundantMirrorListId`, as the ID is derived from the URIs.

In both flows the relayed [EIP-712](./eip-712.md) registration signature covers the *effective* MirrorList ID.
Any address may permissionlessly call this function, and the attester identity is taken from `attestations[0].attester`.

`attestations[0]` MUST use the ERC-8176 schema UID and MUST contain exactly one `data` entry per registration: `attestations[0].data[i]` is the active attestation for `registrations[i]`.

Each entry's `data` MUST be exactly 32 bytes long and ABI-decode to `bytes32` equal to the corresponding `descriptorHash`, and each entry MUST be revocable: the registry requires revocation of a displaced attestation, so an irrevocable active attestation would permanently freeze the affected slots.

All entries in `attestations[1..N]` are supplementary and are passed through to EAS without registry-level validation.

The registry stores only the attestation UID per `(attester, contextId)` slot.
The endorsed descriptor hash is carried in the attestation's data, verified at registration time as described above, and is read from the EAS contract when queried.

The `revocations` parameter MAY be empty when no active slots are replaced. When the attester already has an active attestation for any of the supplied context IDs, the corresponding attestation UID MUST appear in the `revocations` batch; otherwise the call reverts with `MissingRevocation`.

Wallets MUST verify the returned attestation per ERC-8176 "Verifying an Attestation" — schema, descriptor hash match, expiration, and revocation status — before trusting an active attestation record.

##### Registration authorization

The EAS delegated attestation signatures cover only the EAS request fields; they do not cover the context IDs or MirrorLists, which are chosen by the transaction sender.
Unless `msg.sender == attestations[0].attester`, the call MUST therefore carry a `registrationSignature`: an [EIP-712](./eip-712.md) signature by the attester over the structure

```
ClearSigningRegistrationBatch(
    DescriptorRegistration[] registrations,
    bytes32 attestationSignaturesHash
)
DescriptorRegistration(
    bytes32 descriptorHash,
    bytes32 contextIdsHash,
    bytes32 mirrorListId
)
```

The signature is verified via ECDSA recovery for EOA attesters and via [ERC-1271](./eip-1271.md) `isValidSignature` for contract attesters.
When `msg.sender == attestations[0].attester`, `registrationSignature` SHOULD be empty.

A single registration signature authorizes the entire batch, so an attester performs one extra signing operation per registration transaction regardless of how many descriptors it covers.
Binding the batch to the EAS delegated attestation signatures makes the registration signature single-use without any additional nonce state: EAS consumes the attester's nonces when the attestations are submitted, so none of the signatures can be replayed.
This prevents a third party from taking an attester's EAS attestation signatures and registering the descriptors under attacker-chosen context IDs or retrieval endpoints.

#### `resolveDescriptors()`

The primary wallet-facing query entry point: resolves all active endorsements for the given attesters across the given context IDs in a single call.

For every resolved attestation record the returned `ResolvedDescriptor` structure carries all the necessary data: the attester, the context ID, the descriptor hash, the backing attestation UID, the attestation's `expirationTime` and `revocationTime`, and the MirrorList contents.
Results are ordered by `attesters` first, then `contextIds`; empty slots produce no entry.

A wallet derives its candidate context IDs locally — the contract key for a calldata transaction, the deployment and domain-separator keys for an EIP-712 message, and factory keys for factory contracts the wallet supports — and obtains the endorsed descriptor hashes, their attestation validity, and the retrieval URIs in a single `eth_call`.
The wallet remains responsible for applying its trust policy across attesters and for verifying the fetched descriptor bytes against `descriptorHash` and the descriptor's `context` section against the transaction.

#### `clearRevokedEndorsements()`

Clears active slots whose backing EAS attestation has been revoked or has expired.

The function is permissionless: anyone may clean up stale slots, allowing a watchdog service to sweep multiple attesters in a single transaction.
`contextIds[i]` lists the context IDs to clear for `attesters[i]`; the two arrays MUST have the same length, and neither `attesters` nor any per-attester context ID list may be empty.

A slot is cleared when it is non-empty and its backing attestation is revoked (`revocationTime != 0`) or expired (`expirationTime != 0` and in the past).
All other slots are skipped rather than causing a revert, so a large sweep cannot be blocked by a single slot changing state between the sweep's preparation and its inclusion in a block.
The function returns the number of attestation records actually cleared, allowing a watchdog to determine via simulation whether a sweep is worth submitting.

For each cleared slot, an `AttestationUpdated` event is emitted with `bytes32(0)` as the new descriptor hash and attestation ID.

#### `publishMirrorLists(string[][] uriLists) returns (bytes32[] mirrorListIds)`

Stores a batch of retrieval URI lists on-chain and returns their content hashes `keccak256(abi.encode(uris))`.
A MirrorList whose content has already been published is not re-written and emits no event, and identical URI sets occupy storage exactly once regardless of how many registrations or attesters reference them.

Any address may publish MirrorLists, and publication may be done independently of attestation: a pinning service, mirror operator, or any other party can maintain the published lists that attesters may reference by ID.
Registrants that prefer a single transaction can instead supply the URIs inline in `createDescriptorAttestations` (see above).

#### `getMirrorListById(bytes32 mirrorListId)`

Returns the URI list for a given MirrorList ID. Returns an empty array if the ID is unknown.

Because the `MirrorListPublished` event carries only the ID, this getter is the standalone way to read a published list's contents.
An attester's current MirrorList for a descriptor is available through `resolveDescriptors` and through `MirrorListUpdated` events.

### MirrorLists

A MirrorList is a `string[]` of retrieval URIs for a descriptor file.
URIs may use any content-addressed scheme: `ipfs://`, `ar://`, `bzz://`, `magnet:`, etc.
The list is identified by `keccak256(abi.encode(uris))` and stored once globally in the registry.

MirrorLists are published via `publishMirrorLists` or inline during registration, and referenced by ID; each attester's per-descriptor state is a single 32-byte pointer to the content-addressed list.
When multiple registrations or attesters use the same retrieval endpoints, the URI strings exist exactly once in contract storage regardless of how many endorsements reference them.

Retrieval via the published URIs is never the source of trust.
Wallets MUST recompute the retrieved descriptor's `descriptorHash` per ERC-8176 and verify that it equals the registered `descriptorHash` before using the descriptor.

## Rationale

### EAS as the sole attestation authority

Having two parallel attestation mechanisms, native and EAS, would mean the same information is stored in two places.

EAS provides on-chain `attest()` and `revoke()` with full lifecycle guarantees.

The registry adds the context-keyed discovery surface that EAS itself does not provide.

For the same reason, the registry stores only the attestation UID per active slot: the endorsed descriptor hash is already carried in the attestation's data — verified at registration time — and is read from EAS when queried, rather than being duplicated in registry storage.

### Content-addressed MirrorLists

With tens of thousands of descriptors and dozens of attesters, sharing identical retrieval endpoints may save a lot of on-chain storage.

A per-attester state is reduced to a single 32-byte pointer.

Any attester can endorse an existing descriptor by providing only their signatures and the `mirrorListId` — no URI data needs to be re-transmitted or stored.

Publishing MirrorLists is decoupled from attestation, so running mirrors can be a separate role: infrastructure operators maintain the published URI lists while attesters only ever sign 32-byte list IDs.

Once a MirrorList is published, it cannot be modified.

Changing an attester's active MirrorList for a descriptor requires calling `createDescriptorAttestations` with a new `mirrorListId`, which in turn requires a fresh EAS attestation.

This ensures that every change to retrieval endpoints is fully backed by the same attestation chain as the descriptor itself and prevents any party from silently redirecting downloads without a verifiable on-chain record.

### Registration gas costs

The cost of populating the registry is dominated by components external to it.
A batched `createDescriptorAttestations` call costs roughly 350,000–400,000 gas per descriptor: approximately 200,000 gas for the EAS attestation, approximately 150,000 gas for the one-time storage of a content-addressed MirrorList, and the remainder for calldata and registry-level validation.
Each context ID adds roughly 25,000 gas for the registry slot and the event.

As a round reference point: registering 10,000 descriptors averaging two deployments each requires on the order of 4 billion gas — about 40 ETH at a 10 gwei gas price, or roughly 0.004 ETH per descriptor.

Four properties keep this cost manageable:

1. It is a one-time bootstrap cost. Subsequent releases only re-attest descriptors whose content changed.

2. Registrations and MirrorList publications are batched: a release is at most two transactions (one, when MirrorLists are supplied inline) and one registration signature regardless of descriptor count, amortizing per-transaction overhead.

3. Subsequent attesters endorsing already-registered descriptors skip the MirrorList storage entirely (publication is idempotent), reducing their cost by roughly a third.

4. Costs scale linearly with descriptor and deployment counts, so attesters can prioritize high-value protocols and expand coverage incrementally.

These figures are order-of-magnitude estimates only: actual costs depend on the EAS contract implementation, URI lengths, the number of deployments per descriptor, and prevailing gas prices.

### Mainnet-only deployment

The registry is deployed on Ethereum mainnet only.

Context IDs encode `chainId`, so a single registry can serve wallets operating on any EVM chain.

The ERC-7730 descriptors format is designed to cover all deployments on all EVM-compatible chains.

## Backwards Compatibility

This ERC introduces a new contract interface and makes no modifications to ERC-7730 or ERC-8176.

ERC-7730 descriptor files require no changes. The context section fields used for context ID derivation are already present in all compliant descriptors.

## Security Considerations

### Malicious registration

Any address can call `createDescriptorAttestations` with their own EAS attestation.

Wallets only query active slots for their configured `trustedAttesters`.

An unknown attacker's address will never appear in a trusted list, so their slot is invisible to wallets.

Registration is permissionless precisely because it is harmless without trusted attester endorsement.

### Attester key compromise

If a trusted attester's key is compromised, an attacker can create a malicious EAS attestation and call `createDescriptorAttestations`.

Attesters SHOULD use secure setup multisig accounts or governance contracts rather than EOAs.

Wallet vendors SHOULD publish their trusted attester addresses and key rotation procedures.

### EAS attestation revoked after slot update (staleness)

An attester may revoke their EAS attestation directly after calling `createDescriptorAttestations`. The registry's active slot continues to point to the revoked attestation until the attester replaces it or anyone calls `clearRevokedEndorsements`.

Wallets MUST check the revocation and expiration status of the backing attestation (per ERC-8176) and so are not affected by stale slots; clearing is a housekeeping mechanism, not a safety requirement.

### Atomicity of slot replacement and revocation

When `createDescriptorAttestations` replaces an existing active slot, the registry requires that every displaced attestation UID is present in the supplied `revocations` batch. This prevents a permissionless relay from silently updating a slot while leaving a stale active attestation on EAS.

### Incorrect or hijacked context ID bindings

The registry cannot verify `contextIds` against the off-chain JSON descriptor, so an attester could — maliciously or mistakenly — associate a descriptor for contract A under the context ID for contract B.
Wallet-side validation of the descriptor's `context` section against the actual transaction (a MUST, see above) ensures such a mis-binding results in the descriptor being rejected rather than misleading formatting being shown.

Without further protection, a third party observing an attester's EAS delegated attestation signature (for example, in the public mempool) could front-run the registration and bind the descriptor to attacker-chosen context IDs or retrieval endpoints, consuming the attester's EAS nonce and blocking the honest registration.
The mandatory `registrationSignature` (see Registration authorization) closes this vector: the transaction sender cannot assert any `contextIds` or `mirrorListId` that the attester did not explicitly sign.
