## Abstract

ERC-7750 defines a standard interface for submitting, executing, and verifying autonomous agent tasks on EVM-compatible blockchains. Each task carries a payment denominated in ETH or any ERC-20 token; a **1 % protocol fee** is immediately routed to the Opacus treasury on submission, while the remaining 99 % (the "net amount") is escrowed and released to the agent upon proof submission.

## Motivation

AI agents operating on-chain require a trustless payment and verification primitive that is:

* **Chain-agnostic** – works with ETH, USDC, or any fungible token.
* **Proof-verifiable** – agents must supply a keccak256 proof hash upon completion.
* **Fee-normalised** – a uniform 1 % fee funds protocol sustainability without governance overhead.

Existing task-queue and payment-channel patterns do not cover the full lifecycle (submit → proof → release / cancel). ERC-7750 fills this gap.

## Specification

The key words "MUST", "MUST NOT", "SHOULD", "MAY" in this document are to be interpreted as described in RFC 2119.

### Structs

```solidity
struct Task {
    address submitter;
    address agent;
    address token;        // address(0) = native ETH
    uint256 grossAmount;
    uint256 feeAmount;    // 1% of grossAmount → treasury
    bytes32 proofHash;
    TaskStatus status;
    uint64  createdAt;
    uint64  completedAt;
    bytes   payload;
}
```

### Methods

#### `submitTask`

```solidity
function submitTask(
    address agent,
    bytes calldata payload,
    address token,
    uint256 amount
) external payable returns (bytes32 taskId);
```

MUST collect `amount * 100 / 10000` as fee and forward it to `OPACUS_TREASURY` atomically.  
MUST emit `TaskSubmitted(taskId, submitter, agent, token, grossAmount, feeAmount)`.

#### `completeTask`

```solidity
function completeTask(
    bytes32 taskId,
    bytes32 proofHash,
    bytes calldata result
) external;
```

MUST only be callable by `task.agent`.  
MUST emit `TaskCompleted(taskId, proofHash, result)`.  
MUST release `netAmount` to the agent.

#### `cancelTask`

```solidity
function cancelTask(bytes32 taskId) external;
```

MUST only be callable by `task.submitter` while status is `Pending`.  
MUST refund `netAmount` to submitter (fee is non-refundable).  
MUST emit `TaskCancelled(taskId, refundTo)`.

#### `getTask`

```solidity
function getTask(bytes32 taskId) external view returns (Task memory);
```

### Events

```solidity
event TaskSubmitted(bytes32 indexed taskId, address indexed submitter, address indexed agent, address token, uint256 grossAmount, uint256 feeAmount);
event TaskCompleted(bytes32 indexed taskId, bytes32 proofHash, bytes result);
event TaskCancelled(bytes32 indexed taskId, address refundTo);
```

### ERC-165

Compliant contracts MUST return `true` for interface ID `0x4e697472` (`"Nitr"`).

## Rationale

* **Immediate fee collection** – forwarding fees on submission (not completion) prevents fee-evasion via cancel-and-refund.
* **Basis-points arithmetic** – integer overflow-safe at 256-bit. Fee = `gross × 100 / 10_000`.
* **Proof hash** – off-chain execution results are anchored on-chain without storing arbitrary bytes.

## Backwards Compatibility

No existing ERC is modified. ERC-7750 uses its own interface ID for ERC-165 discovery.

## Security Considerations

* **Reentrancy** – implementations MUST use a reentrancy guard on all value-transferring functions.
* **ETH mismatch** – the `submitTask` implementation MUST revert if `msg.value != amount` when `token == address(0)`.
* **Agent impersonation** – `completeTask` MUST verify `msg.sender == task.agent`.
* **Treasury address** – `OPACUS_TREASURY` is an immutable constant; a governance upgrade path is recommended for long-term deployments.
