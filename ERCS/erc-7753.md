## Abstract

This ERC defines an on-chain escrow primitive where funds are released only upon submission of a `proofHash` by the creator. Each escrow has an expiry (`ttlSeconds`); if the creator does not call `releaseEscrow` before expiry, the counterparty or creator can reclaim funds. 1% of the locked amount is collected as a protocol fee at creation time. An optional arbitrator address can resolve disputes.

## Motivation

Autonomous agents that transact on behalf of users need a trustless mechanism to hold payment until a task is proven complete. Existing escrow patterns rely on centralised off-chain arbitration or require custom contracts per use-case. This ERC standardises:

- **Proof-gated release** — funds move only when the creator submits a `proofHash` attesting to delivery.
- **Automatic expiry** — creators and counterparties can recover funds without needing the other party to cooperate after a deadline.
- **Dispute resolution** — a designated arbitrator can settle contested escrows, enabling decentralised freelance and agent-task markets.
- **Protocol fee alignment** — 1% bonded fee creates spam resistance and funds treasury operations.

## Specification

The key words "MUST", "MUST NOT", "SHOULD", "MAY" in this document are to be interpreted as described in RFC 2119.

### Enums

```solidity
enum EscrowStatus { Locked, Released, Refunded, Disputed, Resolved }
```

### Structs

```solidity
struct Escrow {
    address  creator;
    address  counterparty;
    address  token;           // address(0) = native ETH
    uint256  grossAmount;
    uint256  feeAmount;       // 1% forwarded to treasury at creation
    uint256  netAmount;       // 99% held in contract
    bytes32  conditionHash;   // keccak256(conditionDescription)
    bytes32  proofHash;       // submitted by creator at release
    EscrowStatus status;
    uint64   lockedAt;
    uint64   settledAt;
    uint64   expiresAt;       // block.timestamp + ttlSeconds
    string   description;
}
```

| `ESCROW_FEE_BPS` | `100` | Protocol fee in basis points (1%) |

### Methods

#### `createEscrow`

```solidity
function createEscrow(
    address counterparty,
    string  calldata description,
    address token,
    uint256 gross,
    uint64  ttlSeconds
) external payable returns (bytes32 escrowId);
```

MUST collect `gross * 1%` as fee and forward to `OPACUS_TREASURY` atomically.
MUST lock remaining 99% in the contract.
MUST revert if `gross == 0`.
MUST revert if `token == address(0)` and `msg.value != gross`.
MUST set `expiresAt = block.timestamp + ttlSeconds`.
MUST emit `EscrowCreated(escrowId, creator, counterparty, token, gross, fee)`.
MUST return a unique `escrowId` derived from `keccak256(creator, counterparty, timestamp, nonce)`.

#### `releaseEscrow`

```solidity
function releaseEscrow(bytes32 escrowId, bytes32 proofHash) external;
```

MUST only be callable by the escrow `creator`.
MUST only succeed when `status == Locked`.
MUST transfer `netAmount` to `counterparty`.
MUST set `proofHash`, `status = Released`, and `settledAt`.
MUST emit `EscrowReleased(escrowId, proofHash, counterparty)`.

#### `refundEscrow`

```solidity
function refundEscrow(bytes32 escrowId) external;
```

MUST only succeed when `status == Locked`.
MUST allow `counterparty` to trigger a refund at any time.
MUST allow `creator` to trigger a refund only after `block.timestamp > expiresAt`.
MUST transfer `netAmount` back to `creator`.
MUST emit `EscrowRefunded(escrowId, creator)`.

#### `disputeEscrow`

```solidity
function disputeEscrow(bytes32 escrowId) external;
```

MUST only be callable by `creator` or `counterparty`.
MUST only succeed when `status == Locked`.
MUST set `status = Disputed`.
MUST emit `EscrowDisputed(escrowId, caller)`.

#### `resolveDispute`

```solidity
function resolveDispute(bytes32 escrowId, bool releaseToCounterparty) external;
```

MUST only be callable by the designated `arbitrator`.
MUST only succeed when `status == Disputed`.
MUST transfer `netAmount` to `counterparty` if `releaseToCounterparty == true`, otherwise to `creator`.
MUST emit `EscrowResolved(escrowId, releaseToCounterparty)`.

#### `getEscrow`

```solidity
function getEscrow(bytes32 escrowId) external view returns (Escrow memory);
```

MUST return the full `Escrow` struct.

## Rationale

- **Creator-only release** — the creator (payer) controls whether proof is accepted, preventing the counterparty from self-releasing.
- **Counterparty-initiated refund** — the counterparty can voluntarily abandon a job and return funds without waiting for expiry.
- **`proofHash` not validated on-chain** — the contract stores the hash for auditability; validation of the underlying artifact is intentionally off-chain to keep gas costs low and remain storage-agnostic.
- **Optional arbitrator** — deployers may leave `arbitrator = address(0)` for a non-disputable escrow, reducing attack surface in automated agent contexts.
- **`ttlSeconds` as `uint64`** — avoids overflow beyond year 584 billion while saving storage vs `uint256`.

## Backwards Compatibility

No existing ERC is modified. This ERC is composable with [ERC-20](./eip-20.md) tokens and with [ERC-7751](./eip-7751.md) agent registrations; the `escrowId` returned by `createEscrow` MAY be embedded in an agent task record as a payment reference.

## Security Considerations

- **Reentrancy** — `createEscrow`, `releaseEscrow`, `refundEscrow`, and `resolveDispute` transfer value and MUST be protected by a reentrancy guard.
- **Arbitrator trust** — the arbitrator is fully trusted. Deployers SHOULD use a multi-sig or DAO-controlled address.
- **`proofHash` collision** — `bytes32(0)` is the default; implementations MUST reject a release with `proofHash == bytes32(0)` if the protocol requires non-empty proof.
- **Fee-on-transfer tokens** — `grossAmount` is the amount approved; fee-on-transfer [ERC-20](./eip-20.md) tokens will result in `netAmount` being less than expected. Callers SHOULD use standard tokens only.
- **Clock manipulation** — `expiresAt` relies on `block.timestamp`. Minor miner manipulation (±15 s) is acceptable for escrow TTLs measured in hours or days.
