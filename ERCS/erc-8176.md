## Abstract

This ERC defines a detached JSON attestation envelope format for [ERC-7730](./eip-7730.md) clear signing descriptor files. An attestation envelope is a standalone signed document asserting that a specific party has reviewed a descriptor with a specific content hash.

Each envelope is a self-contained JSON object distributable and verifiable independently of the descriptor itself. An envelope identifies the attesting account via a CAIP-10 identifier and carries lifecycle fields (`issuedAt` and optional `expiresAt`). Signatures are verified using [EIP-191](./eip-191.md) personal message recovery (for EOA attesters) or [ERC-1271](./eip-1271.md) `isValidSignature` (for contract attesters). The entire envelope, minus the `signature` field itself, is committed by the signature, so lifecycle metadata cannot be altered without invalidating it.

Wallets fetch attestations alongside descriptors and maintain their own trust model for which attesters to accept.

## Motivation

[ERC-7730](./eip-7730.md) defines a JSON format that enables wallets to display human-readable transaction details before a user signs. Wallets fetch these descriptor files from registries and rely on them to format transaction data correctly. If an attacker can contribute a misleading descriptor to a registry — or substitute one in transit — the wallet may display deceptive information, undermining the security guarantees that clear signing is designed to provide.

Several distinct attack classes motivate this proposal:

- **Registry or repository poisoning.** A malicious contributor submits a subtly incorrect descriptor — for example, one that hides a token approval amount — which passes cursory review but misleads end users.
- **Man-in-the-middle.** A descriptor is modified in transit by a network-layer attacker (malicious wifi, hostile proxy, TLS interception). HTTPS protects data against passive eavesdroppers but does not authenticate the content itself or identify who vouches for its correctness.
- **Supply-chain substitution.** An attacker compromises a distribution node that stores or serves artifacts — for example, a registry mirror, a CDN edge cache, or a pinning gateway — and replaces legitimate descriptors with forged copies.

Existing mechanisms do not address these scenarios adequately:

- **ERC-7730 context binding** verifies that a descriptor targets the correct contract address and chain, but it does not indicate whether the display formatting has been reviewed by a trusted party. A descriptor can have valid context binding while still containing misleading field labels or hidden parameters.
- **HTTPS** protects data in transit but does not authenticate the content itself or identify who vouches for its correctness.
- **Git commit signatures** prove authorship within a repository but do not survive extraction. Once a descriptor file is served via an API or downloaded as a standalone file, the git signature is lost.

This ERC provides a detached attestation mechanism that travels independently of the descriptor file, allowing wallets to verify which trusted parties have reviewed a descriptor regardless of how either is distributed.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

### Attestation Envelope Schema

An attestation envelope is a JSON object with the following shape:

```json
{
  "$type": "erc-8176/v1",
  "descriptorHash": "0x<64 hex>",
  "attester": "eip155:<chainId>:<address>",
  "attesterType": "EOA",
  "signature": "0x<hex>",
  "issuedAt": 1713456000,
  "expiresAt": 1716048000
}
```

| Field            | Type      | Required | Description                                                                                    |
|------------------|-----------|----------|------------------------------------------------------------------------------------------------|
| `$type`          | `string`  | REQUIRED | Envelope type and version. MUST be the literal value `"erc-8176/v1"` for this version.         |
| `descriptorHash` | `string`  | REQUIRED | `0x`-prefixed lowercase hexadecimal Keccak-256 hash of the canonicalized descriptor.           |
| `attester`       | `string`  | REQUIRED | CAIP-10 account identifier of the attesting account. MUST match `eip155:<chainId>:<address>`.  |
| `attesterType`   | `string`  | REQUIRED | MUST be `"EOA"` or `"contract"`. Selects the signature verification path.                      |
| `signature`      | `string`  | REQUIRED | `0x`-prefixed hex-encoded signature bytes.                                                     |
| `issuedAt`       | `integer` | REQUIRED | Unix timestamp in seconds when the envelope was issued.                                        |
| `expiresAt`      | `integer` | OPTIONAL | Unix timestamp in seconds after which verifiers MUST treat the envelope as expired.            |

The envelope MUST NOT contain additional top-level fields in this version.

The `attester` is the account making the attestation claim. For `"EOA"` attesters, the attester itself signs via [EIP-191](./eip-191.md). For `"contract"` attesters, some party authorized by the contract's [ERC-1271](./eip-1271.md) logic produces the signature; the envelope does not identify that party, since only the contract's `isValidSignature` determines validity.

### Descriptor Hash Computation

To compute the `descriptorHash`:

1. Let `D` be the parsed JSON descriptor object.
2. Serialize `D` to a byte string using RFC 8785 (JCS — JSON Canonicalization Scheme).
3. Compute the Keccak-256 hash of the resulting byte string.
4. Encode the hash as a `0x`-prefixed lowercase hexadecimal string (66 characters total).

Implementations MUST use Keccak-256 as specified in the Ethereum Yellow Paper (i.e., the pre-standardization variant used throughout the Ethereum ecosystem), NOT the NIST SHA3-256 standard.

### Envelope Hash Computation

To compute the envelope hash used as the signing input:

1. Let `E` be the envelope object with every field populated except `signature`.
2. Serialize `E` to a byte string using RFC 8785 (JCS).
3. Compute the Keccak-256 hash of the resulting byte string. The result is 32 bytes.

Because every envelope field except `signature` is included in the JCS bytes, the signature commits to the entire envelope. Any modification to `$type`, `descriptorHash`, `attester`, `attesterType`, `issuedAt`, or `expiresAt` invalidates the signature.

### Message Format

The message to be signed is a UTF-8 encoded string with the following format:

```abnf
message    = label LF "hash=" hash-value LF
label      = %s"ERC-8176 attestation v1"
hash-value = "0x" 64HEXDIG
LF         = %x0A
HEXDIG     = DIGIT / "a" / "b" / "c" / "d" / "e" / "f"
```

Expressed as a template:

```
ERC-8176 attestation v1\nhash=0x{envelopeHash}\n
```

where `{envelopeHash}` is the 64-character lowercase hexadecimal encoding of the 32-byte envelope hash from the previous section.

The primary domain separator is `$type: "erc-8176/v1"` inside the envelope. Because `$type` is a required envelope field, the JCS-canonicalized bytes that produce the envelope hash always contain this literal identifier. The message label `"ERC-8176 attestation v1"` provides a human-readable cue for signers inspecting the raw message.

### Signing Algorithm

To produce an attestation envelope (producer-side):

1. Compute `descriptorHash` as defined in [Descriptor Hash Computation](#descriptor-hash-computation).
2. Assemble the envelope with every field populated except `signature`: `$type`, `descriptorHash`, `attester`, `attesterType`, `issuedAt`, and (if applicable) `expiresAt`.
3. Compute the envelope hash as defined in [Envelope Hash Computation](#envelope-hash-computation).
4. Construct the message string as defined in [Message Format](#message-format).
5. Produce the signature:
   - **EOA attesters:** sign the message using [EIP-191](./eip-191.md) personal message signing with the attester's private key.
   - **Contract attesters:** an authorized party produces signature bytes that the contract's [ERC-1271](./eip-1271.md) `isValidSignature` will accept for the EIP-191 personal message hash of this message. The specific signing scheme is determined by the contract implementation (for example, an authorized party signs the EIP-191 message via ECDSA and the contract verifies the recovered address against its authorized-signer set).
6. Attach the resulting signature bytes as the envelope's `signature` field.

### Signature Verification

To verify an attestation envelope:

1. Parse the envelope and confirm `$type` equals `"erc-8176/v1"`. Reject otherwise.
2. Compute `descriptorHash` from the descriptor being verified. Confirm it equals the envelope's `descriptorHash` field. Reject otherwise.
3. Parse `attester` as a CAIP-10 identifier. Reject if the namespace is not `eip155` or the format is malformed.
4. If `expiresAt` is present and the current time exceeds it, treat the envelope as expired.
5. Compute the envelope hash by reconstructing the envelope with every field except `signature` and hashing per [Envelope Hash Computation](#envelope-hash-computation).
6. Construct the message string as defined in [Message Format](#message-format).
7. Compute the [EIP-191](./eip-191.md) personal message hash: `keccak256("\x19Ethereum Signed Message:\n" + len(message) + message)`.
8. Verify the signature according to `attesterType`:
   - If `"EOA"`: recover the attester address from `signature` using ECDSA recovery. Verification succeeds if and only if the recovered address equals the address component of the `attester` CAIP-10 identifier.
   - If `"contract"`: call [ERC-1271](./eip-1271.md) `isValidSignature(hash, signature)` on the contract at the extracted address, on the chain referenced in the `attester` identifier, at the current block. Verification succeeds if and only if the call returns the ERC-1271 magic value (`0x1626ba7e`).

If the referenced chain is unavailable (no RPC endpoint configured, network error), the verification result for that envelope is "inconclusive". Implementations MUST NOT treat an inconclusive result as a verification failure.

Contract attester verification reflects the contract's state at the current block. An envelope that verifies today MAY fail to verify later if the contract's `isValidSignature` logic changes (for example, a signing key rotation inside the contract). This is intentional: contract attesters retain operational control over their attestations through their own signature-validation state.

### Attester Chain vs Descriptor Chain

The `attester`'s CAIP-10 chain identifies where the attesting account exists. It is independent of the chain(s) a descriptor targets. An attester whose account lives on one chain MAY issue envelopes for descriptors that target another chain. Verifiers MUST NOT reject an envelope solely because the attester's chain differs from any chain referenced in the descriptor.

### Supersession and Retraction

When multiple valid envelopes exist for the same `(attester, descriptorHash)` pair, the envelope with the greatest `issuedAt` value is authoritative. Earlier envelopes from the same attester for the same descriptor are superseded.

An attester MAY retract a prior attestation by issuing a new envelope with the same `attester` and `descriptorHash`, a later `issuedAt`, and an `expiresAt` value less than or equal to `issuedAt` (i.e., an envelope that is already expired at the moment of issuance). Supersession selects the retraction envelope; its expired state causes verifiers to conclude that the attester has no currently valid endorsement for that descriptor.

Retraction propagates only through distribution channels the attester can update. Out-of-band retraction (for example, following key compromise, or third-party revocation) is out of scope for this ERC.

### Lifecycle Guidance

Attesters SHOULD set `expiresAt` to a bounded future timestamp to limit exposure if the attester loses operational capability (key loss, organizational change, discontinued service). Envelopes without `expiresAt` remain valid indefinitely unless actively retracted; wallets receiving long-lived envelopes SHOULD weigh this property when deciding trust policy.

### Distribution

Attestation envelopes are distributed alongside descriptors. Wallets fetch envelopes from sources they trust and maintain their own trust model for which attesters to accept. This ERC does not mandate a filename convention, directory layout, registry topology, or discovery mechanism.

### Multi-Attester Semantics

Multiple attesters MAY independently issue envelopes over the same descriptor. Each envelope is verified independently. A failure of one envelope MUST NOT invalidate other envelopes or the descriptor itself.

Wallets are responsible for maintaining their own set of trusted attesters and the policy that determines how many attestations (and from which attesters) are required before the descriptor's formatting instructions are presented to the user.

## Rationale

### Signing the Entire Envelope

The signature commits to every envelope field except `signature` itself. This prevents a network attacker from forging or rewriting unsigned metadata — for example, extending `expiresAt` on a retracted envelope. Any field modification changes the envelope hash and invalidates the signature.

### Keccak-256

Keccak-256 is the hash function used throughout the Ethereum ecosystem — in the EVM, in address derivation, in EIP-191, and in ABI encoding. Every wallet and Ethereum library already implements it. Using a different hash function (e.g., SHA-256) would introduce an unnecessary dependency for wallets that do not otherwise need it.

### RFC 8785 (JSON Canonicalization Scheme)

JSON objects have no defined key ordering, and JSON serializers differ in whitespace, Unicode escaping, and number formatting. To produce a deterministic byte string for hashing, a canonicalization scheme is required. RFC 8785 is a published standard that defines deterministic JSON serialization with well-specified rules for key ordering, number formatting, and string escaping. It is more widely implemented and better specified than ad-hoc approaches like "sort keys and minimize whitespace."

### EIP-191 and Not EIP-712

[EIP-712](./eip-712.md) typed data signing is designed for on-chain verifiable structured data with a domain separator. It requires defining a type schema and domain — complexity that provides no benefit here, since the signed data is a simple message verified off-chain. [EIP-191](./eip-191.md) personal message signing is simpler, widely supported by every wallet, and sufficient for this use case. Domain separation is achieved through the message label (`"ERC-8176 attestation v1"`) rather than through an EIP-712 domain struct.

### CAIP-10 with `eip155` Namespace

An attestation identifies the attesting account by both chain and address, not by address alone. This is necessary because contract attesters verify via ERC-1271, which requires specifying which chain hosts the contract. CAIP-10 is the established standard for chain-scoped account identifiers.

This version restricts the namespace to `eip155` — covering all EVM chains — since wallets already implement EIP-191 recovery and ERC-1271 validation for EVM accounts, and no new cryptographic capability is required.

### Short `$type` Tag Instead of a Hosted JSON Schema

The envelope is a small primitive object. A hosted JSON Schema file would add asset maintenance to the ERC without meaningful validation value for a handful of scalar fields. The `$type: "erc-8176/v1"` tag provides version identification without requiring a dereferenceable URI, and avoids confusion with JSON Schema's `$schema` keyword (which is reserved for JSON Schema pointers).

## Backwards Compatibility

This ERC is independent of [ERC-7730](./eip-7730.md). Descriptors do not change; attestation envelopes are a separate document type.

## Security Considerations

### Threat Model

| Threat                                   | Description                                                                                                                       | Mitigation                                                                                                                                                                          |
|------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Descriptor modification                  | Attacker modifies the descriptor content after it was attested.                                                                   | Recomputed `descriptorHash` does not match the value committed by the envelope; signature verification fails.                                                                       |
| Envelope modification                    | Attacker rewrites any field of a distributed envelope (for example, to advance `expiresAt`).                                      | The signature commits to every envelope field except `signature` itself; any modification changes the envelope hash and invalidates the signature.                                  |
| Attester impersonation                   | Attacker forges an envelope claiming a trusted CAIP-10 identifier as `attester`.                                                  | ECDSA recovery (EOA) or ERC-1271 validation (contract) fails; the recovered address or contract response does not authenticate the claimed attester.                                |
| MITM on transport                        | A network-layer attacker (hostile proxy, TLS interception) modifies descriptor or envelope bytes in transit.                      | Integrity is cryptographic and self-contained; any in-transit modification changes `descriptorHash` or the envelope hash and invalidates the signature.                             |
| Supply-chain substitution (distribution) | Attacker compromises a distribution node (registry mirror, CDN edge cache, pinning gateway) and replaces legitimate envelopes with forged ones. | Wallets verify signatures against their own trusted attester list; forged envelopes from unknown attesters are rejected regardless of delivery channel.                 |
| Downgrade (no attestation fetched)       | Attacker delivers the descriptor without any envelope, hoping the wallet trusts it unattested.                                    | Wallets that maintain a trusted attester list can detect the absence of expected attestations and act accordingly. Trust policy is a wallet concern.                                |
| Stale attestation                        | Attester has lost operational capability (key loss, organizational shutdown) but old envelopes continue to circulate.             | `expiresAt` bounds exposure. Attesters SHOULD set a bounded TTL.                                                                                                                    |
| Replay across descriptors                | Attacker copies an envelope signed over one descriptor and presents it alongside a different descriptor.                          | `descriptorHash` is committed by the signature. A mismatch between the envelope's `descriptorHash` and the descriptor being verified causes rejection.                              |

### Key Compromise

If an attester's private key is stolen, the attacker can issue envelopes indistinguishable from legitimate ones — signatures will verify correctly against the attester's CAIP-10 identifier. The envelope format has no way to distinguish forged-with-stolen-key from authentic.

`expiresAt` bounds exposure: forgeries remain acceptable only until the TTL lapses. This is a soft mitigation, not a fix.

Active retraction is typically unavailable in this case. The compromised attester cannot credibly invalidate their own key through the envelope format alone — the attacker holds the key and can forge competing envelopes. Ending the compromise requires an out-of-band channel the attacker does not control (a root key, an auditor-registry entry, an on-chain revocation).

### Retraction Limitations

Retraction propagates only through channels the attester can update. An envelope copy hosted on a mirror outside the attester's control continues to circulate until its `expiresAt`. Third-party or propagated revocation (the case where someone other than the attester must act) is out of scope for this ERC and is expected to be addressed by a future companion ERC defining an auditor trust registry.

### Contract Attester State Changes

Contract attesters are verified via ERC-1271 at the current block. Their endorsement reflects the contract's state at verification time, not at issuance time. This is intentional: a contract attester retains the ability to invalidate its own prior attestations by changing the contract's signature-validation state. Implementations that cache contract-attester verification results MUST bound staleness accordingly.

### Wallet Trust Responsibility

A valid signature proves that a specific account attested to a descriptor's content. It does not prove that the account is trustworthy or that the descriptor is correct. Wallets are responsible for maintaining the set of attesters they trust, the policy that determines how many attestations are required, and any additional checks (freshness, expiration margin, cross-attester agreement) before applying a descriptor's formatting instructions.

### Explicit Non-Protections

This ERC does **not** protect against:

- **Malicious attesters.** If an attester acts maliciously — signing a known-incorrect descriptor — the envelope will still verify. Trust decisions are wallet policy.
- **Attester key compromise** — see [Key Compromise](#key-compromise) above.
- **Semantic attacks on descriptor content.** A descriptor may pass signature verification yet be semantically misleading (for example, labeling a field "Amount" when it represents something else). Integrity verification does not validate semantic correctness.
- **Supply-chain compromise on the attester side.** If the attester's signing pipeline (build server, CI, HSM, operational tooling) is compromised and produces a valid signature over a malicious descriptor, verifiers cannot distinguish this from a legitimate attestation. Attesters are responsible for their own operational security.

### Implementation Pitfalls

- **Keccak-256 vs. SHA3-256.** Ethereum uses the pre-standardization Keccak-256. The NIST-standardized SHA3-256 produces different output for the same input. Implementations MUST use Keccak-256.
- **RFC 8785 compliance.** Partial or incorrect RFC 8785 implementations (incorrect number serialization, Unicode handling, or key ordering) will produce different canonical forms and cause hash mismatches. Implementations SHOULD use well-tested RFC 8785 libraries and validate against the RFC 8785 test vectors.
- **ERC-1271 chain availability.** Contract attester verification requires an RPC connection to the referenced chain. Implementations MUST handle chain unavailability gracefully and MUST NOT treat it as a verification failure.
- **CAIP-10 parsing.** The CAIP-10 format is `namespace:chainId:address`. Implementations MUST validate that the namespace is `eip155` for this version and that the address is a valid EVM address. Malformed identifiers SHOULD cause the envelope to be rejected with a warning.

### Privacy

Attester addresses are publicly visible in envelopes. Attester identities are not confidential — they are intentionally disclosed to enable trust decisions. Participants should be aware that issuing an envelope publicly links their address to the attestation.
