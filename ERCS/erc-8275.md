## Abstract

This ERC defines a set of three decoupled standard interfaces for a decentralized AI agent service marketplace: **Registry** (service discovery via events, with optional node-role registration), **Escrow** (trust-minimized payment with multi-pool reward distribution), and **Reputation** (event-derived scoring with natural decay and attestation-weighted verification). The three interfaces are independently deployable and communicate exclusively through standardized on-chain events, following a "simple rules → emergence" architecture. An additional node-role taxonomy — **Origin**, **Router**, **Hybrid** — maps each participant class to a separate reward pool, enabling economic settlement across the full attestation → proof → settlement pipeline (ERC-8273 → ERC-8274 → ERC-8275 → ERC-8183).

## Motivation

The AI agent standard ecosystem has seen rapid growth — ERC-8004 (agent identity), ERC-8263 (onchain inference proof anchoring with OCP), ERC-8300 (attestation-gated actions), ERC-8240 (trust infrastructure with IAttestation → IAccountability surface), ERC-8126 (security scoring), ERC-8273 (attestation framing), ERC-8274 (proof verification), and ERC-8183 (agentic commerce) — each address critical layers of the agent stack. However, no existing standard covers the middle layer: **how users discover the right agent, pay for services safely, and how multi-role participants in the attestation pipeline receive economic rewards**.

Current agent applications require bespoke integration for service discovery and payment, resulting in fragmented UX and duplicated engineering. Moreover, a production attestation-to-settlement pipeline involves at least three distinct participant classes — Origin (record producers), Router (aggregators/settlers), and Hybrid (both) — each providing value that deserves separate economic accounting. This ERC fills the gap by standardizing the discovery-payment-reputation-reward loop that every agent marketplace needs, with explicit support for multi-role economic settlement.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

### IAgentRegistry — Service Discovery and Node Registration Layer

Agents register with metadata (IPFS CID or arbitrary URI). Discovery is event-driven: clients listen to `AgentRegistered` events and build their own indices. No search API is defined — global discovery behavior emerges from local indexing of global events.

**ERC-8004 identity linking:** Agents SHOULD reference their ERC-8004 token ID in the metadata field. When an agent has a canonical ERC-8004 identity, the `agentId` in this registry SHOULD be the ERC-8004 token ID to enable cross-standard identity resolution. For agents without an ERC-8004 identity, a self-generated `agentId` is permitted.

**Metadata schema:** Agents SHOULD expose metadata conforming to the schema defined in ERC-8263 (agent card fields) and ERC-8004 (tokenURI identity anchor). Discovery via `/.well-known/agent.json` follows the convention described in ERC-8263. This gives consumers a deterministic extraction path instead of implementation-specific field guessing.

```solidity
interface IAgentRegistry {
    /// @notice Register an agent service
    /// @param metadata Agent metadata (IPFS CID or URI)
    /// @return agentId Globally unique identifier
    function registerAgent(bytes calldata metadata) external returns (bytes32 agentId);

    /// @notice Update agent metadata
    function updateAgent(bytes32 agentId, bytes calldata metadata) external;

    /// @notice Deregister an agent (irreversible)
    function deregisterAgent(bytes32 agentId) external;

    /// @notice Get agent information
    function getAgent(bytes32 agentId) external view returns (
        address owner,
        bytes memory metadata,
        uint64 registeredAt,
        bool active
    );

    /// @notice ERC-165 interface detection
    function supportsInterface(bytes4 interfaceId) external view returns (bool);
}
```

**Events:**

```solidity
event AgentRegistered(bytes32 indexed agentId, address indexed owner, bytes metadata);
event AgentUpdated(bytes32 indexed agentId, bytes metadata);
event AgentDeregistered(bytes32 indexed agentId);
```

**Design decisions:**
- Event-driven discovery: no on-chain search API. Clients index `AgentRegistered` events locally. This follows the principle that global search behavior emerges from local indexing + event communication, analogous to ant pheromone trails.
- Permissionless registration: anyone MAY register. Reputation is determined by the market.
- Open metadata: `metadata` MAY be any format (JSON/IPFS/Hash). No specific schema is mandated.

### INodeRegistry — Node Role Registration (OPTIONAL Extension)

Implementations MAY support a node-role taxonomy that maps participants in the attestation-to-settlement pipeline to separate reward pools. This extension is OPTIONAL but RECOMMENDED for implementations that involve multi-role participants (Origins, Routers, Hybrids). The taxonomy was contributed by TMerlini based on the production ccip-router mesh topology.

```solidity
enum NodeType { Origin, Router, Hybrid }

interface INodeRegistry {
    /// @notice Register a node with a role
    /// @param url Node endpoint URL
    /// @param nodeType Role classification
    /// @return nodeId Unique node identifier
    function registerNode(string calldata url, NodeType nodeType) external returns (bytes32 nodeId);

    /// @notice Get node information
    function getNode(bytes32 nodeId) external view returns (
        address owner,
        string memory url,
        NodeType nodeType,
        uint64 registeredAt,
        bool active
    );

    /// @notice Get contribution count for a node
    /// @dev Aggregated from Escrow settlement events
    function getContributionCount(bytes32 nodeId) external view returns (uint64 count);
}
```

**Node Type Semantics:**

| NodeType | Description | Reward Pool |
|----------|-------------|-------------|
| **Origin** | Record producers — agents that generate attestations at the edge | Pool A — per-attestation reward |
| **Router** | Aggregators/settlers — nodes that relay, aggregate, and settle attestations | Pool B — per-settlement reward |
| **Hybrid** | Nodes performing both roles | Pro rata across Pool A and Pool B |

**Events:**

```solidity
event NodeRegistered(bytes32 indexed nodeId, address indexed owner, string url, NodeType nodeType);
event NodeUpdated(bytes32 indexed nodeId, string url, NodeType nodeType);
event NodeDeregistered(bytes32 indexed nodeId);
```

**Contribution accounting:** Implementations SHOULD expose a `GET /contributions` endpoint that aggregates record counts by `sourcePeer`. This off-chain attribution primitive feeds the on-chain reward distribution logic described in IAgentEscrow.

### IAgentEscrow — Escrow Payment Layer

Funds are locked in escrow, released only on client confirmation. Deadline-protected refunds and pluggable dispute resolution are included. Multi-pool reward distribution supports the Origin/Router/Hybrid taxonomy.

```solidity
interface IAgentEscrow {
    /// @notice Create an escrow order
    /// @param agentId Service provider ID
    /// @param paymentToken Payment token (address(0) = native ETH)
    /// @param amount Payment amount
    /// @param deadline Service completion deadline (Unix timestamp)
    /// @param metadata Order metadata (service description hash, etc.)
    /// @return orderId Unique order identifier
    function createOrder(
        bytes32 agentId,
        address paymentToken,
        uint256 amount,
        uint64 deadline,
        bytes calldata metadata
    ) external payable returns (bytes32 orderId);

    /// @notice Confirm service completion and release payment
    /// @dev Only the order creator MAY call this
    function confirmOrder(bytes32 orderId) external;

    /// @notice Initiate a dispute
    /// @dev Either party MAY call this; funds are locked pending arbitration
    function disputeOrder(bytes32 orderId, bytes calldata reason) external;

    /// @notice Arbitrate a dispute
    /// @dev Only registered arbitrators MAY call this
    /// @param resolution 0 = refund buyer, 1 = release to seller, 2 = split evenly
    function resolveDispute(bytes32 orderId, uint8 resolution) external;

    /// @notice Refund an expired order
    /// @dev Any party MAY call this after the deadline
    function refundExpiredOrder(bytes32 orderId) external;

    /// @notice Get order status
    function getOrder(bytes32 orderId) external view returns (
        bytes32 agentId,
        address client,
        address agent,
        address paymentToken,
        uint256 amount,
        uint64 deadline,
        uint8 status, // 0=Pending, 1=Confirmed, 2=Disputed, 3=Refunded, 4=Resolved
        bytes memory metadata
    );

    /// @notice Distribute reward across node-type pools
    /// @dev Called on `OrderConfirmed` — splits reward per the node taxonomy
    /// @param orderId Settled order
    /// @param contributions Array of (nodeId, contributionWeight) tuples
    function distributeReward(
        bytes32 orderId,
        bytes32[] calldata nodeIds,
        uint16[] calldata contributionWeights
    ) external;
}
```

**Events:**

```solidity
event OrderCreated(bytes32 indexed orderId, bytes32 indexed agentId, address indexed client, uint256 amount);
event OrderConfirmed(bytes32 indexed orderId);
event OrderDisputed(bytes32 indexed orderId, bytes calldata reason);
event OrderResolved(bytes32 indexed orderId, uint8 resolution);
event OrderRefunded(bytes32 indexed orderId);
event RewardDistributed(bytes32 indexed orderId, bytes32 indexed nodeId, uint256 amount, NodeType poolType);
```

**Design decisions:**
- Trust-minimized: funds are held by the escrow contract, not by either party.
- Timeout protection: orders past their `deadline` MAY be refunded by anyone.
- Pluggable arbitration: arbitrator registration is managed independently and is not coupled to the Escrow interface.
- Multi-pool distribution: the `distributeReward` function maps each contributing node's `NodeType` to its respective reward pool (Origin → Pool A, Router → Pool B, Hybrid → pro rata). This enables economic settlement across the full attestation pipeline without requiring each participant class to deploy its own escrow contract.

### IAgentReputation — Reputation Layer

Reputation data is derived entirely from Escrow events and, optionally, from verified attestation history. No manual submission is required or permitted. Inactive agents naturally decay — pheromone evaporation applied to trust scores. **Attestation-weighted scoring** combines escrow outcomes with provably verified execution history for a stronger signal.

```solidity
interface IAgentReputation {
    /// @notice Get agent reputation information
    function getReputation(bytes32 agentId) external view returns (
        uint64 completedOrders,
        uint64 disputedOrders,
        uint64 totalVolume,
        uint64 lastActiveAt,
        uint16 score // 0-10000, computed from order history + optionally attestation weight
    );

    /// @notice Get decay weight (defined by the reputation provider)
    function getDecayWeight(bytes32 agentId) external view returns (uint16 weight);

    /// @notice Verify an order outcome proof (optional, for advanced reputation systems)
    /// @dev SHOULD accept ERC-8263 anchorProof(agentId, proofHash) + OCP record(input_hash) as valid proofs
    function verifyOutcome(bytes32 orderId, bytes calldata proof) external view returns (bool);
}
```

**Attestation-Weighted Scoring Formula (RECOMMENDED):**

The reputation score for an agent SHOULD incorporate both escrow outcomes and verified attestation history:

```
score = (completedOrders / totalOrders) * BASE_WEIGHT
      + (verifiedAttestations / totalAttestations) * ATTESTATION_WEIGHT
      + volumeNormalized * VOLUME_WEIGHT
```

Where:
- `BASE_WEIGHT + ATTESTATION_WEIGHT + VOLUME_WEIGHT = 10000`
- `verifiedAttestations` = count of `verifyOutcome()` calls returning `true` for this agent
- `volumeNormalized` = `totalVolume / maxVolume` across all registered agents
- RECOMMENDED weights: `BASE_WEIGHT = 6000`, `ATTESTATION_WEIGHT = 2500`, `VOLUME_WEIGHT = 1500`

Implementations MAY customize these weights. Different dApps MAY choose different reputation providers.

**ValidatorType Enum (for attestation-gated scoring):**

When evaluating attestation proofs via `verifyOutcome()`, implementations SHOULD distinguish between verification paradigms to avoid collapsing different assurance models into a single score:

```solidity
enum ValidatorType { ReExecution, Attestation, Judgment }

enum ClaimType { ReExecution = 0, Attestation = 1, Judgment = 2 }
```

- **ReExecution (0):** The validator re-executes the computation and compares outputs. Highest assurance, highest cost.
- **Attestation (1):** The validator cryptographically verifies an EIP-712 attestation signature without re-execution. Medium assurance, low cost.
- **Judgment (2):** A trusted validator vouches for the outcome. Lowest assurance, lowest cost. MUST include a `recordPointer` to the outcome-linked record for cryptographic traceability.

**Dual Discriminator for Claim Schema:**

Verification consumers MUST be able to distinguish between claim types without inspecting validator internals. Implementations SHOULD use two complementary mechanisms:

1. **EIP-712 type string:** On-chain schema discrimination — a judgment claim structurally cannot verify against a re-execution schema
2. **Explicit `claimType` enum (0/1/2):** Off-chain indexer readability without additional on-chain round-trips

**Design decisions:**
- Passive computation: reputation is computed entirely from `OrderConfirmed`, `OrderDisputed`, and `OrderResolved` events, plus optional `verifyOutcome` results. No Sybil-attackable manual reviews.
- Natural decay: agents with no recent activity naturally lose weight (Dormant state), similar to pheromone evaporation — inactive paths fade from the system.
- Provider-agnostic: `IAgentReputation` is a query interface. Third parties MAY implement custom scoring algorithms.
- Composable: reputation scores MAY be consumed as input to ERC-8126 security assessments.
- **ERC-8263 proof anchoring:** The `verifyOutcome` hook is designed for composability with on-chain proof layers. Implementations SHOULD accept ERC-8263 `anchorProof(agentId, proofHash)` anchors combined with OCP `record(input_hash)` verification as valid outcome proofs. A reputation engine weighting both escrow outcomes and verified execution history produces a stronger signal than either alone.
- **ValidatorType gating:** Scores derived from ReExecution verifications SHOULD carry higher weight than those from Judgment verifications. Implementations SHOULD publish the `ValidatorType` mix for each agent so consumers can assess the assurance model behind the score.

### Architecture: Three-Layer Decoupling with Node Taxonomy

```
┌─────────────┐   events   ┌─────────────┐   events   ┌──────────────┐
│  Registry   │◄──────────►│   Escrow    │◄──────────►│  Reputation  │
│ (Discovery) │            │  (Payment)  │            │  (Scoring)   │
│             │            │             │            │              │
│ INodeReg.   │            │ distribute  │            │ verifyOutcome│
│ (OPTIONAL)  │            │ Reward()    │            │              │
└─────────────┘            └─────────────┘            └──────────────┘
      │                         │                          │
      ▼                         ▼                          ▼
 AgentRegistered          OrderCreated              getReputation
 NodeRegistered           OrderConfirmed            Read-only
 AgentUpdated             OrderDisputed             
 AgentDeregistered        OrderResolved
                          RewardDistributed
                          
         ┌─────────────────────────────────────────┐
         │  Node Taxonomy → Reward Pool Mapping    │
         │  Origin  → Pool A (per-attestation)     │
         │  Router  → Pool B (per-settlement)      │
         │  Hybrid  → Pro rata A + B               │
         └─────────────────────────────────────────┘
```

- Three layers are **independently deployable** and independently upgradeable.
- The OPTIONAL `INodeRegistry` extension adds node-role registration without modifying existing interfaces.
- Layers do not know each other's contract addresses.
- Communication is exclusively through **standardized events**.
- Any layer MAY be replaced or customized without affecting the others.

**ERC-8240 composability:** The IAgentRegistry and IAgentEscrow interfaces compose naturally with ERC-8240's IAttestation → IAccountability surface. Where ERC-8240 defines *who is trusted* and *who is accountable*, ERC-8275 defines *how to find them* and *how to pay them*. The two standards stack vertically without overlap — ERC-8240 governs the trust relationships between agents and users, while ERC-8275 handles the economic discovery-payment-settlement loop.

**ERC-8273/8274/8275/8183 pipeline:** The editor-confirmed ERC sequence defines a complete attestation-to-commerce pipeline:
- **ERC-8273:** Attestation framing — defines the attestation envelope
- **ERC-8274:** Proof verification — defines the verifier interface (`IAgentVerifier`, `claimType`)
- **ERC-8275:** Settlement — defines payment distribution across the three reward pools (this ERC)
- **ERC-8183:** Agentic commerce — defines the marketplace consumer interface

## Rationale

### Why event-driven discovery instead of a search API?

On-chain search is expensive and creates centralization pressure (who runs the indexer?). By emitting standardized events and letting clients build their own indices, discovery becomes permissionless and scalable. This mirrors how ant colonies achieve global foraging optimization through local pheromone deposition — no central planner required.

### Why three decoupled interfaces?

Separating discovery, payment, and reputation allows each layer to evolve independently. A dApp MAY use a custom reputation algorithm while using the standard Registry and Escrow. An escrow implementation MAY be upgraded without breaking existing agent registrations. Decoupling is the architectural guarantee against vendor lock-in.

### Why passive reputation instead of manual reviews?

Manual reputation systems are vulnerable to Sybil attacks and collusion. By deriving reputation exclusively from on-chain escrow events (which carry economic weight), reputation becomes a cryptoeconomic signal rather than a social one.

### Why natural decay?

Without decay, inactive agents retain high scores indefinitely, misleading users. Decay ensures that reputation reflects *recent* performance. This is modeled on pheromone evaporation in biological systems — trails that are not reinforced fade over time.

### Why a node-role taxonomy?

A production attestation-to-settlement pipeline involves multiple participant classes — Origins produce attestations, Routers aggregate and settle them, and Hybrid nodes do both. Collapsing all participants into a single reward pool would fail to compensate Origin nodes for their distinct contribution (edge attestation vs. settlement routing). The three-tier taxonomy, derived from the production ccip-router mesh topology (`gateway.ensub.org` + 3 routers), ensures economic rewards track actual value contribution.

### Why ValidatorType-gated reputation scores?

Different verification paradigms provide different levels of assurance. A ReExecution validator that recomputes the inference and compares outputs provides stronger evidence than a Judgment validator vouching for an outcome. Collapsing these into a single score without type information would mislead consumers about the assurance model behind the reputation. The `ValidatorType` enum + dual discriminator pattern allows consumers to filter by assurance model without inspecting validator internals.

### Why attestation-weighted scoring?

Escrow outcomes alone are a weak signal — an agent could complete many low-value orders without ever executing a verifiable computation. Attestation history (ERC-8263 + OCP) provides cryptographic proof that the agent actually executed the claimed computation. Weighting both signals produces a reputation score that reflects both *reliability* (escrow outcomes) and *competence* (verified execution history). The recommended weights (6000/2500/1500) prioritize delivery reliability while giving meaningful weight to provable execution history.

## Backwards Compatibility

No backward compatibility issues found. This ERC defines new interfaces that do not conflict with existing standards. The OPTIONAL `INodeRegistry` extension does not modify `IAgentRegistry` — implementations not using the node taxonomy remain fully compatible.

## Security Considerations

### Escrow fund safety

Implementations MUST ensure that funds can only be released via `confirmOrder` (by the client) or `resolveDispute` (by a registered arbitrator). Reentrancy protection MUST be applied to all fund-transferring functions, including `distributeReward`.

### Reputation Sybil resistance

Since reputation is derived from economic activity (escrow orders), Sybil attacks require capital commitment. Implementations SHOULD set minimum order amounts to raise the cost of reputation manipulation. Attestation-weighted scoring further raises the bar — an attacker must produce verifiable attestations, which requires actual computation.

### Event spamming

`AgentRegistered` and `NodeRegistered` events are permissionless and could be spammed. Client indexers SHOULD implement spam filtering (e.g., minimum registration stake, age-weighted filtering, or cross-referencing with Escrow activity).

### Arbitrator trust

The arbitrator mechanism introduces a trust assumption. Implementations SHOULD support multi-signature or decentralized arbitration schemes.

### Reward pool fairness

The `distributeReward` function relies on off-chain contribution accounting (`GET /contributions`) to determine reward weights. Implementations SHOULD publish contribution data with cryptographic integrity guarantees (e.g., signed contribution records) to prevent manipulation of reward distribution.

### ValidatorType assurance misrepresentation

A malicious agent could register only with Judgment-type validators to appear verified while maintaining low actual assurance. Consumers SHOULD inspect the `ValidatorType` mix before interpreting a reputation score. Implementations SHOULD publish a `getValidatorBreakdown(agentId)` view that returns the count of verifications by `ValidatorType`.
