## Abstract

This proposal introduces IERC721Value, an extension to ERC-721 that enables each non-fungible token to be associated with a redeemable amount of native ETH.

Unlike existing approaches relying on external vaults or wrapped assets, the redeemable value is defined within the contract’s accounting model and can only be recovered by burning the token.

This standard defines a minimal interface for querying and redeeming per-token ETH value, while preserving full compatibility with ERC-721.

## Motivation

ERC-721 defines ownership of non-fungible assets but does not provide a standardized way to associate native ETH value with a token in a non-custodial and irreversible manner.

Today, value-bearing NFTs rely on:
- external vault contracts,
- wrapped assets,
- or off-chain accounting assumptions.

These approaches introduce fragmentation, additional trust assumptions, and inconsistent integration patterns across wallets and applications.

This proposal introduces a reusable primitive where each NFT carries a redeemable ETH value that is:

- directly accounted for within the token contract,
- inseparable from the token’s identity,
- and recoverable only through an explicit burn operation by the token owner.

This enables a wide range of applications, including:

- value-backed digital or physical assets,
- licenses or rights with embedded economic guarantees,
- non-custodial escrows represented as NFTs,
- and long-term value containers with constrained liquidity.

Standardizing this pattern allows wallets, indexers, and applications to uniformly detect, display, and interact with NFTs that carry intrinsic redeemable ETH value.

## Specification

This ERC extends ERC-721 and introduces the following interface:

- `valueOf(uint256 tokenId) → uint256`

    Returns the amount of native ETH that is redeemable for the given token.

    The returned value MUST reflect the amount of ETH that the current owner is entitled to receive upon burning the token.

- `burn(uint256 tokenId)`

    Destroys the token and redeems its associated ETH value.

    Requirements:

    - MUST revert unless `msg.sender` is the current owner of `tokenId`.
    - MUST permanently destroy the token.
    - MUST transfer the redeemable ETH value associated with `tokenId` to the current owner.
    - MUST ensure that the redeemed value cannot be claimed again.

### Value Assignment

Implementations MUST define how ETH value is assigned to each token.

The value MAY be:

- uniform across all tokens,
- variable per token,
- or determined by custom minting logic.

The value assigned to a token MUST be consistently reported by `valueOf(tokenId)` and MUST be redeemable at burn time.

### Accounting Model

Implementations MAY use different internal accounting strategies, including:

- per-token value storage,
- pooled reserve models,
- or deterministic value assignment.

However:

- The redeemable value MUST be clearly defined per token.
- The implementation MUST ensure that all reported values are honorably redeemable under its accounting model.

### ETH Balance Considerations

The total ETH balance of the contract MAY exceed or differ from the sum of redeemable values due to:

- direct ETH transfers to the contract,
- surplus accumulation,
- or implementation-specific accounting.

Therefore:

- `valueOf(tokenId)` MUST NOT be inferred from the contract’s raw ETH balance.

## Rationale

This design focuses on defining a minimal, implementation-agnostic primitive for associating redeemable ETH value with ERC-721 tokens.

The standard does not prescribe how value is stored internally, allowing flexibility across different implementations, including fixed-value collections and variable-value tokens.

The introduction of `valueOf(tokenId)` provides a uniform interface for external systems to query redeemable value without relying on contract-specific logic.

Redemption is restricted to the current owner and requires token destruction. This ensures that value recovery is explicit, irreversible, and tightly bound to the non-fungible identity.

Unlike ERC-721 transfer approvals, which allow delegated token management, redemption represents an irreversible economic action. Restricting it to the owner avoids ambiguity and reduces risk of unintended value extraction by third parties.

By separating redeemable value from raw contract balance, the standard avoids incorrect assumptions about solvency and allows implementations to support surplus handling and external funding.

## Security Considerations

Implementations must ensure:

- Proper authorization: only the token owner can trigger value redemption.
- Reentrancy protection during ETH transfers.
- Correct accounting to ensure that all redeemable values are backed and payable.

Developers should carefully handle:

- ETH transfers using safe patterns (e.g., checks-effects-interactions).
- Edge cases where ETH may be sent directly to the contract outside minting flows.
- Situations where surplus ETH exists beyond redeemable obligations.

Failure to properly manage accounting or redemption logic may result in insolvency or unintended loss of funds.
