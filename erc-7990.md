## Abstract

This EIP proposes a new token standard — **DfaFi** (Direct Fractional Auction Finance) — for the tokenization of fractional ownership in real-world assets including bonds, equities, and other financial instruments. 

The standard introduces two core innovations:

1. **Batch primary issuance**: A single blockchain transaction mints ownership shares for an unlimited number of holders using a Direct Fractional Auction (DFA) mechanism, replacing thousands of individual mint transactions.

2. **Batch secondary market clearing**: A Batch Double Auction (BDA) mechanism accumulates buy and sell orders and clears them at a single uniform price in one transaction, eliminating front-running, HFT advantages, and price manipulation.

When BDA does not clear (no matching orders), the issuer or an oracle updates a NAV price. The contract always provides a current price via `getCurrentPrice()`, returning the more recent of the last BDA clearing price or NAV update.

---

## Motivation

### Problem 1: Cost of tokenizing large asset sets

Existing NFT standards (ERC-721, ERC-1155) require one transaction per token. Tokenizing 10,000 wine bottles, bond certificates, or equity shares requires 10,000 mint transactions. At Ethereum gas prices, this costs $500,000–$1,000,000 in fees — making tokenization economically unviable for most real-world assets.

Compressed NFTs (cNFT) reduce minting costs via Merkle trees but introduce critical operational problems: dependency on off-chain indexers, no native DeFi composability, and expensive "unwrapping" fees for secondary market participation.

### Problem 2: Absence of true market price discovery

Current tokenized asset protocols (BlackRock BUIDL, Ondo Finance, Backed Finance) fix token prices at $1.00 NAV or peg to off-chain NAV calculations. This creates:

- **No secondary market**: BUIDL tokens cannot be traded; price is permanently fixed at $1.
- **Premium/discount problem**: Like futures vs spot, a tokenized asset should trade at a price that reflects blockchain-specific factors (liquidity, settlement speed, 24/7 availability). Existing solutions cannot discover this price.
- **Weekend manipulation**: Protocols holding real equities are vulnerable to front-running when news breaks on weekends — DeFi traders can position before traditional markets open Monday.
- **Continuous NAV updates**: Competitors must update NAV every second for each asset, requiring a separate smart contract per asset. DfaFi requires NAV updates only when BDA fails to clear.

### Problem 3: Exclusion from high-minimum instruments

Eurobonds require minimum investments of $200,000. Pre-IPO equity rounds require $1,000,000+. These instruments are structurally inaccessible to retail investors. No existing tokenization standard enables trustless collective ownership that allows groups of small investors to jointly acquire and hold such instruments.

## Specification

### Definitions

- **DFA (Direct Fractional Auction)**: A discriminatory price auction where each bidder pays their own submitted price. Prevents low-ball manipulation. Off-chain computation determines allocation; result is submitted on-chain in a single transaction.

- **BDA (Batch Double Auction)**: A uniform-price double auction where buy and sell orders accumulate over a defined batch window and clear at a single optimal price. All matched participants pay/receive the same clearing price.

- **Ownership Matrix**: A mapping from holder addresses to share counts, stored on-chain. Updated atomically in batch operations.

- **NAV Price**: Issuer or oracle-confirmed net asset value per share, used when BDA does not produce a clearing price.

### Interface

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.28;

interface IDfaFi {

    // ─── EVENTS ──────────────────────────────────────────────────────────
    event BatchMinted(uint256 totalShares, uint256 holderCount, uint256 timestamp);
    event CouponPaid(uint256 totalAmount, uint256 perShareAmount, uint256 timestamp);
    event OrderSubmitted(address indexed trader, uint256 shares, uint256 price, bool isBuy);
    event BDACleared(uint256 clearingPrice, uint256 sharesTraded, uint256 timestamp);
    event NAVUpdated(uint256 navPrice, uint256 timestamp);
    event SharesTransferred(address indexed from, address indexed to, uint256 shares);

    // ─── PRIMARY ISSUANCE (DFA) ───────────────────────────────────────────
    /**
     * @notice Mint all shares in ONE transaction
     * @dev DFA algorithm runs off-chain; results submitted here
     * @param buyers Array of winning bidders from DFA auction
     * @param shares Array of shares allocated to each buyer
     */
    function batchMint(
        address[] calldata buyers,
        uint256[] calldata shares
    ) external;

    // ─── INCOME DISTRIBUTION ─────────────────────────────────────────────
    /**
     * @notice Distribute coupon or dividend to ALL holders in ONE transaction
     * @dev Reads ownership matrix and distributes proportionally
     */
    function payCoupon() external payable;

    // ─── SECONDARY MARKET (BDA) ───────────────────────────────────────────
    /**
     * @notice Submit buy or sell order to BDA order book
     */
    function submitOrder(
        uint256 shares,
        uint256 price,
        bool isBuy
    ) external payable;

    /**
     * @notice Clear accumulated BDA orders at uniform clearing price
     * @dev BDA algorithm runs off-chain; clearing price submitted here
     * @param clearingPrice Uniform price at which all matched trades execute
     * @param matchedBuyers Buyers who trade at clearing price
     * @param matchedSellers Sellers who trade at clearing price
     * @param matchedShares Shares traded per matched pair
     */
    function clearBDA(
        uint256 clearingPrice,
        address[] calldata matchedBuyers,
        address[] calldata matchedSellers,
        uint256[] calldata matchedShares
    ) external;

    // ─── NAV (when BDA doesn't clear) ────────────────────────────────────
    /**
     * @notice Update NAV price when BDA does not produce a clearing price
     * @param navPrice Price per share confirmed by issuer or oracle
     */
    function updateNAV(uint256 navPrice) external;

    /**
     * @notice Get current price — last BDA price or NAV, whichever is more recent
     * @return price Current price per share
     * @return isBDAPrice True if price comes from BDA clearing, false if NAV
     */
    function getCurrentPrice() external view returns (uint256 price, bool isBDAPrice);

    // ─── REDEMPTION ───────────────────────────────────────────────────────
    /**
     * @notice Redeem all shares at maturity
     */
    function redeem() external payable;

    // ─── OWNERSHIP ───────────────────────────────────────────────────────
    /**
     * @notice Get ownership details for a holder
     * @return shares Number of shares held
     * @return sharePercent Ownership percentage in basis points (10000 = 100%)
     * @return shareStart Range start in ownership matrix
     * @return shareEnd Range end in ownership matrix
     */
    function getOwnership(address holder) external view returns (
        uint256 shares,
        uint256 sharePercent,
        uint256 shareStart,
        uint256 shareEnd
    );
}
```

### Key Properties

**1. Single-transaction batch mint**

The `batchMint()` function MUST mint shares for all buyers in a single atomic transaction. The ownership matrix is updated once. Total gas cost is O(n) in the number of buyers, not O(n) in total shares.

**2. Uniform clearing price**

The `clearBDA()` function MUST apply the same `clearingPrice` to all matched pairs. Partial fills are permitted. All unmatched orders MUST be cleared from the pending order book after each BDA round.

**3. Proportional income distribution**

The `payCoupon()` function MUST distribute `msg.value` proportionally to all holders based on their share count at the time of the call. Distribution MUST occur in a single transaction.

**4. Price continuity**

`getCurrentPrice()` MUST return a non-zero price whenever either a BDA clearing or NAV update has occurred. The more recent value takes precedence.

## Rationale

### Why discriminatory auction (DFA) for issuance?

A discriminatory (pay-as-bid) auction for primary issuance prevents bidders from colluding to submit artificially low prices. Each bidder's optimal strategy is to bid their true valuation, producing efficient price discovery. This is standard for sovereign bond auctions (US Treasury, ECB).

### Why uniform price auction (BDA) for secondary market?

A uniform clearing price ensures all participants in a batch are treated equally regardless of order submission time or size. This eliminates front-running (speed provides no advantage), HFT advantages, and slippage. The batch window can be configured from 1 second (for liquid assets) to 1 day (for illiquid bonds).

### Why NAV fallback instead of continuous updates?

Competitors (Ondo Finance, Backed Finance) must update NAV continuously — requiring a separate oracle feed per asset updated every second. DfaFi requires NAV updates only when BDA fails to produce a clearing price. This reduces oracle costs by orders of magnitude and enables one contract to manage an entire portfolio of assets.

### Why not use ERC-1155?

ERC-1155 supports multiple token types in one contract but still requires individual transfer transactions. It does not support batch auction clearing, uniform price settlement, or income distribution in a single transaction. DfaFi is purpose-built for financial instruments with income streams.

---

## Backwards Compatibility

DfaFi does not inherit from ERC-20, ERC-721, or ERC-1155. Implementers MAY choose to add ERC-20 compatible view functions (`balanceOf`, `totalSupply`) for compatibility with existing DeFi infrastructure, but this is not required.

## Reference Implementation

A reference implementation is deployed on Ethereum Sepolia testnet:

- **Contract address**: `0xDf9229f0b7C62b3457b380cB40b4bcEB1976413d`
- **Demo asset**: "Chateau Petrus 2000 Wine Bond" (CPW2000)
- **Total shares**: 10,000
- **Coupon rate**: 8% (800 basis points)

The implementation demonstrates:
- 10,000 shares minted in 1 transaction
- Coupon distributed to all holders in 1 transaction
- BDA secondary market with uniform clearing price
- NAV update when BDA does not clear
- `getCurrentPrice()` returning the more recent of BDA or NAV price

## Security Considerations

1. **Off-chain computation trust**: Both DFA allocation and BDA clearing are computed off-chain. Implementations SHOULD use a multi-validator consensus mechanism (such as ERC-8004 oracle nodes) to prevent manipulation of submitted results.

2. **NAV oracle risk**: The `updateNAV()` function gives the issuer or oracle significant pricing power when BDA does not clear. Implementations SHOULD use decentralized oracle consensus for NAV updates.

3. **Batch window manipulation**: The BDA batch window length is a security parameter. Shorter windows reduce manipulation risk but increase gas costs. Implementations SHOULD document the chosen batch window.

4. **Holder enumeration**: The `payCoupon()` function iterates over all holders. Gas costs scale linearly with holder count. For very large holder sets, implementations SHOULD use a pull-payment pattern as an alternative.
